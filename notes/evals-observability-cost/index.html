评测、观测与成本控制

用 trace、回归集和成本指标让 AI 应用持续变好。

EvalsCost

问题背景

AI 应用上线后，团队最常见的困惑不是“模型会不会回答”，而是“不知道它为什么变好或变差”。产品同学说用户反馈更差，算法同学说离线分数更高，后端同学说接口没有错误，财务同学说账单涨了，运营同学说人工接管更多。每个人都看到了系统的一小片，却没有一条能把输入、上下文、模型调用、工具执行、最终结果、用户反馈和成本串起来的链路。没有这条链路，优化很容易变成猜测。

评测、观测和成本控制经常被分开做。评测被当成上线前的一组题库，观测被当成日志和监控，成本被当成月底账单。这样做在普通软件里已经有问题，在 AI 应用里问题更大。因为模型行为不是固定代码路径，离线评测通过不代表线上分布稳定，线上指标变差也不一定能指向具体 prompt 或检索问题，成本上涨更可能来自上下文策略和失败重试，而不是单纯的调用量增加。三者如果不放在同一个闭环里，团队就无法持续改进。

我见过一个内部数据分析助手，早期只用几十条人工问题做评测。后来接入真实数据库、权限系统和多轮对话，用户开始问更复杂的问题，例如“比较华南和华东最近三个月渠道退款变化，顺便解释异常点”。系统有时会给出漂亮答案，但实际 SQL 查询了错误时间窗口；有时查询正确，却在总结里把渠道和区域混在一起；有时为了拿到足够背景连续调用多个工具，单次成本是普通问题的十几倍。离线题库看不出来，普通接口监控也看不出来。只有把 trace、评测样本和成本指标打通，才能知道失败发生在意图解析、工具参数、数据解释还是生成阶段。

AI 应用的成本也不能只看总额。总额上涨可能是业务增长，也可能是缓存失效、上下文膨胀、模型路由错误、重试风暴、工具循环或评测环境误打生产模型。平均成本下降也不一定是好事，如果系统因为过度压缩上下文导致答案质量下降，短期账单好看，长期人工修正成本更高。成本控制必须和质量指标一起看。好的目标不是“越便宜越好”，而是在给定质量和风险门槛下，找到可解释、可预测、可优化的成本结构。

观测同样要从“服务可用”升级到“行为可解释”。普通后端观测关注 QPS、错误率、延迟、CPU、内存。AI 应用还要关注输入分布、上下文来源、检索命中、模型版本、prompt 版本、schema 校验、工具调用、拒答、澄清、人工接管、用户反馈和 token 使用。一次请求如果只记录最终回答，失败时几乎无法定位。一次请求如果有完整 trace，就可以回放并转成评测样本。

评测、观测和成本控制的关系可以这样理解：评测是上线前的假设检验，观测是上线后的事实采集，成本是每次行为消耗的资源边界。评测告诉我们“候选版本在已知样本上是否更好”，观测告诉我们“真实流量里发生了什么”，成本告诉我们“这种行为是否可持续”。三者共同组成 AI 应用的工程反馈系统。没有评测，改动不可比较；没有观测，失败不可归因；没有成本，成功不可扩展。

这篇文章讨论的是如何把这三件事放到同一个工程设计里。重点不是堆工具名，也不是追求复杂平台，而是先把数据结构、指标口径和工作流想清楚。一个小团队可以从 JSON trace、Markdown 样本和简单报表开始；一个大团队可以建设统一评测平台和模型网关。但无论规模大小，核心问题相同：每次模型行为都要能被记录，每次失败都要能进入样本，每次优化都要能同时看质量、风险和成本。

核心概念

第一个核心概念是 trace。这里的 trace 不是一行日志，而是一次 AI 任务的结构化执行记录。它应该覆盖从用户输入到最终输出的关键事件：输入解析、上下文构建、检索结果、模型调用、schema 校验、工具调用、策略判断、最终答案、反馈和成本。trace 的目标不是保存所有原文，而是保存足以解释行为的证据和版本。敏感内容可以脱敏或摘要，但事件边界、版本号和资源消耗不能丢。

第二个概念是评测样本。好的评测样本不是简单的“问题和答案”，而是一个可重放场景。它应该包含输入、用户角色、时间、外部数据 fixture、期望行为、禁止行为、评价规则和标签。对于 RAG，样本要包含期望引用或证据；对于 Agent，样本要包含允许工具和禁止工具；对于结构化提取，样本要包含字段断言；对于开放生成，样本要有人工 rubric。样本越接近真实 trace，越容易从线上失败回流。

第三个概念是指标分层。AI 应用指标不能只有一个“准确率”。至少要分成质量、可靠性、安全、成本和体验五类。质量看答案是否正确、证据是否充分、任务是否完成；可靠性看 schema 失败、工具错误、重试和超时；安全看越权、泄露、拒答和高风险动作；成本看 token、模型调用、工具调用、缓存命中和单位业务成本；体验看延迟、澄清次数、人工接管和用户反馈。不同指标之间会冲突，必须一起看。

指标层典型指标主要用途常见误区
质量样本通过率、事实正确率、引用准确率判断回答和决策是否可信用一个总分掩盖关键失败
可靠性schema 失败率、工具错误率、重试率发现工程链路问题只看 HTTP 错误率
安全越权率、泄露率、拒答命中率保护数据和业务边界把安全写进 prompt 就结束
成本input/output token、模型费用、P95 成本控制资源消耗和预算只看月度总账单
体验P95 延迟、澄清率、人工接管率、负反馈判断用户是否愿意持续使用只看点赞率

第四个概念是成本归因。一次 AI 请求的成本通常由多个部分组成：输入 token、输出 token、检索成本、重排成本、工具调用、模型路由、缓存未命中、失败重试、评审模型和日志存储。成本归因要按任务、租户、版本、模型和失败类型切分。只看平均成本无法发现长尾。例如百分之五的复杂请求可能贡献一半模型费用；某个 prompt 版本可能让输出变长；某个工具异常可能触发多次重试。成本控制首先是看清楚钱花在哪里。

第五个概念是反馈闭环。线上 trace 中的失败不应该停留在监控告警里，而应该进入样本候选池。人工接管、用户差评、schema 失败、工具错误、成本异常、拒答争议都可以成为候选样本。候选样本经过清洗、标注和去重后，进入回归集或探索集。下一次发布时，它们用于验证修复。这样评测集会随着真实使用增长，而不是永远停留在上线前的想象。

第六个概念是版本维度。评测、观测和成本都必须能按版本切分。版本不只是代码版本，还包括模型、prompt、检索索引、schema、工具目录、策略和评分器。没有版本维度，指标变化无法解释。比如负反馈率上升，到底是新模型、旧缓存、某个租户的新文档，还是评分器变了？如果每条 trace 都带版本，排查会快很多。

最后是预算意识。预算不是月底发现超支后的财务动作，而是系统设计的一部分。每个任务类型都应该有成本预算和质量底线。预算可以体现在上下文长度、最大工具调用次数、模型路由、重试次数、缓存策略和降级策略里。当请求超过预算时，系统可以选择澄清、摘要、分批处理、使用便宜模型、进入异步队列或交给人工，而不是无上限地继续调用。

架构/流程图解说明

把评测、观测和成本控制放在一起，可以画成一个持续循环：

真实请求
  |
  v
运行时 trace：输入、上下文、模型、工具、策略、输出、成本
  |
  v
在线指标：质量反馈、错误率、延迟、人工接管、单位成本
  |
  v
失败和异常候选池：差评、接管、越权、schema 失败、成本长尾
  |
  v
样本治理：脱敏、标注、去重、归因、分级
  |
  v
评测集：核心集、回归集、安全集、成本集、探索集
  |
  v
候选改动：prompt、模型、检索、工具、策略、缓存
  |
  v
离线评测和成本模拟
  |
  v
灰度上线并继续观测

这个循环里，trace 是底座。没有 trace，线上失败无法还原；没有失败还原，评测集无法增长；评测集不增长，发布评测会越来越脱离真实；发布评测脱离真实，线上问题会越来越多。很多 AI 应用的质量停滞，并不是模型没有进步，而是团队没有把真实失败转成工程资产。

架构上可以拆成四个组件：运行时埋点、评测平台、指标仓库和成本归因服务。运行时埋点负责在请求链路中生成结构化事件；评测平台负责回放样本、运行候选版本、执行评分器；指标仓库负责聚合线上指标和离线评测结果；成本归因服务负责把 token、模型单价、工具资源和重试开销映射到任务维度。小团队可以把这些做成几个表和命令行，大团队可以做成统一平台，但边界最好一开始就清楚。

一次请求的 trace 可以看成事件流，而不是一个大 JSON。事件流更适合长任务、多工具和多轮对话。每个事件有类型、时间、版本、父事件、输入摘要、输出摘要和成本。模型调用是事件，检索是事件，工具调用是事件，策略拒绝也是事件。最终可以把事件聚合成一次 run 的摘要，用于查询和报表。

评测平台和生产运行时要共享尽可能多的代码路径。不要为了评测写一套简化逻辑，只直接调用模型。否则评测只测到了 prompt，而没有测到权限过滤、上下文预算、工具网关、schema 校验、缓存和降级。正确做法是用 fixture 替换外部世界，用稳定模型配置或候选模型配置执行同一套 runtime。这样离线失败才接近线上失败。

成本归因要接在模型网关和工具网关两侧。模型网关知道模型、token、单价、缓存和延迟；工具网关知道查询次数、外部 API 成本、数据库扫描、超时和重试。最终归因时要把这些成本挂到 run_id、task_type、tenant、release_id 上。只有这样才能回答“哪个版本让成本上涨”“哪类问题最贵”“哪个租户的使用方式需要单独优化”。

告警也要分层。服务错误率高当然要告警，但 AI 行为异常也要告警。例如 schema 失败率突然升高，可能是模型输出格式漂移；引用缺失率升高，可能是检索异常；人工接管率升高，可能是策略过严或模型变差；单位请求成本升高，可能是缓存失效或工具循环；拒答率异常下降，可能是安全策略失效。告警要尽量指向可行动的版本和任务类型。

工程实现

先从 trace 数据结构开始。下面是一个简化的结构，适合 RAG、Agent 和普通 LLM workflow 共同使用。实际系统可以按需要拆表，但概念上要保留 run、event、usage、feedback 的关系。

type AIRun struct {
    RunID          string            `json:"run_id"`
    App            string            `json:"app"`
    TaskType       string            `json:"task_type"`
    TenantID       string            `json:"tenant_id"`
    UserHash       string            `json:"user_hash"`
    ReleaseID      string            `json:"release_id"`
    PromptVersion  string            `json:"prompt_version"`
    ModelRoute     string            `json:"model_route"`
    SchemaVersion  string            `json:"schema_version"`
    IndexVersion   string            `json:"index_version"`
    StartedAt      time.Time         `json:"started_at"`
    FinishedAt     time.Time         `json:"finished_at"`
    Status         string            `json:"status"`
    FailureType    string            `json:"failure_type,omitempty"`
    Labels         map[string]string `json:"labels"`
}

type AIEvent struct {
    RunID        string          `json:"run_id"`
    EventID      string          `json:"event_id"`
    ParentID     string          `json:"parent_id,omitempty"`
    Type         string          `json:"type"`
    Name         string          `json:"name"`
    StartedAt    time.Time       `json:"started_at"`
    DurationMS   int             `json:"duration_ms"`
    InputRef     string          `json:"input_ref,omitempty"`
    OutputRef    string          `json:"output_ref,omitempty"`
    Metadata     json.RawMessage `json:"metadata,omitempty"`
    Usage        Usage           `json:"usage"`
}

type Usage struct {
    InputTokens   int     `json:"input_tokens"`
    OutputTokens  int     `json:"output_tokens"`
    CacheHit      bool    `json:"cache_hit"`
    EstimatedCost float64 `json:"estimated_cost"`
}

type Feedback struct {
    RunID      string    `json:"run_id"`
    Source     string    `json:"source"` // user, reviewer, policy, automated
    Rating     string    `json:"rating"` // positive, negative, needs_review
    Category   string    `json:"category"`
    Note       string    `json:"note"`
    CreatedAt  time.Time `json:"created_at"`
}

这里有几个设计取舍。UserHash 代替明文用户 ID，方便隐私保护；InputRef 和 OutputRef 指向受控存储，避免在事件表里直接放敏感正文；ReleaseID 和各资产版本让指标能切分；Usage 直接挂在事件上，方便计算每个模型调用和每个工具调用的成本。FailureType 不要只填异常类型，还要填行为失败类别，例如 schema_invalid、missing_citation、tool_permission_denied、cost_budget_exceeded。

事件类型要控制住，不要让每个工程师随意发明。可以先定义一组稳定枚举：

事件类型说明必填元数据
input.normalized输入清洗、语言识别、附件解析输入长度、附件类型、风险标签
retrieval.completed检索和重排完成查询、候选数、入选证据、索引版本
context.built上下文装配完成token 估算、材料来源、截断原因
model.called模型调用模型、prompt、token、采样参数
schema.validated结构化输出校验schema 版本、错误字段
tool.called工具执行工具名、参数摘要、状态、重试
policy.decided安全、权限、预算策略决策、规则、风险等级
response.finalized最终输出输出类型、引用数、降级原因

有了 trace，就可以设计样本回流。一个线上失败进入样本候选池时，不应该自动成为评测样本。先要脱敏，再要归因，再要人工或半自动补充期望行为。下面是一个候选样本到回归样本的流程：

negative feedback or alert
  -> load trace and related artifacts
  -> redact sensitive fields
  -> classify failure type
  -> create minimal reproducible case
  -> attach expected behavior and forbidden behavior
  -> run current stable version to confirm failure
  -> add to regression set after review

样本格式可以这样设计：

id: cost-rag-042
source_run_id: run_7c19
task_type: policy_qa
failure_type: missing_citation
input:
  user_role: support_agent
  question: "这个客户能不能按旧合同继续续费？"
fixtures:
  retrieval:
    index_version: policy_20260313
    documents:
      - doc: contract_policy_2025
      - doc: renewal_policy_2026
expect:
  must_cite:
    - renewal_policy_2026
  must_not_cite:
    - contract_policy_2025
  answer_should:
    - 说明旧合同条款已被新续费政策覆盖
  max_cost: 0.04
tags:
  - regression
  - citation
  - policy_conflict

这个样本同时覆盖质量和成本。它不仅要求引用新政策，还要求单次成本不超过预算。这样可以避免一种常见情况：为了修复引用问题，把上下文塞得很大，答案对了但成本不可接受。评测应该让质量和成本一起进入取舍。

评分器实现上，可以分成确定性评分、语义评分和成本评分。确定性评分检查 schema、枚举、工具参数、引用 ID、权限和预算；语义评分处理开放文本的充分性和可读性；成本评分比较 token、调用次数、重试和单价。确定性评分优先级最高，不能被语义高分抵消。比如发生越权访问时，最终回答再好也应该失败。

成本计算要可配置，因为不同模型和供应商价格会变，内部工具成本也可能按资源估算。一个简单公式如下：

run_cost =
  sum(model_input_tokens * input_unit_price)
  + sum(model_output_tokens * output_unit_price)
  + sum(rerank_requests * rerank_unit_price)
  + sum(tool_calls * tool_unit_price)
  + storage_cost
  + retry_extra_cost

更重要的是按维度聚合：

select
  release_id,
  task_type,
  model_route,
  percentile_cont(0.95) within group (order by estimated_cost) as p95_cost,
  avg(estimated_cost) as avg_cost,
  avg(duration_ms) as avg_latency,
  sum(case when failure_type != '' then 1 else 0 end) * 1.0 / count(*) as failure_rate
from ai_runs
where started_at >= now() - interval '1 day'
group by release_id, task_type, model_route;

这个查询不追求完整语法适配所有数据库，重点是口径：按版本、任务和模型路由看成本与失败率。成本控制不能脱离失败率。如果某个版本成本降低但失败率升高，需要继续看人工接管成本和用户体验，而不是直接庆祝。

最后要把指标接入日常工作流。PR 或发布候选应该附上离线评测报告；灰度期间监控看板应该默认按 release_id 展示；每天或每周要有失败样本治理；成本报表要能下钻到任务类型和 prompt 版本。工程团队不需要每天开大会讨论所有指标，但要让异常能自动浮现，并能指向具体负责人。

测试评测

评测体系要从“上线前跑一次”变成“贯穿开发、发布和线上回流”。开发阶段适合跑小而快的阻断集，保证基本契约不坏；发布阶段跑完整回归和成本模拟；灰度阶段跑在线对比和影子评测；上线后把真实失败转成新样本。不同阶段的目标不同，不能用同一套慢评测阻塞所有开发，也不能只用快速冒烟测试决定全量发布。

阶段评测重点样本规模运行频率
本地开发schema、核心路径、少量回归小每次改动
CI契约、阻断级安全、关键样本中每个 PR
发布候选完整核心集、事故集、成本集大每次发布
灰度影子对比、真实反馈、指标漂移在线抽样持续
线上运营新失败归因、样本治理、趋势分析增量每日或每周

评测样本要按用途分层。核心集保持稳定，用来比较版本；回归集记录修过的线上问题；安全集覆盖越权、注入、敏感数据和高风险建议；成本集覆盖长输入、复杂工具链、多轮对话和缓存未命中；探索集用于观察新能力，不一定阻断发布。把这些混成一个总集，会导致指标解释困难。核心集下降是严重问题，探索集波动则可能只是能力边界还没稳定。

评分方式也要分层。确定性断言适合结构化输出、工具调用、权限、引用 ID 和预算。人工 rubric 适合开放文本、业务语气和建议质量。LLM judge 可以辅助大规模初筛，但要校准，不能单独决定高风险发布。成本评分要使用同一套价格表和模型路由配置，否则不同时间的报告不可比。所有评分器都要有版本，因为评分器变化也会导致分数变化。

一个较完整的评测报告应该包含以下内容：

  • 候选版本和基线版本的资产差异，包括模型、prompt、schema、索引、工具和策略。
  • 各样本集通过率、关键指标变化和阻断项列表。
  • 失败样本的最小复现链接，能够看到 trace、上下文、模型输出和评分证据。
  • 成本对比，包括平均成本、P95 成本、最大成本、模型调用次数、重试次数和缓存命中率。
  • 延迟对比，包括端到端 P95、模型 P95、工具 P95 和排队时间。
  • 人工评审结论，尤其是高风险样本和线上事故样本。
  • 发布建议，包括可发布、带观察发布、需要修复或需要降级。

评测要特别注意“答案正确但过程错误”的情况。比如 Agent 最终总结了正确客户名称，但其实没有调用客户查询工具，而是从用户输入里猜的；RAG 答案说对了政策，但引用了错误文档；数据分析答案趋势正确，但 SQL 时间窗口错了。只评最终文本会放过这些问题。trace-level 评分可以检查过程证据，让系统学会走可靠路径。

成本评测要设计长尾样本。很多应用平均问题很短，但真实用户会粘贴长日志、长合同、多轮对话和多份附件。成本集至少要覆盖：最大允许输入、检索结果很多、工具返回分页、模型第一次 schema 失败需要重试、缓存冷启动、用户连续追问。每个样本应该设置 max_cost 或 max_calls，超过预算即使答案正确也要标记。

灰度评测要利用影子输出。稳定版本给用户回答，候选版本在后台生成输出和 trace，然后比较差异。比较可以分成结构化字段差异、引用差异、成本差异和人工抽检。影子评测特别适合模型升级，因为真实流量分布比离线样本丰富。要注意影子模式下不能执行生产写操作，所有副作用工具必须 mock 或 dry-run。

评测结果进入发布决策时，要避免两个极端。一个极端是只看分数，分数高就发；另一个极端是看到任何非阻断失败就不敢发。工程上更合理的做法是定义阻断项、观察项和接受项。阻断项必须修复，观察项可以灰度并设置告警，接受项记录为已知取舍。比如平均成本上升百分之五但引用准确率明显提高，可能是可接受的；高风险样本越权一次，则必须阻断。

失败模式

第一类失败是评测集陈旧。上线初期的样本来自团队想象，真实用户使用一段时间后，问题分布会变化。如果评测集不吸收线上失败，离线分数会越来越好看，线上体验却不一定提高。解决方法是建立样本回流机制，把负反馈、人工接管、异常成本和策略争议定期治理成回归样本。

第二类失败是 trace 过粗。只记录 prompt 和最终答案，看起来已经比没有日志好，但仍然不足以定位问题。一次 RAG 失败需要知道检索到了什么、丢弃了什么、引用了什么；一次 Agent 失败需要知道工具参数和策略判断；一次成本异常需要知道每次模型调用和重试。trace 过粗会让团队最后还是回到猜测。

第三类失败是指标口径不一致。离线评测用一个成本价格表，线上报表用另一个；评测通过率按样本算，线上成功率按请求算；人工接管在产品侧叫 handoff，运行时叫 escalation。口径不一致会让团队争论数字，而不是解决问题。指标字典和事件枚举应该像 API 一样维护。

第四类失败是过度依赖 LLM judge。模型评审器可以提高评测覆盖，但它也会漂移，也会偏好某些表达，也可能被样本内容影响。安全、权限、schema、引用 ID、工具参数、成本预算这些硬约束，不应该交给 LLM judge。开放文本质量可以让它辅助，但关键发布要有人类抽检和确定性规则兜底。

第五类失败是成本优化伤害质量。为了降低 token，团队把上下文裁得太狠，短期成本下降，引用准确率和用户信任下降；为了减少模型调用，团队取消了澄清步骤，导致错误自动执行增加；为了提高缓存命中，缓存键漏掉权限维度，出现数据边界风险。成本控制必须带着质量和安全指标一起看，不能单独优化。

第六类失败是成本长尾不可见。平均请求成本正常，但少数复杂任务非常贵。它们可能来自长文档、多轮工具循环、schema 反复重试或检索返回过多。长尾成本如果不处理，会在业务增长后突然变成预算问题。P95、P99、最大值和按任务分组的成本分布，比一个平均值更有用。

第七类失败是线上反馈无法转样本。用户点了差评，但系统没有保存当时 trace；人工接管了，但没有记录接管原因；客服说答案错了，但找不到原始上下文。反馈不能转样本，就不能进入回归，下一次还会发生。反馈入口要和 run_id 绑定，人工接管界面要提供失败分类，敏感数据要有合规的脱敏和留存策略。

第八类失败是版本维度缺失。团队看到某天负反馈升高，但那天同时改了 prompt、升级了模型、更新了索引，还启用了新缓存。没有版本维度，无法快速归因。每条 trace 都应该带上所有行为资产版本。评测报告也要记录评分器版本。版本信息不完整，等于主动放弃可解释性。

第九类失败是告警不可行动。比如告警写“AI 质量下降”，但没有说明哪个任务、哪个版本、哪个指标、从多少变到多少、样本链接在哪里。这样的告警只会制造噪音。好的告警应该指向具体维度，例如“release rel_0313 在 policy_qa 任务上的 schema_invalid 从 0.4% 升到 3.2%，主要错误字段是 citations”。这类告警能直接触发排查。

上线 checklist

下面这份 checklist 用于检查一个 AI 应用是否已经具备基本的评测、观测和成本控制能力。它不要求一次做完所有平台化能力，但关键字段和流程最好从第一天保留。

  • 每次请求都有 run_id，用户反馈、人工接管、告警和评测样本都能回到这个 run_id。
  • Trace 记录了输入摘要、上下文来源、模型调用、工具调用、schema 校验、策略判断、最终输出和成本。
  • Trace 中包含 release、prompt、model、schema、index、tool catalog、policy 和 scorer 等版本信息。
  • 敏感输入和输出有脱敏或受控引用，不在普通日志里裸写隐私数据。
  • 事件类型和失败类型有稳定枚举，团队对指标口径有共同定义。
  • 核心评测集、事故回归集、安全集、成本集和探索集分开管理，不用一个总分代表全部质量。
  • 评测样本可以重放，包含必要 fixture、用户角色、时间、期望行为和禁止行为。
  • 评分器分为确定性规则、语义评分和成本评分，高风险硬约束不依赖 LLM judge。
  • 发布报告能对比候选版本和基线版本的质量、可靠性、安全、延迟和成本。
  • 线上看板能按 release_id、task_type、tenant、model_route 切分指标。
  • 成本报表包含平均成本、P95 成本、最大成本、token、模型调用次数、工具调用次数和重试成本。
  • 每类任务有预算边界，包括最大上下文、最大工具调用、最大重试和降级策略。
  • 用户负反馈、人工接管、schema 失败、工具错误和成本异常能进入样本候选池。
  • 样本候选有脱敏、去重、归因、标注和审核流程，重要失败会进入回归集。
  • 灰度期间有影子评测或在线对比，候选版本的成本和质量能与稳定版本并排查看。
  • 告警能指向具体版本、任务、指标和样本链接，而不是只报一个模糊异常。
  • 每周或每个发布周期有人检查新增失败样本和成本长尾，并形成改进项。

如果只能先做三件事，我会优先做 trace、回归样本和成本分位数。Trace 让失败可解释，回归样本让修复可验证，成本分位数让预算风险提前暴露。这三件事打通后，很多高级能力会自然长出来。反过来，如果没有这三件事，直接引入复杂评测平台也容易变成漂亮报表。

上线前还要做一次数据留存和合规检查。AI trace 可能包含用户问题、企业文档片段、工具返回、模型输出和人工备注。不是所有内容都应该长期保存，也不是所有人都应该能看。要区分调试所需和合规边界，给敏感字段设置脱敏、加密、访问控制和过期策略。观测不是把所有东西都扔进日志，而是在可解释和可保护之间做工程设计。

总结

评测、观测和成本控制是 AI 应用的同一个闭环，不是三个独立项目。评测负责在改动前建立信心，观测负责在真实流量里发现事实，成本控制负责让系统在质量和预算之间保持可持续。三者通过 trace、样本和版本连接起来，团队才能知道一次变化为什么有效、为什么失败、值不值得继续。

真正可维护的 AI 应用，不会只追求一个更聪明的 prompt，也不会只盯着月底账单。它会把每次请求变成可解释的运行记录，把每次失败变成可回归的样本，把每次成本异常变成可归因的指标。这样，模型升级、检索优化、工具扩展和产品策略变化都能进入同一个工程反馈系统。

最后要接受一个现实：AI 应用不会被一次评测永久证明正确。用户问题会变，数据会变，模型会变，业务边界也会变。我们能做的是建立持续学习的机制，让系统每次出错都更容易被看见、被解释、被修复。评测、观测与成本控制的价值就在这里，它们让 AI 应用从不可预测的黑盒，逐步变成可以运营、可以迭代、可以负责的工程系统。
