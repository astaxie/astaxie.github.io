Agent 记忆系统设计

短期上下文、长期记忆和项目知识需要分层管理。

MemoryAgent

问题背景

Agent 记忆经常被讲成一个很轻松的功能：把用户说过的话存起来，下次拿出来用。真正做工程系统时会发现，这个说法太粗。Agent 的记忆不只是“记住偏好”，它还要记住任务状态、文件结构、工具经验、项目规则、失败原因、用户纠正、权限边界和最终交付。更麻烦的是，这些信息的生命周期完全不同。有些只在当前工具调用后有用，有些要贯穿整个任务，有些只适用于某个仓库，有些必须长期跟随用户，但又不能被错误地套用到所有场景。

如果没有分层，记忆系统会很快变成两个极端。第一个极端是“全塞上下文”。系统把历史对话、检索片段、项目说明、用户偏好和工具日志都放进 prompt，短期看省事，长期看成本高、噪声大、模型注意力混乱。第二个极端是“只存长期摘要”。系统每次任务结束写一段总结，下次相似任务再召回。这样会丢掉大量过程细节，也很难知道这条记忆是否仍然有效。工程 Agent 需要的是分层记忆，而不是一个万能记事本。

记忆系统的难点还在于它会影响行动。普通问答系统拿错一条知识，可能只是回答不准确；Agent 拿错一条记忆，可能会跳过检查、修改错误目录、调用不该调用的工具，或者把用户一次性的临时指令当成永久偏好。比如用户在一次写作任务里说“不要用脚本生成正文”，这是一条强约束，但它不应该影响下次生成测试数据的任务。再比如用户说“这个项目最终只要简短汇报”，这可能是当前任务的汇报风格，不代表遇到安全风险时也要省略说明。

记忆还必须能被撤销。人会改变习惯，项目会重构，命令会迁移，权限会收紧。一个记忆系统如果只会追加，不会过期、覆盖和降权，半年后就会积累大量“曾经正确”的事实。Agent 最危险的记忆往往不是明显错误，而是过期事实。它看起来合理，有历史证据，也能被检索召回，但已经不适用于当前代码和组织流程。

所以，Agent 记忆系统的目标不是尽可能多地记，而是把不同类型的信息放在合适的层级，用合适的写入门槛、检索方式和失效机制管理。短期上下文负责当前推理，工作记忆负责当前任务，项目知识负责可复用事实，长期记忆负责稳定偏好和经验，审计日志负责回放和纠错。每一层都应该有明确边界。

核心概念

我通常把 Agent 记忆分成五层：即时上下文、工作记忆、任务档案、项目知识和长期用户记忆。即时上下文是模型当前能看到的输入，适合放最近用户指令、当前计划和少量关键证据。工作记忆是任务运行时的结构化状态，适合放待办、观察、假设和验收。任务档案是任务结束后的可回放记录，适合放最终改动、测试结果和关键决策。项目知识是跨任务复用的仓库事实，适合放目录结构、构建命令、规范和常见坑。长期用户记忆是跨项目复用的偏好，适合放语言、沟通密度、风险偏好等稳定信息。

记忆层典型内容生命周期写入策略检索方式
即时上下文当前指令、最近观察、下一步动作单轮到少数几轮自动组装，不长期保存直接注入 prompt
工作记忆计划、假设、约束、验收状态当前任务调度器维护按任务 ID 读取

用户记忆偏好、沟通风格、稳定禁忌长期明确表达或多次确认用户级检索，低数量注入

这五层不是数据库表的硬分类，而是设计边界。即时上下文追求短小准确，工作记忆追求可恢复，任务档案追求可审计，项目知识追求可复用，用户记忆追求稳定和保守。把它们放在同一个向量库里不是不可以，但元数据和策略必须能表达层级差异。否则系统会把一次工具日志和长期偏好放在同等地位，检索时很难做正确判断。

记忆的另一个核心维度是作用域。作用域可以按用户、组织、工作区、仓库、目录、任务类型、会话和时间划分。作用域越宽，写入门槛越高；作用域越窄，自动写入可以更积极。比如“这个仓库的文章源文件在 content/articles”是项目知识，可以自动写入仓库级；“用户不喜欢长篇解释”如果只出现一次，就不应该写成用户级长期偏好；“这次任务只允许改四个文件”是会话约束，任务结束后不应作为项目事实长期生效。

记忆还要带证据和置信度。证据说明它从哪里来，置信度说明系统有多相信它。用户明确说出的约束，置信度高；从文件结构推断出的规范，置信度中等；从一次命令失败推断出的工具限制，置信度较低，除非多次验证。没有证据的记忆只能作为提示，不能作为自动行动依据。工程上要把“可参考”和“可执行”分开。

最后要区分“事实记忆”和“策略记忆”。事实记忆描述世界，例如某个测试命令、某个目录、某个 API。策略记忆描述如何做事，例如先读规范再写文章、失败两次后缩小复现范围、改权限前请求确认。事实记忆容易过期，策略记忆更抽象但也容易误用。一个成熟的系统会在检索时根据任务类型选择不同组合，而不是每次都召回全部相似内容。

架构/流程图解说明

可以把 Agent 记忆系统画成两条流水线：写入流水线和读取流水线。写入流水线从对话和工具事件里提取候选记忆，经过分类、作用域判断、敏感信息过滤、去重、确认和存储。读取流水线从当前任务出发，先确定需要哪些记忆，再做作用域过滤、语义召回、时效检查、冲突合并，最后生成一份短小的任务简报。

这张图里最重要的是“记忆需求计划”。很多系统一上来就按用户输入做向量搜索，这会召回大量语义相近但行动价值不高的内容。更稳的做法是先判断任务类型。写代码时需要项目命令、模块边界和历史失败；写文章时需要内容规范、元数据来源和风格要求；做线上运维时需要权限、变更窗口和回滚流程；回答概念问题时可能只需要用户语言偏好。检索不是越多越好，注入到模型里的记忆越少越清楚。

读取结果也不要原样塞进上下文。系统应该生成一份“记忆简报”，按优先级排列。第一部分是硬约束，例如不能修改某目录、必须请求确认、不得联网。第二部分是项目事实，例如构建命令、测试命令、文章元数据来源。第三部分是历史风险，例如某个测试不稳定、某个脚本会改产物。第四部分是软偏好，例如最终汇报要简短。每条都带来源和置信度，模型才能知道哪些必须遵守，哪些只是参考。

冲突处理需要单独设计。假设项目知识里有一条“使用 npm”，最近任务档案里有一条“迁移到 pnpm”，用户又在当前任务里说“按 README 来”。系统不能简单取相似度最高的一条。更好的规则是：当前用户指令优先于长期记忆；更近且有文件证据的项目事实优先于旧任务总结；明确废弃的记忆不参与自动注入；冲突无法解决时列为需要确认，而不是让模型猜。

在数据流上，审计日志要独立存在。长期记忆可能会被压缩、合并、降权，但审计日志应该保留原始事件引用，至少能回答“这条记忆为什么存在”。当用户质疑 Agent 为什么这么做，系统可以回到证据，而不是说“根据历史偏好”。这种可解释性对团队环境尤其重要，因为很多记忆其实来自某个成员的一次指令，不一定代表整个团队规范。

工程实现

落地时可以从一个简单的统一记录模型开始，但不同层级使用不同表或集合。下面是一个记忆记录的基础结构：

type MemoryRecord struct {
    ID          string
    Layer       string // working, task_archive, project_knowledge, user_memory
    Kind        string // fact, constraint, preference, procedure, failure, decision
    Summary     string
    Scope       Scope
    Evidence    []EvidenceRef
    Confidence  float64
    Sensitivity string // normal, private, secret
    Status      string // active, tentative, superseded, revoked, expired
    CreatedAt   time.Time
    UpdatedAt   time.Time
    ExpiresAt   *time.Time
}

type Scope struct {
    UserID     string
    OrgID      string
    Workspace  string
    Repo       string
    PathPrefix string
    TaskType   string
}

这个结构看起来普通，但关键在约束。Layer 决定生命周期，Kind 决定写入和检索策略，Scope 防止乱用，Evidence 支撑可解释性，Status 支撑撤销和覆盖。不要把这些都塞进一段自然语言摘要里。摘要是给模型读的，字段是给系统判断的。

写入策略可以用规则加模型共同完成。模型擅长从自然语言里提取候选，例如用户说“以后这个仓库的文章都从 js/articles.js 拿元数据”，模型可以提取为项目事实。规则负责判断能不能自动写入：如果作用域是当前任务，自动写入工作记忆；如果作用域是项目且证据来自文件和命令，可以写项目知识；如果作用域是用户长期偏好，最好要求明确确认；如果包含密钥、客户数据或个人隐私，则默认不写长期记忆。

一个候选写入策略表可以这样设计：

候选类型自动写入条件需要确认条件默认过期
会话约束用户当前任务明确要求无任务结束
项目命令文件证据加成功执行命令有外部副作用30 到 90 天后复验
用户偏好多次稳定出现且低风险会影响行为或输出边界不自动过期，但可撤销
失败经验有命令输出和定位结论失败原因只是推测14 到 30 天
流程经验多次任务验证会扩大权限或跳过检查90 天复验

读取实现要从过滤开始，而不是从向量相似开始。一个查询请求可以长这样：

{
  "user_id": "u1",
  "workspace": "astaxie.github.io",
  "repo": "astaxie.github.io",
  "task_type": "content_authoring",
  "needs": ["constraint", "fact", "procedure", "failure"],
  "exclude_status": ["revoked", "expired", "superseded"],
  "max_records": 12,
  "min_confidence": 0.6
}

系统先按用户、工作区、仓库、任务类型和状态过滤，再做语义召回和排序。排序不能只看相似度，还要看作用域距离、时间新鲜度、证据强度、风险级别和用户显式程度。当前任务越接近某条记忆的作用域，权重越高；证据越强，权重越高；太旧的事实权重降低；用户原话高于模型推断。

记忆注入建议使用固定格式，避免模型误解优先级：

记忆简报
硬约束：
- 当前任务只允许修改 content/articles 下四个目标 Markdown 文件。来源：用户当前指令。

项目事实：
- 文章 front matter 需要与 js/articles.js 中同 slug 元数据一致。来源：content/ARTICLE_STYLE.md、check_articles.py。

历史风险：
- 长文校验只检查最低汉字数和固定章节，不能代表文章质量。来源：项目校验脚本观察。

软偏好：
- 最终汇报保持简短。来源：用户当前指令。

这里的格式不是为了好看，而是为了让模型在行动时区分强弱。硬约束必须进入不变量检查；项目事实可以指导操作；历史风险提醒验证；软偏好只影响表达。实际系统里，硬约束最好同时进入程序层检查，不能只放进 prompt。

更新和撤销也要有明确 API。用户说“这条以后不用了”，系统应该能按摘要和作用域检索候选记忆，让用户确认后标记 revoked。新事实覆盖旧事实时，不要删除旧记录，而是把旧记录标为 superseded，并建立覆盖关系。这样审计时能看到变化过程，也能避免旧记录继续参与自动注入。

最后，记忆系统需要隐私和安全边界。默认不要长期保存密钥、访问令牌、客户个人信息、未公开商业数据和高敏操作细节。即使用户在上下文里提供了这些信息，也不代表系统可以把它们写入长期记忆。可以保存一个抽象事实，例如“该仓库需要私有包权限”，但不要保存具体令牌。记忆越持久，安全门槛越高。

测试评测

记忆系统的评测要分读取、写入、更新和安全四块。只测“能不能召回相似内容”远远不够。一个记忆系统可能召回率很高，但总是把会话约束当长期偏好；也可能相似度很好，但在权限受限任务里注入了不该出现的项目秘密。评测样本必须覆盖生命周期和作用域。

测试类型样本例子预期结果
作用域隔离A 仓库要求 pnpm，B 仓库使用 npmB 仓库任务不注入 A 的命令
会话约束过期本次任务只改四个文件下次无关任务不继续限制四个文件
用户偏好确认用户一次说“简短点”不写成永久偏好，或标为低置信
项目事实更新README 从 npm 改为 pnpm旧命令降权或被覆盖
冲突处理历史记忆与当前指令冲突当前指令优先，并记录冲突
敏感过滤用户输入访问令牌不写长期记忆，日志脱敏

读取评测可以建立一组任务输入和期望记忆简报。比如输入“在 astaxie.github.io 新增文章”，期望召回文章元数据来源、正文位置、校验命令和写作风格；不应该召回 Go 服务部署流程、另一个仓库的测试命令或用户私人偏好。评分时既看命中，也看误召回。误召回在 Agent 场景里很贵，因为每条错误记忆都有可能改变行动。

写入评测要看候选筛选。给系统一段任务轨迹，里面包含用户硬约束、工具失败、无关闲聊、敏感信息和最终测试结果，检查系统生成哪些记忆。好的写入器应该保留可复用事实和关键失败，丢弃闲聊，对敏感信息脱敏，对用户长期偏好保持保守。这里可以人工标注黄金集，也可以用规则先过滤，再让模型做候选解释。

更新评测要模拟时间变化。先写入“测试命令是 npm test”，再给出新的 package.json 和成功运行的 pnpm test，系统应该能降低旧记忆权重或建立覆盖关系。再过一段时间，如果没有复验，项目事实应该进入需要确认状态。时间不是装饰字段，它直接影响记忆可信度。

安全评测要专门做红队样本。用户可能要求“记住这个密码”，工具输出可能包含环境变量，日志里可能有内部 URL。系统要区分“任务中临时使用”和“长期保存”。对于高敏数据，最安全的默认策略是不写入长期记忆，并在审计日志里做脱敏。评测时要检查存储层，而不是只看最终 prompt，因为泄漏常常发生在后台记录里。

线上指标可以关注记忆命中率、误注入率、撤销次数、用户纠正次数、记忆过期命中次数和敏感拦截次数。命中率高不一定好，误注入率低才关键。用户经常纠正“不是这个项目”“我不是这个意思”，说明作用域和写入策略有问题。撤销次数增加不一定是坏事，可能说明用户开始信任系统可以管理记忆；但撤销后仍被召回就是严重缺陷。

失败模式

第一类失败是记忆泛化过度。系统把一次会话里的局部要求写成长期偏好，例如把“这次不要解释太多”变成“用户永远喜欢极短回答”。这种失败会让 Agent 越用越窄，像被过去的偶然指令绑住。解决办法是扩大作用域必须有更高证据门槛，尤其是用户级记忆要保守。

第二类失败是项目事实过期。仓库迁移了包管理器，目录结构重构了，测试命令改了，但记忆仍然注入旧信息。模型看到旧命令后会优先使用，因为它看起来像系统给出的权威上下文。解决办法是给项目事实设置复验时间，并在读取时结合当前文件证据。对会直接驱动命令的记忆，最好在执行前做轻量验证。

第三类失败是记忆噪声。系统把每次任务总结都写入长期库，几个月后相似搜索返回一堆低价值片段。模型读了很多“本次已完成”“测试通过”“用户要求简短”之类的内容，反而忽略真正重要的约束。解决办法是引入写入预算和去重合并，把任务档案和可复用知识分开。

第四类失败是证据断裂。记忆摘要写着“不要修改构建产物”，但找不到它来自哪次用户指令、哪个文件或哪个规则。遇到冲突时，系统无法判断谁更可信，只能靠模型猜。解决办法是所有可执行记忆必须带证据引用，没有证据的内容只能低权重提示。

第五类失败是敏感信息长期化。Agent 在调试时看到了令牌、客户数据或内部地址，然后把它们写进长期记忆或向量索引。即使后续不注入 prompt，存储本身也可能成为安全风险。解决办法是写入前做敏感检测、脱敏和层级限制，对长期记忆使用更严格的白名单。

第六类失败是撤销无效。用户说“忘掉这个规则”，系统只是追加一条“不要使用旧规则”的新记忆，旧规则仍然能被召回。正确做法是修改旧记录状态，建立撤销事件，并让读取查询默认排除已撤销内容。撤销应该影响检索层，而不是只影响展示层。

上线 checklist

  • 是否明确区分即时上下文、工作记忆、任务档案、项目知识和长期用户记忆。
  • 每条可复用记忆是否包含作用域、证据、置信度、状态和更新时间。
  • 会话约束是否会在任务结束后失效，而不是进入长期记忆。
  • 用户级长期偏好是否需要明确表达或多次验证。
  • 项目事实是否有复验机制，过期后是否降权或请求确认。
  • 检索是否先做作用域、权限和状态过滤，再做语义召回。
  • 注入给模型的记忆是否分清硬约束、项目事实、历史风险和软偏好。
  • 冲突记忆是否有优先级规则，当前用户指令是否最高优先级。
  • 敏感信息是否在写入长期存储前被拦截或脱敏。
  • 用户是否能查看、撤销和修正长期记忆。
  • 是否有轨迹样本测试写入、读取、覆盖、撤销和安全策略。

总结

Agent 记忆系统的价值，不在于让系统显得“更懂你”，而在于让协作可以延续。它应该帮 Agent 少走重复弯路，记住项目边界，复用经过验证的流程，并在风险出现时提醒自己。但记忆越能影响行动，越需要被治理。没有作用域、证据、状态和过期机制的记忆，迟早会从助手变成负担。

工程上最稳的路线，是先把记忆分层，再逐步自动化。当前任务靠工作记忆维持，任务结束写档案，只有少量有证据、有作用域、未来可复用的内容进入项目知识或长期用户记忆。读取时生成短小的记忆简报，写入时保持保守，冲突时尊重当前用户指令。这样的系统不会追求一次记住所有东西，但会让每一条被记住的东西更值得信任。
