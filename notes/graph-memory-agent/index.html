Agent 记忆和 GraphRAG

长期 Agent 需要把任务、偏好、工具和结果写回可检索的图谱记忆。

Agent MemoryGraphRAG

问题背景

Agent 做一次性任务时，记忆问题并不明显。用户给出目标，模型读取当前上下文，调用工具，产出结果，任务结束。但当 Agent 需要长期协作，问题就来了：它要记住用户偏好、项目约束、工具使用经验、失败过的方案、某个仓库的结构、某个接口的脾气，以及上次为什么没有继续某条路线。把这些内容全部塞进 prompt 既昂贵又混乱；只靠对话历史又很脆弱，因为历史里有试探、错误、临时决定和已经过期的结论。

很多产品把 Agent memory 做成一个简单的文本列表，比如“用户喜欢 Go”“用户不喜欢长回答”“项目使用 PostgreSQL”。这对轻量个性化有用，但对工程 Agent 远远不够。工程协作里的记忆不是孤立偏好，而是带关系的事实：用户在某个项目里偏好 Go；这个偏好不一定适用于前端；某个工具在这个仓库里失败过，因为环境变量缺失；某个测试不能并行跑，因为会争用临时端口；某次迁移被暂停，是因为数据回填没有完成。这些记忆如果不带范围、证据和时效，很容易变成错误自动化。

GraphRAG 适合处理 Agent 记忆，是因为长期记忆天然有图结构。任务依赖工具，工具依赖权限，项目包含模块，模块关联测试，用户偏好只在某些上下文有效，过去的失败经验指向具体命令和错误。向量检索能找到相似片段，但很难稳定回答“这个经验是否适用于当前任务”。图谱可以把“适用范围”和“证据路径”显式表示出来，再配合向量召回处理模糊描述。

Agent 记忆还有一个比普通知识库更敏感的点：它会反过来影响行为。RAG 知识库答错了，最多是误导用户；Agent 记忆错了，可能会自动跳过检查、调用危险工具、修改错误文件，甚至把一次临时授权当成永久权限。因此记忆系统必须有写入门槛、置信度、撤销机制和可审计记录。记忆不是模型的私人笔记，而是可治理的运行状态。

一个具体场景：用户连续几周让 Agent 维护一个博客仓库。第一次任务里，Agent 发现文章源文件在 content/articles，生成页面由构建脚本处理；第二次任务里，用户要求只改 Markdown，不碰构建产物；第三次任务里，校验脚本要求中文汉字数量；第四次任务里，用户明确不希望用脚本生成正文。一个好的 Agent 应该能记住这些项目级约束，并在下次类似任务开始时主动应用。但它也要知道这些记忆只适用于这个仓库，不应该迁移到另一个项目。

核心概念

Agent 记忆至少分成五类：身份偏好、项目事实、任务经验、工具经验和结果记忆。身份偏好描述用户长期习惯，例如沟通风格、语言偏好、风险偏好。项目事实描述某个工作空间的结构、约定、命令和模块边界。任务经验记录某类任务的成功流程和失败坑。工具经验记录命令、API、浏览器、数据库等外部工具的可用性和限制。结果记忆记录某次任务做了什么、为什么做、有没有后续风险。

记忆类型生命周期写入来源检索触发风险
用户偏好长期，但需要确认明确表达、重复行为所有任务的沟通和决策过度概括用户意图
项目事实随仓库变化文件读取、命令结果、用户说明当前工作区任务代码变化后过期
任务策略中期，可被复盘修正成功任务总结、失败诊断相似任务开始前把偶然流程当最佳实践
工具经验短中期工具返回、错误日志调用同类工具前环境变化导致误判
结果记忆按任务保存最终改动、测试结果后续维护、回滚、解释保存过多噪声

GraphRAG 在这里的作用不是把所有记忆都变成知识图谱，而是让关键关系可检索、可解释、可更新。记忆节点可以包括 User、Workspace、Repository、Task、Constraint、Tool、Command、Artifact、Failure、Preference。关系边可以包括 applies_to、observed_in、caused_by、verified_by、supersedes、conflicts_with、requires_confirmation。

记忆必须有作用域。没有作用域的偏好是危险的。例如“不要写脚本生成正文”可能只是某一次内容创作任务的要求，不应该影响代码生成任务；“测试需要 pnpm test”可能只适用于一个前端仓库；“用户希望简短汇报”可能适用于最终总结，不代表中间遇到风险时也不解释。作用域可以是用户级、组织级、仓库级、目录级、任务类型级，也可以是一次会话级。写入时不确定作用域，就应该保守地放窄。

记忆还要有证据。证据可以是用户原话、文件内容、命令输出、测试结果或人工确认。没有证据的记忆只能作为低置信提示，不能驱动自动行动。比如 Agent 推断“这个项目不用 TypeScript”，只是因为当前读到的是 JavaScript 文件，这种推断不应该写成强事实；只有看到 package.json、构建配置和用户确认后，才可以提高置信度。

另一个核心概念是“写入策略”。Agent 每完成一个任务都会产生大量上下文，但大部分不值得长期保存。记忆写入应该经过筛选：是否未来可复用，是否有明确作用域，是否有证据，是否会影响行为，是否包含敏感信息，是否需要用户确认。未经筛选的自动记忆会变成垃圾场，检索时反而拖累决策。

架构/流程图解说明

一个可运行的 Agent 图记忆系统，可以由“观察、候选、审核、存储、检索、注入、反馈”七段组成。

用户目标 + 当前上下文 + 工具结果
        |
        v
记忆观察器：识别偏好、约束、事实、失败、成果
        |
        v
候选记忆生成：内容、作用域、证据、置信度、敏感级别
        |
        v
写入策略：自动保存 / 需要确认 / 丢弃 / 仅会话保存
        |
        v
图记忆库 + 向量记忆库 + 原始证据仓库
        |
        v
任务开始时检索：意图识别 -> 作用域过滤 -> 图扩展 -> 重排
        |
        v
记忆注入：约束、警告、已知流程、相关证据
        |
        v
Agent 行动与结果复盘，再写回记忆

这里有两条线要分开看。第一条是写入线，它负责从运行过程里提炼记忆。第二条是读取线，它负责在下一次任务开始前找出真正相关的记忆。很多系统只重视读取，忽略写入质量；最后图谱里充满“可能”“似乎”“上次看到”的低质量节点。长期看，写入质量比检索算法更重要。

读取线通常分三步。第一步按作用域过滤：当前用户、当前仓库、当前任务类型、当前权限。第二步用向量召回找语义相近的记忆，例如“只改文章源文件”和“不要改构建产物”可能表达不同但含义接近。第三步用图扩展找关联证据，例如某个约束由哪次用户指令产生、在哪些任务里被验证、有没有被后来的记忆覆盖。

记忆注入不要把全部召回内容塞给模型。更好的方式是生成一段“运行前简报”，分成硬约束、软偏好、已知风险、可复用流程和需要确认的事项。例如：

硬约束：
- 当前仓库的文章正文必须手写 Markdown，不能用脚本或循环模板生成。
- 文章源文件位于 content/articles，校验脚本会比对 js/articles.js 的 front matter。

已知风险：
- 中文长文任务容易满足脚本校验但不满足用户对深度内容的要求，需要人工组织结构。
- 只允许修改目标 Markdown 文件，避免触碰构建产物。

需要确认：
- 无。用户本次已经给出目标 slug 和校验命令。

图解里还有一个“撤销”通道。用户可以说“忘掉这个偏好”“这只适用于上次”“以后不要按那个流程”。系统要能找到对应记忆节点，标记为废弃或缩小作用域，而不是简单追加一条相反记忆。否则检索时两条记忆同时出现，模型又要自己猜哪条有效。

工程实现

实现时我会把记忆库拆成三类存储：结构化图表、向量索引、证据日志。图表保存节点和边，向量索引保存可语义召回的记忆文本，证据日志保存原始来源。三者通过稳定 ID 关联。不要只把图谱存在向量库的 metadata 里，也不要只用图数据库做全文相似；两者各有长处。

一个简化的记忆节点结构如下：

type MemoryNode struct {
    ID          string
    Type        string // preference, constraint, project_fact, tool_experience, task_result
    Summary     string
    Scope       MemoryScope
    Confidence  float64
    Sensitivity string // public, private, secret
    Status      string // active, tentative, superseded, revoked
    EvidenceIDs []string
    CreatedAt   time.Time
    UpdatedAt   time.Time
    ExpiresAt   *time.Time
}

type MemoryScope struct {
    UserID     string
    Workspace string
    Repo       string
    PathPrefix string
    TaskType   string
}

边结构要支持时效和证据：

type MemoryEdge struct {
    FromID     string
    ToID       string
    Type       string // applies_to, verified_by, caused_by, supersedes, conflicts_with
    EvidenceID string
    Confidence float64
    CreatedAt  time.Time
}

写入流程可以按四个阶段走。第一阶段是候选提取，从用户明确指令、工具失败、测试成功、最终总结里找可能记忆。第二阶段是归类和作用域判断，例如“只修改上述四个 Markdown 文件”应该是会话级硬约束，不一定写长期记忆；“这个仓库文章元数据来自 js/articles.js”可以写项目事实。第三阶段是去重和合并，避免每次任务都写一条相同约束。第四阶段是策略决策，高风险或长期偏好需要用户确认。

候选记忆可以设计成这样：

{
  "summary": "在 astaxie.github.io 中，文章 front matter 需要与 js/articles.js 的对应 slug 元数据保持一致。",
  "type": "project_fact",
  "scope": {
    "workspace": "astaxie.github.io",
    "repo": "astaxie.github.io",
    "path_prefix": "content/articles"
  },
  "evidence": [
    {"kind": "file", "uri": "content/ARTICLE_STYLE.md"},
    {"kind": "command", "uri": "python3 scripts/check_articles.py"}
  ],
  "confidence": 0.92,
  "write_policy": "auto"
}

检索流程要避免“记忆污染当前任务”。我通常会先根据当前任务生成一个检索计划，而不是无差别搜索所有记忆。计划里包括任务类型、工作区、需要的记忆类别和排除项。比如用户要求修复 CI，就优先检索项目命令、历史失败、工具权限和最近改动；用户要求写文章，就优先检索内容规则、风格要求和文件约束；用户只是问概念，不应该注入一堆仓库操作经验。

一个任务开始前的查询计划可以是：

{
  "task_type": "content_authoring",
  "workspace": "astaxie.github.io",
  "needs": ["project_fact", "constraint", "tool_experience"],
  "avoid": ["old_task_result", "unconfirmed_preference"],
  "max_items": 12,
  "require_evidence": true
}

重排时不要只看向量相似度。记忆和当前任务的适配度至少由五个分数组成：语义相似、作用域匹配、证据强度、时效、状态。一个过期但语义很像的记忆应该被压下去；一个作用域完全匹配但表达略有差异的项目事实应该被提上来。简单公式可以是：

score = 0.30 * semantic
      + 0.25 * scope_match
      + 0.20 * evidence_quality
      + 0.15 * freshness
      + 0.10 * status_weight

对于会影响外部世界的记忆，要加确认门。比如“用户允许自动提交代码”“这个工具可以删除远端分支”“部署失败时直接回滚”。这些记忆即使曾经出现过，也不能直接触发危险操作。图谱里可以把它们标成 requires_confirmation，检索注入时只作为提醒，不作为授权。

工程上还要做记忆压缩。长期运行后，任务结果会很多。可以把相同项目、相同任务类型的一组结果压缩成“经验摘要”，同时保留原始证据。压缩不是覆盖，而是新增一个摘要节点，并通过 summarizes 边指向原始任务。这样检索时先看摘要，必要时再展开证据。

记忆合并需要单独设计，否则图谱会被近义节点撑爆。合并不能只靠文本相似度，因为“不要自动提交代码”和“提交前需要用户确认”语义相近但强度不同；“这个仓库使用 pnpm”和“这个仓库曾经使用 yarn”也不能直接合并。我的做法是先按作用域和类型分桶，再在桶内做候选合并，最后根据证据和状态决定是合并、覆盖、冲突还是保留并列。合并结果要写成新节点或新边，保留原始证据，不要直接删除旧节点。

一个合并判定表可以这样用：

新记忆旧记忆判定处理
同一仓库使用 pnpm test同一仓库使用 pnpm test重复增加验证证据，提高置信度
同一仓库使用 pnpm同一仓库使用 yarn可能覆盖查找时间和配置证据，建立 supersedes
本次不要跑测试项目提交前要跑测试作用域不同新记忆仅会话保存
用户喜欢简短回答用户要求详细解释架构场景不同分别绑定沟通场景

记忆读取也要支持“反证”。如果当前任务里读到的文件、命令输出或用户指令与旧记忆冲突，系统不应该继续把旧记忆当硬约束。比如记忆说测试命令是 pnpm test:unit，但当前 package.json 已经删除这个脚本，Agent 应该降低记忆置信度，转而读取当前配置，并在任务结束后写入修正。长期记忆不是比当前证据更高级的真理，它只是帮助 Agent 少走弯路的历史线索。

在多 Agent 协作里，还要区分私有记忆和共享记忆。一个 Agent 在本地调试得到的工具经验，未必能直接给另一个 Agent 使用；一个团队确认的项目事实，才适合进入共享层。共享记忆最好要求更高证据门槛，例如必须来自仓库文件、CI 结果、用户确认或合并后的文档。这样可以避免某个 Agent 的误判污染所有执行者。

记忆删除也不是简单物理删除。用户要求“忘掉”某条内容时，系统要区分隐私删除、行为撤销和历史废弃。隐私删除意味着证据和向量都要清除；行为撤销意味着以后不能再用这条记忆指导行动，但可以保留一条审计记录说明它已撤销；历史废弃意味着旧事实曾经成立，现在被新事实覆盖，适合保留 superseded 状态。三者混在一起，会导致要么删不干净，要么失去必要的历史解释。

记忆系统还要处理“沉默偏好”。用户没有明说，但多次纠正 Agent，例如“不要这么啰嗦”“这类改动要先跑测试”“别动无关文件”。这些信号有价值，但不能直接变成强规则。可以先写成低置信候选，累计多次同类证据后提升置信度，或者在下一次相关任务开始时用一句话确认：“我看到这个仓库之前要求只改目标文件，这次也按这个边界执行吗？”这种确认会让记忆更可靠，也让用户知道系统正在学习什么。

最后，记忆注入要有预算。长期记忆再准确，也会占用模型注意力。可以给每次任务最多注入若干条硬约束、若干条风险提示和少量流程经验，其余通过工具按需查询。对于复杂任务，Agent 可以在计划阶段再检索第二批记忆，而不是一开始把所有相关内容塞进去。记忆系统的目标是降低认知负担，不是制造另一种上下文噪声。

测试评测

Agent 记忆的评测要比普通 RAG 更像行为测试。因为最终目标不是答得好听，而是让 Agent 在正确时机使用记忆，在不确定时克制，在记忆冲突时请求确认。可以准备一组回放任务，把同一个 Agent 在有记忆和无记忆两种条件下运行，比较它的计划、工具调用、文件修改和最终回答。

评测项样本设计通过标准
作用域准确两个仓库有相反约定只在匹配仓库注入对应记忆
过期处理新记忆废弃旧流程不继续使用 superseded 记忆
写入质量任务里有临时指令和长期事实只保存可复用且有证据的内容
行为改善重复任务包含历史坑Agent 避免重复失败
克制能力检索到低置信偏好输出需要确认而不是直接执行
隐私安全记忆含敏感路径或密钥片段不进入普通上下文和日志

一个回放测试可以这样设计。第一轮，用户让 Agent 在仓库 A 里运行测试，Agent 发现 npm test 会因环境缺失失败，正确命令是 pnpm test:unit。第二轮，在同一仓库里用户要求改组件，Agent 应该优先使用 pnpm test:unit。第三轮，在仓库 B 里用户也要求改组件，Agent 不应该套用仓库 A 的命令。这个测试看似简单，实际能暴露作用域过滤、证据质量和记忆注入三个问题。

写入评测也要独立做。给 Agent 一段任务轨迹，里面包含用户长期偏好、一次性限制、错误推断、工具失败和最终成功。要求记忆系统输出候选记忆。人工标注哪些应该保存、保存到什么作用域、是否需要确认。这个评测能防止系统把所有东西都记下来，也能防止它错过真正有价值的项目事实。

冲突评测很关键。比如早期记忆是“这个项目用 yarn”，后来的证据显示项目迁移到 pnpm。系统应该生成 supersedes 边，默认使用新记忆，并在必要时说明迁移来源。如果只是两条都存着，模型可能在一次回答里同时建议 yarn install 和 pnpm install。

还要做“无记忆基线”对比。很多记忆系统看起来复杂，但实际没有改善任务结果，只是增加上下文噪声。每月抽取一批重复任务，用相同模型分别在关闭记忆和开启记忆下运行，比较完成时间、工具失败次数、用户纠正次数、测试通过率。记忆系统的价值应该体现在更少重复探索、更少违反项目约定、更快定位入口，而不是让回答显得更个性化。

可观测指标包括：候选记忆数量、自动写入比例、用户确认比例、记忆命中率、命中后被模型使用比例、因记忆导致的失败次数、被撤销记忆数量、平均作用域宽度。尤其要看“命中但未使用”和“使用后出错”。前者说明注入内容不清晰或太多，后者说明记忆质量或作用域有问题。

失败模式

第一种失败是“过度记忆”。Agent 把每个临时要求都写成长期偏好，下一次任务就开始自作主张。比如用户某次说“这次不要跑测试”，系统记成用户不喜欢测试，以后就跳过验证。这类问题只能靠写入策略和作用域控制解决。

第二种失败是“无证据自信”。模型在任务中推断出某个项目事实，但没有文件或命令证据，就写入高置信记忆。下一次任务基于这个记忆行动，错误会被放大。所有自动写入的事实都应该能追溯到证据，推断类记忆要标成 tentative。

第三种失败是“图谱越来越旧”。仓库结构变了，命令改了，工具权限调整了，但记忆还在。Agent 记忆要有过期时间和验证机制。项目事实可以在相关文件变更时触发复查；工具经验可以在下一次成功调用后刷新；长时间未命中的记忆可以降权。

第四种失败是“跨用户泄露”。在多用户系统里，把一个用户的偏好或项目路径注入另一个用户的上下文，是严重事故。记忆库的权限模型不能只靠应用层约定，存储查询本身就要带用户和租户隔离。向量索引也要分区或强过滤。

第五种失败是“记忆压制探索”。系统记住过去某个方案失败，于是永远不再尝试。但当依赖升级、环境修复或需求变化后，旧失败可能已经不适用。失败记忆应该记录原因和条件，而不是简单写成“不要做某事”。检索注入时也应该表达为风险提示，而不是绝对禁令。

第六种失败是“摘要丢失责任”。为了节省上下文，把多个任务压缩成一句经验，但没有保留原始证据。后来经验错了，没人知道它来自哪次任务，也不知道该怎么修。压缩记忆必须保留 summarizes 关系，支持展开。

第七种失败是“用户无法纠正”。如果用户说“你记错了”，系统却没有可见的记忆条目和撤销入口，用户会迅速不信任 Agent。长期记忆应该可以展示、编辑、禁用和解释。越是会影响行为的记忆，越要让用户能看见。

上线 checklist

  • 明确记忆类型：偏好、项目事实、任务经验、工具经验、结果记录分别处理。
  • 每条长期记忆都有作用域、证据、状态、置信度、敏感级别和更新时间。
  • 写入策略区分自动保存、用户确认、仅会话保存和丢弃。
  • 高风险授权、外部副作用和安全相关内容不能仅凭记忆自动执行。
  • 检索前先计算当前用户、工作区、仓库和任务类型，避免跨域注入。
  • 重排同时考虑语义相似、作用域匹配、证据质量、时效和状态。
  • 记忆注入使用简报格式，区分硬约束、软偏好、风险和待确认事项。
  • 支持记忆撤销、废弃、缩小作用域和冲突合并。
  • 向量索引、图存储、证据日志都执行同一套权限隔离。
  • 为写入候选建立人工标注评测集，防止过度记忆和漏记关键事实。
  • 为行为改善建立回放测试，比较开启记忆前后的工具失败和用户纠正。
  • 记录记忆命中、使用、导致失败、被撤销等运行指标。
  • 定期清理低置信、长期未命中、证据失效的记忆。
  • 给用户提供可查看、可解释、可删除的记忆管理界面。
  • 在最终回答中说明关键行为是否受长期记忆影响，必要时列出依据。

总结

Agent 记忆不是把聊天记录塞进一个向量库。真正可用的长期记忆要知道自己从哪里来、适用于哪里、现在是否仍然有效、会影响什么行为。GraphRAG 的价值在于把记忆之间的关系显式化：偏好作用于任务，任务验证工具，失败由条件触发，新事实覆盖旧事实，敏感内容需要确认。这样 Agent 才能把过去经验转化成稳健行动，而不是带着一堆模糊印象继续工作。

我会把 Agent 记忆当成一套小型知识治理系统来做。先从项目事实和工具经验开始，因为它们最容易验证，也最能减少重复探索；再逐步加入用户偏好和任务策略；最后才考虑自动压缩和跨项目迁移。每一步都要有评测和撤销机制。长期 Agent 的能力不只来自更大的上下文窗口，也来自它能否负责任地记住、忘记和解释自己的经验。
