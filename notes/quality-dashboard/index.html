AI 质量看板

质量看板要把准确率、延迟、成本和人工接管放在一起。

DashboardQuality

问题背景

AI 应用上线以后，团队最容易低估的是“质量”这个词的混乱程度。产品同学说质量下降，可能指用户觉得答案啰嗦；业务负责人说质量不稳定，可能指同一个问题今天答对明天答错；工程同学看日志却发现接口成功率很高、模型没有报错、平均延迟也不差。大家都在说质量，却拿着不同证据。最后会议会变成几个截图互相传阅，谁的案例更吓人谁就赢，工程行动反而不清楚。

传统 Web 系统的看板通常从错误率、延迟、吞吐和资源占用开始。AI 系统当然也需要这些指标，但远远不够。一次模型请求 HTTP 200，并不代表答案可用；一次答案被用户复制，也不代表事实正确；一次离线评测通过，也不代表真实流量里没有权限、时效、上下文冲突和安全边界问题。AI 质量看板必须把“用户是否完成任务”“答案是否可靠”“系统是否在可接受成本内交付”“人是否需要接管”放在同一张桌面上看。

很多团队会先做一个漂亮仪表盘：请求量、成功率、平均延迟、token 数、成本曲线、满意度按钮。第一周大家很兴奋，第二周没人打开。原因不是图表不够多，而是图表没有回答上线决策。一个有用的质量看板应该能让值班同学在十分钟内判断：今天是不是出了质量事故，事故影响哪个租户或功能，是否由模型版本、prompt 版本、检索数据、工具依赖、流量结构或用户行为变化引起，应该回滚、降级、扩样评测还是派人复核。不能指导行动的图表只是墙纸。

AI 质量还有一个特点：它不是单点数字，而是多目标约束。更高准确率可能带来更高延迟，更强模型可能带来更高成本，更保守安全策略可能提高拒答率，更激进的工具调用可能提高任务完成率但增加失败路径。质量看板如果只展示一个综合分，会诱导团队做错误优化。比如把所有低置信度问题都拒答，事实错误率下降了，任务完成率也被一起打穿；把模型换成更便宜版本，成本下降了，人工接管和用户重试却上升了。质量看板要能暴露这些交换关系。

对社区和小团队来说，另一个现实问题是资源有限。不是每个功能都能建设完整的评测平台、标注平台和 BI 系统。我的建议是先把质量看板做成生产反馈闭环，而不是一次性做成大平台。第一阶段抓住关键路径：统一 trace、记录 prompt 和模型版本、抽样保存输入输出、接入人工复核、按功能聚合质量门槛。只要这些基础事实准确，后面加图表、告警和自动评测都比较自然。反过来，如果事实层乱了，仪表盘越多，噪声越大。

质量看板的核心目标，是让工程团队不再靠感觉讨论模型质量。它要把线上请求、离线评测、人工反馈、成本延迟和发布版本串起来，让每一次质量波动都能被定位，让每一次模型、prompt、检索或工具改动都能被量化比较。它不是汇报用的装饰，而是 AI 应用的驾驶舱。

核心概念

第一个概念是质量事件。一次用户请求会产生很多事实：输入是什么、走了哪个功能、命中了哪个模型路由、用了哪些检索片段、调用了哪些工具、花了多少时间和成本、输出是否通过结构化校验、用户是否采纳、是否触发投诉或人工复核。看板不要直接从散落日志里拼指标，而要先沉淀统一的质量事件。事件是后续评测、聚合、告警和复盘的共同语言。

第二个概念是任务成功。AI 应用的质量不能只看答案文本本身，还要看用户任务是否完成。客服助手的任务成功可能是“用户没有继续追问同一问题，工单没有升级”；代码助手的任务成功可能是“建议被接受并通过测试”；知识库问答的任务成功可能是“答案引用了正确文档，用户没有点击负反馈”；数据分析代理的任务成功可能是“生成的 SQL 执行成功，图表被用户保留”。不同功能必须定义自己的成功信号，不能用一个通用点赞按钮糊弄过去。

第三个概念是分层质量。最底层是系统可用性，包括错误率、超时率、流式中断率、工具失败率。中间层是生成可用性，包括格式正确率、引用覆盖率、事实一致性、拒答合理性。上层是业务有效性，包括采纳率、人工修改率、复购或留存、投诉率。看板要让这三层同时出现。只看上层业务指标，反馈太慢；只看底层工程指标，又看不到答案真的好不好。

第四个概念是人工接管。人工接管不是失败的耻辱，而是质量系统的一部分。真实业务里，总会有高风险、低置信度、权限复杂或信息不足的请求需要人来兜底。看板应该记录接管率、接管原因、接管时机、人工修改幅度和最终处理结果。一个系统如果从不接管，可能不是质量好，而是风险没有被识别；一个系统如果接管率很高，可能是模型弱，也可能是产品边界定义不清。

第五个概念是评测校准。离线评测集、自动裁判和线上反馈之间必须互相校准。自动裁判说通过，不代表用户满意；用户点踩，不一定代表事实错误，也可能是语气不合适、答案太长、等待太久。看板要展示评测指标和人工复核之间的相关性。比如裁判模型给出的事实分低于 0.6 时，人工判错比例是否真的上升；引用缺失是否真的导致更多投诉；延迟超过十秒后，用户取消率是否明显增加。没有校准，自动评测会变成另一个黑盒。

第六个概念是切片。全局平均质量通常没有诊断价值。质量看板必须支持按租户、功能、渠道、语言、模型版本、prompt 版本、知识库版本、风险等级、输入长度、工具路径、发布时间和用户类型切片。很多问题只发生在一个切片里：中文效果好英文差，新文档检索好旧文档差，短问题好长问题差，内部用户好外部客户差。切片能力决定了看板能不能从“质量下降”走到“哪一段链路出了问题”。

指标层级代表指标主要用途误用风险
系统可用性错误率、超时率、流式中断率、工具失败率值班、容量、事故发现HTTP 成功被误认为答案成功
生成可用性格式通过率、引用命中率、事实一致性、拒答合理性模型和 prompt 回归自动裁判漂移导致虚假稳定
业务有效性采纳率、重试率、人工修改率、投诉率产品决策、客户复盘受流量结构和界面变化影响
成本效率单任务成本、重试成本、人工接管成本路由和套餐策略只降成本不看质量损失
风险控制敏感输出率、权限越界率、人工升级率安全和合规过度拒答造成任务失败

我不建议一开始就追求一个“AI 质量总分”。总分可以作为管理层入口，但工程看板必须保留分项。一个健康的质量看板应该像体检报告，而不是像考试分数。血压、血糖、肝功能和心电图不能简单相加，AI 质量也一样。看板的价值在于让团队知道该看哪一项、该找谁、该改哪里。

架构/流程图解说明

质量看板的架构可以从请求链路开始，而不是从 BI 页面开始。每一次 AI 请求在入口创建 trace，业务服务补充功能和租户信息，检索层记录证据，模型网关记录版本和 usage，后处理记录校验结果，前端或业务回调记录用户行为，人工复核系统记录标签。所有事实进入事件流，再进入明细表和聚合表。看板只是这些事实的一个消费端。

  v
AI 入口服务：创建 trace_id、feature、tenant、risk_level
  |
  v
执行链路
  |-- 检索：query、doc_version、chunk_id、score、permission_scope
  |-- 模型网关：model_id、prompt_version、input_tokens、output_tokens
  |-- 工具调用：tool_name、latency、success、error_code
  |-- 后处理：schema_valid、citation_valid、safety_result
  |
  v
质量事件流
  |-- request_finished
  |-- eval_finished
  |-- user_feedback
  |-- human_reviewed
  |
  v
明细仓库 + 指标聚合 + 告警规则
  |
  v
质量看板、发布门禁、复盘报告、样本挖掘

这张流程图里最重要的是事件之间要通过同一个 trace_id 串起来。否则用户投诉在客服系统里，模型 usage 在供应商账单里，检索日志在向量库里，prompt 版本在代码仓库里，人工复核在表格里，最终没有人能复盘一次失败请求到底发生了什么。很多质量看板失败，不是图表问题，而是 trace 没有贯通。

第二个关键点是区分明细和聚合。明细数据用于排查，聚合指标用于趋势。明细要尽量保留足够上下文，但要做好脱敏、权限和保留周期；聚合要稳定、可比较、能被告警使用。不要在看板查询时临时解析大段 JSON，也不要让每个图表自己定义一次指标口径。推荐做一层指标定义，把“引用正确率”“人工接管率”“有效任务成本”这些口径固化下来，发布时版本化。

第三个关键点是把发布信息放进同一条链路。每次模型升级、prompt 调整、检索索引重建、工具参数变化、后处理规则改动，都应该带上版本号。质量曲线发生变化时，看板才能直接标出发布事件。没有版本标记，排查会变成翻聊天记录：“昨天下午谁改了 prompt？”“索引是不是重建过？”“供应商是不是切了模型？”这种方式在小团队也许还能撑一阵，在多功能、多租户系统里很快失控。

一个实用的看板首页可以按三层组织。第一层是全局健康：请求量、任务成功率、P95 延迟、单任务成本、人工接管率、严重安全事件。第二层是质量回归：与过去七天、上一个发布版本、离线基准相比的变化。第三层是可行动切片：下降最明显的功能、租户、模型路由、prompt 版本、知识库版本。首页不要塞满所有图，而要帮助值班同学快速找到下一步。

首页判断路径
  1. 今天是否异常：红线指标是否破阈值
  2. 异常在哪：按 feature / tenant / route / version 排名
  3. 为什么异常：查看 trace 样本、失败类型、发布标记
  4. 怎么处理：回滚、降级、扩样评测、人工复核、创建 issue

质量看板不是替代日志系统、链路追踪系统或 BI 系统。它应该站在 AI 应用语义之上，把底层观测数据重新组织成工程决策语言。底层工具可以是 ClickHouse、BigQuery、Postgres、Prometheus、OpenTelemetry 或任何团队熟悉的栈；关键是指标口径和事件模型要稳定。

工程实现

我通常会先落一个最小可用的事件模型。它不追求覆盖所有细节，但必须覆盖定位质量问题所需的关键维度。下面是一个 Go 服务里可以直接使用的结构示例，真实项目可以再拆成请求事件、评测事件和反馈事件。

type QualityEvent struct {
	TraceID          string            `json:"trace_id"`
	TenantID         string            `json:"tenant_id"`
	UserIDHash       string            `json:"user_id_hash"`
	Feature          string            `json:"feature"`
	Route            string            `json:"route"`
	RiskLevel        string            `json:"risk_level"`
	PromptVersion    string            `json:"prompt_version"`
	ModelID          string            `json:"model_id"`
	KnowledgeVersion string            `json:"knowledge_version"`
	InputTokens      int               `json:"input_tokens"`
	OutputTokens     int               `json:"output_tokens"`
	LatencyMs        int               `json:"latency_ms"`
	FirstTokenMs     int               `json:"first_token_ms"`
	CostMicros       int64             `json:"cost_micros"`
	SchemaValid      bool              `json:"schema_valid"`
	CitationValid    bool              `json:"citation_valid"`
	SafetyResult     string            `json:"safety_result"`
	TaskOutcome      string            `json:"task_outcome"`
	HandoffReason    string            `json:"handoff_reason,omitempty"`
	ErrorCode        string            `json:"error_code,omitempty"`
	Labels           map[string]string `json:"labels,omitempty"`
	CreatedAtUnixMs  int64             `json:"created_at_unix_ms"`
}

这里有几个取舍。UserIDHash 不保存原始用户标识，避免看板扩散敏感数据。KnowledgeVersion 单独记录，因为很多质量问题来自索引和文档版本，而不是模型。TaskOutcome 不用布尔值，而用枚举，例如 completed、partial、handoff、abandoned、blocked，这样后续可以区分失败原因。Labels 留给业务扩展，但核心维度不要都塞进 labels，否则查询和治理会变难。

事件写入要走异步队列，但不要影响主链路。模型请求已经很慢，不能为了看板再阻塞用户响应。常见做法是在请求结束时把质量事件放入本地 buffer 或消息队列，失败时落降级日志，再由后台任务补偿。为了避免“成功请求有事件，失败请求没事件”的偏差，入口和退出都应该记录。请求开始事件可以很轻，完成事件补充结果。发生 panic、超时或连接断开时，也要尽量写一条 request_aborted。

指标层要把事件转成稳定口径。比如任务成功率不能简单等于 TaskOutcome == completed 的比例，因为某些请求属于系统主动拒答，拒答可能是正确行为。可以按风险等级定义不同口径：低风险问答看完成率，高风险操作看安全通过和人工接管是否合理，结构化生成看 schema 通过和用户采纳。指标定义最好放在配置或代码里评审，不要让 BI 图表自由发挥。

一个质量聚合表可以按小时、功能、路由和版本生成：

字段含义
bucket_hour聚合小时，方便看趋势和告警
feature功能名，例如 knowledge_qa、contract_review
route模型路由，例如 fast、balanced、accurate
prompt_versionprompt 或工作流版本
request_count请求数，用于判断样本量
task_success_rate任务成功率，按功能口径计算
schema_valid_rate结构化输出通过率
citation_valid_rate引用校验通过率
handoff_rate人工接管率
p95_latency_msP95 完整延迟
avg_cost_micros平均请求成本

接下来是样本抽取。看板只展示趋势不够，必须能从每个异常指标跳到样本。比如 citation_valid_rate 降了五个百分点，看板应该能直接打开最近二十条引用失败的 trace，展示用户问题、检索片段、模型答案、引用校验结果、prompt 版本和文档版本。工程师不应该再去多个系统手动拼证据。样本页要支持脱敏展示，敏感租户默认只展示元数据，由授权人员查看内容。

自动评测可以作为质量事件的一部分。线上每条请求都跑重型评测通常不现实，成本和延迟都不可接受。更合理的是分层抽样：高风险功能高比例抽样，新版本发布后短时间提高抽样比例，异常切片触发定向抽样，低风险稳定功能降低抽样。评测结果写回同一条 trace。这样看板可以看到“线上用户反馈”和“自动裁判”在同一个请求上的差异。

告警规则不要只看绝对阈值，还要看变化率和样本量。比如任务成功率低于 90% 要告警，但如果某个小功能一小时只有三条请求，百分比没有意义。推荐规则里同时包含请求数下限、相对跌幅和持续时间：

alerts:
  - name: qa_task_success_drop
    feature: knowledge_qa
    min_requests: 200
    window: 30m
    baseline: 7d_same_hour
    condition: task_success_rate_drop > 0.08
    severity: page
  - name: qa_handoff_spike
    feature: contract_review
    min_requests: 50
    window: 1h
    condition: handoff_rate > 0.25
    severity: ticket

最后是权限和数据治理。质量看板会接触用户输入、模型输出和业务结果，不能当普通监控随便开放。至少要做到租户隔离、字段脱敏、访问审计、数据保留周期和导出控制。对外部客户的复盘报告，可以从聚合指标和已脱敏样本生成；内部排障才允许查看原文。很多 AI 项目忽略这一点，等看板做好以后才发现安全团队不允许使用真实内容，前期设计就白费了。

测试评测

质量看板本身也需要测试。否则团队会把一个有 bug 的看板当事实来源。第一类测试是事件结构测试。每个业务入口都应该有单元测试或集成测试，确保成功、失败、超时、取消、人工接管几条路径都会产生事件，关键字段不为空，枚举值合法，版本号能正确传递。尤其是失败路径，经常因为提前返回而漏写事件，最后看板只反映“成功请求的质量”。

第二类测试是指标口径测试。给定一组固定事件，聚合任务应该产出预期指标。这个测试很朴素，但很重要。比如十条请求里两条合理拒答、一条人工接管、七条完成，任务成功率到底怎么算，必须用样例写清楚。以后有人改口径时，测试会强制他解释变化。指标测试最好保留业务命名，不要只写数学函数。

第三类测试是端到端 trace 测试。构造一个包含检索、模型、工具、后处理、用户反馈的完整请求，跑完后检查看板明细能否从 trace_id 找到所有阶段。这个测试能发现链路断裂、字段命名不一致、队列丢消息和权限配置问题。看板上线前至少要有几条代表性流程：普通成功、模型超时、工具失败降级、引用校验失败、人工接管。

第四类测试是评测器校准。自动裁判如果参与看板，就要定期和人工标注对比。可以每周抽样一批请求，让人工标注事实正确、引用正确、语气合适、是否完成任务，再计算自动评测的准确率、召回率和一致性。评测器不是越严格越好，也不是越宽松越好，它要服务工程决策。如果裁判把大量可接受答案判错，会造成无意义告警；如果裁判漏掉关键错误，会让团队误以为发布安全。

第五类测试是看板可用性演练。拿一次真实或模拟事故，要求值班同学从看板出发定位原因。比如把某个 prompt 版本的引用格式改坏，观察看板能否在一小时内暴露 citation_valid_rate 下降，并能切到对应版本和样本。再比如把某个工具接口延迟增加，观察任务成功率、P95 延迟和人工接管是否按预期变化。演练能暴露看板设计里的断点：指标有了但样本跳不过去，样本有了但版本看不到，告警有了但没人知道处理动作。

第六类测试是数据质量监控。事件流延迟、丢失率、重复率、聚合任务成功率、明细和聚合对账都要监控。质量看板依赖数据管道，管道自身故障会伪装成业务质量变化。比如队列积压导致看板延迟三小时，团队可能误判当前质量；重复消费导致请求量翻倍，成本和错误率都会失真。看板的数据质量指标应该显示在看板自己首页，不要藏起来。

一个评测发布门禁可以这样设计：

门禁项通过标准阻断动作
离线回归核心样本任务成功率不低于当前线上版本阻止发布，要求补充样本解释
线上金丝雀新版本切片任务成功率无显著下降自动回滚路由权重
成本延迟单任务成本和 P95 延迟不超过预算降低流量或切回旧模型
安全指标权限越界、敏感输出、错误拒答无新增高危案例安全复核后再放量
人工复核高风险样本人工一致性达到阈值进入人工审批队列

这些测试不需要一次做完。可以先从事件结构、指标口径和端到端 trace 开始，再逐步加入自动评测和演练。关键是承认看板也是生产系统，它产生的结论会影响发布、回滚和客户沟通，因此必须被验证。

失败模式

第一个失败模式是把看板做成报表中心。很多指标看起来完整，但没有负责人、没有阈值、没有处理动作。任务成功率下降，没人知道是否该回滚；成本上涨，没人知道是路由还是流量；人工接管上升，没人知道是模型问题还是业务规则变化。看板的每个核心指标都应该有 owner、解释文档和行动建议。没有行动的指标应当降级为探索图表，不要放在首页。

第二个失败模式是样本偏差。只收集用户点踩样本，会高估问题严重程度；只收集成功完成样本，会低估失败；只评测高频问题，会忽略长尾风险；只看内部用户，会误判外部客户体验。质量看板需要明确采样策略，并展示样本覆盖率。评测集也要持续从生产日志里更新，不然会变成历史版本的纪念馆。

第三个失败模式是平均值幻觉。全局任务成功率 95% 看起来很好，但可能企业客户只有 80%，免费用户 99%；中文 97%，英文 70%；短问题 98%，长文档 75%。AI 质量问题经常藏在切片里。看板首页可以展示全局数字，但必须让工程师一键下钻到异常切片。否则平均值越稳定，团队越容易忽略真正受影响的用户。

第四个失败模式是自动裁判漂移。模型供应商升级、prompt 变化、业务标准变化都会影响裁判结果。今天裁判认为引用严格匹配才算对，明天裁判接受语义引用，指标就会突然变好，但系统没有真的变好。裁判版本要记录，裁判样本要回放，裁判和人工一致性要持续监控。对高风险功能，不要让单一裁判模型决定上线。

第五个失败模式是指标互相打架但没人解释。比如安全拒答率上升，可能是策略更安全，也可能是误拒；人工接管率下降，可能是模型更好，也可能是接管入口坏了；成本下降，可能是路由优化，也可能是答案变短不完整。看板需要展示相关指标，而不是孤立数字。质量、延迟、成本、人工接管必须并排出现，才能避免局部优化。

第六个失败模式是把敏感内容暴露给过多人员。为了排障方便，团队可能把用户原文和模型输出直接放进看板。短期很爽，长期会变成合规和信任问题。正确做法是默认脱敏、按角色授权、访问留痕，并提供可复现的最小样本。工程师排查不一定需要完整用户身份，很多时候 trace、版本、片段和错误类型就够了。

第七个失败模式是缺少发布标记。没有发布标记的质量曲线很难解释。一次指标下降可能来自新模型、prompt、索引、工具、前端入口、用户增长或外部事件。每次发布都要写入事件维度，并在看板趋势上标注。发布系统和质量看板要打通，否则看板永远只能告诉你“变坏了”，不能告诉你“从哪里开始变坏”。

第八个失败模式是没有保留失败现场。为了省存储，团队只保留聚合指标，删除输入输出和中间证据。等客户投诉时，只能说“那小时成功率确实低”，却无法解释具体回答为什么错。明细保留当然要控制成本和权限，但关键样本、异常样本和发布窗口样本必须保留足够长。AI 质量复盘离不开现场。

上线 checklist

  • 明确每个功能的任务成功定义，不使用一个全局点赞率替代所有质量指标。
  • 为 AI 请求打通 trace_id，确保检索、模型、工具、后处理、反馈和人工复核能串联。
  • 记录 model_id、prompt_version、knowledge_version、route、release_version 等版本维度。
  • 定义质量事件结构，覆盖成功、失败、超时、取消、降级和人工接管路径。
  • 建立明细表和聚合表，指标口径进入代码或配置评审，不让每张图自己定义指标。
  • 首页只放能指导行动的核心指标：任务成功、质量校验、P95 延迟、成本、人工接管、安全事件。
  • 所有核心指标支持按功能、租户、模型路由、prompt 版本、知识库版本和风险等级切片。
  • 每个异常指标能跳到代表性 trace 样本，并展示输入、输出、证据、版本和失败原因。
  • 自动评测结果带上裁判版本，并定期与人工标注校准。
  • 告警规则同时考虑阈值、相对变化、样本量和持续时间。
  • 看板数据管道有自身监控，包括事件延迟、丢失率、重复率、聚合失败和对账结果。
  • 敏感内容默认脱敏，明细访问按角色授权，并保留访问审计。
  • 发布流程接入质量门禁，新模型、prompt、索引和工具改动都能在看板上标注。
  • 为常见异常准备处理动作：回滚、降级、暂停放量、提高抽样、人工复核、创建缺陷。
  • 定期做质量事故演练，验证值班同学能从看板定位到具体版本和样本。

总结

AI 质量看板不是把日志画成图，也不是给管理层看的漂亮大屏。它是生产系统的一部分，负责把线上行为、离线评测、人工判断、版本发布和工程成本连接起来。一个合格的看板应该让团队少争论感觉，多讨论证据；少追逐单一分数，多理解质量、延迟、成本和风险之间的取舍；少在事故后翻聊天记录，多从 trace 和版本切片直接定位问题。

建设顺序也不必复杂。先统一质量事件和 trace，再定义功能级任务成功指标，然后做明细样本、聚合趋势和核心告警，最后逐步加入自动评测、发布门禁和人工校准。每一步都要服务真实工程动作。看板如果不能帮助发布、回滚、降级、复盘和样本挖掘，就应该重新设计。

我更愿意把质量看板看作 AI 应用的“共同事实层”。模型会换，prompt 会改，业务会变，评测集会更新，但团队需要一个稳定位置来观察这些变化对用户任务的影响。把准确率、延迟、成本和人工接管放在一起，不是为了让页面更热闹，而是为了让 AI 系统在真实生产环境里可解释、可治理、可持续迭代。
