Agent 规划循环怎么设计

规划、执行、观察和修正需要稳定协议，而不是只靠长 prompt。

PlanningAgent

问题背景

很多团队第一次做 Agent，都会从一个很长的系统提示词开始：你要先分析任务，再制定计划，必要时调用工具，完成后总结。这个方向没有错，但只靠提示词很快会遇到边界。模型会在简单任务上表现得像有规划能力，在复杂任务上却把计划当成一段漂亮文字；它会忘记哪些步骤已经做过，会把工具失败解释成任务失败，也会在用户中途追加约束时继续沿着旧计划走。真正进入工程环境以后，Agent 不是一段 prompt，而是一个持续运行的控制系统。

规划循环要解决的核心问题，是把“想做什么、正在做什么、观察到了什么、下一步为什么改变”从模型的隐式推理里拿出来，变成可记录、可检查、可恢复的协议。人类工程师写代码时也会这样做：先确认目标和约束，列出待办，读代码，提出假设，改一小块，跑测试，发现失败后定位原因，再更新判断。区别在于人类可以靠工作记忆和经验维持这个循环，而 Agent 的上下文会膨胀、压缩、丢失，还会被工具返回的大量文本淹没。

一个没有稳定规划循环的 Agent，最常见的失败不是不会调用工具，而是不会停下来重新看局势。比如用户要求“只修改四个 Markdown 文件”，Agent 一开始记住了，但中途发现构建脚本里有校验逻辑，顺手改了脚本；用户要求“不要脚本生成正文”，Agent 仍然写了一个生成器来省事；测试失败后，Agent 没有区分是实现错误、测试环境缺依赖，还是校验命令本身需要参数。每一个小错误都不是模型完全不懂，而是循环里缺少明确的状态边界。

规划循环还有一个现实约束：它必须能处理不完整信息。用户不会每次都给出完整需求，仓库也不会把所有约定写进 README。Agent 需要先读、再问、再做，但也不能把所有不确定性都抛回给用户。好的规划循环应该允许低风险探索，要求高风险确认，并在每次行动前知道自己是否仍然满足用户给出的硬约束。这里的“规划”不是一次性生成全量路线图，而是在有限观察下持续维护一个可执行的任务模型。

在工程产品里，我更愿意把 Agent 规划循环理解成一个小型调度器。模型负责提出候选动作和解释理由，宿主程序负责维护状态、约束、预算、权限和日志。这样做的好处是，模型的创造性不会被消灭，但关键的执行纪律不再完全依赖模型自觉。计划可以被展示给用户，可以被压缩进上下文，可以在失败后回放，也可以作为评测样本的一部分。

核心概念

一个可用的规划循环至少有五个概念：目标、约束、计划、观察和决策。目标回答“最后要交付什么”；约束回答“哪些事情不能做、哪些条件必须满足”；计划回答“当前准备用哪些步骤逼近目标”；观察回答“刚才的工具调用、文件读取、测试结果说明了什么”；决策回答“下一步继续、修正、回滚、询问还是结束”。这几个概念必须分开存储，不能揉成一段自然语言。

概念主要来源生命周期需要被用户看到吗工程风险
目标用户输入、任务上下文整个任务通常需要目标理解偏差会导致全局跑偏
硬约束用户明确要求、权限系统、仓库规则整个任务或会话必须可见违反后通常不可接受
软偏好用户习惯、项目风格、历史经验可调整视情况展示过度套用会损害结果
计划项Agent 分解结果动态变化对复杂任务应展示计划过细会僵化，过粗会失控
观察工具返回、测试日志、文件内容随行动累积关键观察应总结观察未结构化会污染上下文
决策模型和调度器共同产生每轮更新关键变更应说明决策不可追踪会难以复盘

规划循环不是瀑布模型。很多人把计划写成十个步骤，然后要求 Agent 顺序执行，这在真实仓库里很脆弱。读到代码之后，原计划可能要改变；跑完测试之后，风险排序可能要改变；用户插入一句“不要改配置”，计划也要立刻收缩。更适合 Agent 的结构是“滚动计划”：只把最近两三步写得具体，把后面的方向保持为可调整的假设。

循环里还有一个重要概念叫“不变量”。不变量是每一轮行动前都要检查的条件。例如只允许修改某些文件、不能联网、不能删除用户改动、必须保留已有 API、最终要跑指定命令。这些条件不应该只在开头读一次，而应该在每次执行写操作、调用外部服务、提交结果前重新检查。很多 Agent 事故，本质上都是不变量没有进入执行层。

还需要区分“任务计划”和“工具计划”。任务计划是面向用户目标的，例如“创建四篇长文并校验”；工具计划是面向执行环境的，例如“读取元数据、写 Markdown、运行校验”。模型可以同时维护两层，但 UI 和日志里最好分开。用户关心交付物和风险，工程系统关心命令、文件、权限和退出码。混在一起会让计划既不便于用户理解，也不便于程序执行。

最后，规划循环必须有结束条件。Agent 很容易陷入“再检查一下”的惯性，尤其是测试失败、输出不确定或上下文很长时。结束条件可以包括：所有硬性验收通过；剩余问题被明确记录且不阻塞；没有权限继续；需要用户做业务判断；预算耗尽但状态可恢复。没有结束条件的 Agent 会把谨慎变成拖延，把自动化变成噪声。

架构/流程图解说明

一个工程 Agent 的规划循环可以拆成下面这条主线：

用户目标
  |
  v
任务建模：目标、范围、硬约束、验收标准
  |
  v
上下文收集：读取文件、检索记忆、检查状态
  |
  v
计划生成：近期步骤、风险点、需要确认的分支
  |
  v
行动选择：读、写、运行、询问、等待、结束
  |
  v
工具执行：带权限、预算、超时和可观测日志
  |
  v
观察归纳：事实、失败、证据、对计划的影响
  |
  v
计划修正：继续、重排、缩小范围、回滚或升级
  |
  +---- 回到行动选择，直到验收通过或需要人工决策

这张图里最容易被忽视的是“观察归纳”。工具返回不是天然观察。rg 返回一百行结果，只是原始材料；测试输出一屏错误，也只是原始材料。观察需要回答三个问题：这次行动有没有达到预期；它对目标有什么新信息；它是否改变约束、风险或下一步。没有这层归纳，Agent 会把工具输出原样塞回上下文，随着循环次数增加，关键事实反而被稀释。

状态机可以更具体一些：

NEW -> UNDERSTAND -> EXPLORE -> PLAN -> EXECUTE -> VERIFY -> REPORT
              ^          |          |        |         |
              |          v          v        v         v
              +------ REPLAN <--- BLOCKED <- FAILED <-+

UNDERSTAND 阶段解析目标和约束；EXPLORE 阶段允许只读调查；PLAN 阶段生成可执行步骤；EXECUTE 阶段做有副作用的动作；VERIFY 阶段跑验收；REPLAN 阶段根据观察调整；BLOCKED 表示缺少权限、信息或人工判断；REPORT 表示交付。注意 FAILED 不等于任务结束，它只是说明上一轮行动失败。只有失败无法被修复、继续会违反约束，或者需要用户选择时，才进入 BLOCKED。

规划循环的运行记录建议分三层保存。第一层是事件日志，记录每个工具调用、参数、输出摘要和退出状态；第二层是任务状态，记录当前目标、约束、计划项、观察和验收状态；第三层是用户可读摘要，说明当前进展和风险。事件日志用于调试，任务状态用于下一轮决策，用户摘要用于协作。三者的粒度不同，不能互相替代。

一个真实流程例子：用户要求给一个 Go 服务加限流。Agent 先进入 UNDERSTAND，识别交付物是限流实现和测试，硬约束是保持现有 API。进入 EXPLORE 后读取路由、中间件、配置和测试结构。进入 PLAN 后决定先加配置字段，再加中间件，再写集成测试。执行第一步后发现配置系统不支持热更新，观察归纳为“如果把限流放进配置，会引入重启语义变化”。于是进入 REPLAN，改成在现有中间件初始化阶段读取静态配置，并把热更新作为非目标。这个修正如果只靠模型上下文记忆，容易在后面忘掉；如果进入结构化状态，就能在最终总结和测试设计里体现。

工程实现

实现规划循环时，我会先定义任务状态，而不是先写 prompt。下面是一个简化的数据结构，重点是把可执行计划、约束和观察拆开：

type TaskState struct {
    ID          string
    Goal        string
    Scope       Scope
    Constraints []Constraint
    Acceptance  []AcceptanceCheck
    Plan        []PlanItem
    Observations []Observation
    Phase       Phase
    Budget      Budget
    UpdatedAt   time.Time
}

type PlanItem struct {
    ID        string
    Title     string
    Intent    string
    Status    string // pending, running, done, skipped, blocked
    Evidence  []string
    RiskLevel string
}

type Observation struct {
    Source    string // file, command, user, memory
    Fact      string
    Impact    string // confirms_plan, changes_plan, blocks, irrelevant
    Evidence  string
    CreatedAt time.Time
}

这类结构不需要一开始就很复杂，但要从第一天支持两个能力：可序列化和可压缩。可序列化意味着进程重启、上下文压缩、用户刷新页面后任务还能恢复；可压缩意味着当上下文太长时，系统可以把事件日志压成观察摘要，而不是让模型重新读几万行输出。实际实现里，事件日志可以放对象存储或数据库，任务状态放关系数据库，短期上下文放内存，三者用任务 ID 串起来。

每一轮循环可以写成一个调度函数：

func Step(ctx context.Context, state TaskState) (TaskState, Action, error) {
    state = EnforceInvariants(state)
    prompt := BuildDecisionPrompt(state)
    decision := ModelDecide(ctx, prompt)
    action := ValidateAction(decision.Action, state.Constraints, state.Budget)
    if action.RequiresApproval {
        state.Phase = PhaseBlocked
        return state, action, nil
    }
    result := Execute(ctx, action)
    observation := SummarizeResult(result, action.Expectation)
    state = ApplyObservation(state, observation)
    state = UpdatePlan(state)
    state = CheckAcceptance(state)
    return state, NextAction(state), nil
}

这里有几个工程细节很关键。第一，EnforceInvariants 不能交给模型自由发挥，它应该由宿主程序检查。例如写文件前验证路径是否在允许范围，运行命令前验证是否需要网络或高风险权限。第二，ValidateAction 要把模型输出转成受控动作，模型说“删除临时文件”不等于系统就执行 rm。第三，SummarizeResult 要尽量结构化，把退出码、关键错误、影响范围和下一步建议拆开，而不是生成一段泛泛的失败说明。

计划项的粒度要能被验证。坏计划写成“完善系统”“优化代码”“检查问题”；好计划写成“读取配置入口，确认限流参数放在哪里”“新增中间件单元测试覆盖超限返回”“运行 go test ./... 验证无回归”。每个计划项都应该有可观察结果，完成状态不靠模型自称，而靠文件变化、测试结果或用户确认。

规划循环还需要处理用户中途输入。用户可能追加限制、纠正理解、要求停止，或者提供新的业务背景。宿主程序应该把用户输入作为高优先级事件，先更新目标和约束，再让模型重新决策。不要简单把新消息追加到对话末尾后继续执行旧动作。尤其是已经排队的工具调用，如果用户说“暂停”或“不要改数据库”，队列必须能取消。

对长任务来说，检查点非常实用。每完成一个风险较高的阶段，保存任务状态和工作区差异摘要。检查点不是 Git commit，也不一定要修改代码仓库，它可以是 Agent 内部的“已知状态”：读过哪些文件、做过哪些改动、哪些测试通过、哪些假设被推翻。发生上下文压缩或进程恢复时，Agent 可以从检查点继续，不必靠一段模糊总结重新推理。

一个适合落地的计划项 JSON 例子如下：

{
  "id": "p3",
  "title": "为内容校验补齐目标文章",
  "intent": "创建四个 Markdown 文件，并确保 front matter 与元数据一致",
  "status": "running",
  "risk_level": "medium",
  "acceptance": [
    "只修改 content/articles 下四个目标文件",
    "运行指定 check_articles.py 命令通过",
    "每篇包含固定章节和标题图"
  ],
  "evidence": [
    "content/ARTICLE_STYLE.md",
    "js/articles.js",
    "scripts/check_articles.py"
  ]
}

这个例子里，计划项同时包含意图、风险和验收。模型可以根据它组织行动，程序可以根据它检查边界，用户也能看懂为什么当前正在做这件事。规划循环越工程化，Agent 越不需要在每轮靠“我会小心”这种空话维持可靠性。

测试评测

规划循环的评测不能只看最终答案。一个 Agent 最后把任务做完，不代表循环设计可靠；它可能走了很多危险路径，只是运气好没有爆炸。评测应该覆盖过程质量，包括是否正确识别约束、是否在失败后修正计划、是否避免无授权动作、是否按验收结束、是否能在上下文压缩后恢复。

评测维度样本设计通过标准常见缺陷
目标理解给出含多个交付物的任务状态中列出完整交付物只抓住第一个需求
约束保持中途加入“只读”或“只改某目录”后续动作被限制旧计划继续执行
失败修正工具返回可修复错误观察指出原因并调整反复执行同一命令
权限升级需要外部副作用请求确认而非直接执行把高风险动作包装成普通步骤
恢复能力人为截断上下文根据状态继续任务重新探索或丢掉约束
结束判断验收已通过停止并简短汇报无限补充检查

测试样本要像真实任务，而不是问答题。比如构造一个小仓库，要求 Agent 修改三个文件，但其中一个测试会因为环境变量缺失失败。好的 Agent 应该识别失败原因，说明实现可能已经通过相关测试，只是环境缺失，并根据约束决定是否继续；差的 Agent 会不断改代码，试图让一个环境问题变成实现问题。再比如给一个前端项目，用户先说“实现功能”，执行中途再说“不要改样式文件”，系统要验证后续写操作是否避开样式文件。

过程评测可以采集事件轨迹，给每个动作打标签：读取、分析、写入、执行、询问、汇报。然后检查轨迹是否满足规则。例如写入动作前是否已经收集足够上下文；高风险命令前是否有权限；测试失败后是否产生新观察；最终汇报是否引用验收结果。这种轨迹评测比只看输出文本更接近真实可靠性。

还可以设计“扰动测试”。在同一个任务中随机插入无关工具输出、重复文件、旧错误日志、用户纠正语句，观察 Agent 是否能保持计划稳定。真实环境里上下文从来不干净，日志里有历史失败，仓库里有废弃文件，用户表达也会变化。规划循环如果没有结构化状态，很容易被最新的一段文本带跑。

指标上，我会关注五个数字：平均重规划次数、无效工具调用比例、约束违规次数、验收前停止比例、人工介入次数。重规划次数不是越少越好，复杂任务应该有合理修正；无效工具调用比例越高，说明计划和观察质量差；约束违规是硬指标；验收前停止说明结束判断太乐观；人工介入次数要结合风险看，高风险任务多确认是正常的。

最实用的回归测试，是保存失败任务的完整轨迹。每次线上出现 Agent 跑偏，不要只改 prompt，而要把那条轨迹做成回放样本：初始目标、工具响应、用户插话、权限状态、期望决策。以后每次改模型、改 prompt、改调度器，都跑一遍轨迹回放。Agent 系统没有这种回归集，很难持续变稳。

失败模式

第一类失败是计划表演化。Agent 会写出一份看起来很完整的计划，但行动和计划没有绑定。它说先读规范，实际直接写文件；说最后跑测试，实际因为上下文太长忘掉；说只改目标文件，实际改了辅助脚本。这类问题不能靠要求模型“严格执行计划”解决，需要调度器把计划项和动作关联起来，每个动作都标明服务于哪个计划项。

第二类失败是观察漂移。工具结果进入上下文后，模型提取了错误重点。例如测试输出里真正的错误是端口被占用，模型却盯着最后一行断言失败；文件里已经有实现，模型却因为搜索词不准认为没有实现。观察漂移会导致重规划错误。缓解办法是对关键工具输出做结构化摘要，并在不确定时允许模型再次读取更小范围的上下文。

第三类失败是过度规划。Agent 把简单任务拆成十几步，每步都请求确认，用户体验很差。不是所有任务都需要显式计划。修改一个错别字、回答一个概念问题、运行一个简单命令，可以直接执行。规划循环应该按风险和复杂度调整显式程度。低风险短任务使用轻量循环，高风险长任务使用完整状态机。

第四类失败是局部最优。Agent 在某个错误方向上投入太多，越修越复杂。比如一个测试失败本来是 mock 数据缺字段，Agent 却开始重构业务逻辑。规划循环需要“退一步”机制：连续失败两次后回到目标和观察，重新审视假设；同一文件反复修改后要求比较差异；影响范围扩大时提高风险级别。

第五类失败是结束过早。模型看到一个测试通过就汇报完成，但用户要求的文档、校验、边界场景还没有处理。结束条件必须来自验收列表，而不是来自模型对“看起来完成”的感觉。每个验收项要有状态：通过、失败、未验证、跳过并说明原因。最终汇报只在所有必要项都有结论后发生。

第六类失败是忽略人类输入优先级。用户中途纠正“不是这个模块”，Agent 仍然继续执行已经生成的计划。工程上要把用户新消息当成中断事件，而不是普通上下文。中断后应该暂停动作队列，重新计算约束，并在必要时说明哪些已做工作可能需要回滚。

上线 checklist

  • 任务状态中是否明确保存目标、范围、硬约束、软偏好和验收标准。
  • 每个有副作用动作是否绑定到计划项，并经过路径、权限、预算和风险检查。
  • 工具结果是否被压缩成观察，观察里是否说明对计划的影响。
  • 用户中途输入是否能中断待执行动作，并触发重新规划。
  • 失败是否区分可修复错误、环境阻塞、权限不足和需求不清。
  • 是否支持检查点和恢复，尤其是上下文压缩后的长任务。
  • 最终汇报是否逐项对应验收标准，而不是只说“已完成”。
  • 是否有轨迹回放测试覆盖典型跑偏案例。
  • 是否记录无效工具调用、重规划次数、约束违规和人工介入。
  • 是否为低风险任务提供轻量路径，避免所有任务都被流程拖慢。

总结

Agent 规划循环的重点，不是让模型写出更漂亮的计划，而是让计划成为运行时协议。目标、约束、计划、观察和决策必须有结构化位置；工具调用必须经过权限和不变量检查；失败必须进入观察并推动重规划；结束必须对齐验收。这样设计出来的 Agent 仍然会犯错，但错误会留下证据，会被系统拦截，会进入回归样本，而不是变成一次无法解释的随机行为。

我更相信小而稳定的循环，而不是一段试图包办所有智慧的长 prompt。模型擅长理解语义、提出方案和总结观察；宿主程序擅长保存状态、执行检查、控制权限和回放轨迹。把这两者分工清楚，Agent 才能从“会聊天的工具调用器”变成真正能和工程团队协作的执行系统。
