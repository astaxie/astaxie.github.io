用 Codex 做大重构

大重构要拆成可验证的小批次，避免同时改变行为和结构。

CodexRefactor

问题背景

大重构最容易出问题的地方，不是改代码的那一刻，而是人和工具都误以为自己已经理解了系统。一个服务跑了几年以后，目录结构、命名、接口边界和业务规则通常不是同一时期形成的。你看到一个旧的 manager、一个重复的 service、一堆传来传去的 context，很自然会想把它们整理干净。Codex 这类编码 Agent 让这种冲动更强，因为它能很快读文件、改接口、补测试、跑命令，好像只要给一个足够明确的目标，它就能把混乱收拾成整洁。

但大重构不能只靠速度。重构的核心约束是“改善结构而不改变外部行为”。如果在同一批改动里同时改目录、改接口、改缓存策略、改错误码、改数据库读写顺序，最后测试绿了也不一定说明安全。测试可能只覆盖了常见路径，日志可能没有暴露边界差异，线上调用方可能依赖了一个没人写进文档的老行为。Codex 能加速机械迁移，也能帮助理解调用链，但它不会自动知道哪些行为是不能动的不变量，除非我们把不变量显式地写出来，并在每个批次里反复验证。

我在实际使用里会把“用 Codex 做大重构”理解成一个工程调度问题，而不是一个 prompt 技巧问题。提示词当然重要，但更重要的是工作单元如何切分、上下文如何喂给模型、哪些文件允许写、哪些命令必须跑、每一步如何留下证据，以及什么时候停下来让人做判断。尤其在老项目里，重构计划经常会遇到半路发现：原来这个包被一个离线任务引用，原来这个字段名字虽然奇怪但写入了报表，原来这个错误包装被前端拿来判断登录态。没有分批和证据，Agent 会把这些细节抹平。

大重构的另一个风险是“结构正确性”遮蔽“业务正确性”。比如把一组散落的用户权限判断抽成统一的 Authorizer，从代码审美看是进步；但原来某个后台接口故意允许超级管理员绕过组织边界，另一个批量导出接口故意在权限不足时返回空列表而不是错误。统一抽象如果没有把这些差异建模出来，就会把历史语义折叠成一个看起来合理的新规则。Codex 可以很快完成抽象迁移，但它无法凭空判断这些差异是 bug、债务还是业务契约。

所以我会先问三个问题。第一，这次重构要保护的外部行为是什么，是 API 响应、数据状态、性能、日志字段、错误类型，还是构建产物？第二，能不能把重构拆成多个行为保持的小批次，每个批次只解决一种结构问题？第三，Codex 在每个批次里需要什么证据才能继续，而不是靠“看起来完成了”往前滚？这三个问题答清楚，Codex 就能成为一个很有用的执行者；答不清楚，它只会把模糊的风险更快地扩散到更多文件。

核心概念

做大重构时，我最看重四个概念：不变量、切片、护栏和证据。不变量是重构期间必须持续成立的事实，例如公共 API 路径不能变、数据库 schema 不动、序列化字段名不变、错误码不变、同一输入的业务决策不变。切片是每次提交能独立验证的一小块改动。护栏是限制 Codex 行动范围的规则，包括只写哪些文件、禁止哪些工具、必须先读哪些规范、什么时候需要停下来。证据是每个切片完成后的可检查产物，比如测试结果、差异报告、调用图、日志样本、基准数据。

概念在大重构里的作用需要落到哪里常见误区
不变量保证重构不改变用户可见行为任务说明、测试断言、评审清单只写“保持行为不变”，没有具体字段
切片降低回滚和定位成本分支计划、提交粒度、Agent 子任务按文件切，而不是按语义切
护栏防止 Agent 过度发挥写入范围、命令白名单、禁止事项只在开头提醒一次，后续不检查
证据证明每一步可继续测试输出、diff 摘要、人工观察让 Agent 自己宣布完成
回滚点让失败能停在局部小提交、开关、兼容层一口气改完整个包后才测试

不变量要写得具体。比如“接口行为不变”太粗，Codex 很难据此判断改动是否安全；更好的写法是“GET /api/projects/:id 的状态码、字段名、空值语义、权限不足时的错误码保持不变；内部日志可以调整，但 request_id、project_id、latency_ms 三个字段必须保留”。当不变量具体到字段、错误、顺序、时间复杂度，模型才有能力围绕它们设计测试和比对。

切片也不要简单等同于文件数量。一个大文件里可能包含多个业务语义，一个小文件也可能是所有调用的枢纽。好的重构切片应该满足：输入和输出清楚、影响面能枚举、验证命令能在几分钟内跑完、失败时能还原。比如“把用户查询路径从旧 DAO 迁移到 repository 接口”是一个合理切片；“顺便把用户、订单、账单三个模块都改成新接口”就太大；“只重命名三个变量”又可能太小，不能产生结构收益。

护栏要体现在交互协议里，而不是停留在人的期待里。我通常会要求 Codex 在动手前列出本批次可写文件、不可碰的目录、预计测试命令和回滚方式。改动过程中如果发现需要扩大范围，先说明原因，不直接改。这个规则对人类工程师也适用，但对 Agent 更重要，因为 Agent 很容易为了让编译通过而顺手修改调用方、测试夹具、配置文件，最后把本来局部的重构变成全局行为改动。

证据是大重构的节拍器。每个批次结束时，我们不问“代码是不是更漂亮”，先问“有什么证据证明行为没有漂移”。证据可以是单元测试，也可以是 golden file、API 快照、数据库迁移 dry run、性能基准、类型检查、日志字段对比、甚至是手工构造的输入输出样本。只要证据足够具体，Codex 就可以帮助补齐它；如果证据模糊，Codex 只会补一些看起来相关但不一定有约束力的测试。

还有一个概念叫兼容层。大重构中最稳的策略往往不是直接替换旧实现，而是先建立一个兼容接口，让旧调用方和新实现同时存在，然后逐步迁移调用点。兼容层看似多写代码，实际上给了我们验证窗口：可以在同一输入下同时跑新旧实现，比较输出差异；也可以用开关切回旧路径。Codex 很适合写这类机械桥接代码，但前提是我们明确告诉它桥接期间哪些行为必须双写、哪些差异可以接受、何时删除旧路径。

架构/流程图解说明

我把 Codex 参与大重构的流程拆成七层：目标建模、系统勘探、重构地图、切片执行、行为比对、合并收敛、上线观察。这个流程不是为了增加仪式感，而是为了把“模型理解系统”这件事变成可验证的工程动作。

重构目标
  |
  v
目标建模：写清楚要改善的结构问题、不能改变的外部行为
  |
  v
系统勘探：读取入口、调用图、测试覆盖、配置、历史兼容点
  |
  v
重构地图：确定切片顺序、可写范围、兼容层、回滚点
  |
  v
切片执行：Codex 在受限范围内完成一批小改动
  |
  v
行为比对：测试、快照、基准、日志字段、人工样本验证
  |
  v
合并收敛：删除临时桥接、统一命名、补文档、减少分支
  |
  v
上线观察：灰度、指标、错误预算、回滚预案

目标建模要把“为什么要重构”说清楚。常见目标有几类：降低重复逻辑、分离领域和基础设施、替换过期依赖、拆开循环引用、统一错误处理、提高测试可控性。不同目标的切法不同。如果目标是分离领域逻辑和数据库访问，第一批不应该先改目录名，而应该先把领域函数的输入输出抽出来；如果目标是替换 HTTP 客户端，第一批应该先建立适配接口和测试替身，而不是全局搜索替换。

系统勘探是 Codex 很擅长的部分，但也最容易跑偏。我会让它回答具体问题，而不是泛泛总结代码库。比如：这个模块有哪些外部入口；哪些函数被跨包调用；哪些结构体参与 JSON 序列化；哪些测试覆盖了错误路径；哪些配置项影响行为；是否存在反射、全局变量、初始化顺序、定时任务、消息队列消费者。勘探输出最好是表格和引用路径，这样后续切片能回到证据，而不是依赖模型记忆。

重构地图需要表达依赖顺序。下面是一个把“散落的订单状态判断”迁移成统一状态机的简化地图：

批次目标可写范围行为验证回滚方式
1建立状态机类型和只读映射domain/order_state.*、测试文件旧状态到新状态映射快照删除新文件
2在查询路径旁路调用新状态机并记录差异查询服务、差异日志同一订单双算结果一致关闭差异日志开关
3迁移写入路径到状态机写入服务、领域测试非法状态迁移用例开关切回旧判断
4删除旧判断并收敛命名旧工具函数、文档全量测试和快照回退单个提交

这张表的价值不在于格式，而在于每一批都有行为验证和回滚方式。Codex 执行第二批时，不应该顺手删除旧判断，因为第二批的目标是双算对比；执行第四批时，才进入清理阶段。把阶段说清楚，可以减少 Agent 为了“完成重构”提前收尾的冲动。

在多人或多 Agent 场景下，架构图里还需要资源锁。两个 Codex 会话同时改同一个核心包，通常不是好主意。即便它们改的是不同文件，也可能在语义上冲突，比如一个在抽象接口，另一个在删除旧接口。我的做法是把切片和资源绑定：每个会话只拥有一组明确文件和一个明确语义目标；如果发现需要改共享文件，先回到主会话协调。这比事后处理大面积冲突可靠。

工程实现

一个可落地的 Codex 大重构任务，最好从“重构任务单”开始。任务单不是给老板看的计划，而是给 Agent 执行的控制面。它包含目标、不变量、切片、文件范围、测试命令、证据格式和停止条件。

refactor:
  name: order-state-machine-migration
  goal: 将订单状态判断从散落的 if/switch 收敛为领域状态机
  invariants:
    - API 响应字段和状态码不变
    - 订单状态写入顺序不变
    - 已完成订单不能重新进入支付中
    - 日志字段 request_id/order_id/state_before/state_after 保留
  writable:
    - internal/order/domain/**
    - internal/order/service/query.go
    - internal/order/service/query_test.go
  forbidden:
    - migrations/**
    - api/openapi.yaml
    - frontend/**
  checks:
    - go test ./internal/order/...
    - go test ./internal/api/... -run TestOrderAPIContract
  stop_when:
    - 需要修改数据库 schema
    - 需要改变公开 API 字段
    - 发现旧逻辑与注释冲突且无测试覆盖

这份任务单可以直接进入 Codex 对话，也可以由人手动维护在 PR 描述里。重点是 stop_when。大重构里最危险的不是失败，而是遇到未知事实后继续猜。比如 Codex 发现某个旧逻辑看起来明显不合理，它可能会主动“修正”。但重构批次里，未知业务差异应该先记录，再交给人判断是否另开 bugfix。重构和修 bug 可以连续发生，但不应该混在同一个不可分割的 diff 里。

执行时，我会让 Codex 先做只读勘探，并产出一个调用面清单：

入口当前行为风险需要的保护测试
POST /orders/:id/pay支付成功后从 pending 到 paid写路径，影响资金状态重复支付、超时回调、幂等键
POST /orders/:id/cancel部分状态允许取消用户可见，错误码敏感已发货、已完成、已退款
OrderList 查询根据状态展示标签读路径，前端依赖文案空状态、未知状态、分页
定时关单任务超时订单自动取消后台任务，时间敏感边界时间、并发锁

这类清单比“我已经理解订单模块”有价值。接下来每一批改动都可以回到表格看影响面。如果第一批只是引入状态机类型，就不应该碰定时任务；如果第三批要迁移写路径，就必须覆盖幂等和并发。

具体代码层面，我偏向先写新抽象，不急着替换调用方。比如状态机可以这样开始：

type OrderState string

const (
    OrderPending  OrderState = "pending"
    OrderPaid     OrderState = "paid"
    OrderShipped  OrderState = "shipped"
    OrderCanceled OrderState = "canceled"
    OrderRefunded OrderState = "refunded"
)

type Transition struct {
    From   OrderState
    Event  string
    To     OrderState
    Reason string
}

type StateMachine struct {
    allowed map[OrderState]map[string]Transition
}

func (m StateMachine) Apply(from OrderState, event string) (Transition, bool) {
    byEvent, ok := m.allowed[from]
    if !ok {
        return Transition{}, false
    }
    next, ok := byEvent[event]
    return next, ok
}

这个结构不是为了炫技，而是把原本散落在多个 if 里的状态迁移变成可枚举数据。Codex 可以继续补表、补测试、迁移调用点。但在迁移前，我会要求它先从旧代码提取一份“旧行为样本”，例如把每个入口在不同状态下返回什么错误列成表。这样新状态机不是凭理想规则设计，而是以旧行为为基线。

迁移调用方时，建议使用“旁路对比”而不是直接替换。查询路径可以同时执行旧判断和新状态机，结果不一致时只打日志，不影响响应。写路径如果副作用太大，可以先在测试和 staging 环境双算。旁路对比的日志要结构化：

{
  "event": "order_state_diff",
  "order_id": "o_123",
  "entry": "cancel_order",
  "old_allowed": true,
  "new_allowed": false,
  "state": "paid",
  "reason": "new machine requires pending before cancel"
}

有了这种日志，Codex 后续可以帮助分析差异样本，判断是新规则写错，还是旧逻辑存在历史债务。没有差异证据，大家只能在 PR 里争论“这个状态应该能不能取消”。

在 Codex 交互上，我会使用比较硬的任务语句。例如：“本轮只做第一批：新增状态机数据结构和测试，不修改任何调用方。完成后运行 go test ./internal/order/domain，如果需要改服务层先停下来说明。”这种写法比“帮我重构订单状态”有效得多。每轮结束后，再让 Codex 总结：改了哪些文件、保留了哪些行为、跑了什么测试、哪些风险未解决。这个总结不是形式主义，它会成为下一轮上下文的一部分。

测试评测

大重构的测试不能只靠原有测试套件。原有测试能说明“当前覆盖过的行为没有坏”，但重构最大的风险往往在没覆盖的隐含契约里。因此我会把测试分成五类：基线测试、契约测试、差异测试、性能测试和人工样本测试。

基线测试是项目已有的单元测试、集成测试、类型检查、lint。它们是最低门槛，不代表足够。Codex 每次切片后都应该跑与改动范围相关的基线测试，而不是等最后跑全量。小批次测试失败时定位成本低，最后全量失败时往往已经混入太多变化。

契约测试保护外部可见行为。对 HTTP API，可以保存请求和响应快照；对消息消费者，可以保存输入消息和输出事件；对 CLI，可以保存参数、stdout、stderr 和退出码。契约测试不一定要求所有字段完全相同，有些时间戳、追踪 ID 可以做归一化，但业务字段、错误语义和状态码应该严格。

差异测试适合新旧实现共存阶段。把同一组输入喂给旧实现和新实现，比较输出。输入样本可以来自 fixtures、生产脱敏样本、历史 bug 案例和随机生成。Codex 可以帮助生成样本矩阵，但人要审查样本是否覆盖业务边界。例如订单状态迁移至少要覆盖正常流、重复事件、非法回退、并发事件、未知状态、超时任务和外部回调乱序。

测试类型适合阶段关注点示例
基线测试每个切片编译和已有功能go test ./internal/order/...
契约测试改入口前后外部响应不变API golden snapshot
差异测试新旧并存语义是否漂移旧判断和新状态机双算
性能测试替换核心路径延迟、分配、查询次数benchstat 对比
人工样本测试高风险业务历史特殊案例运营提供的异常订单

性能评测不要放到最后才做。有些结构改善会引入额外分配、锁竞争、数据库查询次数变化。Codex 写出的抽象如果过度通用，可能在热路径里创建大量临时对象。对 Go 服务，我会至少看三件事：基准延迟、内存分配和数据库调用次数。数据库调用次数尤其重要，因为单元测试绿了但 N+1 查询增加，线上才会暴露。

测试输出也要被记录。不是简单说“测试通过”，而是留下命令和结果。对于大重构 PR，我希望看到类似下面的验收记录：

checks:
  - go test ./internal/order/...: pass
  - go test ./internal/api/... -run TestOrderAPIContract: pass
  - go test ./internal/order/domain -run TestStateMachineGolden: pass
  - benchstat old.txt new.txt: p95 no regression above 3%
manual_samples:
  - sample_o_1024_cancel_after_paid: old=false new=false
  - sample_o_2048_timeout_pending: old=canceled new=canceled

这种记录让评审者能快速判断风险，也让回滚后复盘有材料。Codex 可以代跑命令和整理结果，但不要让它绕过失败。测试失败时，先分类：是重构引入的真实行为差异，还是测试依赖旧结构，还是原测试本身过度耦合实现。只有第三类才适合改测试；前两类要么修实现，要么回到需求判断。

失败模式

用 Codex 做大重构，最常见的失败模式是“顺手修复”。模型看到旧代码有明显坏味道，会主动清理命名、调整错误处理、合并分支、删除看似无用的兼容逻辑。这些动作单独看都合理，但如果不在当前切片目标里，就会扩大验证面。解决办法是把“禁止顺手修复”写进本轮任务，并要求所有额外发现进入后续清单。

第二个失败模式是“测试迎合实现”。Codex 为了让测试通过，可能会修改测试期望，而不是保护旧行为。尤其当测试失败提示很明确时，模型容易把断言改成新输出。评审时要特别看测试 diff：是增加了旧行为覆盖，还是把旧断言放松了。如果契约测试发生变化，必须有业务理由，而不是“为了适配新结构”。

第三个失败模式是“抽象过早”。大重构经常从一个真实重复点开始，最后被抽成一个覆盖所有场景的巨大接口。Codex 有时会偏好完整性，生成过多配置项和泛型参数。抽象应该从实际迁移路径长出来。先服务两三个真实调用点，再决定接口边界；不要为了未来可能的场景设计一套框架。

第四个失败模式是“上下文漂移”。长会话里，Codex 可能忘记早期约束，或者在上下文压缩后只记得目标不记得边界。应对方式是把关键约束放在每轮任务开头，或者维护一份短的重构任务单。每轮开始前重申本轮范围，每轮结束后更新证据和风险。

第五个失败模式是“合并后才发现冲突”。多个切片并行时，单独看都通过测试，合并后出现命名冲突、接口重复、行为覆盖不一致。解决办法是减少共享文件并行写入，并在每个切片结束后尽早集成到主分支。大重构不适合长期分叉，越晚合并越难判断问题来自哪里。

第六个失败模式是“没有上线观察”。很多人以为重构没有业务功能，就不需要灰度和指标。实际上重构最需要观察，因为它的目标就是让用户看不出变化。没有变化也要有证据：错误率没有上升、延迟没有退化、关键日志字段还在、队列积压没有变化、客服反馈没有异常。否则线上沉默不代表安全，只代表还没看见。

上线 checklist

上线前我会用一份很朴素的清单，不追求漂亮，只追求能防住常见事故。

  • 重构目标已经写清楚，评审者知道这次解决的结构问题是什么。
  • 外部行为不变量已经列出，包括 API、消息、数据库、日志、错误、性能。
  • 每个切片都有独立提交或独立 diff，失败时能局部回退。
  • 没有把重构和业务修复混在同一个不可拆分改动里。
  • 所有新增抽象都有至少两个真实调用点，或者有明确的后续迁移计划。
  • 旧实现和新实现并存期间的兼容层有删除计划，不会永久留成双轨。
  • 契约测试、差异测试、基线测试和必要性能测试已经运行并记录结果。
  • 测试 diff 没有无理由放松断言，没有为了新实现删除历史边界用例。
  • 日志字段、指标名称和告警规则已经检查，不会因为命名重构失去可观测性。
  • 灰度范围明确，知道先放哪些租户、哪些流量、哪些后台任务。
  • 回滚方式实际可执行，不依赖“再让 Codex 改回去”。
  • 上线后观察窗口和负责人明确，关键指标有基线值。
  • 所有“重构中发现但本次不修”的问题已经记录到后续清单。

如果这个清单里有多项答不上来，我宁愿继续拆小，而不是让 Codex 一次性推完。Agent 提升的是执行吞吐，不会降低系统复杂度本身。越是能快速修改，越要慢一点确认边界。

总结

Codex 很适合参与大重构，但它最适合承担的是“受约束的高吞吐执行者”，不是“凭感觉重塑系统的总设计师”。人要负责定义目标、不变量、切片和验收证据；Codex 负责读代码、整理调用面、生成兼容层、迁移调用点、补测试、运行命令和汇总差异。这个分工清楚，重构会快很多，也稳很多。

大重构的关键不是把代码一次改到理想状态，而是在每一步都能回答：这一步改变了什么结构，保护了什么行为，有什么证据，失败时怎么退。只要这四个问题持续成立，Codex 就能把原本漫长的机械劳动压缩到可控时间里；如果这四个问题缺失，再强的模型也只是更快地制造一个难以评审、难以回滚、难以上线的大 diff。
