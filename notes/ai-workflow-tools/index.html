AI 工作流与本地工具组合

DeepChat、CLI、编辑器和浏览器自动化如何进入日常研发流程。

WorkflowTools

问题背景

AI 工具进入研发流程以后，很多讨论会停在“哪个模型更强”“哪个客户端更好用”。这些问题当然重要，但真正影响效率的，往往不是单个工具，而是工具之间怎么组合。一个工程师每天要在编辑器、终端、浏览器、文档、Issue、日志平台、数据库、设计稿和聊天工具之间切换。如果 AI 只停留在一个聊天窗口里，它能帮忙解释问题，却很难进入完整工作流。

我更关心的是本地工具组合：DeepChat 这类多模型对话客户端适合做长上下文讨论和资料整理；CLI 适合执行确定性命令、搜索代码、跑测试、批量处理文件；编辑器适合小步修改和理解局部上下文；浏览器自动化适合验证前端、复现用户路径、检查文档和控制台错误；Git 和 CI 负责把变更纳入团队流程。这些工具各有边界，组合得好，AI 就像一个能参与工程循环的协作者；组合得差，AI 只是一个会写片段的搜索框。

日常研发里最浪费时间的部分，常常不是写代码，而是上下文搬运。你在浏览器里看到报错，把报错复制到聊天工具；AI 建议改某个文件，你再回编辑器搜索；改完之后去终端跑测试；测试失败再把日志复制回来；前端还要切浏览器点一遍；最后写提交说明和 PR 描述。每一步都很小，但累积起来就是大量切换成本。AI 工作流的目标不是让模型替人做所有决定，而是减少这种低价值搬运，让工程师把注意力放在判断、设计和验收上。

另一个问题是可控性。很多人把 AI 工具接入开发后，很快遇到两个极端：一种是只让 AI 回答问题，不让它碰代码，效率提升有限；另一种是给 AI 太大权限，让它一次性改很多文件，结果 review 成本很高。更稳定的做法，是把工作拆成可验证的小闭环：先让 AI 读上下文并提出假设，再让它做小范围修改，然后用测试、lint、截图或日志验证，最后由人做取舍。工具组合的关键，不是“自动化程度越高越好”，而是“每一步都有清楚输入、输出和验收方式”。

在团队场景里，工具组合还会影响协作标准。如果每个人都用不同方式和 AI 交互，prompt、上下文、命令、验证结果都散落在个人机器上，团队很难复用经验。好的工作流会把常用命令、代码规范、评测脚本、故障排查路径、PR 模板和文档入口沉淀下来，让 AI 读得到，也让人看得懂。AI 越强，越需要清晰的工程边界；否则它会把模糊需求快速放大成模糊实现。

这篇文章讨论的不是某个工具的测评，而是一套本地 AI 研发工作流的搭建方式。重点是 DeepChat、CLI、编辑器和浏览器自动化如何分工，如何共享上下文，如何把探索、实现、验证和交付连起来。不同团队可以换成自己熟悉的工具，但原则相同：聊天适合推理和整理，终端适合确定性执行，编辑器适合代码编辑，浏览器适合真实体验验证，版本控制适合沉淀结果。

核心概念

第一个概念是上下文入口。AI 工具必须知道当前任务是什么、代码在哪里、约束是什么、已有决策是什么。上下文入口可以是 README、AGENTS.md、架构文档、Issue、错误日志、测试失败输出，也可以是 DeepChat 里整理过的任务说明。没有入口，AI 会靠猜；入口太多又会淹没重点。工程上要为常见任务提供稳定入口，比如“修 bug 先读哪些文件”“改 API 先看哪份契约”“前端验证要打开哪个本地地址”。

第二个概念是执行边界。聊天工具擅长讨论和推理，但不适合直接承担所有文件操作；CLI 擅长执行命令，但不应该替代设计判断；编辑器擅长精确修改，但不适合跨系统验证；浏览器自动化擅长验证交互，但不应该成为业务逻辑来源。每个工具都要有边界。边界越清楚，越容易把 AI 放进流程，而不是让它在所有地方都半懂不懂地乱跑。

第三个概念是可验证小步。AI 工作流不应该追求一次性完成一个巨大任务。更好的节奏是：提出假设，定位文件，修改一小段，运行验证，记录结果，再进入下一步。这个节奏看起来慢，但对复杂代码库更快，因为它降低了返工概率。尤其在有用户改动、脏工作区、多人协作和 CI 门禁的项目里，小步验证比大改后祈祷可靠得多。

第四个概念是工具状态。浏览器有登录态，编辑器有未保存文件，终端有当前目录和环境变量，DeepChat 有对话上下文，Git 有工作区状态。这些状态都会影响 AI 的判断。一个常见事故是 AI 在错误目录跑命令，或者基于旧浏览器页面截图判断问题。工作流里要显式检查状态，比如当前分支、当前端口、当前页面、测试数据库、环境变量和未提交文件。

第五个概念是可审计输出。AI 参与研发后，团队需要知道它改了什么、为什么改、怎么验证。最终产物不应该只是一堆文件变化，还应该有简短的变更说明、测试结果、残余风险和回滚方式。可审计不是为了仪式感，而是为了让 review 和维护成本可控。

第六个概念是人机分工。工程师应该把重复搜索、样板修改、测试执行、文档整理和多路径验证交给工具，把需求判断、架构取舍、安全边界、上线节奏和最终验收留给人。AI 可以提出建议，但不应该替团队决定风险偏好。一个成熟的工作流，会让模型更快暴露信息，而不是让模型独自承担责任。

架构/流程图解说明

一个实用的 AI 本地研发工作流，可以看成四层：对话层、执行层、验证层、沉淀层。对话层负责把模糊任务变成可执行计划；执行层负责在仓库和系统里做动作；验证层负责用测试、浏览器和日志证明结果；沉淀层负责把结论写回代码、文档、Issue 或 PR。

这张图里最容易被忽视的是验证层。很多 AI 编程演示会停在“代码改完了，测试也通过了”。但前端体验、浏览器兼容、登录态、权限路径、网络请求、加载状态和视觉布局，经常只能通过真实页面发现。对后端服务来说，也需要在本地或测试环境打一次真实请求，检查日志和指标。验证层是把“看起来能工作”变成“确实按用户路径工作”的关键。

上下文流动也需要设计。推荐把任务上下文拆成三类：稳定上下文、临时上下文、验证上下文。稳定上下文放在仓库文档里，例如架构、命令、代码规范和领域词汇；临时上下文来自当前 Issue、错误日志和用户说明；验证上下文来自本次运行的测试输出、截图和控制台。AI 每次工作时优先读取稳定上下文，再接收临时上下文，最后根据验证上下文修正判断。这样不会把所有历史聊天都当成真相。

在多人团队里，可以把常用流程固化成“工作流卡片”。比如：

工作流卡片不是限制工程师，而是减少重复解释。团队新人和 AI 都可以从这些卡片里理解“这类任务怎样算做完”。

工程实现

落地时可以从一个本地工作区约定开始。仓库里维护几类文件：面向人的 README，面向 AI 和工程协作的 AGENTS.md，面向任务的 Issue 或 Markdown 说明，面向验证的脚本和测试命令。不要把所有规则写进聊天 prompt；应该把长期有效的规则放进仓库。这样换客户端、换模型、换工程师时，流程仍然存在。

一个简单的 AGENTS.md 可以包含：

# Agent Guide

## Project shape
- Web app: `src/`
- API server: `server/`
- Shared types: `packages/contracts/`

## Common commands
- Install: `pnpm install`
- Unit tests: `pnpm test`
- Lint: `pnpm lint`
- Dev server: `pnpm dev`

## Rules
- Do not edit generated files under `dist/`.
- Prefer existing UI components from `src/components/ui/`.
- For API changes, update contract tests.
- For frontend changes, verify in browser at `http://localhost:3000`.

这类文件看起来普通，但对 AI 工作流很有用。它把项目形状和禁区提前说清楚，减少模型在错误目录里探索。更重要的是，它让团队把“我们平时怎么做事”写下来。AI 不是唯一受益者，新同事也会受益。

任务执行时，我喜欢使用一个固定的任务记录结构。它不一定要变成系统，可以是 Issue 模板，也可以是本地 Markdown：

task:
  id: ui-filter-empty-state
  goal: 修复筛选后空状态文案不显示的问题
  constraints:
    - 不改接口
    - 复用现有 EmptyState 组件
    - 保持移动端布局不变
  context:
    - issue: "#482"
    - entry_files:
        - src/pages/orders/ListPage.tsx
        - src/components/OrderFilter.tsx
  verification:
    - pnpm test -- OrderFilter
    - pnpm lint
    - browser: http://localhost:3000/orders
  done_when:
    - 空结果显示明确文案
    - 清除筛选按钮可用
    - 控制台无错误

这个结构把目标、约束、上下文和验收放在一起。DeepChat 可以先帮你把 Issue 整理成这个结构；CLI 和编辑器可以根据 entry_files 开始搜索；浏览器自动化可以根据 verification 打开页面。关键是，AI 不再只收到一句“修一下 bug”，而是收到一个可执行工作单。

CLI 层的实现要坚持确定性。搜索用 rg，文件列表用 rg --files，测试用项目已有命令，格式化用仓库配置。不要让 AI 用临时脚本去绕过现有工具，也不要让它在不理解的情况下全局替换。一个良好的命令序列通常是：

1. git status --short
2. rg "EmptyState|empty state|No orders" src
3. sed -n '1,220p' src/pages/orders/ListPage.tsx
4. sed -n '1,220p' src/components/OrderFilter.tsx
5. pnpm test -- OrderFilter
6. pnpm lint
7. pnpm dev
8. browser verify /orders with filter applied

这不是固定模板，而是一种纪律：先确认工作区状态，再找入口，再读文件，再改，再验证。很多 AI 造成的麻烦，来自跳过前两步，直接根据文件名猜实现。

编辑器层要关注修改粒度。AI 很擅长生成大段代码，但工程上更希望它改小一点。一次修改最好能用一句话说明：补一个空状态分支，修一个工具参数映射，增加一个错误码处理，调整一个测试期望。小粒度修改有三个好处：review 容易，回滚容易，失败后定位容易。如果一个任务确实很大，就拆成多次提交或多个垂直切片。

浏览器自动化层可以分两类。一类是开发者验证，例如打开本地页面、检查按钮、查看控制台和截图。另一类是回归脚本，例如 Playwright 测试。不要把这两类混在一起。开发者验证适合快速确认真实体验，回归脚本适合长期保护关键路径。AI 可以先用浏览器自动化探索问题，等修复稳定后，再补一条正式测试。

DeepChat 的位置更像工作台。它适合保留长对话，比较不同方案，整理会议纪要，生成测试用例草稿，分析日志片段，甚至让多个模型互相校验结论。但它不应该成为唯一事实来源。最终事实应该回到仓库、测试结果、截图、PR 和文档里。聊天记录可以帮助思考，不能替代工程记录。

一个完整例子：用户反馈“订单列表筛选后白屏”。工作流可以这样走：

1. 在 DeepChat 里整理现象：筛选条件、账号类型、浏览器、是否必现。
2. 在 CLI 里搜索订单列表和筛选组件，确认入口文件。
3. 读代码发现空数组分支没有传给 EmptyState，而是继续渲染表格。
4. 在编辑器里补空状态分支，复用已有组件，不改接口。
5. 跑单测，发现现有测试只覆盖有数据场景。
6. 补一个空结果测试，确认按钮能清除筛选。
7. 启动本地服务，用浏览器自动化实际选择筛选条件。
8. 检查控制台无错误，截图记录空状态。
9. 最终在 PR 描述里写清楚修改、测试和验证路径。

这套流程里，AI 没有“神奇地解决问题”，它只是加速了每个环节：整理上下文更快，搜索入口更快，写测试更快，浏览器验证更稳定。人的责任仍然是判断空状态文案是否符合产品语气、是否应该清除筛选、是否影响移动端体验。

测试评测

AI 工作流本身也需要评测。否则团队只会凭感觉说“好像更快”。可以从四个维度评估：任务完成时间、返工率、缺陷逃逸率和上下文沉淀质量。完成时间只看快不快；返工率看 AI 引入了多少需要人工修正的问题；缺陷逃逸率看上线后是否有回归；上下文沉淀质量看本次经验是否变成了文档、测试或工作流规则。

具体可以维护一张轻量记录表：

任务使用工具节省时间返工原因验证方式是否沉淀
修订单空状态CLI、编辑器、浏览器约 30 分钟初版漏移动端单测、截图增加测试
调整 RAG rerank 参数DeepChat、CLI约 1 小时样本切片不足eval 回归更新评测集
写 API 迁移文档DeepChat、编辑器约 40 分钟术语不统一人工 review更新术语表

这个记录不是为了考核个人，而是为了发现流程问题。如果大量返工都来自“AI 不知道项目规则”，那就该补 AGENTS.md 和代码注释；如果返工来自“验证不充分”，就该补测试和浏览器脚本；如果节省时间主要出现在文档整理，而代码任务收益不明显，就说明当前代码上下文入口可能不够好。

测试层面要区分 AI 生成的测试和人认可的测试。AI 可以快速补测试，但测试是否表达了真实业务风险，仍然需要工程师判断。一个坏测试会制造虚假的安全感。比如只检查按钮存在，不检查点击后的状态；只 mock 成功响应，不覆盖错误码；只快照整个页面，不验证关键文案。AI 工作流应该鼓励补“能抓住这次 bug 的测试”，而不是补“看起来覆盖率上升的测试”。

前端验证建议至少包含三项：关键路径截图、控制台错误检查、响应式布局抽查。后端验证至少包含三项：单元测试、集成请求、日志或指标检查。Agent 和 RAG 验证还要包含评测样本回归，尤其是历史失败样本。所有验证结果都应该在最终说明里出现，哪怕只是简短一句“已跑 pnpm test，浏览器验证 /orders 空状态无控制台错误”。

失败模式

第一种失败模式是工具太多但没有流程。装了 DeepChat、多个 CLI agent、编辑器插件、浏览器插件和一堆自动化脚本，却没有定义什么时候用哪个。结果是同一个问题在多个工具里重复解释，上下文散落，最后还要人手动合并结论。工具数量不是能力，流程才是能力。

第二种失败模式是上下文过载。很多人以为给 AI 越多文件越好，结果模型在无关信息里迷路。更好的做法是先给任务目标和入口文件，让 AI 自己按证据扩大范围。稳定规则放文档，当前问题给最小必要上下文，验证结果实时补充。上下文应该像索引，而不是垃圾堆。

第三种失败模式是忽视工作区状态。AI 在脏工作区里修改文件，如果不先看 git status，很容易覆盖用户改动或把无关变化混进任务。团队要把“先看工作区状态”变成习惯。对于多人协作仓库，还要明确哪些文件是生成物、哪些文件不能手改、哪些迁移需要单独提交。

第四种失败模式是只跑单测不看真实页面。前端尤其容易出现这种问题。组件测试通过，但真实页面里 CSS 被父容器影响；API mock 正常，但真实接口字段为空；桌面布局正常，移动端按钮溢出。浏览器自动化不是可选装饰，它是前端 AI 工作流的基本验证工具。

第五种失败模式是让 AI 做模糊产品决策。比如“让页面更高级”“优化一下回答质量”“把流程做顺”。这些话需要先拆成可验证目标。页面更高级可能意味着信息密度、视觉层级、加载状态和可访问性；回答质量可能意味着事实正确率、引用准确率、拒答策略和语气。没有指标，AI 只能输出看似合理的改动。

第六种失败模式是没有沉淀。AI 帮你查了半小时，最后只留下一个口头结论。下次遇到类似问题，又从零开始。任何重复出现的问题，都应该沉淀到文档、测试、脚本或工作流卡片里。AI 工具组合的复利，来自这些沉淀，而不是单次对话。

第七种失败模式是权限边界不清。浏览器有登录态，CLI 可以访问本地文件，某些工具可以调用云服务。团队必须明确哪些数据可以发给模型，哪些命令需要人工确认，哪些环境不能让 AI 操作。尤其涉及客户数据、生产数据库、密钥和付费 API 时，要默认保守。

上线 checklist

把这套工作流推广到团队时，最需要警惕的是“每个人都觉得自己已经会用了”。AI 工具的个人体验很容易给人一种错觉：只要自己能和模型聊明白，团队流程就自然会变好。事实正好相反，个人效率提升如果没有沉淀，团队协作可能更混乱。有人让 AI 大范围改文件，有人只让 AI 写测试，有人把验证结果留在聊天记录里，有人把敏感日志直接贴进外部模型。上线 checklist 的意义，就是把这些隐含习惯变成显式约束。

团队可以先选择两三类高频任务做试点，而不是要求所有研发活动都接入 AI。例如前端缺陷修复、接口兼容性调整、RAG 样本回归，这些任务都有明确入口和验证方式，适合形成标准路径。试点期间要记录失败案例：AI 读错了哪个约束，漏了哪个测试，误用了哪个组件，在哪一步需要人工纠偏。记录这些问题，比炫耀节省了多少时间更重要，因为它们会告诉你下一步应该补文档、补测试，还是收紧工具权限。

另外，团队要接受一个现实：AI 工作流不是一次配置完成的系统，而是一套会随代码库演进的操作规程。新目录、新服务、新设计系统、新安全要求出现后，入口文档和验证清单也要更新。否则模型会按旧规则工作，人也会按旧习惯 review。把工作流维护纳入日常工程治理，才能让工具组合长期稳定，而不是刚上线时热闹，几个月后又回到各用各的状态。

  • 仓库有清楚的 AI 协作入口文档，说明项目结构、常用命令、禁区和验证方式。
  • 本地 CLI 工作流默认先检查 git status --short，避免混入用户未提交改动。
  • 常见任务有稳定入口，例如前端页面入口、API handler、契约文件、评测样本和日志查询方式。
  • DeepChat 中形成的长期决策会回写到仓库文档或 Issue，而不是只留在聊天记录里。
  • 代码修改保持小步提交或小步 diff，每一步都有可运行的验证。
  • 前端变更必须经过真实浏览器路径验证，至少检查页面、控制台和关键响应式尺寸。
  • 后端变更必须跑相关单测或集成请求，并检查错误处理和日志。
  • RAG、Agent、模型相关变更必须跑历史失败样本或核心 eval。
  • AI 生成的测试要由人确认是否覆盖真实业务风险。
  • 自动化工具的权限边界明确，不把生产密钥、客户原文和敏感数据随意发给模型。
  • PR 或最终说明包含变更摘要、验证命令、浏览器验证结果和残余风险。
  • 团队定期复盘 AI 工作流的返工来源，把高频问题沉淀为文档、测试或脚本。
  • 对新同事和新模型都使用同一套工作流入口，避免经验只存在个人习惯里。

总结

AI 研发效率不是靠单个神奇工具堆出来的，而是靠一套能闭环的本地工作流。DeepChat 适合讨论和整理，CLI 适合确定性执行，编辑器适合精确修改，浏览器自动化适合真实体验验证，Git 和 CI 适合沉淀协作结果。每个工具都做自己擅长的事，组合起来才会稳定。

这套工作流的核心不是“让 AI 全自动开发”，而是让探索、修改、验证和交付之间少一点搬运，多一点证据。工程师仍然要负责判断目标、风险和质量；AI 负责加速搜索、生成、执行和检查。只要边界清楚，小步验证，结果可审计，AI 就能进入日常研发，而不是停留在演示里。

真正有复利的是沉淀。一次问题修完以后，最好留下测试、文档、脚本或工作流卡片。下次 AI 和人再进来，就不用重新摸索。长期看，优秀团队不会只是“更会问 AI”，而是会把 AI 能利用的工程上下文建设得更好。工具会不断变化，但清晰上下文、确定性执行、真实验证和可审计交付这几件事，会一直有效。
