Agent Eval Harness

Agent 测试需要模拟工具、固定环境和可回放轨迹。

AgentEvals

问题背景

很多团队第一次给 Agent 做评测，会自然沿用问答系统的办法：准备一批问题，跑一遍模型，把最终回答拿去人工打分，或者交给另一个模型判断。这种方式对普通问答有一定价值，但放到 Agent 上很快失效。Agent 的错误经常不在最后一段文字里，而在中间路径里。它可能查了错误租户的数据，可能把只读任务变成写操作，可能在工具超时后没有重试，可能没有拿到用户确认就继续执行，也可能为了让最终回答看起来完整而编造了工具结果。最终文字如果写得客气、流畅、结构清楚，反而会把这些工程风险遮住。

真正的 Agent 评测要回答的是“它有没有按可接受的方式完成任务”。这句话里有三个关键词：任务、方式、可接受。任务意味着测试样本不能只有一句 prompt，还要包含用户身份、权限、时间、上下文、工具返回、外部系统状态和期望产物。方式意味着不能只看答案，要看计划、工具选择、参数、重试、分支、停止条件和人工确认。可接受意味着系统允许多条合理路径，但要明确哪些行为永远不能发生，例如跨租户读取、调用生产写接口、泄漏隐私字段、跳过审批、把低置信证据说成确定结论。

如果没有专门的 eval harness，Agent 评测会很容易退化成脆弱的演示脚本。开发者在本地跑一次，看起来成功；CI 上因为时间、检索索引、工具数据或模型随机性变化而失败；线上又因为权限和真实数据形态不同而走出另一条路径。结果是团队无法判断一次失败到底是模型退化、提示词改坏、工具 schema 变化、fixture 过旧，还是样本本身写得不清楚。评测本来应该增加信心，最后却变成噪声来源。

Eval harness 的定位就是把这些变量收束起来。它提供一个可重复的运行壳：固定时间，固定用户，固定工具返回，固定权限边界，固定模型参数或模型网关配置，记录完整轨迹，然后用一组规则和评审器判断结果。它不负责证明 Agent 永远正确，也不负责覆盖所有真实世界变化。它负责让一批重要场景可以稳定回放，让团队在改模型、改 prompt、改工具、改运行时之前知道影响在哪里。

这里还有一个现实问题：Agent 的质量不是单点指标。一个客服 Agent 可能最终回复正确，但多调用了三个昂贵工具；一个代码 Agent 可能修复了 bug，却改了不该改的文件；一个运营 Agent 可能生成了可用文案，却没有引用活动规则；一个内部助理可能成功创建任务，却没有把风险操作交给用户确认。不同场景的门槛不同，harness 必须允许多维断言，而不是所有样本都套一个“答案相似度”。

我更愿意把 Agent eval harness 看成“可执行的质量契约”。产品把风险边界写成样本，工程把工具和环境写成 fixture，运行时把轨迹写成事件，评分器把可接受行为写成断言。每次发布前跑这些契约，团队就能讨论具体事实：哪个样本失败，哪个步骤偏离，哪个工具参数错，哪条策略没有触发。这样的讨论才有工程价值。

核心概念

一个可用的 Agent eval harness 至少要包含六个概念：评测用例、环境快照、工具替身、执行轨迹、判定器和报告。评测用例描述任务和期望；环境快照固定运行条件；工具替身模拟外部系统；执行轨迹记录 Agent 做过什么；判定器负责判断路径和产物；报告把结果聚合给开发、产品和运维看。

概念作用容易做错的地方工程判断
评测用例定义用户目标、输入、权限、预期产物只写 prompt，不写上下文和禁止行为用例要能独立解释任务发生的世界
环境快照固定时间、租户、记忆、索引版本、模型参数允许实时数据进入，导致结果漂移能冻结的冻结，不能冻结的标明可变
工具替身返回可控数据，模拟失败和延迟mock 太干净，任何参数都返回正确结果mock 要验证参数，也要模拟真实脏数据
执行轨迹记录计划、调用、返回、策略、产物只保存最终 answer，看不到中间错误事件流比日志字符串更适合评分
判定器检查硬约束、语义质量和成本用 LLM judge 替代所有规则硬规则先行，开放质量再交给评审
报告对比版本、定位失败、辅助上线只给总分，不能行动每个失败要有证据和归因线索

评测用例不是普通测试里的 fixture 文件名集合，而是一个小型世界描述。它要说清楚谁在请求、请求发生在什么时间、用户拥有哪些权限、哪些工具可用、工具返回怎样的数据、哪些动作必须发生、哪些动作禁止发生、最终产物有哪些最低要求。Agent 的输入如果只是一句“帮我整理续约风险”，那这个任务在真实环境里有无数解释。只有把客户、合同、工单、权限和时间固定下来，评测才有可比性。

工具替身是 harness 的核心。Agent 与普通生成任务最大的差异是它会行动。行动需要外部系统，而外部系统在测试里不能随便真实调用。好的工具替身不是简单返回一段 JSON，而是一个带约束的模拟服务：它能检查参数是否包含正确租户，能模拟分页，能返回部分失败，能记录调用顺序，能区分 dry-run 和 commit，能在高风险调用上要求确认 token。这样才能测出 Agent 是否真的尊重工具协议。

执行轨迹要结构化。不要依赖模型在最终回答里自述“我查询了客户系统”。评测应该读取运行时产生的事件：plan.created、tool.called、tool.result、policy.checked、user.confirmation.required、artifact.written、run.completed。每个事件包含时间、父步骤、工具名、参数摘要、状态、错误类别和关联 artifact。结构化轨迹让评分器可以写确定性断言，也让失败排查不必从一大段自然语言里猜。

判定器可以分为三类。第一类是硬规则，例如禁止工具是否被调用、租户是否一致、写操作是否经过确认、最终状态是否成功。第二类是产物规则，例如 Markdown 是否包含指定字段、代码 patch 是否只改允许路径、JSON 是否符合 schema、引用是否能追溯到工具返回。第三类是语义评审，例如摘要是否覆盖主要风险、建议是否可执行、语气是否适合客户。三类判定器不要混在一起，否则一次失败会变得很难解释。

还有一个概念叫“可接受差异”。Agent 不是传统确定性函数，同一任务可能有多条好路径。Harness 不应该强迫每个步骤逐字相同，也不应该把某次黄金输出当成唯一正确答案。更好的做法是锁住关键约束：必须查合同和工单，不能发外部邮件，必须引用证据，必须在折扣建议前提醒审批，最终产物要包含风险、证据、建议和待确认项。这样既保留模型改进空间，又守住工程边界。

架构/流程图解说明

一个实用的 Agent eval harness 可以按流水线理解：用例仓库提供样本，runner 创建隔离运行环境，tool proxy 拦截所有工具调用，Agent runtime 执行任务，trace collector 收集事件，scorer 读取轨迹和产物，reporter 输出版本对比。关键是评测必须经过与生产一致的 runtime，不能绕过权限、策略和工具网关直接调用模型。

这张图里最容易被低估的是 Tool Proxy。很多团队在评测里直接把工具函数替换成本地函数，短期很快，长期会偏离生产。生产里工具调用通常要经过 schema 校验、权限检查、速率限制、审计、幂等键和错误分类。评测里如果绕过这些边界，就测不到 Agent 在真实协议下的表现。Tool Proxy 的职责是保留协议边界，同时把外部副作用变成可控 fixture。

第二个关键点是临时工作区。很多 Agent 会读写文件、生成 patch、创建草稿、更新状态。评测不能让不同样本共享同一个目录或同一份数据库状态。Runner 应该为每个用例创建独立工作区，注入只读输入，收集输出 artifact，最后销毁或归档。这样样本之间不会互相污染，也能把失败现场保存下来。

第三个关键点是报告不是简单 pass/fail。对工程团队最有用的是版本差异。例如同一个用例在基线版本通过，新版本失败；或者两者都通过，但新版本工具调用数从三次涨到九次；或者新版本不再触发人工确认，虽然最终产物仍然完整。报告应该把这些变化列出来，让上线讨论围绕具体风险展开。

工程实现

实现上可以从一个朴素的用例格式开始。不要一开始就发明过度抽象的 DSL。评测样本应该让工程师和产品同学都能读懂，同时足够结构化，方便 runner 执行。下面是一个“内部客户续约分析”的用例，重点不是字段名，而是它把任务世界、工具返回和判断规则放在同一个地方。

id: agent-renewal-risk-001
title: 生成客户续约风险摘要
now: "2026-03-04T10:00:00+08:00"
user:
  id: "u_173"
  tenant: "acme-cn"
  role: "account_manager"
prompt: "帮我整理星河制造下季度续约风险，先给我内部摘要，不要直接发给客户。"
environment:
  memory_snapshot: "empty"
  index_version: "kb_20260304_0900"
  model_profile: "release_candidate"
tools:
  crm.customer.search:
    assert_args:
      tenant: "acme-cn"
      name_contains: "星河制造"
    response: "fixtures/renewal/customer.json"
  contract.list:
    assert_args:
      customer_id: "cus_8848"
    response: "fixtures/renewal/contracts.json"
  ticket.search:
    response: "fixtures/renewal/tickets.json"
expect:
  required_tools:
    - "crm.customer.search"
    - "contract.list"
    - "ticket.search"
  forbidden_tools:
    - "email.send"
    - "crm.discount.apply"
  must_include:
    - "合同到期时间"
    - "未关闭工单"
    - "折扣审批"
  require_citations: true
  max_tool_calls: 8

这个例子有几处设计很实际。now 固定了“下季度”的解释，user 固定权限和租户，environment 让模型和索引版本可追溯，assert_args 不只是返回 fixture，还会检查 Agent 是否查了正确客户，forbidden_tools 明确禁止对外发送和直接申请折扣，max_tool_calls 防止路径发散。评测不是为了让 Agent 背答案，而是为了让它在一个固定世界里做出合理行动。

Runner 的数据结构可以很简单。核心是把样本、运行结果和评分结果分开。样本是稳定资产，运行结果是某个版本的产物，评分结果是对运行结果的解释。三者分开后，基线对比、失败重跑、报告归档都会清楚。

type EvalCase struct {
    ID          string
    Title       string
    Now         time.Time
    User        EvalUser
    Prompt      string
    Environment EvalEnvironment
    Tools       map[string]ToolFixture
    Expect      Expectations
}

type EvalRun struct {
    RunID        string
    CaseID       string
    AgentVersion string
    ModelProfile string
    Events       []TraceEvent
    Artifacts    []ArtifactRef
    Usage        UsageSummary
    Status       string
}

type CheckResult struct {
    Name     string
    Passed   bool
    Severity string
    Message  string
    Evidence []string
}

工具替身要按真实协议实现。假设生产工具调用统一经过 ToolInvoker，评测里就不要让 Agent 直接调用本地函数，而是替换 ToolInvoker 的实现。这个实现先记录调用事件，再校验工具是否在样本允许列表里，再校验参数，最后返回 fixture 或注入错误。对于写工具，它可以只接受 dry-run，或者要求样本里明确配置确认事件。

type HarnessToolInvoker struct {
    Fixtures map[string]ToolFixture
    Recorder TraceRecorder
}

func (h *HarnessToolInvoker) Invoke(ctx context.Context, call ToolCall) (ToolResult, error) {
    h.Recorder.Record("tool.called", map[string]any{
        "tool": call.Name,
        "args": Redact(call.Args),
    })

    fixture, ok := h.Fixtures[call.Name]
    if !ok {
        return ToolResult{}, fmt.Errorf("unexpected tool: %s", call.Name)
    }
    if err := fixture.Assert(call.Args); err != nil {
        return ToolResult{}, err
    }
    return fixture.Result(), nil
}

这里的 Assert 很重要。没有参数断言的 mock 会让评测失真。Agent 查错客户、漏传租户、把日期范围传反，如果 mock 仍然返回正确数据，最终回答就可能通过。工具替身应该像严格的服务端一样拒绝不合约的请求，并把拒绝写进 trace。这样失败才会指向具体参数，而不是最后变成“回答质量不够”。

时间和随机性也要被 harness 接管。Agent 里所有“今天”“本周”“最近三个月”都应该从注入的 clock 读取，而不是直接调用系统时间。模型温度、随机种子、重试策略也要在评测 profile 里固定。对于无法完全确定的模型输出，可以多跑几次，用“关键约束通过率”而不是逐字一致性判断稳定性。重要的是把不确定性显式化，不要让它偷偷进入 CI。

产物检查要贴近任务类型。代码 Agent 要检查 diff 范围、测试结果、格式化、是否修改禁止路径；数据分析 Agent 要检查查询条件、聚合口径、图表字段；办公 Agent 要检查收件人、附件、审批状态；RAG Agent 要检查引用和证据。通用 harness 只提供事件、工具和报告框架，具体判定器应该按领域扩展。

一组比较可靠的评分顺序是：先检查运行状态，再检查禁止行为，再检查必要行为，再检查产物结构，再检查语义质量，最后检查成本和延迟。这个顺序符合排障习惯。如果 Agent 已经调用了禁止工具，就没有必要再让 LLM judge 判断摘要好不好；如果产物 JSON 都不合法，也不必先讨论语气。

检查阶段示例断言失败处理
运行状态run.completed，未 panic，未超时标记基础设施或运行时失败
禁止行为未调用生产写工具，未跨租户读取阻断发布，必须修复
必要行为调用合同、工单、客户工具归因到计划或工具选择
产物结构Markdown 字段齐全，JSON schema 通过归因到输出约束或后处理
语义质量风险排序合理，建议可执行进入人工或模型评审
成本延迟工具调用数、token、耗时未超预算作为性能回归处理

报告层要支持基线对比。每次评测保存 agent_version、prompt_version、model_profile、tool_schema_version 和 dataset_version。当某个样本从通过变失败时，报告要显示变化发生在哪个阶段：工具参数不同、检索证据不同、策略事件缺失、最终产物缺字段。不要只给红色叉号。红叉没有证据，最后还是要人工翻日志。

还要给样本生命周期留位置。线上事故转成样本后，先进入候选集；稳定复现并明确期望后，进入事故回归集；反复通过且覆盖核心能力后，可以进入核心集。不同集合在 CI 里的运行频率不同：提交时跑核心小集，合并前跑核心加事故集，每天跑完整集，发布前跑接近生产配置的灰度集。这样可以兼顾速度和覆盖。

并发执行也要提前设计。评测集一大，单线程跑会拖慢发布，但并发太高又会让模型网关、fixture 服务和本地资源互相干扰。Runner 可以按样本声明的资源类型分组：纯文本样本可以高并发，文件写入样本使用独立临时目录，依赖同一模拟数据库的样本串行，调用真实只读服务的样本设置较低并发。并发策略要写进报告，因为同一个样本在资源饱和时失败，和 Agent 行为失败不是一回事。

缓存策略也要谨慎。为了省钱，很多团队会缓存模型输出或工具返回。评测里可以缓存 fixture 读取和静态 artifact，但不要默认缓存被测模型结果，否则新版本可能实际没有被执行。确实需要缓存时，缓存键必须包含样本 ID、模型 profile、prompt 版本、工具 schema 版本、运行时版本和随机参数。报告里也要标明哪些结果来自缓存，哪些结果是真实运行。质量门禁通常应该使用真实运行结果，缓存更适合开发阶段的快速调试。

最后，harness 要给人工评审留接口。自动规则能抓住边界问题，但开放任务仍然需要人判断。比较好的方式是把人工评审做成结构化表单，而不是让评审者写一段自由评论。表单可以包含目标完成度、证据充分性、风险提示、表达清晰度、是否需要澄清、是否可直接交付。人工结论写回 CheckResult，和自动评分一起进入报告。这样人的判断也能沉淀成数据，而不是散落在评审会议里。

测试评测

Eval harness 自身也要测试。它是质量门禁的一部分，不能默认可信。第一类测试是样本解析测试。缺字段、日期格式错误、工具名拼错、fixture 文件不存在、期望字段类型不对，都应该在运行前失败，并给出清楚错误。不要让样本错误伪装成 Agent 失败。

第二类测试是工具替身测试。要构造正确参数、错误租户、缺少必填字段、未知工具、写工具未确认等场景，确认 proxy 会按预期返回结果或拒绝。尤其要测参数断言的负例。很多 harness 一开始只测正例，结果 mock 写得过宽，线上才发现 Agent 传错参数。

第三类测试是评分器单元测试。可以手工构造 EvalRun，不启动真实 Agent，直接喂给评分器。比如构造一个包含 email.send 的事件流，确认禁止工具评分失败；构造一个调用了合同工具但状态为 timeout 的事件流，确认必要工具不算通过；构造一个产物有引用字段但引用 ID 不存在的 artifact，确认引用检查失败。评分器的边界越清楚，报告越可信。

第四类是端到端回放测试。选择少量稳定样本，启动完整 Agent runtime，通过 Tool Proxy 跑完，断言 trace、artifact 和报告都能生成。这类测试数量不必多，但要覆盖完整链路。每次运行时事件模型变化、工具 schema 变化、策略系统变化，都应该有端到端样本暴露兼容性问题。

评测指标不要只看通过率。通过率当然重要，但它会掩盖成本、稳定性和人工接管。一个发布候选如果通过率略升，但平均工具调用数翻倍，或者高风险任务人工确认率下降，就要谨慎。下面是一份更适合工程讨论的报告摘要：

还可以做 mutation testing。故意把样本里的租户改错、去掉必要工具、替换过期证据、关闭确认策略，看评分器能不能抓住。如果评分器抓不住这些人为注入的问题，就说明门禁不够强。Mutation 不需要每天跑，但在新增评分器或高风险领域样本时很有价值。

最后要管理 flaky。Agent 评测中偶发失败不可避免，尤其是语义评审和外部模型调用。不要简单重跑到通过。每个 flaky 样本都应该标记原因：模型随机、fixture 不稳定、评审器漂移、运行时超时、样本期望过窄。短期可以隔离到观察集，长期要修复根因。评测系统如果长期允许“重跑就过”，它的上线门槛很快会失去信用。

失败模式

第一个失败模式是只评最终答案。Agent 走错路径但最终说得像成功，报告仍然通过。这会奖励语言包装，而不是可靠执行。解决办法是把工具调用、策略事件和 artifact 纳入硬断言。

第二个失败模式是 mock 过于理想。fixture 数据干净、字段齐全、没有分页、没有权限裁剪、没有重复记录，Agent 在评测里表现很好，上线遇到真实数据就失败。解决办法是让 fixture 带脏数据，并用参数断言模拟真实服务端。

第三个失败模式是样本写得太窄。把某次成功输出逐字当成黄金答案，模型稍微调整表达就失败。这会阻碍合理改进。解决办法是用结构和行为约束替代逐字匹配，只在法规、合同、代码等必须精确的场景使用严格比对。

第四个失败模式是样本写得太宽。只要求“回答包含风险”，不要求查证据、不要求引用、不限制工具，Agent 靠猜也能过。解决办法是把关键工程动作写进 required_tools、must_include、require_citations 和禁止行为。

第五个失败模式是评审器漂移。LLM judge 版本变化后，同一输出分数变化，团队误以为 Agent 质量变化。解决办法是把 judge 版本纳入报告，对关键样本保留人工标注，并且让硬规则承担安全门禁。

第六个失败模式是评测和生产 runtime 分叉。评测跑的是简化链路，生产跑的是完整链路，结果评测无法发现权限、审计、幂等和策略问题。解决办法是让评测经过同一套 runtime，只替换外部副作用和数据源。

第七个失败模式是报告不可行动。报告只显示总分下降，没有样本、事件和证据链接。解决办法是每个失败都必须包含评分器名称、严重程度、事件引用、artifact 引用和建议归因。

上线 checklist

  • 样本 front door 已定义：新样本必须包含用户、时间、权限、工具 fixture、必要行为和禁止行为。
  • Runner 使用生产同源 Agent Runtime，评测没有绕过权限、策略、工具网关和后处理。
  • Tool Proxy 会校验参数、记录调用、模拟错误，并阻断未声明工具。
  • 每个样本运行在独立工作区，文件、数据库状态和缓存不会跨样本污染。
  • 时间、模型 profile、索引版本、prompt 版本和工具 schema 版本写入评测报告。
  • 禁止工具、跨租户、未确认写操作、无证据确定结论都属于阻断级失败。
  • 评分器有单元测试，至少覆盖正例、负例和边界例。
  • 关键端到端样本能在 CI 中稳定回放，失败时保存 trace 和 artifact。
  • 报告支持与基线对比，能列出新增失败、修复样本、成本变化和工具调用变化。
  • Flaky 样本有隔离和修复流程，不能靠无限重跑获得通过。
  • 线上事故可以转入候选样本池，并经过标注后进入事故回归集。
  • 发布门槛按集合和严重程度定义，而不是只看总通过率。

总结

Agent eval harness 的价值不在于制造一个漂亮分数，而在于把 Agent 的行为变成可回放、可比较、可审计的工程事实。它让团队看到任务是在什么环境里执行的，工具被怎样调用，策略是否触发，产物是否有证据，成本是否可接受。没有这些事实，Agent 评测很容易停留在“答案看起来不错”的阶段。

真正可用的 harness 要把变量固定住，把副作用隔离掉，把轨迹记录下来，把判定拆成硬规则和开放评审。它既要允许模型用不同方式完成任务，也要明确不可违反的边界。这样的系统建设起来不轻，但它是 Agent 从 demo 走向生产的必要基础。只要团队开始把线上事故、人工接管和高风险场景沉淀成样本，评测就会从一次性验收变成持续改进的工程循环。
