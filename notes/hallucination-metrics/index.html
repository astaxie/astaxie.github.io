幻觉指标怎么定义

幻觉要拆成无来源、错来源、过度推断和事实冲突。

HallucinationMetrics

问题背景

很多团队说“模型有幻觉”，但真正开始修的时候才发现，这句话没有工程含义。客服系统里把退款期限说错是幻觉，知识库问答里引用了不存在的文档也是幻觉，代码助手把一个私有函数名编出来还是幻觉。它们对用户的伤害不同，根因也不同。如果统一塞进一个 hallucination rate，最后只能得到一个好看或难看的数字，不能指导检索、提示词、模型、知识库和产品策略的改动。

我更愿意把幻觉看成系统没有履行证据契约。用户不是要求模型“像人一样想象”，而是要求系统在给定权限、知识、工具和时间点下，讲出可以被追溯的事实。事实来自文档、数据库、工具返回、用户输入、会话历史或明确允许的常识。只要答案越过这些边界，就需要被分类、计数和复盘。这个定义听起来严格，但生产系统需要严格，因为用户不会区分“模型猜错了”和“应用设计没有兜住模型”。

幻觉指标难做，还有一个原因是最终答案通常不是一个原子事实，而是一串声明。比如系统回答“你们的订单没有发货，因为仓库缺货，预计明天补发，可以在订单页申请补偿”。这里至少包含四个可检查声明：订单状态、未发货原因、补发时间、补偿入口。第一句可能来自订单工具，第二句可能来自库存系统，第三句可能没有证据，第四句可能来自帮助文档。只给整段答案打对错，会遮住真正的问题。

另外，幻觉和“不完整”不是同一件事。一个答案漏掉关键限制，是 coverage 问题；一个答案把没有证据的内容说成事实，才是 hallucination。二者可能同时发生，但修复方式不同。不完整通常要改召回、上下文预算或答案结构；幻觉则要改证据约束、生成策略、拒答规则和后处理校验。如果指标没有把它们分开，团队会把所有质量问题都推给模型，错过系统层面的修复机会。

在 RAG 和 Agent 场景中，幻觉指标还必须覆盖中间链路。最终答案没有明显错误，不代表链路可靠。模型可能引用了正确文档但得出错误结论；也可能答案正确但引用指向无关段落；还可能工具调用失败后用经验补全。只检查最终文本，会放过这些隐患。生产系统应该把 trace、证据、声明、引用和策略判断放在同一张评测表里，这样每一次失败都能落到可改的组件上。

很多幻觉事故不是“完全瞎编”，而是很接近事实。模型拿旧版本文档回答当前问题，拿相邻产品的规则回答目标产品，拿测试环境的配置解释生产故障，拿用户问题里的假设当成已证事实。这类错误最危险，因为读起来合理，人工抽查也容易被流畅表达骗过。指标设计必须能识别来源错位和边界扩张，而不是只抓不存在的名词。

所以我建议不要问“幻觉率是多少”，而要问四个更具体的问题：有多少声明没有来源，有多少声明引用了错误来源，有多少声明超出了来源能支持的范围，有多少声明和可信事实冲突。把这四类拆开后，工程动作才会清楚：无来源优先加证据约束和拒答；错来源优先修引用映射和检索过滤；过度推断优先收紧答案模板；事实冲突优先处理知识版本和冲突检测。

本文讨论的是生产系统的幻觉指标，不是学术论文里的通用 factuality benchmark。生产指标要服务上线决策、回归阻断、事故复盘和日常运营。它可以借用 LLM judge，但不能把所有判断都交给另一个模型。能用确定性规则检查的地方，就不要让裁判模型发挥；必须语义判断的地方，也要把输入、证据和判定理由记录下来，方便复核和校准。

核心概念

我把幻觉拆成四个一级类型：无来源、错来源、过度推断、事实冲突。它们不是互斥的哲学分类，而是便于工程归因的故障标签。一条声明可以同时命中多个标签，比如“功能 X 已在 2026 年 3 月全量上线”引用了一个 2025 年草案，并且当前发布系统显示它只灰度到 20%。这既是错来源，也是事实冲突。评测报告里可以保留多个标签，但阻断规则需要定义优先级。

无来源是最常见的幻觉。系统输出了一个事实性声明，但 trace 中找不到任何可支持它的证据。这里的“来源”不一定是文档，也可以是工具返回、结构化数据库、用户上传文件或对话上下文。关键在于声明能否回跳到系统可见的证据。比如“续费后额度会立即恢复”如果没有命中文档或计费工具返回，就属于无来源声明，即使它在现实中可能是真的。

错来源指答案给出了引用或来源，但来源不支持该声明，或者来源属于错误对象。它比无来源更隐蔽，因为界面上看起来有引用，用户会降低警惕。常见例子是引用了同名项目的文档、引用了旧版本政策、引用了包含关键词但结论相反的段落、引用了摘要而不是原文。错来源指标要检查“引用存在”之外的“引用支持度”，否则系统会学会贴一个看似相关的链接。

过度推断是从真实证据出发，但结论跨过了证据边界。文档写“部分地区可能延迟”，模型回答“所有海外订单都会延迟”；工具返回“工单状态为待审核”，模型回答“审核一定会在今晚完成”；日志显示“错误率升高”，模型回答“数据库已经损坏”。这类问题往往不是检索失败，而是生成阶段没有区分事实、解释、概率和建议。指标需要识别语气和范围的膨胀。

事实冲突是声明和可信事实源直接矛盾。冲突源可以是黄金数据集、结构化工具返回、最新文档、权限系统或已知业务规则。比如订单工具返回“已退款”，模型回答“尚未退款”；权限系统显示用户没有访问项目 A 的权限，模型却引用项目 A 的内部报告；最新发布说明写“接口已废弃”，模型仍建议使用旧接口。事实冲突通常应当高权重阻断，因为它说明系统没有服从更强证据。

类型判断问题典型根因首选修复

定义指标前，先定义声明。一个 answer 需要被切成 claim，claim 是可以单独判断真假的最小事实单元。不是所有句子都是 claim。“下面是排查建议”不是事实声明，“请先确认网络连通性”是建议，也不一定需要证据；“当前失败原因是证书过期”就是事实声明，需要证据。声明抽取可以用规则加模型混合完成，但抽取结果必须可审计，否则后面的分数没有基础。

我通常给 claim 加五个字段：text、claim_type、subject、scope、required_evidence_type。claim_type 可以是事实、因果、时间、数量、权限、建议、预测。subject 标记声明针对哪个对象，避免同名对象错配。scope 标记范围和条件，例如“仅企业版”“2026 年 3 月后”“北京区域”。required_evidence_type 标记需要文档、工具、数据库还是用户上下文支持。这个结构让后续 evaluator 不再只做文本相似。

还要区分强证据和弱证据。工具返回、数据库快照、签署后的政策文档通常是强证据；草案、历史讨论、用户问题里的假设、模型自己的中间推理是弱证据。强证据和弱证据冲突时，系统应该优先服从强证据。幻觉评测不能把所有上下文片段当成同等来源，否则旧文档和当前状态会互相抵消，最终让裁判无从判断。

一个可操作的指标集合可以这样定义：

指标分母分子说明

不要只看 grounded_answer_rate。这个指标适合给产品和运营看，但工程修复要看 claim 级别的分解。如果一个答案有十个声明，九个正确一个编造，答案级指标只告诉你“失败”；claim 级指标能告诉你失败发生在因果声明、时间声明还是数量声明。对工程团队来说，后者才有用。

架构/流程图解说明

幻觉评测的关键是把答案拆成可验证单元，并把每个单元和证据链对齐。推荐流程不是“拿问题和答案给裁判模型打分”，而是先收集应用 trace，再做声明抽取、证据归因、冲突检查和裁判复核。流程可以画成这样：

这里有两个设计点。第一，评测系统必须拿到 trace，而不是只拿最终 answer。trace 至少包含检索候选、进入上下文的片段、每个引用的 doc_id 和 chunk_id、工具调用参数和返回、系统提示词版本、模型版本、时间、用户权限。没有 trace，就只能凭最终文本猜测来源，很多错来源和过度推断会漏掉。

第二，声明和证据的匹配要分两步。先用确定性信息缩小范围，例如答案里的引用 ID、claim subject、doc version、工具名；再用语义判断证据是否支持声明。直接把全部上下文扔给 judge，很容易让裁判被无关材料干扰，也不利于定位是哪条引用错了。

图解说明看起来复杂，但实际落地可以从最小闭环开始：先在离线评测里保存 trace，抽取关键声明，检查引用是否存在和是否支持声明。等这个闭环稳定，再加入强事实源冲突检查、过度推断判定和线上抽样。不要一开始就追求全自动完美判断，幻觉评测真正的价值来自持续复盘和稳定口径。

工程实现

工程上我建议把幻觉评测拆成四个模块：trace 标准化、claim 抽取、evidence verifier、metric aggregator。它们可以在同一个服务里，也可以拆成批处理任务。重要的是模块之间传结构化数据，不要用一段自然语言解释在组件之间传来传去。

下面是一个简化的 Go 数据结构，足以支撑第一版评测：

type Trace struct {
    AnswerID   string
    UserScope  []string
    Model      string
    PromptHash string
    Retrieved  []Chunk
    Context    []Chunk
    ToolCalls  []ToolCall
    Citations  []Citation
    Answer     string
}

type Claim struct {
    ID                   string
    Text                 string
    Type                 string // fact, causal, temporal, quantity, permission, recommendation
    Subject              string
    Scope                string
    RequiredEvidenceType string // doc, tool, db, user_context
}

type Verification struct {
    ClaimID      string
    Label        string // grounded, unsupported, wrong_source, over_inference, contradiction
    Severity     string // low, medium, high, blocking
    EvidenceIDs  []string
    CitationIDs  []string
    StrongSource string
    Reason       string
}

Trace 的 Context 和 Retrieved 要分开。很多系统会召回十几段材料，但只有其中几段进入最终上下文。claim 的证据只能从模型可见内容、工具返回和显式引用里找；如果某段材料只是被召回但没有进入上下文，就不能证明模型基于它回答。Retrieved 仍然有价值，它能帮助定位“检索到了但组装丢了”的问题。

Claim 抽取不要过度依赖句号。中文技术回答经常一条句子里塞多个事实，比如“当前失败由证书过期触发，影响 API 网关和回放任务，修复方式是轮换服务账号”。这至少有三个声明。第一版可以让模型抽取 claim，但要用 schema 限制输出，并保留原文 span，方便人工检查。高风险领域可以叠加规则，比如金额、日期、权限、状态字段必须单独成 claim。

一个 claim 抽取结果可以长这样：

{
  "answer_id": "ans_20260306_001",
  "claims": [
    {
      "id": "c1",
      "text": "当前同步失败由服务账号权限收紧触发",
      "type": "causal",
      "subject": "同步失败",
      "scope": "当前事故",
      "required_evidence_type": "doc"
    },
    {
      "id": "c2",
      "text": "所有历史回放任务都会失败",
      "type": "quantity",
      "subject": "历史回放任务",
      "scope": "所有",
      "required_evidence_type": "tool"
    }
  ]
}

Verifier 的第一层是引用和证据存在性检查。如果 claim 带了引用，就检查引用 ID 是否在 trace 中存在，引用文档是否对当前用户可见，引用版本是否有效。没有引用的 claim，则根据 required_evidence_type 在上下文和工具返回里找候选证据。候选选择可以用 embedding 或 BM25，但最终判定要考虑 subject 和 scope，不能只看语义相似。

第二层是支持度判断。我常用四档：supports、partially_supports、contradicts、irrelevant。supports 表示证据足以支持原声明；partially_supports 表示证据支持部分事实但不支持范围或语气；contradicts 表示相反；irrelevant 表示只是关键词相关。无来源对应没有 supports 或 partially_supports 的候选；错来源对应引用存在但是 irrelevant 或 contradicts；过度推断常见于 partially_supports；事实冲突对应强证据 contradicts。

一个具体例子更容易说明。文档写的是：“权限收紧后，使用旧服务账号的回放任务需要重新授权。”模型回答：“权限收紧后，所有回放任务都会失败。”证据确实相关，但不支持“所有”和“都会失败”。这不是无来源，也不完全是错来源，而是过度推断。Verifier 应该给出：

{
  "claim_id": "c2",
  "label": "over_inference",
  "severity": "high",
  "evidence_ids": ["doc:replay-runbook#service-account"],
  "reason": "证据只限定旧服务账号路径，答案扩大为所有回放任务"
}

第三层是强事实源冲突检查。对于订单状态、权限、余额、库存、部署版本、事故状态这类结构化事实，工具和数据库应该比文档更强。评测系统可以配置 source_priority，例如 tool.order_status > doc.help_center > model_prior。当答案声明和高优先级工具返回冲突时，直接标 contradiction，不必让 LLM judge 再猜。

source_priority:
  order_status:
    - tool:order_api
    - db:order_snapshot
    - doc:help_center
  permission:
    - tool:authz_check
    - db:acl_snapshot
    - doc:runbook

Metric aggregator 要同时输出宏观指标和可修复列表。宏观指标用于看趋势，可修复列表用于行动。聚合维度至少包括 release、model、prompt_hash、retriever_version、doc_version、feature、risk_level、claim_type。这样当 contradiction_rate 突然升高时，可以判断是某个模型版本的问题，还是某批文档更新导致的。

阻断规则要保守而明确。我的经验是：高风险样本中的事实冲突和错来源直接阻断；无来源按关键声明阻断，非关键背景信息可以降级；过度推断根据范围扩张程度设阈值；低风险样本允许少量非关键 claim 失败，但必须进入队列。不要让单一总分决定上线，因为平均数会掩盖严重事故。

gates:
  high_risk:
    contradiction_rate: 0
    wrong_citation_rate: 0
    unsupported_key_claim_rate: 0
  normal:
    grounded_answer_rate_min: 0.94
    over_inference_rate_max: 0.03
  regression:
    max_new_blocking_failures: 0
    max_metric_drop: 0.02

线上也要采样，但线上采样和离线评测目标不同。离线评测保护已知关键路径，线上采样发现新问题。线上不要把所有用户内容都送去裁判模型，先做脱敏和风险分层。可以只抽取包含日期、金额、权限、强结论、外部承诺的回答，优先检查这些高风险 claim。对用户隐私敏感的业务，可以只记录 claim 类型、证据 ID 和标签，把原文留在受控日志里。

测试评测

幻觉指标本身需要校准。没有校准的自动评测，容易制造新的幻觉：看板显示质量提高，人工复核却发现裁判变松了。第一步是做一个人工标注的小集合，覆盖无来源、错来源、过度推断、事实冲突和非幻觉失败。每条样本至少两个人标，分歧样本要讨论出标注规则，而不是简单投票。这个集合不一定大，五十到两百条高质量 claim 就能暴露很多口径问题。

第二步是评测 verifier 的准确率。不要只看整体准确率，要看每个标签的 precision 和 recall。错来源通常最难，因为它需要理解引用是否支持声明；过度推断也难，因为范围词很细。可以把 blocking 标签的 precision 放得更高，避免误杀发布；把线上告警标签的 recall 放得更高，宁愿多进入人工队列。不同用途的阈值不必相同。

第三步是做反事实测试。把证据删掉，看系统是否拒答；把证据换成相邻对象，看错来源指标是否升高；把范围词从“部分”改成“全部”，看过度推断是否被抓住；把工具返回改成相反状态，看事实冲突是否阻断。反事实样本比自然样本更尖锐，能测试指标是否真的理解边界。

第四步是把指标和真实事故对齐。每次线上出现幻觉事故，都要问：这次事故在现有指标里会被哪个标签捕获？如果捕获不了，是 trace 缺失、claim 抽取失败、证据判定失败，还是阻断规则太宽？事故复盘最后应该产出一个新的评测样本或一条新的判定规则。否则复盘只是写文档，不能变成系统免疫力。

还要测试成本和延迟。claim 级评测可能很贵，尤其是每条声明都调用 judge。可以做分层策略：确定性规则先筛；只有高风险或不确定 claim 调用模型；低风险样本夜间批处理；CI 只跑阻断集合。评测系统本身如果太慢，工程师会绕开它。质量基础设施要可靠，也要便宜。

人工复核界面也算测试的一部分。复核人员需要看到用户问题、答案、claim、引用、证据片段、标签、判定理由和一键改标签。不要只给一段 answer 和一个分数。好的复核界面能让标注规则持续进化，坏的复核界面会让大家在聊天软件里争论，最后没有结构化数据沉淀。

失败模式

第一种失败模式是把引用存在当成 grounded。很多产品界面只要答案末尾有几个链接，就让用户觉得可信，但链接可能不支持声明。工程上必须检查 claim 到 citation 的支持关系，而不是 answer 到 document 的粗粒度相关性。

第二种失败模式是裁判模型被答案说服。模型写得越流畅，judge 越容易给高分，尤其当证据很长、冲突很细时。解决办法是先缩小证据候选，把强事实源结构化展示给 judge，并要求输出 unsupported_claims 和 contradicting_evidence，而不是只给分。

第三种失败模式是把用户问题里的假设当事实。用户问“既然订单已经退款，为什么钱没到账”，工具其实显示未退款。系统如果顺着问题回答“到账延迟”，就是事实冲突。评测时要把 user_claim 和 verified_fact 分开，不能把对话上下文全部当可信证据。

第四种失败模式是知识版本错位。旧政策、草案、迁移前 runbook、过期 FAQ 都可能被召回。错来源指标要记录 doc_version 和 valid_time。对于时间敏感场景，样本也要有 as_of 时间，否则很难判断答案按哪个时间点才算正确。

第五种失败模式是过度拒答。为了降低幻觉，系统可能对很多可回答问题都说“证据不足”。这在看板上会让幻觉率下降，但用户任务完成度也下降。因此幻觉指标必须和 refusal precision、refusal recall、answer coverage 放在一起看。安全不是沉默，安全是该说的时候有证据地说。

第六种失败模式是只在离线评测里表现好。团队可能针对 Golden Dataset 调 prompt，已知样本通过了，线上新问题仍然乱答。解决办法是保留线上抽样、变体样本和事故回流机制，并对样本做版本化，不要让评测集变成背题集。

第七种失败模式是 trace 不完整。没有工具返回、没有引用映射、没有上下文快照，就无法判断 claim 为什么失败。上线前如果不能完整记录 trace，幻觉评测只能停留在表面。很多团队到事故发生后才发现日志里只有最终答案，这时已经很难复盘。

第八种失败模式是所有 claim 权重一样。答案里一个无关背景句无来源，和退款金额说错，不应该同等严重。评测需要 severity 和 risk_level。阻断规则关注关键声明，趋势指标再覆盖全部声明。这样既不会放过严重问题，也不会被低价值噪声拖垮。

上线 checklist

  • [ ] 每个生产回答都有 answer_id、trace_id、model、prompt_hash、retriever_version 和 doc_version。
  • [ ] Trace 中保存进入上下文的 chunk、工具调用返回、引用 ID 映射和用户权限范围。
  • [ ] Claim 抽取输出结构化字段，至少包含 text、type、subject、scope 和原文位置。
  • [ ] Verifier 能区分 unsupported、wrong_source、over_inference、contradiction，不只输出一个总分。
  • [ ] 强事实源优先级已经配置，订单、权限、金额、状态等字段不会被文档或模型覆盖。
  • [ ] 高风险样本的事实冲突、错来源、关键无来源声明全部设为发布阻断。
  • [ ] LLM judge 有固定校准集，并记录 judge 版本、提示词版本和判定理由。
  • [ ] 线上采样经过脱敏和权限控制，复核界面能回看证据但不扩大可见范围。
  • [ ] 每次幻觉事故都会回流为样本、规则或文档治理任务。
  • [ ] 看板同时展示幻觉指标、拒答指标、任务完成度、成本和延迟，避免单指标优化。

总结

幻觉不是一个可以靠单一百分比解决的问题。生产系统要把它拆成声明、证据、引用、冲突和范围边界，再用无来源、错来源、过度推断、事实冲突四类标签承接工程修复。这样定义以后，指标不再只是质量汇报，而是能直接告诉团队应该修检索、修文档、修工具、修 prompt 还是修拒答策略。

真正有用的幻觉指标必须贴着系统 trace 走。没有 trace，就没有可复盘的来源；没有 claim，就没有可定位的错误；没有强事实源优先级，就会被旧文档和流畅表达带偏；没有人工校准，自动裁判会慢慢漂。把这些基础设施补齐，模型升级、知识库更新和产品策略变化才不会把质量控制变成猜测。

最后要记住，降低幻觉不是让系统少回答，而是让系统在证据允许的地方坚定回答，在证据不足的地方清楚说明边界，在证据冲突的地方服从强事实源。指标如果能推动这种行为，它才算定义对了。
