用 GitHub Pages 做个人知识库

静态站适合沉淀长期内容，也适合被 AI 工具读取和引用。

GitHub PagesKnowledge Base

问题背景

个人知识库最常见的问题，不是没有地方写，而是写下来的东西很难长期使用。很多工具在刚开始时很顺手：打开快、格式好看、支持标签、可以双链、还能同步。可是一旦内容积累几年，真正的问题会浮出来：文章能不能被搜索引擎索引，能不能被 Git 管理，能不能在本地批量处理，能不能被 AI 工具读取，能不能在十年后迁移，能不能把公开内容和私密草稿分开。知识库如果只满足“当下写得舒服”，往往会在长期维护上付出代价。

我越来越倾向于把公开或准公开的长期知识，放到 GitHub Pages 这类静态站里。原因很简单：Markdown 是朴素格式，Git 是可靠历史，静态 HTML 易于发布，URL 是长期引用，仓库结构能被本地工具和 AI Agent 直接读取。它不是最花哨的知识管理方案，却很适合工程师的工作方式。你可以用编辑器写，可以用脚本检查，可以用 Pull Request 审阅，可以用站点构建流程发布，也可以让 AI 在本地仓库里读规范、改文章、跑校验。

用 GitHub Pages 做知识库，不等于把所有笔记都公开。更合理的分层是：私密原始材料放在本地或私有仓库，经过整理、脱敏和验证的长期内容发布到静态站；临时想法先在草稿区，成熟后再进入公开内容；项目上下文、架构决策、常见问题和教程文章可以变成稳定 URL。这样，知识库既是个人外部表达，也是给未来自己和 AI 工具使用的上下文接口。

很多人对静态站的误解是“只能展示，不能管理”。其实对工程师来说，静态站的管理能力来自仓库。front matter 是结构化元数据，目录是信息架构，构建脚本是质量门禁，Git history 是变更记录，Issue 和 PR 是工作流，CI 是发布前检查。你不一定需要一个复杂后台。只要把内容源文件组织好，静态站就能承载相当严肃的知识工程。

当然，GitHub Pages 也不是银弹。它不适合保存高度私密的原始资料，不适合多人实时协作编辑，不适合频繁变动的事务数据，也不适合需要复杂权限的企业知识库。它适合的是另一类内容：稳定、可引用、需要长期维护、对结构和质量有要求、希望被浏览器和 AI 工具都能读取的知识。个人技术文章、工程手册、学习笔记、项目复盘、工具清单、公开课程资料，都属于这一类。

这篇文章讨论的重点，不是如何点击按钮开通 Pages，而是如何把 GitHub Pages 当成一个可维护的个人知识库系统来设计。我们会看内容模型、目录结构、构建流程、元数据、AI 读取方式、测试评测、失败模式和上线检查项。一个好的知识库不是靠主题模板堆出来的，而是靠长期稳定的内容协议和工程习惯养出来的。

核心概念

第一个核心概念是“源文件优先”。知识库的真正资产不是生成后的 HTML，而是 Markdown、图片、元数据、脚本和构建配置。HTML 可以随时重新生成，主题可以更换，部署平台也可以迁移；源文件如果混乱，后面所有展示层都会受影响。因此，设计知识库时应该先设计内容源文件，而不是先挑页面效果。

第二个概念是“稳定 URL”。知识库内容一旦被别人引用、被搜索引擎收录、被 AI 工具作为来源读取，URL 就变成了契约。随意改路径，会让旧链接失效，也会让长期知识失去可引用性。我的做法是给每篇文章一个稳定 slug，URL 和 slug 一一对应，标题可以改，分类可以调整，但 slug 尽量不变。如果确实需要迁移，要保留重定向或旧路径说明。

第三个概念是“front matter 即内容数据库”。很多静态站把 front matter 当成几个展示字段：标题、日期、标签。但如果把知识库当系统，front matter 就是最轻量的内容数据库。它可以记录 slug、URL、摘要、分类、来源、日期、封面图、标签、状态、作者、引用材料、是否推荐。只要字段稳定，就可以生成列表页、搜索索引、RSS、站点地图，也可以让 AI 工具按元数据筛选内容。

第四个概念是“构建时质量门禁”。静态站没有数据库约束，所以更需要构建时检查。重复 slug、缺失图片、front matter 字段不一致、中文字数不足、标题层级跳跃、站内链接失效、标签拼写不统一，都应该在发布前被发现。对个人知识库来说，质量门禁不必复杂，但必须自动化。靠人工记忆维护几十篇文章还可以，维护几百篇文章一定会漏。

第五个概念是“面向 AI 的可读性”。过去我们写知识库主要给人看，现在还要考虑 AI 工具读取。AI 读取并不意味着写给机器看，而是要让内容结构清楚、来源明确、标题语义稳定、段落有上下文、代码块和表格不混乱。静态站天然适合这件事，因为 Markdown 源文件可以直接作为模型上下文，HTML 页面也能被浏览器工具提取。相比很多封闭笔记系统，仓库化知识库更容易进入 Agent 工作流。

第六个概念是“公开层和私密层分离”。个人知识库经常混合灵感、账号信息、客户材料、未公开项目、公开文章草稿。把这些全部放进一个公开 Pages 仓库，会带来明显风险。更稳的方式是至少分三层：私密 inbox 保存原始材料，工作草稿保存半成品，公开站点保存经过整理的内容。AI 可以在本地同时读取多个层，但发布流程只能从允许公开的目录取内容。

下面这张表总结了 GitHub Pages 知识库和常见笔记工具的差异：

维度GitHub Pages 知识库本地笔记应用云端协作文档
长期格式Markdown、静态资源、Git取决于应用，可能有私有格式取决于平台导出能力
发布能力天然公开 URL 和静态页面通常需要额外发布共享方便但公开站点弱
版本历史Git 精确记录每次变更应用内历史不一定完整平台内历史，迁移成本高
AI 读取直接读仓库和 HTML需要本地连接或导出需要 API 或复制粘贴
权限模型仓库级公开或私有本机为主细粒度权限较好
适合内容长期文章、手册、公开知识私密草稿、日记、零散收集团队协作材料

这些差异说明，GitHub Pages 不必替代所有笔记工具。它更适合作为“长期知识发布层”。私密收集、临时草稿、团队协作仍然可以用别的工具，但最终沉淀出来的稳定内容，最好进入一个可版本化、可引用、可构建检查的地方。

这里最重要的边界是“源仓库”和“生成站点”的边界。源仓库负责表达知识，生成站点负责展示知识。不要把生成结果当成主要编辑对象，也不要在 HTML 里手工修内容。只要所有内容修改都回到 Markdown 和元数据，知识库就能保持可迁移。

从一条想法变成一篇文章，我通常会让它经过五个阶段：

  1. 收集：把原始链接、摘录、会议记录、代码片段放到私密 inbox。
  2. 整理：删除不能公开的内容，补充自己的判断，形成草稿。
  3. 结构化：写 front matter，确定 slug、标题、摘要、分类、标签和图片。
  4. 发布前检查：跑脚本检查元数据、字数、链接、图片、构建输出。
  5. 维护：根据反馈和新经验更新文章，保留 Git 历史。

AI 工具可以参与每个阶段，但权限不同。在收集阶段，AI 可以帮忙摘要，但不能默认发布。在整理阶段，AI 可以提出结构和缺口。在结构化阶段，AI 可以对照已有元数据生成 front matter。在检查阶段，AI 可以运行脚本和修复格式问题。在维护阶段，AI 可以根据 issue 或评论提出补丁。这个流程让 AI 成为知识库维护者，而不是随意往公开站写内容的黑盒。

对于站点结构，我建议保持简单稳定：

content/
  ARTICLE_STYLE.md
  articles/
    local-tool-stack.md
    github-pages-knowledge-base.md
assets/
  article-visuals/
    local-tool-stack.svg
    github-pages-knowledge-base.svg
js/
  articles.js
scripts/
  check_articles.py

这类结构的好处是，内容、资源、元数据和检查脚本一眼可见。AI Agent 进入仓库后，也容易知道该读哪里。相比把文章散落在多层目录里，稳定路径能降低检索和自动化成本。

这条流程看似传统，但它给知识库带来了工程纪律。每篇文章不是“写完就算”，而是经过元数据检查、内容长度检查、构建检查和 diff 审查。长期看，这些小门槛能显著提升知识库质量。

工程实现

工程实现的第一步，是定义一份稳定的文章数据模型。Markdown front matter 是人写的，构建脚本和站点列表通常还会有一份 JSON 或 JavaScript 元数据。两份数据如果没有约束，迟早会不一致。解决办法有两种：一种是以 Markdown front matter 为唯一来源，构建时生成索引；另一种是保留外部元数据文件，但发布检查必须验证两边一致。两种都可以，关键是不能靠人工记忆同步。

一篇文章的最小模型可以这样表达：

type ArticleMeta = {
  slug: string
  url: string
  title: string
  summary: string
  categoryKey: string
  category: string
  categoryLabel: string
  source: string
  date: string
  image: string
  tags: string[]
}

这组字段满足几个用途。slug 用于文件名和稳定标识，url 用于公开访问，title 和 summary 用于列表页和搜索结果，categoryKey 用于程序筛选，category 和 categoryLabel 用于展示，source 用于追溯内容来源，date 用于排序，image 用于视觉入口，tags 用于主题聚合。字段不多，但足够支撑一个长期知识库。

Markdown 文件可以遵循这样的结构：

---
slug: github-pages-knowledge-base
url: /notes/github-pages-knowledge-base/
title: 用 GitHub Pages 做个人知识库
summary: 静态站适合沉淀长期内容，也适合被 AI 工具读取和引用。
categoryKey: workflow
category: Workflow
categoryLabel: AI 工作流与工具
source: NOTES/WORKFLOW
date: 2026-02-13
image: /assets/article-visuals/github-pages-knowledge-base.svg
tags:
  - GitHub Pages
  - Knowledge Base
---

## 问题背景
...

这个结构对人和机器都友好。人可以快速看懂文章属性，机器可以稳定解析字段。图片引用放在正文开头，既满足页面渲染，也让 Markdown 阅读时保留视觉语义。

目录分类要克制。知识库早期很容易设计十几个分类，结果每个分类只有两三篇文章。分类应该代表读者理解路径，而不是作者脑海里的所有细节。标签可以更细，分类要少而稳。比如一个 AI 工程知识库可以先分为 Agent、RAG、MCP、Workflow、Evaluation、Product。文章多起来后，再通过标签和系列页细分。

搜索索引可以先用静态 JSON。构建时抽取标题、摘要、标签、正文前几段和 URL，生成一个轻量索引。浏览器端可以做简单模糊搜索，本地 AI 工具则可以直接读取 Markdown。不要一开始就上复杂搜索服务。对个人站点来说，静态索引足够覆盖大部分需求，维护成本低，也不增加服务端依赖。

图片资源要有命名协议。每篇文章的标题图放在 /assets/article-visuals/<slug>.svg，正文引用同一路径。这样检查脚本可以根据 slug 验证图片是否存在，站点也能统一生成预览卡片。图片不是装饰细节，它是内容资产的一部分。命名协议稳定后，AI 生成或更新文章时也不容易写错路径。

链接管理是另一个关键实现点。站内链接应该尽量使用最终 URL，而不是相对文件路径，因为文章发布后读者看到的是 URL。站外链接要保留标题和访问日期，重要引用可以在正文里说明来源。对 AI 工具来说，链接周围的上下文很重要。不要只贴一个裸链接，最好说明这个链接支持了哪一个判断。

一个实用的检查脚本应该覆盖以下规则：

检查项目的失败示例修复方式
front matter 必填字段保证站点列表和文章页可渲染缺少 summary补齐字段
slug 与文件名一致保证稳定标识文件名是 a.md，slug 是 b修改文件名或 slug
url 与元数据一致避免死链/note/foo/ 写成 /notes/foo/对照索引修正
image 存在并被正文引用保证视觉资源完整front matter 有图，正文没引用增加 Markdown 图片
中文字数下限避免空壳文章只有提纲补充完整正文
标题层级保证阅读结构## 后直接 ####调整层级
重复 slug避免路由冲突两篇文章同 slug重命名其中一篇

这些检查看起来简单，但能挡住大量低级错误。尤其当 AI 参与写作时，front matter 字段、图片路径、标签拼写、正文长度都容易出错。把规则交给脚本，而不是靠人反复审查，是工程化知识库的基本态度。

为了让 AI 更好地维护知识库，可以在仓库里放一份写作规范，例如 content/ARTICLE_STYLE.md。这份规范要写清楚 front matter 格式、正文长度、必须包含的章节、语气、检查命令和禁止事项。AI 在动手写文章前先读规范，就像开发者改代码前先读 README 和测试。规范越具体，输出越稳定；规范只写“请写高质量文章”，结果通常会飘。

还有一个很有用的做法，是为每篇文章保留“来源轨迹”。source 字段可以指向笔记目录或主题集合，正文中也可以在关键段落说明依据来自项目实践、读书笔记、线上故障或公开资料。来源轨迹不是学术引用那么重，但它能帮助未来更新文章。当你半年后重新打开文章，知道它当初为什么这么写，比只看到一段漂亮文字更重要。

测试评测

GitHub Pages 知识库的测试评测，应该同时覆盖内容质量和工程质量。内容质量回答“这篇文章有没有用”，工程质量回答“这个站点能不能长期维护”。两者缺一不可。只有内容，没有检查，后期会变乱；只有检查，没有深度，站点会变成格式正确的空壳。

工程质量测试可以自动化。第一类是元数据一致性：每个 Markdown 的 front matter 是否完整，是否与索引文件一致，日期格式是否正确，标签是否为数组，图片路径是否存在。第二类是结构完整性：正文是否包含标题图，是否有必要章节，标题层级是否合理。第三类是构建完整性：站点能否生成，生成后的链接是否可访问，静态资源路径是否正确。第四类是内容底线：中文汉字数是否达到要求，是否存在明显重复段落，是否残留占位符。

内容质量测试更适合半自动。可以让 AI 或规则先检查文章是否覆盖问题背景、核心概念、工程实现、测试、失败模式、上线 checklist 和总结，再由人判断论证是否扎实。对长期知识库来说，文章不需要每篇都像论文，但必须有明确读者、有实践经验、有可执行建议。只讲理念、不讲实现的内容，很难在几个月后继续发挥价值。

我会为知识库准备一组固定评测任务：

评测任务输入期望输出自动指标人工关注点
新增文章slug、元数据、写作规范合规 Markdown字数、front matter、图片、章节是否有真实工程细节
更新旧文旧文章、新事实、变更说明最小修改 diff链接、构建、无重复是否保留原文风格
生成索引全部文章元数据分类页、搜索数据无重复 slug、排序正确分类是否符合读者路径
链接巡检站内外链接失效链接报告HTTP 状态、锚点存在是否需要替换引用
AI 问答文章集合、用户问题带来源回答引用命中率是否过度推断

其中“AI 问答”值得单独说。知识库既然要服务 AI 工具，就应该评测 AI 能否从中找到正确材料。可以准备十几个真实问题，例如“本地 AI 工具栈为什么需要事件日志”“GitHub Pages 知识库如何避免元数据不一致”“文章发布前要检查哪些项”。让 AI 只能基于知识库回答，观察它引用了哪些文章、是否漏掉关键段落、是否把旧内容当成新事实。这类评测能反向暴露文章结构问题。如果 AI 总是找不到某个概念，可能是标题不清楚、标签不合理，或者段落没有明确主题句。

视觉评测也不要忽略。静态站文章写得再好，如果移动端标题溢出、代码块横向撑破、表格无法阅读、图片路径错误，读者体验会很差。至少要在桌面和手机宽度预览首页、文章页、分类页和搜索页。对长文来说，目录、段落间距、代码块样式和表格滚动尤其重要。

发布后的评测可以看几个指标：哪些文章被访问，哪些站内搜索没有结果，哪些外部链接带来读者，哪些文章被 AI 工作流频繁读取，哪些旧文章需要更新。个人站不一定要重分析，但这些信号能帮助你决定下一步写什么、修什么。知识库不是一次性项目，它需要一个小而稳定的反馈回路。

失败模式

第一类失败是内容源和展示索引不一致。比如 js/articles.js 里写了一个标题，Markdown front matter 里是另一个标题；列表页显示的日期和文章页不同；标签在两个地方拼写不一致。这类问题会让站点看起来不专业，也会让 AI 读取时混淆。解决办法是明确单一事实来源，或者用检查脚本强制一致。

第二类失败是 URL 频繁变化。作者在整理分类时顺手改了路径，外部引用和搜索结果全部失效。个人站早期常见这种问题，因为内容不多时改路径成本低，久而久之就养成习惯。解决办法是把 slug 当作公开契约。标题、摘要、正文都可以迭代，slug 不要轻易改。确实要改时，保留旧 URL 的跳转。

第三类失败是知识库变成草稿堆。什么都往公开目录放，会议摘录、临时命令、未验证判断、复制来的资料混在一起。短期看内容增长很快，长期看读者和 AI 都难以判断可信度。解决办法是区分 inbox、draft、published。公开文章必须经过整理，有明确结论和来源；草稿可以粗糙，但不要直接进入发布层。

第四类失败是过度追求主题和页面效果。静态站很容易花很多时间调主题、动画、卡片和首页，却忽略内容模型。视觉当然重要，但知识库最核心的资产是可维护内容。主题应该服务阅读和检索，而不是反过来决定写作结构。尤其不要为了展示效果，把关键内容写死在 HTML 或 JavaScript 组件里，导致源文件不可读。

第五类失败是检查脚本缺席。没有脚本时，文章越多，低级错误越多。AI 参与写作后，这个问题更明显，因为模型可能自信地写错路径、漏掉字段、改变标签格式。解决办法是把常见要求变成命令，并在发布前运行。检查脚本不是对人的不信任，而是给人省注意力。

第六类失败是公开边界不清。把私密笔记仓库和公开 Pages 仓库混在一起，或者让 AI 在没有策略的情况下从私密材料生成公开文章，都会带来泄露风险。解决办法是目录和仓库层面分离，并在发布流程中加入敏感词、密钥、客户名和内部链接检查。公开知识库要长期运营，边界必须保守。

第七类失败是文章缺少维护机制。技术内容会过期，工具版本会变化，个人观点也会更新。如果没有更新时间、来源和复审机制，旧文章会继续被搜索和 AI 引用。解决办法是给重要文章加上更新记录，定期检查高访问或高引用内容。维护不是重写全部，而是在关键事实变化时及时修正。

第八类失败是把 AI 当成唯一作者。AI 很适合整理结构、补齐清单、检查一致性，但知识库的价值来自作者的判断。没有实践经验和取舍，只靠 AI 生成长文，文字可能流畅，却没有长期参考价值。解决办法是让 AI 做工程助手：读取规范、生成初稿、跑检查、指出缺口；最终判断、案例和结论仍然来自作者。

上线 checklist

把 GitHub Pages 用作个人知识库前，我会逐项检查：

  • 仓库结构：内容、资源、元数据、脚本、构建配置分区清楚，路径稳定。
  • 文章模型：front matter 字段固定，slug、url、title、summary、date、image、tags 含义明确。
  • URL 策略：slug 一旦公开尽量不改，必要迁移有重定向或旧链接说明。
  • 图片协议：标题图和正文引用路径一致，资源文件存在，命名跟 slug 绑定。
  • 写作规范：仓库内有明确风格文件，说明长度、章节、语气、检查命令和禁止事项。
  • 发布检查：有命令验证元数据、正文长度、图片引用、链接和构建结果。
  • 私密边界：inbox、draft、published 分开，公开目录不保存敏感原始材料。
  • Git 流程：重要文章通过分支或 PR 审查，提交信息能说明修改意图。
  • 本地预览：发布前能在本地打开站点，检查桌面和移动端阅读体验。
  • 搜索索引：标题、摘要、标签和正文片段能进入搜索或 AI 检索流程。
  • 维护机制：高价值文章有更新时间，过期内容能标记或修订。
  • AI 协作：AI 写作前必须读取规范和元数据，写作后必须运行检查，不跳过人工判断。

如果是从零开始，我建议先发布十篇以内的核心文章，验证模型和流程，再扩大规模。不要一开始就迁移几百篇旧笔记。旧笔记通常格式不统一、质量参差、公开边界不清。先用少量高价值内容跑通结构、检查和发布，再批量整理，风险会小很多。

总结

GitHub Pages 做个人知识库的价值，不在于它免费，也不在于它能快速搭一个页面，而在于它把知识变成了可版本化、可检查、可引用、可迁移的工程资产。Markdown 保持内容朴素，Git 保持历史可信，静态站提供长期 URL，脚本提供质量门禁，AI 工具可以直接读取仓库并参与维护。这几个部分组合起来，比单独的漂亮笔记界面更适合长期积累。

一个好的个人知识库，应该同时服务三类读者：现在的自己、未来的自己、以及需要基于这些内容工作的工具。现在的自己需要写作顺手，未来的自己需要来源和历史，AI 工具需要结构化元数据和清晰正文。GitHub Pages 不是唯一方案，但它在这三者之间取得了很实用的平衡。

长期看，知识库的质量取决于更新机制，而不是首次搭建时的兴奋感。稳定 slug、清晰 front matter、自动检查、公开边界、复审节奏，这些看起来朴素的工程习惯，决定了内容能不能跨年份继续有用。把 GitHub Pages 当成一个小型知识系统来维护，它就不只是个人主页，而会成为你和 AI 工具共同使用的长期工作记忆。
