Agent 产品设计的真实入口

用户不需要一个万能机器人，而需要可预期的工作流入口。

ProductAgent

问题背景

很多 Agent 产品一开始都会把入口做成一个聊天框。聊天框很自然，用户可以随便说，模型也可以随便答，看起来最通用。但真正放进日常工作以后，问题会很快暴露：用户不知道该怎么描述任务，不知道系统能做到哪一步，不知道哪些动作会自动执行，也不知道结果错了以后该从哪里修。万能入口降低了启动成本，却把能力边界、上下文选择、权限确认和结果验收都推给了用户。

用户真正需要的通常不是“一个懂所有事情的机器人”，而是“在我当前工作流里，有一个可预期的帮手接住下一步”。写文档的人希望在当前段落旁边改写、补证据、生成摘要；销售希望在客户页面里整理沟通记录、生成跟进计划、更新 CRM 字段；运维希望在告警详情页里归纳影响范围、查看最近变更、生成处置建议；开发者希望在 PR、issue、代码文件和 CI 失败之间切换，而不是每次都重新解释仓库背景。这些场景的共同点是：入口应该贴着工作对象，而不是悬浮在一个空白对话框里。

Agent 产品设计的真实入口，是用户正在处理的对象和已经形成的工作意图。对象可以是一篇文档、一个客户、一条告警、一个工单、一段代码、一个表格、一场会议。意图可以是审核、补全、迁移、诊断、发布、总结、追踪。好的入口会把对象、意图、权限和可用动作一起呈现，让用户知道“这个 Agent 在这里能帮我做什么，做到什么程度，哪些地方需要我确认”。这比让用户从零写 prompt 稳定得多。

聊天式入口不是不能用，而是不应该成为唯一入口。聊天更适合探索、澄清和跨对象协商；工作流入口更适合高频、可重复、有副作用的任务。一个成熟产品往往两者都需要：在对象页给出明确动作，在对话里允许用户调整目标；在执行前展示计划和影响范围，在执行后把产物落回原来的业务系统。这样 Agent 不会变成另一个需要维护的孤岛，而是嵌入用户已经使用的工具链。

这里还有一个信任问题。用户并不会因为系统写着“AI 自动化”就放心把权限交出去。信任来自可预期性：入口明确、范围明确、步骤可见、预览可改、失败可恢复、责任可追踪。产品设计如果只强调模型能力，而不设计这些边界，用户会在低风险任务里试用，在高价值任务前停下。真正能进入生产流程的 Agent，往往不是最会聊天的，而是最清楚自己何时该行动、何时该请用户看一眼的。

核心概念

Agent 产品入口可以用几个概念来拆：工作对象、入口位置、意图模板、能力边界、权限姿态、预览确认、交接方式和反馈闭环。它们决定了用户是否能自然启动任务，也决定了系统是否能稳定完成任务。

概念产品问题工程映射设计要点
工作对象用户现在到底在处理什么document、ticket、customer、repo、alert入口必须能读取对象 ID 和当前状态
入口位置用户在什么时刻需要帮助页面按钮、侧栏、选中文本菜单、自动建议入口越贴近对象，prompt 负担越低
意图模板用户想完成哪类工作summarize、rewrite、diagnose、prepare、publish模板要保留可配置项，不要写死

预览确认用户如何看见影响范围diff、dry-run、draft、checklist副作用动作必须先可视化

反馈闭环如何让产品持续变好thumbs、correction、trace、eval case用户修正要进入评测和规则

第一个核心判断是：入口应该围绕对象，而不是围绕模型。比如在知识库产品里，“帮我做点什么”的入口很弱，“基于当前文档生成发布摘要”“检查这篇文章缺少哪些引用”“把选中段落改成社区公告风格”就清楚得多。对象给了上下文，动作给了目标，用户只需要调整少数参数。模型的自由度变小了，产品的确定性反而提高。

第二个判断是：能力要以工作流表达，而不是以技术能力表达。用户不关心系统是否使用长上下文、向量检索、函数调用或多 Agent；用户关心“能不能帮我把会议纪要变成三条跟进任务，并在发送前让我确认”。产品文案、按钮和状态都应该围绕业务动作设计。技术能力留在系统内部，用来决定上下文组装、工具选择和权限策略。

第三个判断是：副作用必须分级。读取、总结、草稿生成、写入系统、通知他人、执行命令，这些动作的风险完全不同。如果产品把它们都包装成“让 Agent 处理”，用户无法建立稳定预期。更好的方式是定义权限姿态：只读模式可以自动运行；草稿模式可以生成但不写回；确认模式必须展示差异；自动模式只给经过评测和权限授权的低风险流程。

第四个判断是：用户需要能中途介入。Agent 产品不能只有开始和结束两个状态。长任务需要展示当前阶段、已使用资料、下一步动作、阻塞原因和可取消入口。用户发现方向不对时，应该能改目标、收窄范围、删除某个资料来源、要求先停在草稿。中途控制不是“高级功能”，而是信任机制的一部分。

最后，入口设计要和评测设计连在一起。每一个入口其实都是一个可评测任务族。入口越明确，评测样本越容易收集，失败越容易归因。相反，一个万能聊天框会把所有场景混在一起，数据看起来很多，却很难知道到底哪个工作流需要改。

架构/流程图解说明

一个面向产品入口的 Agent 架构，可以把“用户从哪里进来”和“运行时怎么执行”分开。前者属于产品表面，后者属于能力平台，中间通过结构化的工作流请求连接。

这个流程里，入口控制器非常关键。它不是一个简单按钮，而是负责把产品上下文转成结构化请求。比如用户在文档里选中三段文字，点击“改成技术博客风格”，入口控制器应该传入文档 ID、选区范围、当前用户、目标风格、可写权限和返回位置。运行时不需要猜“这段文字从哪里来”，也不需要决定是否能直接覆盖原文，这些都由入口提供。

意图路由负责把入口动作映射到工作流模板。模板不是死板脚本，而是一份可配置协议：需要哪些上下文、允许哪些工具、默认权限姿态、输出形态、确认策略、评测用例。比如“生成客户跟进计划”需要客户资料、最近沟通记录、合同阶段和团队规则；默认输出应该是草稿任务列表，不应该自动发邮件；如果要写入 CRM，就必须展示字段差异。

上下文构建要尽量“少而准”。很多 Agent 产品为了显得聪明，会把用户所有历史资料都塞进去，结果模型被旧信息污染。入口已经提供了对象和意图，系统应该围绕它们取上下文：当前对象状态、用户选区、最近相关活动、组织规则、少量可引用知识。上下文清单要能展示给用户或进入 trace，尤其是在结果有争议时，用户需要知道 Agent 依据了什么。

预览与确认是产品体验的核心。对于文档改写，预览可以是 diff；对于 CRM 更新，预览可以是字段前后对比；对于邮件发送，预览应该包含收件人、正文、附件和发送时间；对于运维动作，预览必须包含命令、目标资源、影响范围和回滚方式。确认不是一个“确定/取消”的弹窗，而是让用户有机会修改 Agent 的产物，并把修改记录变成系统学习材料。

工程实现

工程实现可以从一个入口注册表开始。每个入口声明自己适用的业务对象、可见条件、参数 schema、风险等级和工作流模板。前端根据对象状态渲染入口，后端根据注册表验证请求，运行时只接收结构化任务。

type AgentEntry = {
  id: string;
  label: string;
  objectType: "document" | "customer" | "alert" | "pull_request";
  intent: string;
  visibleWhen: string[];
  parameterSchema: Record<string, unknown>;
  defaultPermission: "read_only" | "draft" | "confirm_required" | "auto_apply";
  workflowTemplate: string;
  outputMode: "inline_diff" | "draft_card" | "task_list" | "timeline_note";
  requiredCapabilities: string[];
};

type AgentRunRequest = {
  entryId: string;
  objectRef: { type: string; id: string; version?: string };
  userRef: { id: string; role: string; tenant: string };
  parameters: Record<string, unknown>;
  returnTarget: { surface: string; anchor?: string };
};

这里的重点是 objectRef 和 returnTarget。前者确保 Agent 处理的是明确对象，不是游离文本；后者确保结果能回到用户原来的工作表面。很多 Agent 原型失败，是因为生成结果停留在聊天窗口里，用户还要手动复制回业务系统。只要这个复制动作发生，高频工作流的价值就会大打折扣，也会引入权限和版本错误。

后端可以把入口请求转成工作流实例：

{
  "run_id": "run_20260418_001",
  "entry_id": "customer.followup.prepare",
  "object_ref": {"type": "customer", "id": "cus_2388", "version": "v17"},
  "intent": "prepare_followup_tasks",
  "permission": "confirm_required",
  "context_policy": {
    "include": ["customer_profile", "last_5_interactions", "open_contracts", "team_playbook"],
    "exclude": ["private_notes_without_permission"],
    "max_tokens": 12000
  },
  "preview_policy": {
    "mode": "task_list_diff",
    "requires_human": true,
    "write_targets": ["crm.tasks", "crm.next_step"]
  }
}

这个实例里没有让模型决定是否读取私人备注，也没有让模型决定是否直接写 CRM。模型可以建议需要更多资料，但权限和预览策略由产品和平台定义。这样做会牺牲一点“自由发挥”，但换来的是稳定、可审计、可解释。

产品界面上，建议把状态分成五类：可启动、准备中、生成中、等待确认、已写回。每个状态要有具体动作，而不是只显示旋转加载。准备中可以展示正在读取哪些资料；生成中可以展示当前阶段；等待确认要展示 diff 和可编辑草稿；已写回要提供回到对象的位置和撤销或修正入口。对于长任务，还要支持暂停、取消和接管。

下面是一个客户跟进场景的完整流程例子：

步骤用户看到的界面系统动作风险控制
进入客户页侧栏出现“准备跟进任务”读取客户 ID、阶段、用户角色只在有读取权限时显示
点击入口参数面板显示时间范围和任务数量构造 AgentRunRequest默认草稿模式
生成草稿展示引用的沟通记录和建议任务检索最近记录，生成任务列表不写入 CRM
用户编辑用户删除一条、改一条截止日期保存人工修正修正进入反馈数据
确认写回展示将创建的任务和字段变化调用 CRM 工具使用幂等键和审计日志
完成返回客户页时间线出现新任务写回结果和 trace 链接支持撤销和备注

这个流程并不复杂，但它把 Agent 从“聊天助手”变成了“客户页里的工作流加速器”。用户不需要描述客户是谁，不需要复制沟通记录，也不需要担心系统偷偷发消息。入口已经把对象、意图、权限和返回位置都固定住了。

对于文档产品，也可以采用类似结构。用户选中一段内容后，入口菜单提供“压缩为摘要”“补充技术背景”“改成发布语气”“检查缺少引用”。每个入口都对应不同上下文策略和输出模式。改写类入口返回 inline diff；引用检查返回问题清单；发布摘要返回草稿块。不要把所有能力塞进一个“AI 帮我”按钮，否则用户每次都要重新解释期望，系统也很难积累稳定评测。

工程上还要处理版本和并发。用户启动 Agent 时，业务对象有一个版本号；写回前必须确认对象没有被别人改过。如果版本变化，系统应该重新生成 diff 或要求用户合并，而不是覆盖。对于外部系统写入，要使用幂等键，避免用户重复点击导致创建多条任务。对于自动模式，要限制在低风险动作上，并且保留可撤销记录。

权限实现不要只做前端隐藏。入口可见性只是第一道门，后端还要基于用户、租户、对象和动作重新校验。Agent Runtime 调用工具时，也要带着工作流实例的权限上下文，不能因为模型请求某个工具就直接执行。最小权限原则在 Agent 产品里尤其重要，因为模型会探索路径，系统必须让探索停在授权边界内。

还有一个容易被忽略的实现细节，是入口配置的生命周期。入口不是上线后永远不变的按钮，它会随着业务规则、工具能力、组织权限和用户习惯变化。配置里最好带版本号、启用范围、负责人和回滚版本。灰度时可以只给一个团队开放，观察接受率和人工修改率；发现模板有问题时，可以把入口降级到草稿模式，而不是直接下线。这样产品团队能以配置方式调整风险，不必每次都发布前端和后端代码。

数据存储上，也建议把入口定义、运行实例、用户修改和写回结果分开。入口定义回答“这个功能应该怎样工作”；运行实例回答“这一次用户让系统做了什么”；用户修改回答“人类在哪些地方不认可模型产物”；写回结果回答“系统最终改变了哪些业务对象”。四类数据分开以后，排障、审计和评测都会清楚很多。否则所有信息混在一条聊天记录里，后面想分析某个入口为什么失败，会非常痛苦。

测试评测

Agent 产品评测要同时看模型质量和产品闭环。一个入口如果生成内容质量不错，但用户需要大量复制、改写、确认和补救，它仍然不是好入口。评测应该覆盖启动成本、上下文准确性、结果可编辑性、写回可靠性和用户接管成本。

评测类型关注点样本来源指标
入口可发现性用户是否知道何时使用可用性测试、点击热区首次启动率、误点率
意图匹配入口动作是否符合用户目标真实任务录屏、访谈参数修改率、放弃率
上下文质量Agent 是否拿到正确资料trace、引用清单、对象版本缺证据率、旧信息命中率
结果质量产物是否可直接进入下一步人工评分、业务结果接受率、编辑距离、返工率
权限安全副作用是否被正确确认审计日志、策略测试越权拦截率、误写率
工作流效率是否减少总操作成本A/B 测试、任务计时完成时间、手动步骤数

我会为每个入口维护一个小型黄金集。比如“客户跟进任务”入口，样本应该覆盖不同客户阶段、沟通记录缺失、合同风险、客户表达模糊、团队规则冲突、无权限读取某些备注等情况。每个样本都要有期望输出形态，而不仅是期望文字。比如应该生成几个任务、是否需要引用来源、哪些字段不能自动写、什么情况下必须要求人工确认。

除了离线评测，还要做产品内观测。每次运行都记录入口 ID、对象类型、权限姿态、上下文清单、预览修改、确认结果、写回目标和用户反馈。特别要关注“用户修改了什么”。如果用户经常删除某类建议，说明意图模板过宽；如果用户经常补充同一类上下文，说明入口没有拿到关键资料；如果用户经常停在确认页不提交，说明风险提示或产物质量不足。

可用性测试也不能省。让真实用户在自己的工作对象上完成任务，观察他们是否理解入口、是否相信预览、是否知道如何撤销、是否能解释 Agent 做了什么。很多问题不会出现在模型评测里，只会出现在界面动作里。比如按钮文案太宽泛，用户以为会生成草稿，结果系统准备写回；或者确认页只展示最终文本，没有展示会更新哪些字段，用户因此不敢点确认。

上线前还应该做“反向评测”：列出入口不应该处理的任务。比如客户页入口不应该处理跨客户统计，文档改写入口不应该读取用户没有权限的私有文档，告警诊断入口不应该直接执行重启命令。拒绝能力是产品能力的一部分。一个总是试图完成所有请求的 Agent，在企业环境里通常不可信。

失败模式

最常见的失败是入口过宽。产品只放一个“问 AI”按钮，用户每次从空白开始输入。短期看实现简单，长期看数据很脏、体验不稳、评测困难。修复方式不是把 prompt 写得更长，而是把高频工作流拆成贴近对象的入口。

失败模式用户感受系统根因改进方向
魔法黑盒不知道 Agent 要做什么，不敢确认缺少步骤、依据和影响范围展示计划、引用、预览和写回目标
能力宣传过大用户要求超出系统边界的任务入口文案空泛，边界未声明用具体动作命名入口
上下文错配结果引用旧资料或无关资料入口没有绑定对象版本和资料策略记录并展示上下文清单
副作用模糊用户以为是草稿，系统实际写入权限姿态没有分级区分只读、草稿、确认、自动
产物孤岛结果停在聊天里，用户手动搬运没有 return target 和写回协议让产物回到原业务对象
无法中断长任务跑偏后只能等待结束运行状态不可控支持暂停、取消、修改目标
反馈浪费用户修了很多，系统没有学习没有收集编辑差异把修正转成评测样本和规则

还有一种失败是“把 Agent 当功能层，而不是产品层”。团队做了能力平台、工具调用、记忆、评测，却没有认真设计入口。结果销售、运营、开发、客服都能打开同一个聊天框，但没有一个岗位觉得它正好贴合自己的工作。Agent 产品不是在现有系统旁边加一个智能面板，而是要把智能能力嵌入业务动作：该出现在选中文本旁边，就不要藏在全局侧栏；该在提交前做检查，就不要等用户事后追问；该生成草稿，就不要直接写入。

另一个问题是“过早自动化”。有些团队为了展示效率，喜欢让 Agent 一键完成写入、发送和发布。低风险场景可以这样做，但高价值流程最好先从草稿和确认开始。产品要通过数据证明某个入口稳定以后，再提升自动化等级。自动化不是开关，而是一条成熟度路径：只读建议、草稿生成、人工确认写回、有限自动写回、异常自动回退。

最后要警惕“入口泛滥”。当每个页面都有十几个 AI 按钮，用户会开始忽略它们。入口设计要基于高频痛点和明确时机，而不是把所有模型能力都暴露出来。一个页面上有两个真正好用的入口，往往比有二十个模糊入口更有效。产品经理应该持续删除低使用、低成功、高误解的入口，把资源集中在能改变工作流的地方。

上线 checklist

  • 每个入口是否绑定明确的工作对象和对象版本。
  • 入口文案是否描述具体工作动作，而不是泛泛写“AI 助手”。
  • 是否为入口定义意图模板、参数 schema、上下文策略和输出形态。
  • 是否区分只读、草稿、确认写回和自动写回四类权限姿态。
  • 副作用动作是否有 diff、dry-run 或字段对比预览。
  • 写回前是否重新校验用户权限、对象版本和幂等键。
  • 结果是否能回到原业务界面，而不是停在聊天窗口。
  • 长任务是否支持暂停、取消、接管和修改目标。
  • 用户是否能看到关键资料来源、引用和影响范围。
  • 是否记录入口 ID、对象类型、上下文清单、用户修改和确认结果。
  • 是否为每个入口准备黄金评测集和拒绝样本。
  • 是否把用户编辑差异转成评测样本、规则或模板改进。
  • 是否对高风险入口设置灰度、审计和回滚流程。
  • 是否定期清理低价值入口，避免按钮泛滥。

总结

Agent 产品设计的关键，不是把聊天框做得更聪明，而是找到用户真实工作流里的入口。入口贴着对象，系统就能拿到可靠上下文；入口表达具体动作，用户就能形成稳定预期；入口绑定权限和预览，副作用就能被治理；入口把产物写回原系统，Agent 才真正节省工作，而不是制造新的搬运任务。

好的 Agent 产品通常是从窄入口长出来的。先选择一个高频、痛、可验证、风险可控的工作流，把对象、意图、上下文、预览、写回和反馈闭环打通。等这个入口稳定以后，再扩展相邻入口，而不是一开始承诺万能。模型能力会继续变化，但产品设计里这些基本问题不会过时：用户在哪里需要帮助，系统能代表用户做什么，哪些动作必须确认，结果如何回到工作现场。把这些回答清楚，Agent 才会从演示走进生产。
