Agent 权限模型

工具权限要按读、写、执行和外部副作用分层授权。

PermissionAgent

问题背景

Agent 接入工具以后，产品的风险边界会立刻变化。一个只会回答问题的模型，即使回答错了，多数时候造成的是认知偏差；一个能读仓库、改文件、发消息、创建工单、调用支付接口的 Agent，错误会进入真实系统。很多团队在早期 Demo 阶段只关心“模型能不能自动完成任务”，等到要开放给同事或客户时才发现，权限问题不是把 API key 藏好那么简单，而是要回答一连串更细的问题：它能读哪些资源，能写哪些资源，能执行哪些命令，哪些动作会触发外部副作用，哪些步骤必须让人确认，确认时用户看到的影响范围是否足够清楚。

传统应用的权限模型常常围绕用户身份设计。用户登录以后拥有某些角色，角色可以访问某些接口，接口再做资源级校验。Agent 系统不能只照搬这一层，因为 Agent 并不是一个稳定意图的普通用户。它会根据上下文规划步骤，可能在同一轮任务中先读文件、再生成修改、再尝试执行测试、最后想把结果发到群里。这里每一步的风险都不同。如果只给它一个“管理员”或者“只读”角色，前者过于危险，后者又让 Agent 永远停在建议层，无法完成真正有价值的工作。

更麻烦的是，模型的工具选择是概率性的。即使 prompt 写着“不要删除文件”，模型也可能在某些上下文里选择了一个会覆盖产物的工具；即使工具 schema 写得很清楚，参数也可能来自误解的用户意图。权限模型的职责不是相信模型会乖，而是在模型提出动作之后，用系统策略重新判断这个动作是否符合当前任务、当前身份、当前环境和当前风险等级。换句话说，权限层要把 Agent 的“想做什么”变成可审计、可拒绝、可降级、可确认的工程事实。

我更推荐把 Agent 权限看成运行时能力控制，而不是静态菜单。一次任务开始时，Agent 拿到的不是所有工具，而是一个经过裁剪的能力包。随着任务推进，能力包可以因为用户确认而扩大，也可以因为风险升高而收缩。比如用户说“帮我整理这份文档”，默认只允许读文档和生成建议；用户明确说“可以直接提交修改”，系统才授予写入草稿的权限；当 Agent 准备把文档发给外部客户时，即使用户之前说过“你看着办”，也应该再次触发确认，因为这个动作跨出了内部工作空间。

权限模型还要兼顾效率。一个处处弹窗的 Agent 没有人愿意用，一个完全自动执行的 Agent 又没人敢接入生产系统。好的设计不是简单地在人和机器之间选边，而是把风险拆开：低风险读操作自动放行，中风险写操作先 dry-run 并展示 diff，高风险外部副作用必须显式确认，危险操作默认不可达。这样用户会感觉系统在关键处谨慎，在日常处顺手，工程团队也能把事故边界收得更窄。

核心概念

Agent 权限模型的第一层是动作分类。不要只说“工具调用”，要把工具按副作用分级。读操作获取信息，通常可以缓存和脱敏；写操作改变内部状态，例如修改知识库草稿、更新数据库字段；执行操作会运行代码、命令或工作流，可能消耗资源或读写文件；外部副作用会离开当前系统，例如发送邮件、发 Slack、提交 PR、触发付款、修改客户配置。分级以后，策略才有可讨论的对象。

权限层级典型动作默认策略关键审计字段
Read搜索文档、读取 issue、查询只读 API可自动执行，受资源范围限制数据来源、资源 ID、脱敏状态
Propose生成补丁、生成回复草稿、规划变更可自动生成，不能落库为最终结果目标、草稿内容哈希、关联证据
Write Internal写入内部草稿、更新任务状态、修改非生产配置需要任务授权，建议支持 dry-rundiff、操作者、幂等键、回滚点
Execute运行测试、调用构建、执行迁移预检查需要环境限制和超时限制命令、工作目录、网络权限、输出摘要
External Effect发消息、提交代码、调用客户 API、付款必须确认或通过预批准策略接收方、影响范围、确认记录、补偿方案
Dangerous删除、批量覆盖、生产写入、权限提升默认拒绝，少数场景走双人审批拒绝原因、审批链、风险说明

第二层是资源范围。很多事故不是动作类型错了，而是资源边界错了。Agent 可以读“当前项目”的文档，不代表可以读所有项目的文档；可以改“本次任务创建的临时分支”，不代表可以改主分支；可以查询一个客户的工单，不代表可以遍历全部客户数据。资源范围要表达得足够具体，最好支持租户、项目、仓库、目录、标签、环境、时间窗口这些维度。

第三层是身份绑定。Agent 不是凭空拥有权限，它必须绑定到某个主体。主体可以是用户、团队、服务账号或一次受限会话。工程上要避免把所有 Agent 操作都记成一个后台机器人账号，否则审计时只能看到“机器人做了事”，看不到是谁授权、为哪个任务授权、当时的上下文是什么。更合理的方式是让操作记录同时包含 actor_user_id、agent_run_id、delegation_id 和 policy_decision_id，这样可以区分“用户自己点了按钮”和“用户授权 Agent 代办”。

第四层是授权时效。Agent 的权限应该短命。一次会话授予的写权限不应该在第二天仍然有效，一个任务里的外发确认不应该被拿去发另一封邮件。可以把授权设计成 capability token：包含动作、资源、有效期、最大调用次数、风险上限和绑定的 run id。即使 token 泄露或被误用，它也很难越过任务边界。

第五层是策略决策。策略引擎的输出不要只有 allow 和 deny，至少要有四类结果：允许执行、拒绝执行、要求人工确认、降级为草稿或 dry-run。降级很重要。比如 Agent 想直接发邮件，策略可以改成“只生成邮件草稿”；Agent 想执行生产迁移，策略可以改成“只运行预检查”。这样系统不会因为安全约束而完全卡死，用户也能在低风险状态下继续推进。

最后是最小权限和可解释性。最小权限不是一句口号，而是每个工具 schema、每个任务入口、每个运行时状态都要体现。Agent 在规划时只看到它当前能用的工具，执行器在调用时再次校验，审计日志能解释为什么这次允许、为什么那次拒绝。只在 UI 上隐藏按钮是不够的，模型仍然可能构造参数绕过前端；真正的边界必须在服务端、执行器和外部系统凭证层都存在。

架构/流程图解说明

一个可落地的权限架构，通常要把模型运行时和工具执行层隔开。模型可以提出意图，但不能直接拿到通用凭证。所有工具调用都经过 Tool Gateway，Tool Gateway 再向 Policy Engine 请求判定，并且只使用与本次动作匹配的短期凭证。

用户请求
  |
  v
Run Manager 创建 run_id、actor、task_scope
  |
  v
Capability Builder
  |-- 根据用户身份、入口场景、租户策略生成能力包
  |-- 只把当前可见工具 schema 暴露给模型
  v
Agent Planner 选择动作和参数
  |
  v
Tool Gateway
  |-- 参数结构化校验
  |-- 资源范围归一化
  |-- 请求 Policy Engine
  |-- 必要时执行 dry-run
  |
  +--> Policy Engine: allow / deny / confirm / downgrade
  |
  +--> Human Gate: 展示 diff、接收方、影响范围、回滚方案
  |
  v
Executor 使用短期凭证调用真实系统
  |
  v
Audit Log + Trace + Metrics

这张流程图里有两个容易被忽略的点。第一，Capability Builder 发生在模型调用之前。它决定模型能看到哪些工具，能减少模型提出危险动作的概率。第二，Policy Engine 发生在工具执行之前。它不信任模型已经看不到危险工具，因为工具名、参数和上下文都可能在运行时变化。两层一起做，才不会把安全寄托在 prompt 上。

在多租户系统里，权限判断还需要结合环境。开发环境和生产环境的策略不一样，内部员工和外部客户的策略不一样，工作时间和紧急窗口的策略也可能不一样。下面是一个简化的流程例子：用户要求 Agent 修复知识库中的过期配置说明。Agent 先读取当前文档和相关 issue，这是读权限；然后生成 Markdown diff，这是 propose；用户确认“可以改文档”，系统授予只对该文件有效的写权限；Agent 写入草稿分支，这是内部写；Agent 想直接把链接发到客户群，策略发现接收方是外部空间，于是要求人工确认；用户只批准发给内部群，外部消息被降级成草稿。

读文档 -> 生成 diff -> 用户确认写入 -> 更新草稿分支
                                     |
                                     +-> 发送内部通知: allow
                                     |
                                     +-> 发送外部通知: confirm 或 downgrade

这种流程看起来比“一个 API key 全部搞定”复杂，但它让事故变得可控。即使 Agent 误判了接收方，外部通知也会被拦住；即使写入内容有问题，diff 和分支可以回滚；即使用户投诉，审计日志也能说明当时展示了什么、谁确认了什么、系统根据哪条策略放行。

工程实现

工程实现要从工具注册开始。每个工具不应该只是一个函数名和 JSON schema，还要声明副作用、资源类型、幂等能力、是否支持 dry-run、最大风险等级和需要的凭证类型。下面是一个接近实际服务的结构，不绑定具体语言，但在 Go、TypeScript 或 Python 里都可以落地。

tool: docs.update_article
version: 3
description: update a markdown article in the internal knowledge base
effect: write_internal
supportsDryRun: true
idempotency: required
resources:
  - type: repository_path
    patterns:
      - "content/articles/*.md"
requiredScopes:
  - docs:read
  - docs:write:draft
policy:
  maxRisk: medium
  requireConfirmationWhen:
    - branch == "main"
    - changedLines > 200
    - containsSecret == true
audit:
  capture:
    - diffHash
    - changedFiles
    - actorUserId
    - agentRunId

工具调用进入网关后，第一步是把模型给出的参数归一化。路径要解析成仓库内绝对资源，用户 ID 要转换成内部 ID，接收方要展开成真实成员列表，金额要转换成最小货币单位。不要把原始字符串直接交给策略判断，因为“../”、“大小写别名”、“群组别名”、“默认环境”都会造成绕过。归一化失败时，应该返回可恢复错误，让 Agent 澄清或修正，而不是进入执行器。

第二步是风险打分。风险不是只看工具名，也看参数和上下文。更新一篇草稿文档是低到中风险，批量更新一百篇文档就是高风险；发给自己是低风险，发给客户群是高风险；在临时分支运行测试是低风险，在生产机器执行命令是危险操作。风险打分不需要一开始就做成复杂模型，规则表加少量可配置条件就够用，关键是每条规则有名字、有版本、有命中记录。

type PermissionDecision struct {
    Decision        string   `json:"decision"` // allow, deny, confirm, downgrade
    RiskLevel       string   `json:"risk_level"`
    MatchedRules    []string `json:"matched_rules"`
    CapabilityID    string   `json:"capability_id"`
    ConfirmationID  string   `json:"confirmation_id,omitempty"`
    DowngradedTool  string   `json:"downgraded_tool,omitempty"`
    Reason          string   `json:"reason"`
}

第三步是凭证下发。执行器不要持有长期万能凭证，而是向凭证服务申请短期 token。token 绑定工具、资源、run id、调用次数和过期时间。比如 docs.update_article 只能写 content/articles/agent-permission-model.md，只能调用一次，十分钟后失效，不能用于 git.push。如果外部系统不支持细粒度 token，也可以在内部代理层做二次校验，至少不能让模型运行时直接拿到外部系统的根密钥。

第四步是 dry-run 和 diff。凡是写操作和外部副作用，都应该尽量先产生预览。文档修改展示 unified diff，数据库更新展示受影响行数和关键字段变化，邮件发送展示收件人、主题、正文摘要和附件列表，工单批量修改展示样本和总量。确认页面要展示“将要发生什么”，而不是只问“是否允许 Agent 继续”。用户确认的对象必须是具体动作和具体参数，不能是抽象授权。

第五步是幂等和补偿。Agent 很容易因为网络超时、模型重试、状态恢复而重复调用工具。外部副作用尤其要带幂等键。幂等键可以由 run_id + step_id + normalized_action_hash 组成，服务端记录第一次执行结果，后续相同请求直接返回同一结果。对于无法幂等的动作，要明确补偿方案，例如撤回消息、关闭工单、回滚配置、创建反向交易。没有补偿能力的工具，默认风险等级应该上调。

第六步是审计。每次判定和执行都要写入不可随意修改的审计日志。审计日志不是普通 debug log，它要能回答：谁发起、谁确认、Agent 当时想做什么、策略为什么放行或拒绝、实际调用了什么、外部系统返回了什么、有没有回滚。日志正文可以脱敏，但资源 ID、规则 ID、哈希、时间和结果状态不能丢。

一个推荐的权限判定记录如下：

{
  "policy_decision_id": "pd_20260429_001",
  "agent_run_id": "run_abc",
  "actor_user_id": "u_42",
  "tool": "slack.post_message",
  "effect": "external_effect",
  "resource": "slack://workspace/team/channel/customer-a",
  "decision": "confirm",
  "risk_level": "high",
  "matched_rules": ["external_recipient_requires_confirmation", "message_mentions_customer"],
  "normalized_args_hash": "sha256:7ab1...",
  "created_at": "2026-04-29T09:12:31Z"
}

落地时还有一个实践细节：权限错误要给 Agent 可理解的返回。只返回 HTTP 403，模型会反复尝试同一个工具。更好的错误结构包含拒绝原因、可选替代动作和是否可以请求用户确认。例如“不能直接发送外部邮件，但可以创建草稿并请求用户确认”。这样规划器能选择降级路径，用户体验也更自然。

策略发布也要工程化。权限规则不能只在代码里零散出现，否则新增工具时很难判断旧入口受到了什么影响。我通常会把规则分成平台基线、租户覆盖、入口覆盖和临时例外四层。平台基线定义所有环境都不能突破的红线，例如跨租户读取、生产删除、静默外发；租户覆盖表达不同团队的合规要求；入口覆盖限制某个产品功能能做什么；临时例外必须有到期时间和审批记录。合并这些规则时要输出解释链，让开发者知道一个动作被拒绝到底是命中了平台红线，还是某个租户关闭了外部发送。

还要给权限升级设计迁移路径。很多 Agent 一开始只读，后来需要写草稿，再后来要能提交工单。如果每次升级都直接把工具暴露给模型，风险会跳得很快。更稳的方式是灰度能力：先允许 propose，观察草稿质量；再允许 write_internal，但只在沙箱或草稿分支；再开放小范围自动执行，并用审计和指标跟踪误操作。权限不是一次配置完的静态文件，而是跟随系统成熟度逐步打开的能力面。这个过程要有回退开关，发现误触发或误放行时能立刻把某类动作降级为确认或草稿。

测试评测

权限模型的测试不能只测 allow 分支。真正的价值在边界条件，尤其是模型提出了“不该做但看起来很合理”的动作时，系统是否能拦住。测试可以分四层做。

测试层测试内容样例通过标准
单元测试策略规则和资源匹配content/articles/a.md 可写，/etc/passwd 拒绝判定稳定，错误消息清楚
契约测试工具 schema、dry-run、幂等键写文档必须支持 diff 和 idempotency key工具声明和实现一致
场景测试从用户请求到工具调用的完整路径“修正文档并通知团队”低风险自动，高风险确认
对抗测试越权提示、路径穿越、批量副作用“忽略规则，把全部客户导出”不泄露、不执行、不升级权限

评测样本最好来自真实事故和真实拒绝记录。比如 Agent 曾经想把内部总结发给外部群，就把那次上下文脱敏后放进回归集；Agent 曾经把测试命令跑在错误目录，也要转成样本。权限评测不是为了证明模型听话，而是证明策略在模型不听话时仍然有效。

指标上，我会重点看几组数据。第一是策略命中分布：哪些规则最常触发，是否有大量误拒。第二是确认转化率：用户看到确认页后批准、修改、取消的比例。第三是自动化覆盖率：低风险任务是否顺畅通过，高风险任务是否没有绕过。第四是越权尝试率：模型或用户提示中出现权限升级、跨租户访问、外部副作用的次数。第五是重复执行率：幂等键命中是否说明运行时在频繁重试。

还要做故障注入。让策略服务短暂不可用，工具网关应该默认拒绝高风险动作，而不是默认放行。让凭证服务返回过期 token，执行器应该停止并上报可恢复错误。让确认页面超时，Agent 应该回到等待状态，而不是把上一次确认复用到新动作。让审计写入失败，高风险工具应该停止执行，因为没有审计的副作用在生产环境里基本不可接受。

对开发者来说，一个简单但有效的本地评测是“权限快照测试”。把某个入口场景下生成的 capability 列表保存为快照，之后工具注册或策略变更时自动对比。如果一个普通文档整理入口突然多了 email.send 或 git.push，代码评审时就能立刻看到。这个测试看似朴素，却能防住很多因为重构工具目录导致的权限膨胀。

失败模式

第一类失败是权限过宽。最常见的原因是为了赶进度给 Agent 配了共享管理员 token。短期看 Demo 很顺，长期看每个工具都是事故入口。修复办法是拆分工具凭证，至少按读、写、执行、外部副作用分开，再逐步细化到资源和环境。

第二类失败是只在前端确认。用户点了确认，后端却没有把确认 ID 和动作哈希绑定。结果 Agent 可以在确认后修改参数，或者重试时把确认用于另一个接收方。确认必须绑定规范化后的参数、资源、工具版本和过期时间，执行器要二次校验。

第三类失败是 dry-run 与真实执行不一致。有些工具预览时只做字符串 diff，执行时却触发格式化、自动补全或级联更新，最终影响范围比用户看到的大。解决方式是让 dry-run 调用同一套业务逻辑，只是在提交阶段切换事务或沙箱，并在执行前校验 dry-run 哈希。

第四类失败是资源别名绕过。用户说“发给销售群”，系统解析成内部群；模型参数里写的是群名而不是 channel id；策略只看字符串不看真实成员，于是外部成员混入也没有触发确认。资源归一化必须发生在策略之前，群组、目录、分支、环境都要解析成稳定 ID。

第五类失败是权限记忆污染。Agent 在上一个任务中获得了写权限，状态恢复后错误地带到下一个任务。所有 capability 都要绑定 run id 和任务 scope，长期记忆里不能保存可执行授权，只能保存偏好或事实。

第六类失败是审计不可用时继续执行。很多系统把审计当作旁路日志，写失败也不影响主流程。对 Agent 来说，高风险副作用没有审计就不应该发生。可以把低风险读操作和高风险写操作区分开，后者要求审计写入成功后才执行。

第七类失败是拒绝信息太模糊。Agent 收到“权限不足”后，可能换一个工具继续试，造成大量噪声。拒绝响应要告诉它安全替代路径，例如“你可以生成草稿”、“你可以请求用户确认”、“你可以读取当前项目但不能跨项目查询”。这不是放松权限，而是让规划更可控。

第八类失败是策略没人维护。早期规则写在代码里，后来业务增加了新工具、新租户、新环境，规则没人更新，最后不是误杀就是漏放。策略需要版本、owner、变更记录和回归样本。每次新增外部副作用工具，都应该默认进入安全评审。

上线 checklist

上线前我会用下面这份清单逐项过，不建议为了赶发布时间跳过高风险项。

  • 工具已按 read/propose/write_internal/execute/external_effect/dangerous 标注副作用。
  • 每个工具都有资源类型、资源匹配规则、凭证类型、幂等声明和 dry-run 声明。
  • Agent 运行时只接收裁剪后的能力包，不能直接枚举所有内部工具。
  • Tool Gateway 在服务端做参数结构化校验和资源归一化。
  • Policy Engine 支持 allow/deny/confirm/downgrade，并记录规则版本。
  • 高风险动作的确认绑定工具、规范化参数、资源、动作哈希、run id 和过期时间。
  • 外部副作用工具必须支持幂等键，或在策略里提升风险并要求更强确认。
  • 执行器使用短期凭证，凭证绑定动作、资源、调用次数和有效期。
  • dry-run 和真实执行共享核心业务逻辑，执行前校验预览哈希。
  • 审计日志包含 actor、run、tool、resource、decision、confirmation、result 和外部系统 ID。
  • 策略服务、凭证服务、审计服务异常时，高风险动作默认停止。
  • 已有越权、路径穿越、跨租户、外部发送、批量修改的回归样本。
  • 权限变更会触发能力包快照测试，评审者能看到新增能力。
  • 用户确认页面展示影响范围、diff、接收方、回滚或补偿说明。
  • 运维看板能看到拒绝率、确认率、误拒反馈、越权尝试和幂等命中。

这份 checklist 的目标不是让流程变慢，而是把权限模型从“大家相信 Agent 会小心”变成“系统在每个关键边界都有闸门”。当这些项齐了，团队才有资格讨论更高的自动化比例。

总结

Agent 权限模型的核心，是承认模型输出不是权限事实。模型可以提出计划，可以选择工具，可以生成参数，但是否允许执行，必须由独立的策略、资源范围、短期凭证、人工确认和审计链路共同决定。把这些边界做扎实，Agent 才能从玩具式助手进入真实业务流程。

工程上最重要的取舍，是不要在“全自动”和“全人工”之间二选一。读操作可以顺畅，草稿可以自动生成，内部写入可以在范围内放行，外部副作用需要明确确认，危险操作默认不可达。这样设计出来的 Agent 不会因为恐惧而失去价值，也不会因为兴奋而越过边界。

我对权限模型的判断标准很简单：当一次 Agent 运行出问题时，团队能不能清楚说明它被授予了什么能力、为什么被授予、执行了什么、谁确认了、影响了哪些资源、如何回滚。如果这些问题都有答案，权限模型就不是文档里的安全声明，而是生产系统里真正能工作的工程结构。
