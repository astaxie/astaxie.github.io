模型升级评测

换模型前先跑回归、成本、延迟和安全样本。

Model UpgradeEvals

问题背景

模型升级最容易被包装成一个简单决定：新模型更强、更便宜、更长上下文、更好的工具调用、更低延迟，那就换。真实生产系统里，换模型从来不是把配置里的 model 字段改一下。模型是 AI 应用执行链路里的核心依赖，它会影响 prompt 解释方式、输出格式、拒答风格、工具调用参数、token 预算、延迟分布、成本曲线、内容安全和用户信任。一个在演示里明显更聪明的模型，放到既有业务流里，可能让某些老样本退化得很隐蔽。

我见过不少升级事故并不是“新模型整体更差”，而是新模型改变了系统的边界。旧模型很机械，严格按 JSON schema 输出；新模型更会解释，但也更喜欢补充一句自然语言，导致解析器失败。旧模型不太主动调用工具，答案虽然保守但稳定；新模型更积极地调用工具，任务完成率上升，第三方接口费用和超时率也一起上升。旧模型遇到不确定信息会说不知道；新模型更自信，少量幻觉进入高风险流程。单看通用榜单或供应商说明，根本看不出这些变化。

模型升级评测的目标不是证明新模型“更好”，而是判断它是否适合当前系统、当前用户和当前风险边界。这个判断必须包含四类证据：质量是否不退化，延迟是否在体验预算内，成本是否在业务预算内，安全与合规是否没有引入新风险。任何一类证据缺失，都不应该直接全量发布。对 AI 产品来说，模型升级更像数据库迁移或支付通道切换，而不是普通依赖小版本升级。

升级评测还要尊重应用的局部性。一个模型在知识问答上更好，不代表在结构化抽取上更好；在中文客服上更好，不代表在英文合同上更好；在短问题上更好，不代表长上下文代理任务更好。团队需要维护按功能组织的评测集，而不是一个混合大分数。混合大分数会把关键业务的退化藏起来。例如一千条闲聊样本涨了三分，五十条支付争议样本掉了十分，总分仍然好看，但业务风险已经不可接受。

更麻烦的是，模型升级经常和 prompt、工具、路由一起变化。为了适配新模型，工程师会顺手改系统提示词、改工具描述、改输出修复逻辑、改上下文压缩策略。这样做有时必要，但会让评测失去归因能力。到底是模型变好了，还是 prompt 改好了？到底是模型退化，还是工具 schema 不兼容？推荐把升级拆成几个受控阶段：先保持业务流程不变，只替换模型跑回归；再在分支上做适配优化；最后比较“旧模型旧配置”“新模型旧配置”“新模型新配置”三组结果。

小团队也应该建立升级评测纪律。不是所有系统都需要庞大的 eval 平台，但至少要有固定样本、固定指标、固定报告和固定放量规则。最差的方式是靠几个人手工问十几个问题，觉得不错就上线。手工体验可以帮助发现直观差异，但不能覆盖边界样本、历史事故样本和成本延迟。模型能力越强，越容易在普通样本上表现很好；真正决定生产稳定性的，往往是那些少量、难看、业务语义复杂的样本。

核心概念

第一个概念是候选模型。候选模型不只是供应商模型 ID，还包括推理参数、上下文窗口、工具调用模式、结构化输出能力、价格表、区域、限流策略和安全策略。升级评测必须把这些一起版本化。model-x-latest 这种不稳定名称不适合作为生产版本，评测报告里也不能只写“新模型”。要记录可复现的模型标识和调用参数，否则后续无法解释差异。

第二个概念是基准版本。评测不是绝对考试，而是对比。基准版本通常是当前线上模型加当前线上 prompt、工具和后处理。候选版本至少要和基准版本跑同一批样本，使用同一套裁判和同一套统计口径。对于已经发生过事故的场景，还要把事故样本放进固定回归集。每次升级都必须证明这些样本没有再次失败。

第三个概念是评测切片。样本要按功能、语言、风险等级、输入长度、工具路径、租户类型和历史失败原因切片。切片不是为了报表好看，而是为了设置不同门槛。低风险 FAQ 可以允许轻微语气变化，高风险合规回答不能允许事实退化；普通文本生成可以接受输出长度波动，结构化抽取必须保证 schema 通过；离线总结可以慢一点，编辑器补全不能慢。

第四个概念是门禁。门禁是发布前必须满足的条件，而不是发布后再看的建议。门禁可以包含硬性阻断和软性审批。比如安全样本出现高危失败必须阻断；成本上涨超过 30% 需要产品和业务确认；延迟 P95 变差但仍在预算内可以放小流量观察。门禁要写在发布流程里，不能只存在评测文档里。

第五个概念是影子流量。离线样本再完善，也无法覆盖真实用户输入的分布。影子流量让新模型在不影响用户的情况下接收一部分真实请求，产出候选答案和指标。影子结果不返回给用户，但会进入评测和对比。影子流量能发现离线样本没有覆盖的问题，比如某个租户的文档特别长、某个渠道输入特别口语化、某个工具在真实参数下失败。

第六个概念是金丝雀放量。影子流量通过后，也不应该立刻全量。金丝雀放量把少量真实用户切到新模型，并设置自动回滚阈值。离线评测回答“能不能试”，影子流量回答“真实输入下有没有明显风险”，金丝雀回答“真实用户体验是否稳定”。这三个阶段缺一不可，尤其是有写操作、外部工具或高价值客户的系统。

评测维度典型指标最低要求备注
任务质量任务成功率、事实正确率、引用正确率、结构化通过率不低于线上基准，关键切片不得退化不同功能使用不同口径
成本单请求成本、单任务成本、重试成本、工具成本在预算内，超出需审批关注输出变长和工具变多
延迟首字 P95、完成 P95、工具等待时间满足体验 SLO新模型排队策略也要测
安全越权回答、敏感输出、错误拒答、策略绕过高危样本零新增安全裁判要人工抽检
兼容性JSON 解析、函数调用、引用格式、停止词核心格式必须稳定旧解析器是常见事故点
稳定性多次采样方差、重试率、限流率关键场景波动可控温度和种子策略要固定

模型升级评测不是一次跑分，而是一套发布机制。它要求团队提前定义“什么算更好”“什么算不可接受”“如果一项变好一项变坏怎么决策”。没有这些定义，评测报告会沦为挑选对自己有利的数据。

架构/流程图解说明

一个完整的模型升级流程可以分为六段：候选登记、离线回归、适配实验、影子流量、金丝雀放量、全量与回滚观察。每一段都产出证据，并且只有上一段通过才进入下一段。流程不一定要重，但边界必须清楚。

这套流程里有两个不要省略的对照。第一是“新模型旧配置”，它能告诉你模型本身和旧系统是否兼容。第二是“新模型新配置”，它能告诉你经过适配后能达到什么效果。如果只测新模型新配置，升级收益和工程改动混在一起，出了问题也不知道该回滚模型还是回滚 prompt。如果只测新模型旧配置，又可能低估新模型，因为它没有得到适合自己的提示词和工具描述。

数据流上，评测系统需要能复用线上执行链路。不要在评测脚本里重新拼一套简化 prompt，也不要绕过真实检索和后处理。模型升级要验证的是生产路径，不是一个隔离 demo。可以为评测增加开关，比如固定时间、冻结检索版本、禁用写操作工具、使用沙箱 API，但核心流程应该尽量一致。否则离线通过，线上失败，是必然的。

影子流量要特别注意副作用。读操作可以复制请求，写操作不能直接让候选模型执行真实写入。对于会发邮件、改数据库、下单、创建工单的代理任务，影子流量必须使用沙箱工具或 dry run 工具。候选模型可以生成工具调用计划，但执行器只记录参数，不真正执行。评测的目标是比较计划质量和风险，而不是让用户收到两封邮件。

金丝雀阶段要把模型路由做成可回滚配置，而不是发版才能切换。理想情况下，路由系统可以按租户、功能、风险等级和随机桶分流，并能在告警触发时自动把候选权重降为零。模型升级事故通常需要分钟级处理，等重新发版太慢。路由配置也要版本化，方便复盘“哪些用户在什么时间段受影响”。

工程实现

实现上可以从评测样本格式开始。样本不要只存输入和期望答案，还要存业务上下文、切片标签、评测方法和门禁权重。下面是一个 JSONL 样本例子：

{
  "case_id": "kb-refund-0231",
  "feature": "knowledge_qa",
  "risk_level": "medium",
  "locale": "zh-CN",
  "input": "企业版退款周期是多久？如果已经开过发票怎么办？",
  "context_fixture": {
    "knowledge_version": "kb_2026_02_28",
    "tenant_plan": "enterprise"
  },
  "expected": {
    "must_cite": ["refund-policy#enterprise", "invoice-policy#void"],
    "must_not_claim": ["当天到账"],
    "answer_points": ["退款审核周期", "原路退回", "发票作废或红冲"]
  },
  "slices": ["billing", "policy", "citation_required"],
  "gates": {
    "citation_valid": "block",
    "factual_score_min": 0.85,
    "latency_p95_ms": 9000
  }
}

这样的样本比“问题加标准答案”更适合生产系统。context_fixture 可以固定知识库版本和租户条件，避免评测时被实时数据污染。must_cite 和 must_not_claim 让规则校验先抓硬错误，裁判模型再处理语义差异。slices 用于聚合切片，gates 用于发布门禁。样本越接近业务语义，评测越能指导升级。

评测运行器的核心数据结构可以这样设计：

type EvalCase struct {
	CaseID         string
	Feature        string
	RiskLevel      string
	Locale         string
	Input          string
	ContextFixture map[string]string
	Expected       ExpectedSpec
	Slices         []string
	Gates          GateSpec
}

type RunResult struct {
	CaseID        string
	Candidate     string
	Output        string
	TraceID       string
	Usage         Usage
	LatencyMs     int
	FirstTokenMs  int
	SchemaValid   bool
	ToolCalls     []ToolCall
	RuleFindings  []Finding
	JudgeScores   map[string]float64
	Error          string
}

type Comparison struct {
	CaseID          string
	BaselinePass    bool
	CandidatePass   bool
	Regression      bool
	Improvement     bool
	BlockingReasons []string
	CostDeltaPct     float64
	LatencyDeltaPct  float64
}

运行器不要只输出总分，要输出每条样本的对比结果。升级评审最关心的是回归样本：旧版本过了，新版本没过。其次是阻断原因：事实错误、引用错误、结构化失败、安全失败、成本超限、延迟超限。最后才是总体提升。报告首页可以简洁，但必须能下钻到原始输出、裁判解释、trace 和 usage。

评测执行计划要支持矩阵。最少三列：基准、候选原始、候选适配。复杂系统还要按温度、工具策略和上下文长度做小矩阵。但矩阵不能无限膨胀，否则评测周期太长。我的经验是先固定业务关键参数，只比较真正要发布的组合；对有争议的参数，再开小规模实验。评测不是研究论文，目标是安全发布。

runs:
  - name: baseline
    model: current-prod
    prompt_profile: prod
    tool_profile: prod
  - name: candidate_raw
    model: model-2026-03
    prompt_profile: prod
    tool_profile: prod
  - name: candidate_adapted
    model: model-2026-03
    prompt_profile: upgrade-2026-03
    tool_profile: upgrade-2026-03

成本和延迟要在评测运行器里一等对待。不要等质量通过后才查账单。每次 run 都要记录输入 token、输出 token、工具调用次数、重试次数、首字延迟、完成延迟和排队错误。新模型常见的隐性成本是输出更长、工具调用更多、结构化修复减少或增加、上下文窗口变大后团队不自觉塞更多材料。评测报告应该展示“通过样本的成本”和“失败样本的成本”。失败样本也花钱，而且在生产里可能触发重试和人工处理。

安全评测要混合规则、裁判和人工。规则适合检查越权字段、敏感词、禁止工具、schema 限制；裁判适合判断拒答是否合理、回答是否泄露推理过程、是否把不确定信息说成事实；人工适合高风险边界样本。不要把所有安全判断交给同一个新模型或同一供应商裁判。升级模型可能带来安全策略变化，裁判也可能有相同偏差。

发布门禁最好机器可读。评测报告可以给人看，但 CI 或发布系统需要明确结论。比如：

release_gate:
  required:
    - no_blocking_safety_regression
    - critical_slices_success_rate_delta >= 0
    - structured_output_valid_rate >= 0.995
    - p95_latency_ms <= 9000
    - avg_cost_delta_pct <= 25
  approvals:
    - owner: product
      when: avg_cost_delta_pct > 10
    - owner: security
      when: any_safety_warning

这样发布讨论会更具体。不是“我觉得新模型不错”，而是“高风险样本无阻断，结构化通过率从 99.6% 到 99.7%，知识问答切片提升 2.1%，合同审查长文档切片下降 1.4% 但仍在阈值内，平均成本上涨 12%，需要产品确认”。工程团队需要这种可审计的证据链。

测试评测

第一类评测是离线固定回归。它覆盖核心业务样本、历史事故样本、格式边界样本、安全边界样本和高成本样本。固定回归的价值是可比较。每次模型升级都跑同一批样本，团队才能看到趋势。样本集要有版本，新增样本不应该悄悄改变历史报告。可以保留“核心集”和“扩展集”：核心集小而快，放在每次发布门禁；扩展集大而慢，放在夜间或重大升级前。

第二类评测是变形测试。很多 AI 输出没有唯一标准答案，适合用变形关系检查稳定性。比如同一个问题换一种说法，答案要保持关键事实；同一个文档段落打乱无关句子，抽取结果不应变化；同一个用户权限降低，答案应该隐藏受限内容；同一个 JSON schema 字段顺序变化，模型仍应输出合法结构。变形测试能发现模型对表述、顺序和权限提示的脆弱性。

第三类评测是工具调用回放。代理和工具型应用升级模型时，不能只看最终文字。要比较工具调用序列、参数、失败恢复和是否调用了禁止工具。新模型更会使用工具是一把双刃剑。它可能提高完成率，也可能把原本可直接回答的问题变成昂贵调用。工具评测应该区分“必须调用”“可选调用”“禁止调用”。对于写操作工具，评测使用 dry run，记录意图和参数。

第四类评测是结构化输出兼容性。很多生产链路依赖 JSON、XML、函数参数或固定 Markdown 片段。新模型如果多输出一句解释，用户看起来没问题，系统会直接失败。结构化评测要覆盖缺字段、类型错误、枚举值错误、额外字段、转义问题、长文本截断和多语言字段。不要只用解析成功率，还要检查语义字段是否正确。

第五类评测是延迟和限流。离线质量评测通常串行跑，无法代表真实并发。候选模型要在接近生产并发的条件下测首字、完成、排队、超时和重试。供应商对新模型的限流策略可能不同，低并发时很快，高并发时排队严重。金丝雀阶段尤其要监控 P95 和 P99，而不是只看平均。对流式产品，还要测首字和中间 token 间隔，用户体感不只由完成时间决定。

第六类评测是成本回归。每条样本记录实际 usage 后，按功能和切片聚合成本。成本上涨不一定阻断，但必须被解释。比如新模型在合同审查上成本涨 18%，但人工修改率下降 40%，可能值得；在低价值 FAQ 上成本涨 18%，质量没有提升，就不值得。成本评测要结合任务价值，不要机械追求最低价。

第七类评测是人工抽检。自动评测再完整，也要有人看。人工抽检不需要覆盖所有样本，而要覆盖阻断样本、裁判分歧样本、关键客户样本和新模型明显改变风格的样本。抽检表单要结构化，至少包含事实、引用、格式、语气、风险和是否可发布。人工结论也要进入报告，而不是留在会议纪要里。

一个模型升级报告可以包含以下结构：

报告区块需要回答的问题
版本信息候选模型、参数、prompt、工具、知识库、评测集版本是什么
总体结论是否建议进入影子、金丝雀或全量，是否需要审批
关键切片哪些功能提升，哪些功能退化，样本量是否足够
阻断样本旧版本通过新版本失败的案例和原因
成本延迟平均和 P95 变化，是否超过预算
安全兼容安全样本、结构化输出、工具调用是否有新增风险
人工意见抽检结论和未解决分歧

评测不应该只在升级当天运行。模型供应商可能在同名模型背后调整权重或服务策略，外部工具和知识库也会变化。对于关键系统，建议把核心评测集做成定时任务，线上版本每天或每周跑一次小回归。这样即使不主动升级，也能发现依赖变化带来的质量漂移。

失败模式

第一个失败模式是只看供应商基准或公开榜单。通用能力提升不等于你的业务提升。公开榜单不会覆盖你的 prompt、工具、权限、格式和客户语言。模型升级必须跑自己的样本，尤其是历史事故样本。越是业务化的应用，越不能把外部跑分当发布依据。

第二个失败模式是把 prompt 适配和模型替换混在一起。新模型需要适配很正常，但评测必须保留归因。否则上线后出问题，团队不知道是模型不兼容、prompt 新 bug、工具描述误导，还是后处理规则过时。至少保留三组对照，并在报告里明确每组差异。

第三个失败模式是评测集过于干净。真实用户会输入错别字、半句话、截图 OCR 结果、复制来的表格、前后矛盾的需求和权限不足的请求。如果评测集都是文档里整理好的标准问题，升级一定显得很顺利。生产样本挖掘要持续补充脏数据、长尾数据和失败数据。

第四个失败模式是忽略输出风格变化。新模型可能更礼貌、更详细、更自信，也可能更喜欢给建议。风格变化在客服、医疗、法律、金融和开发工具里会产生实质影响。答案多解释几句可能让用户更满意，也可能让界面溢出、成本增加、合规风险上升。风格要作为评测项，而不是只看事实对错。

第五个失败模式是结构化输出只测 happy path。很多系统的解析器对多余文本、尾随逗号、空数组、长字符串和 Unicode 转义很敏感。新模型在正常样本上 JSON 很好，在异常样本上可能加解释、道歉或 Markdown。结构化输出要用严格解析器和业务校验双重测试，不能只看肉眼。

第六个失败模式是影子流量产生副作用。候选模型如果调用真实写工具，可能造成重复发送、重复创建、重复扣费。所有写操作必须 dry run 或沙箱化。影子流量还要注意隐私和数据授权，真实用户输入复制给候选供应商前，要确认合同和数据策略允许。

第七个失败模式是成本评测漏掉重试。新模型单次调用可能更便宜，但如果结构化失败率高、工具超时多、裁判低分触发升级，最终单任务成本会更高。成本要按完整任务算，不只按主模型调用算。失败路径的成本尤其要记录。

第八个失败模式是金丝雀没有自动回滚。团队说先放 5%，但没有实时指标和回滚配置。等客户反馈到群里，问题已经扩大。金丝雀必须有明确阈值：任务成功率下降多少、P95 延迟超过多少、结构化失败率超过多少、安全样本触发多少，就自动切回旧路由并通知负责人。

第九个失败模式是样本泄漏和过拟合。为了让新模型通过评测，团队不断根据失败样本改 prompt，最后评测集分数很好，真实流量提升有限。固定回归集要保护，适配可以参考失败类型，但不要把样本答案硬编码进 prompt。保留一部分盲测样本，对重大升级很有价值。

第十个失败模式是没有回滚窗口。全量发布不是结束。新模型可能在高峰流量、特定客户、月底业务周期或新文档进入后才暴露问题。发布后至少保留一段观察窗口，旧模型路由和相关 prompt 不要立刻删除，评测报告和样本也要归档。不能回滚的升级，本质上不是安全升级。

上线 checklist

  • 记录候选模型的稳定标识、参数、价格、区域、上下文窗口、工具模式、限流和安全策略。
  • 准备当前线上基准版本，包括模型、prompt、工具描述、后处理、知识库和路由配置。
  • 将评测样本按功能、风险、语言、输入长度、工具路径和历史失败原因切片。
  • 固定核心回归集，覆盖历史事故、结构化输出、安全边界、高成本和高价值客户场景。
  • 分别运行基准版本、新模型旧配置、新模型适配配置，保留三组对照。
  • 对每条样本记录输出、trace、usage、首字延迟、完成延迟、工具调用、规则校验和裁判结果。
  • 发布门禁机器可读，明确哪些指标阻断、哪些指标需要产品或安全审批。
  • 成本评测按完整任务计算，包含主模型、工具、重试、修复和裁判成本。
  • 结构化输出使用生产解析器和业务校验，不只看模型文本是否像 JSON。
  • 工具调用评测区分必须调用、可选调用和禁止调用，写操作使用 dry run 或沙箱。
  • 安全评测混合规则、裁判和人工抽检，高危新增失败必须阻断。
  • 影子流量不影响用户结果，且确认数据授权、脱敏和供应商策略。
  • 金丝雀路由支持按功能、租户、风险和随机桶放量，并能分钟级回滚。
  • 金丝雀期间实时监控任务成功、P95 延迟、成本、结构化失败、安全事件和人工接管。
  • 全量后保留旧模型和旧配置一段回滚窗口，归档评测报告和失败样本。

总结

模型升级评测的重点不是追逐最新模型，而是保护生产系统的行为边界。新模型可能带来更强推理、更好工具调用、更低价格或更长上下文，但这些收益只有在你的业务样本、你的延迟预算、你的成本约束和你的安全规则下成立，才算真正可用。升级前跑回归、成本、延迟和安全样本，是工程纪律，不是流程负担。

一套务实的升级机制应该从小处开始：稳定样本格式，复用生产链路，记录对照结果，输出可执行门禁。随后再引入影子流量、金丝雀路由、自动回滚和定时回归。每一步都要回答一个具体问题：是否兼容，是否更好，是否可承受，是否能回滚。只要这些问题被清楚回答，模型升级就不会变成凭感觉下注。

AI 应用会持续换模型，这是未来几年的常态。团队真正需要积累的不是某一次升级经验，而是可重复的升级能力。把模型当成可评测、可路由、可回滚的生产依赖，才能在模型快速演进时既吃到能力红利，又不把用户流程交给运气。
