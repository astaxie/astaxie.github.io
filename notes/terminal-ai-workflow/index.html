终端里的 AI 工作流

CLI 适合把 AI 能力变成可复现、可组合的工程动作。

CLIWorkflow

问题背景

过去几年，很多开发者第一次接触 AI 编程助手，是从聊天窗口开始的。聊天窗口适合探索想法，也适合解释概念，但一旦进入真实工程，问题就变复杂了：代码在仓库里，测试在命令行里，日志在文件里，版本差异在 git 里，部署配置在环境变量里，团队协作在 issue 和 review 里。一个只停留在对话框里的 AI，很容易变成“聪明的旁观者”：它能给建议，却不一定知道当前工作区的真实状态；它能写片段，却不一定能跑测试；它能解释错误，却不一定把修复落到正确文件。

终端里的 AI 工作流要解决的不是“让模型更会聊天”，而是把模型接进工程动作。开发者每天已经在终端里组合 rg、git diff、go test、npm test、curl、jq、make、docker compose、psql、ssh 等工具。终端天然具备几个优势：动作可记录、输入输出可重放、上下文靠近代码、失败可以被命令捕获、产物可以进入版本控制。AI 如果能在这个环境中工作，就不再只是回答问题，而可以参与“读代码、形成假设、修改文件、运行验证、解释结果”的闭环。

很多团队把 AI 接入终端后，第一反应是做一个万能命令：用户输入一句自然语言，AI 直接生成并执行 shell。这个方向看似强大，实际风险很大。自然语言意图经常不完整，shell 的副作用又很强，一条错误命令可能删除文件、修改数据库、泄漏凭据或污染工作区。更成熟的做法不是让 AI 替代终端，而是让 AI 成为终端工作流中的一个受控步骤：它可以读取上下文、提出计划、生成补丁、调用只读命令、在明确授权后执行有副作用动作，并把结果回写为可审查的 diff。

终端工作流也不是只给个人开发者用。团队层面真正有价值的是可复现。一个修复 bug 的 AI 会话，如果只有几段聊天记录，别人很难复盘它为什么这么改。若它留下的是命令轨迹、文件 diff、测试输出和决策说明，reviewer 就能看清楚：问题如何定位，哪些假设被排除，哪些测试覆盖了风险，哪些文件没有被触碰。AI 的能力越强，越需要这种工程化约束，否则团队只会得到更多看似合理但来源不明的改动。

我更愿意把终端 AI 看成一种“工程动作编排器”。它的输入不是一句孤立 prompt，而是当前仓库、目标任务、可用工具、权限边界和验证标准；它的输出也不是一段答案，而是一组可检查的动作和产物。对于 Go、Node、Python、Rust 或前端项目，这套思路都成立。差异只在工具链不同，核心仍然是：先观察，再计划；先小步修改，再验证；先保护用户改动，再考虑自动化。

核心概念

终端里的 AI 工作流至少包含六个核心概念：工作区、上下文、动作、权限、验证和记忆。工作区是实际文件系统和版本控制状态；上下文是被显式纳入推理的代码、日志、文档和命令输出；动作是可以执行或建议执行的命令、补丁和外部调用；权限定义哪些动作允许自动做，哪些必须询问；验证用测试、lint、构建、截图或业务断言确认结果；记忆则把本次经验沉淀到文档、脚本、评测样本或团队规范中。

概念在终端里的表现设计重点常见风险
工作区git 状态、文件树、依赖、环境变量读清楚当前状态，不覆盖用户改动误删未提交文件，混入无关变更
上下文rg、sed、日志、测试输出、文档只装入和任务相关的信息读太少导致猜测，读太多导致噪声
动作shell 命令、文件补丁、API 调用动作要可解释、可回滚、可审查自然语言直接变危险命令
权限只读、工作区写入、网络、部署按副作用分级，敏感动作确认模型越权访问密钥或生产资源
验证单测、集成测试、构建、快照验证要对应改动风险只跑最快测试，漏掉关键回归
记忆README、runbook、脚本、issue把重复流程固化成团队资产每次都靠会话临时记忆

工作区意识是第一位的。终端 AI 不能把仓库当成干净沙盒，真实项目里经常有用户未提交的修改、生成文件、临时实验、分支差异和本地配置。开始改动前应该读取 git status --short，对目标文件做最小范围检查，发现已有改动时要分辨它们是否与任务相关。AI 不应该为了让测试通过而回滚陌生改动，也不应该把格式化命令扩散到整个仓库，除非任务明确要求。

上下文不是越多越好，而是要围绕任务逐层收集。比如用户说“修复登录测试失败”，第一层上下文是失败输出；第二层是相关测试和被测代码；第三层是最近改动；第四层才是框架文档或历史 issue。很多失败不需要全仓库索引，也不需要长篇架构总结。终端工作流的优势在于可以用 rg 和具体命令快速缩小范围，让模型在小而真实的上下文里判断。

动作要有类型。只读动作如 rg、ls、sed、git diff 风险较低；工作区写入动作如修改文件、运行格式化、生成锁文件需要控制范围；环境动作如安装依赖、启动服务、迁移数据库、访问网络需要明确目的；生产动作如部署、删除远端资源、写入真实数据库必须强制人工确认。把动作分级之后，AI 才能在不同任务里保持同样的安全边界。

验证是终端 AI 的分水岭。没有验证，AI 只是生成补丁；有验证，它才参与工程闭环。验证不能只追求“跑过一个命令”。一个改动触及 parser，就应该有输入输出样例；触及权限，就要有拒绝路径；触及 UI，就要有浏览器截图或交互检查；触及性能，就要有基准或至少有可解释的观测。验证也要承认现实约束：有些仓库测试很慢，有些服务缺少依赖，有些命令需要密钥。此时 AI 要明确说哪些验证已完成，哪些因为环境限制未完成。

记忆让终端工作流从个人技巧变成团队能力。如果一次任务发现“本仓库所有内容页都要通过某个校验脚本”，那应该把脚本写进文档或 checklist；如果一次排查发现某类错误的复现路径很固定，就可以沉淀成诊断命令；如果 AI 反复需要读取同一组架构文件，就说明仓库需要更清晰的 CONTEXT.md 或开发者指南。好的终端 AI 工作流会逐渐减少临时解释，增加可执行资产。

架构/流程图解说明

终端 AI 工作流可以拆成一个清晰流水线：接收目标，读取工作区状态，收集上下文，形成计划，执行受控动作，运行验证，整理结果。重点不是每一步都自动化，而是每一步都有明确输入输出，能被用户打断、审查和复盘。

这条流水线里的关键边界是执行层。AI 不应该把“计划”和“执行”混成一团。计划阶段要说明修改哪些模块、为什么这些模块相关、验证将如何证明改动有效。执行阶段要尽量小步，每次改动后看 diff 或跑局部测试。验证阶段要把测试结果和任务目标绑定，而不是简单列出“跑了测试”。交付阶段则要把用户真正关心的事情讲清楚：改了什么、怎么验证、还有什么没法验证。

在实现上，可以把终端 AI 运行时分成四个组件：上下文管理器、工具执行器、补丁管理器和报告器。上下文管理器负责决定读取哪些文件和命令输出；工具执行器负责命令权限、超时、输出截断和错误分类；补丁管理器负责应用修改、保护用户改动、展示 diff；报告器负责把命令结果、文件变更和剩余风险整理成简短交付信息。

上下文管理器和工具执行器之间要有反馈。命令失败不是终点，而是新的上下文。例如 go test ./... 失败，输出显示某个包缺少 fixture，那么下一步不一定是直接修代码，可能是读取测试文件和 fixture 目录；npm test 因依赖缺失失败，下一步要判断是不是需要安装依赖，还是仓库本来就用另一个包管理器。终端 AI 的价值很大一部分来自这种“看到真实反馈后调整路径”的能力。

工程实现

一个可落地的终端 AI 工作流，不需要一开始做成复杂平台。可以先定义一套任务会话结构，让每次工作都有明确状态。下面是一个简化的数据结构，适用于本地 CLI、IDE 内嵌终端或远程开发环境。

session_id: twf_20260218_001
workspace: /repo/product
goal: 修复用户导入 CSV 时日期字段解析失败的问题
constraints:
  - 不修改数据库 schema
  - 只允许改 parser 和相关测试
  - 最终需要跑 parser 单测
state:
  branch: fix/csv-date
  dirty_files:
    - web/import_preview.tsx
permissions:
  read_commands:
    - rg
    - sed
    - git
    - go test
  write_scope:
    - internal/csvparser/**
    - internal/csvparser_test/**
  require_confirmation:
    - network
    - database
    - package_install
verification:
  required:
    - go test ./internal/csvparser/...
  optional:
    - go test ./...

这个结构的重点是把目标、约束、脏文件、权限和验证放在同一处。dirty_files 提醒 AI 不要碰用户正在改的 UI 文件；write_scope 限制补丁范围；required 明确最低验证。实际产品可以从这些字段派生 UI 提示，也可以在执行器里做硬拦截。

工具执行器要处理很多工程细节。首先是超时。AI 调用命令时不能无限等待，尤其是测试、构建和服务启动。每个命令应该有默认超时，超时后保存部分输出并分类为 timeout，让 planner 决定缩小测试范围还是延长等待。其次是输出截断。模型不需要读取几十万行日志，执行器应该保留开头、结尾和匹配错误模式的片段，并允许后续按需打开完整日志。第三是 shell 安全。自然语言生成的命令要经过解析和策略检查，尤其是包含 rm、重定向、管道写入、网络访问、环境变量打印和 sudo 的命令。

一个命令请求可以这样表示：

{
  "kind": "command",
  "cwd": "/repo/product",
  "argv": ["go", "test", "./internal/csvparser/..."],
  "purpose": "验证日期解析修复是否覆盖 parser 包",
  "side_effect": "read_or_build_cache",
  "timeout_ms": 120000,
  "capture": {
    "max_bytes": 20000,
    "highlight": ["FAIL", "panic", "Error", "expected"]
  }
}

这里刻意使用 argv 而不是一整段 shell 字符串，因为结构化参数更容易做权限判断。不是所有命令都能避免 shell，但常见的测试、搜索和格式化命令都可以结构化。purpose 也很重要，它让用户和审计日志知道这条命令为什么被执行。没有目的的命令调用，后面很难复盘。

补丁管理器的职责是让文件修改可控。手写文件、自动格式化和批量替换都应该进入同一套 diff 流程。每次修改前读取目标文件，修改后检查 diff，只展示任务相关变化。如果格式化工具改了大量无关文件，补丁管理器应该提醒并允许只保留目标文件。对于已有用户改动的文件，要么避开，要么在改动前明确说明将如何与已有改动共存。终端 AI 最忌讳“为了完成任务顺手整理一堆文件”。

在 Go 项目里，一个典型修复流程可以是：

  1. 运行失败测试或读取用户提供的错误输出，确定入口。
  2. 用 rg "ParseDate|csv" 找到 parser 和测试。
  3. 阅读最小相关文件，确认日期格式支持范围。
  4. 增加一个失败用例，例如 2026/02/18 或带时区格式。
  5. 修改 parser，优先保持原有 API 不变。
  6. 跑局部测试，若失败就读取失败断言继续缩小。
  7. 跑格式化和更宽测试。
  8. 输出 diff 摘要、测试结果和兼容性说明。

这看起来很普通，但正是终端 AI 应该做好的基本功。它不是一次性猜出答案，而是把工程师平时的诊断动作做得更快、更连续、更可审计。

上下文收集也需要策略。下面是一组我在实践中觉得有效的优先级：

场景第一优先级第二优先级不建议一开始做
测试失败失败输出、对应测试文件被测函数、最近 diff全仓库阅读
编译失败编译器定位行、类型定义调用链、生成代码规则大范围重构
UI 错位截图、DOM、相关组件CSS 约束、响应式断点只看静态代码猜
性能退化基准、慢日志、火焰图最近改动、数据规模盲目换算法
部署失败CI 日志、环境差异构建脚本、镜像层本地随便安装依赖

终端 AI 的上下文预算有限，必须避免把注意力浪费在不相关内容上。rg 比人工猜文件名可靠，测试输出比“可能是这里”可靠，git diff 比主观记忆可靠。AI 最应该擅长的是把这些小证据串起来，而不是在缺证据时写宏大推理。

权限策略要落地到执行器，而不是只写在提示词里。可以把命令分成四级：observe、workspace_write、environment_change、external_side_effect。observe 包括读文件、搜索、查看 diff；workspace_write 包括编辑仓库文件和运行格式化；environment_change 包括安装依赖、启动容器、写本地数据库；external_side_effect 包括推送代码、发请求到生产、修改云资源。不同级别需要不同确认和日志记录。即使是本地仓库，也要把 git push、gh pr merge、kubectl delete 这类动作视为高风险。

对于多语言仓库，工作流要先识别本地约定，而不是套用通用命令。Node 项目可能用 pnpm、npm、yarn 或 bun；Go 项目可能有 make test 封装；Python 项目可能依赖 uv、poetry 或 tox；前端项目可能需要先启动 dev server 再用浏览器检查。终端 AI 应该优先读取 package.json、Makefile、go.mod、pyproject.toml、README 和 CI 配置，找出团队已经定义的入口。尊重本地约定，比凭经验运行“看起来标准”的命令更可靠。

测试评测

评测终端 AI 工作流，不能只看它最终答案是否正确。更重要的是看过程是否安全、是否高效、是否能复现。一个能修好 bug 但顺手改了二十个无关文件的系统，不适合进入团队工作流；一个每次都要读完整仓库才能回答小问题的系统，会浪费上下文和时间；一个从不运行测试的系统，会把风险转给 reviewer。

可以从以下指标评测：

指标解释采集方式合格信号
定位效率从目标到相关文件的步数命令轨迹、读取文件数先读失败输出和相关文件
改动聚焦diff 是否只覆盖任务范围git diff、文件列表无无关格式化和重构
用户改动保护是否识别并避开陌生改动初始和最终 git status不回滚、不覆盖、不混淆
验证完整度测试是否覆盖修改风险命令结果、测试名称局部和必要全局验证都有说明
失败恢复命令失败后是否合理调整失败分类和后续动作能缩小范围或解释阻塞
报告质量交付说明是否可 review最终摘要文件、原因、测试、风险清楚

为终端 AI 建 eval，可以准备一组真实仓库任务。每个任务包括初始工作区、用户目标、允许修改范围、预期 diff 特征和必须通过的验证命令。不要只用玩具任务。真实任务里应包含脏工作区、失败测试、慢命令、缺失依赖、错误提示不明显、多处相似代码等情况。这样才能测出系统在工程现场是否稳定。

一个评测用例可以这样写：

id: terminal-flow-csv-date-001
repo_fixture: fixtures/repos/importer
goal: 修复 CSV 导入时 yyyy/MM/dd 日期解析失败
initial_dirty_files:
  - web/import_preview.tsx
allowed_write:
  - internal/csvparser/**
required_checks:
  - go test ./internal/csvparser/...
assertions:
  touched_files_must_match:
    - internal/csvparser/parser.go
    - internal/csvparser/parser_test.go
  must_not_touch:
    - web/import_preview.tsx
  final_diff_must_include:
    - "yyyy/MM/dd"
  report_must_mention:
    - "go test ./internal/csvparser/..."

这类评测不会要求 AI 的每个 token 相同，而是检查工程边界。只要它保护了用户改动、修改了正确模块、增加了合理测试、跑了指定命令，就算路径略有差异也可以接受。对于终端工作流，允许多条合理路径比追求逐字复现更重要。

还需要做故障注入。比如让 go test ./... 超时，看系统是否会退回局部测试并说明原因；让依赖安装需要网络，看系统是否会先询问；让目标文件已有用户改动，看系统是否会先读取 diff；让测试输出很长，看系统是否能截取关键错误。故障注入能暴露很多提示词里看不出来的问题。

人工评测也不可省。终端 AI 的很多质量维度很难完全自动化，例如计划是否合理、解释是否诚实、有没有过度自信、有没有把未验证内容说成已验证。可以让资深工程师抽样 review 会话轨迹，并给出标签：上下文不足、命令风险、改动过宽、验证不够、交付不清。把这些标签反过来加入系统规则和评测样本，工作流才会越来越稳。

失败模式

终端 AI 的失败模式通常比聊天助手更有破坏性，因为它离文件和命令更近。最常见的第一类失败是上下文不足。系统没有读测试输出，没有看本地约定，只凭文件名和经验改代码。表现是补丁看起来合理，但和真实调用路径不一致。解决办法不是塞更多 prompt，而是强制在修改前收集最小证据：失败命令、相关文件、现有测试、当前 diff。

第二类失败是动作过宽。用户要求修一个边缘 bug，AI 顺手重构模块、改命名、格式化全仓库，导致 reviewer 无法看清真正修复。这个问题要靠写入范围和 diff 审查控制。默认策略应是局部修改，只有在明确发现设计问题且用户同意时才扩大范围。终端 AI 应该把“保持变更小”当作工程纪律。

第三类失败是权限越界。模型可能为了完成任务尝试安装包、访问网络、打印环境变量、连接数据库、推送分支或调用云 CLI。很多动作在开发者电脑上可以成功，但不代表应该自动执行。解决办法是执行器分级和确认机制，尤其是外部副作用动作必须有人工确认。提示词声明“不做危险操作”不够，必须在工具层拦截。

第四类失败是验证幻觉。AI 改完代码后说“应该可以通过”，或者把没有跑完的命令描述成已验证，或者测试失败后仍然给出成功结论。终端系统要把命令结果结构化记录，报告只能引用真实执行结果。若测试无法运行，要明确原因和未覆盖风险，而不是用语言把风险抹平。

第五类失败是工作区污染。包括覆盖用户未提交修改、留下临时文件、生成缓存文件进入 diff、改变锁文件但不说明、启动服务后不停止。解决办法是在会话开始和结束都检查工作区状态，写入前确认目标文件，结束前清理临时进程和无关产物。对长时间任务尤其重要。

第六类失败是记忆不当。AI 把一次任务中的临时状态当成永久规则，或把一个仓库的约定带到另一个仓库。终端工作流需要区分会话记忆、仓库记忆和全局习惯。只有写进仓库文档或配置的规则，才应该被后续会话稳定采用；临时观察只能作为当前任务上下文。

第七类失败是成本失控。终端 AI 如果每次都读取大量文件、跑全量测试、反复启动服务，团队会很快失去耐心。成本不只是 token，也包括 CPU、CI 时间、开发者等待和日志噪声。需要给上下文读取、命令运行和重试次数设预算。预算不是为了少做事，而是让系统在必要时解释为什么需要扩大范围。

上线 checklist

把终端 AI 工作流接入团队前，我会用下面这份 checklist 做最低检查：

  • 工作区保护：会话开始读取 git status，能识别未跟踪、已修改和暂存文件。
  • 写入范围：任务可以声明允许修改路径，系统默认不碰无关文件。
  • 命令分级：只读、工作区写入、环境变更、外部副作用有不同权限策略。
  • 高风险拦截：删除文件、打印密钥、写数据库、推送、部署、云资源修改需要确认。
  • 输出管理：长日志有截断和关键行提取，完整日志可按需查看。
  • 超时管理：测试、构建、服务启动有超时，超时后状态明确。
  • 补丁审查：每次文件修改后能查看 diff，格式化不会扩散到无关文件。
  • 本地约定：优先读取 README、Makefile、CI、包管理器配置，不硬套通用命令。
  • 验证声明：最终报告区分已运行、未运行、失败、因环境受阻的验证。
  • 进程清理：启动的 dev server、容器或后台任务有记录，结束时处理清楚。
  • 审计记录：关键命令、目的、结果、修改文件和测试输出能被追踪。
  • 隐私保护：不主动读取或输出密钥、私有配置、生产数据和无关个人文件。
  • 失败复盘：会话失败也能留下足够上下文，方便工程师接手。
  • 团队反馈：reviewer 可以标注问题类型，并回流到规则和评测样本。

上线时建议先从低风险场景开始，例如代码解释、测试失败诊断、文档更新、局部 bug 修复。不要一开始就让系统自动处理部署、数据库迁移或大规模重构。随着评测样本、权限策略和团队信任逐步成熟，再扩展到更复杂的工程动作。

组织层面还要明确责任边界。AI 可以建议和执行受控动作，但提交代码的人仍然要负责理解 diff 和验证结果。终端 AI 的目标不是消除 review，而是让 review 面对更清晰的证据。一个好的交付说明应该让 reviewer 在一分钟内知道：为什么改、改了哪里、怎么测、还有什么没测。

总结

终端是 AI 进入工程现场最自然的入口之一，因为真实软件开发本来就由命令、文件、测试、日志和版本控制组成。把 AI 放进终端，不是让它随意执行自然语言 shell，而是让它在受控权限下完成可复现的工程动作：观察工作区，收集证据，形成计划，小步修改，运行验证，交付可审查结果。

这套工作流的价值不在炫技，而在纪律。它要求系统尊重用户改动，尊重仓库约定，尊重权限边界，尊重测试结果。它也要求团队把一次次临时经验沉淀为脚本、文档、评测和 checklist。AI 越能动手，工程约束越不能省。

如果做得好，终端 AI 会让开发者少花时间在重复搜索、机械修改和日志整理上，把精力放回判断和设计。它不会替代工程师的责任，但能把很多模糊的“帮我看看”变成清楚的“这是证据、这是补丁、这是验证、这是风险”。这才是 CLI 与 AI 结合最实用的方向。
