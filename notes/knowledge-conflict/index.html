冲突知识处理

当旧文档和新事实冲突时，系统需要给出时间、来源和置信度。

Knowledge ConflictRAG

问题背景

知识库越认真维护，冲突知识越不可避免。旧设计文档说“检索服务使用远端 embedding”，新运维手册说“默认使用本地 embedding”；产品 FAQ 写“免费用户每天可以调用 100 次”，计费配置里实际是 80 次；事故复盘说根因是缓存穿透，后续补充又说明真正触发点是权限策略变更。RAG 系统如果只做相似度召回，很可能把这些材料一起塞进上下文，最后让模型自己调和。模型会尝试给出一个顺滑答案，但它未必知道哪个事实更新、哪个来源更权威、哪个只是历史解释。

冲突知识不是简单的“文档过期”。有些冲突来自时间演进：旧事实在过去正确，新事实在现在正确。有些来自视角差异：产品文档讲用户可见行为，内部配置讲系统限制。有些来自粒度不同：一篇总览说所有环境都使用本地模型，部署说明却规定生产环境走远端服务。有些来自错误：文档写错、复制粘贴没改、抽取器误把猜测当事实。把这些都粗暴归为“旧文档”会丢失重要背景；全部交给模型总结又会把错误包装得更可信。

工程上要承认一个事实：知识库里存放的不只是文本，而是一组带来源、时间、范围和置信度的断言。冲突处理的目标不是让系统永远只有一个答案，而是在面对互相矛盾的断言时，能够做三件事：识别冲突，解释冲突，按问题场景选择合适证据。用户问“现在是多少”时需要当前有效断言；用户问“为什么变了”时需要新旧断言和变更原因；用户问“哪个文档可信”时需要来源优先级和维护记录。

如果没有冲突处理，RAG 的失败会很难定位。回答错了，大家会争论是模型幻觉、召回不准、文档没更新，还是用户问法不清。引入显式冲突模型后，失败可以落到具体层：断言抽取漏了字段，时间范围没解析，来源权威度配置错了，冲突簇没有更新，或者答案生成没有展示不确定性。可定位，才有办法持续改。

核心概念

冲突处理的最小单位不是整篇文档，而是 fact assertion，也就是“某个主体在某个范围内具有某个取值或关系”的断言。比如“生产环境检索服务使用 local-embedding-v3”可以拆成 subject=检索服务，predicate=embedding_provider，object=local-embedding-v3，scope=production，valid_from=某日期，source=部署手册。另一条断言“检索服务使用 remote-embedding-v2”如果 scope 也是 production，时间重叠，predicate 相同，object 不同，就构成潜在冲突。

断言需要几个基础字段：主体、谓词、宾语、范围、时间、来源、证据、置信度、状态。主体和宾语可以是实体，也可以是数值、枚举、文本片段。范围用于表达环境、用户群、产品版本、地域、平台。时间用于表达 valid_from 和 valid_to，也可以只保留 observed_at。来源用于判断权威度，例如配置文件、运行时指标、ADR、运维手册、会议纪要、聊天记录。置信度不是模型自信，而是系统对这条断言作为事实的信任程度。

冲突不一定是坏事。一个健康知识系统应该保留冲突的历史，因为它能解释决策变化。真正危险的是“未标记的冲突”：系统不知道两条断言互斥，回答时随机采样其中一条。冲突处理不是把旧材料删掉，而是给断言建立关系：supersedes、contradicts、narrows、expands、corrects、same_as。这样答案可以说清楚“当前规则是 80 次；旧 FAQ 写 100 次，该 FAQ 已被配置变更修正”。

还有一个重要概念是“冲突簇”。单独比较两条断言不够，因为同一事实可能出现在多个文档、表格、配置和代码注释里。系统需要把同一 subject-predicate-scope 下互斥或相近的断言聚成一组，选出当前首选断言，同时保留其他成员。冲突簇的主键不一定完全一致，实体别名、范围层级和谓词归一化都会影响聚类质量。第一版可以从少数高价值谓词开始，例如 limit、owner、status、provider、depends_on、requires。

架构/流程图解说明

冲突处理链路可以放在实体关系抽取之后、上下文组装之前。它既需要离线计算，也需要在线决策。离线阶段负责从文档和配置里抽取断言、归一化实体、检测冲突簇；在线阶段根据问题意图、用户上下文和时间点选择断言，并生成可解释上下文。

离线抽取不要只依赖自然语言文档。很多最权威的事实来自机器可读来源：配置文件、数据库迁移、Feature Flag、运行时 API、OpenAPI schema、CI 配置。自然语言文档更适合解释“为什么”，但未必是“当前值”的最佳来源。冲突系统应该允许多种来源进入同一个断言仓库，并给来源类型设置默认权威度。比如生产配置高于手册，已接受 ADR 高于会议讨论，运行时观测高于过期截图。

冲突检测的核心是同域比较。subject、predicate 和 scope 相同或可包含，object 不同，且时间区间重叠，就需要判断是否冲突。scope 层级非常重要。production 和 staging 不冲突；all_users 和 enterprise_users 可能是覆盖关系；v2 API 和 v1 API 不应混在一起。时间也不能忽略。旧断言 valid_to 早于新断言 valid_from，不是当前冲突，而是历史演进。

在线阶段要把冲突状态带进上下文。模型不应该只看到两段互相矛盾的原文，而应该看到系统整理后的冲突簇：当前首选断言是什么，哪些断言与它冲突，各自来源、时间、范围和证据是什么。这样模型的任务从“猜哪个对”变成“按给定证据解释”。这会显著降低胡乱调和的概率。

工程实现

实现时可以先建一个 assertion store。它不需要替代向量库或图数据库，而是补充一层事实索引。每条 assertion 必须能回到原始证据，同时有归一化后的结构字段。Go 里可以把断言设计成不可变记录，任何修正都新增版本，而不是原地覆盖。这样评测和审计能回放某个时间点系统为什么做出某个回答。

type Assertion struct {
    AssertionID string
    SubjectID   string
    Predicate   string
    ObjectValue string
    ObjectType  string
    Scope       Scope
    ValidFrom   *time.Time
    ValidTo     *time.Time
    ObservedAt  time.Time
    SourceID    string
    SourceKind  string
    EvidenceID  string
    Confidence  float64
    Status      string
}

type Scope struct {
    Environment string
    Product     string
    Version     string
    Region      string
    UserSegment string
}

type ConflictCluster struct {
    ClusterID          string
    SubjectID          string
    Predicate          string
    ScopeFingerprint   string
    PreferredAssertion string
    Members            []string
    ResolutionReason   string
    UpdatedAt          time.Time
}

谓词归一化是第一道难关。文档里可能写“使用”“依赖”“走”“接入”“默认模型为”，其实都在表达 provider 或 dependency。不要一开始追求覆盖所有谓词。挑最影响回答正确性的十几个谓词做字典和抽取 schema，例如 current_owner、api_limit、storage_backend、embedding_provider、feature_status、requires_permission。抽取器只允许输出这些谓词，未知关系保留为普通文本证据，不进入冲突判定。

范围解析要保守。宁可把 scope 标成 unknown，也不要把“某些租户”“灰度用户”“内部环境”误归为全局。scope unknown 的断言可以参与召回，但不能覆盖 scope 明确的断言。范围之间需要一个包含关系函数：specific scope 优先于 general scope。例如 enterprise_users 的规则可以覆盖 all_users 的默认规则，但不能反过来。如果用户问题没有给 scope，系统可以使用用户上下文；仍然不明确时，回答应该提示“在生产环境是 A，在开发环境是 B”。

冲突判定可以先用规则。对同一 subject-predicate，如果 object 不同、scope 兼容、时间重叠，就生成 conflict candidate；再按来源、证据和谓词类型决定严重程度。数值类谓词可以直接比较，枚举类谓词也容易；文本类谓词需要 embedding 或 LLM 辅助判断。LLM 的角色应是“解释候选是否真实冲突”，不是自由发现所有冲突。这样成本和可控性都更好。

resolution policy 要写清楚。一个常见优先级是：运行时配置或数据库当前值，高于代码 schema；代码 schema 高于已接受 ADR；已接受 ADR 高于运维手册；运维手册高于 FAQ；FAQ 高于会议纪要；会议纪要高于聊天记录。这个排序不是通用真理，要按团队实际调整。关键是它可配置、可审计、可在答案中解释。系统选择某条断言，不是因为它“看起来新”，而是因为它在给定范围和时间上更权威。

一个具体例子：用户问“企业版每天能调用多少次批量导入 API”。断言仓库里有三条：FAQ 写所有用户 100 次，date 是 2026-03；计费配置写 enterprise_users 500 次，valid_from 是 2026-05；一次会议纪要写“可能调到 300 次”，状态是 reviewing。scope 解析识别用户问 enterprise，冲突簇中计费配置更具体且来源权威，答案应该说企业版当前是 500 次，同时提示 FAQ 中的 100 次适用于旧默认说明或需要修正，会议纪要只是讨论记录。模型不需要猜，它只需要把结构化冲突讲明白。

冲突回答策略

冲突处理最终要体现在答案里。很多系统在后台做了复杂检测，但生成时只输出一个值，用户看不到为什么。对于工程知识库，我更倾向于使用三种回答模式：确定当前值、展示差异、请求澄清。确定当前值用于 scope、时间、来源都足够明确的场景；展示差异用于用户询问演进、比较或系统检测到高风险冲突；请求澄清用于范围不足，比如环境、版本、用户群缺失。

确定当前值不等于隐藏冲突。如果候选里存在明显冲突，答案可以先给结论，再给来源解释：“当前生产环境使用 local-embedding-v3，依据是 2026-05-06 的部署配置；旧设计文档中的 remote-embedding-v2 已被 2026-05-01 的迁移记录替代。”这类回答既高效，又给用户足够线索。如果只是给“使用 local-embedding-v3”，用户遇到旧文时仍会困惑。

展示差异适合“为什么改”“什么时候改”“哪个版本”的问题。上下文组装时可以按时间线排列断言：旧事实、变更事件、新事实、剩余例外。对模型提示也要明确：不要合并互斥断言，不要把历史值说成当前值，不要把讨论中的候选方案说成已经上线。生成后可以用规则检查：如果答案里同时出现两个不同 object，却没有时间或范围限定，就标记为潜在混淆。

请求澄清也应该是系统能力的一部分。比如用户问“调用次数是多少”，知识库里有免费版、企业版、内部版三个不同限制。直接回答一个平均值是错误的。系统可以说“这个限制按用户类型不同：免费版 80 次，企业版 500 次，内部版不限。你要确认哪个用户类型？”如果用户上下文里已经有账号类型，就自动套用上下文，并在答案中说明范围。

答案层还可以提供 conflict badge。引用列表里如果某条证据属于冲突簇，展示“当前首选”“历史值”“候选方案”“待修正文档”等标签。用户点击后能看到冲突簇详情。这个功能看起来像产品细节，但它会反过来帮助维护：用户更容易发现哪篇文档需要更新，owner 也能收到明确反馈。

数据修复与人工介入

冲突处理不能只靠自动算法。自动检测可以把问题浮出水面，但很多冲突需要人做业务判断：某条配置是不是临时热修，某份手册是不是应该同步更新，某个灰度规则是不是只面向一个客户，某个会议结论是否已经被取消。系统要为人工介入设计明确工作台，而不是把冲突列表扔给维护者。好的工作台应该展示冲突簇、首选断言、成员证据、来源权威度、时间范围、影响到的问答样本和推荐动作。

推荐动作可以分成五类。第一类是确认首选断言，适用于系统已经选对，但需要 owner 背书。第二类是修正文档，适用于权威来源正确、说明文档落后。第三类是拆分 scope，适用于看似冲突实际是环境或用户群不同。第四类是调整来源权重，适用于团队政策变化，例如运行时配置开始比 ADR 更权威。第五类是标记抽取错误，适用于模型误读否定句、假设句或讨论记录。不同动作产生不同回写，不要都变成一句评论。

人工修复还要能回流到系统学习，但这里的“学习”不一定是训练模型。更多时候是更新规则、字典和评测。比如维护者连续把“计划”“考虑”“可能”标成非事实，抽取 prompt 和后处理规则就应该增加 modality 判断；维护者多次拆分 enterprise 和 free scope，说明范围解析器需要把用户群作为强字段；维护者反复确认配置文件覆盖 FAQ，说明来源权重可以固化。不要把人工反馈只存成审计记录，它应该变成工程改进输入。

冲突修复也需要优先级。不是所有冲突都值得当天处理。可以按影响面排序：进入最终答案的冲突高于仅在候选中的冲突，高频问题相关冲突高于冷门文档冲突，高权威来源互相冲突高于低权威文档冲突，当前事实冲突高于历史演进冲突。这个优先级能让团队把精力放在会误导用户决策的地方。冲突系统如果每天制造一百条低价值提醒，很快就会被关闭。

最后要设计“暂缓决议”。某些冲突短期内无法解决，例如产品策略尚未确定，配置和文档都在变。系统可以把冲突簇标为 unresolved，但答案策略必须随之改变：当前操作问题给出保守回答并提示 owner，历史或探索问题可以展示多种候选。暂缓不是忽略，而是把不确定性显式化，并设置复查时间。

冲突修复还要注意审计语义。人工确认一条首选断言，不应该删除其它断言，因为旧断言可能仍解释历史事故，也可能在某个 as_of 查询里有效。正确做法是给冲突簇增加 resolution record：谁确认、何时确认、依据是什么、适用范围是什么、是否需要下次复查。答案使用首选断言时，可以引用这个 resolution record；审计回放时，也能还原当时为什么没有选择另一条材料。这样冲突系统既能给当前用户明确答案，也不会抹掉知识演进过程。

测试评测

冲突系统的评测要覆盖抽取、检测、解析、选择和表达。只看最终答案会漏掉很多中间错误。第一层评测是 assertion extraction：给定文档片段，系统是否抽出正确 subject、predicate、object、scope、time、source。第二层是 conflict detection：给定一组断言，是否把真正互斥的聚成簇，是否避免把不同环境或不同版本误判为冲突。第三层是 resolution：首选断言是否符合来源和时间策略。第四层是 answer grounding：答案是否把范围、时间、来源讲清楚。

评测样本应该包含“容易误判为冲突”的负例。比如“开发环境使用本地模型，生产环境使用远端模型”不是冲突；“v1 API 限制 100 次，v2 API 限制 200 次”不是冲突；“默认值是 80 次，企业版可申请 500 次”不是冲突，而是层级规则。没有这些负例，系统会过度报警，最后团队对冲突提示失去信任。

自动化测试可以用黄金断言集。把十几个高价值实体的真实规则手工标注成 assertions，再把文档和配置作为输入，检查系统能否还原这些断言。每次修改抽取 prompt、scope 解析、来源优先级，都跑同一套黄金集。对于答案评测，可以写规则：当前事实问题必须包含 preferred assertion 的来源；如果存在冲突成员，答案必须至少包含时间或范围限定；如果 preferred 置信度低于阈值，答案不能使用强确定语气。

线上观测同样关键。需要统计冲突簇数量、未解决冲突数量、高权威来源与低权威文档冲突数量、用户点击冲突详情次数、冲突导致拒答次数。还要记录冲突的老化时间：一个配置和 FAQ 的冲突如果存在三十天，说明内容维护流程有问题。冲突系统不只是问答质量工具，也是文档治理工具。

人工反馈可以设计成结构化动作。用户发现答案冲突时，不要只给“有帮助/没帮助”。更有用的选项是：当前值错误、范围错误、旧文未标记、来源不可信、需要补充例外、答案没说明差异。每个选项都能路由到不同 owner：配置错误找平台，文档过期找作者，scope 解析错误找 RAG 工程。反馈越结构化，回归测试越容易沉淀。

失败模式

第一个失败模式是把最新文档当成最可信。时间新鲜度很重要，但不是唯一标准。聊天记录可能比正式手册新，但它只是讨论；一份新草稿可能提到未来计划，但未上线。resolution policy 必须同时看来源类型、评审状态、时间和范围。否则系统会被“新但不权威”的材料带偏。

第二个失败模式是忽略 scope。很多所谓冲突只是范围不同。生产和开发、免费版和企业版、国内区和海外区、v1 和 v2，都可能有不同事实。scope 解析不清时，系统应该展示差异或追问，而不是强行合并。范围字段越规范，冲突系统越有价值。

第三个失败模式是断言抽取过度。模型看到“我们考虑把限制调到 300 次”，就抽成 api_limit=300；看到“不要再使用远端模型”，就抽成 provider=remote。否定句、建议、假设、反事实和会议讨论都容易出错。抽取 schema 要有 modality 字段，例如 factual、proposal、negation、hypothesis。只有 factual 和 accepted proposal 才能进入强事实冲突。

第四个失败模式是冲突簇不解释原因。系统内部选择了 preferred assertion，但答案和界面没有说明依据，用户仍然不信。冲突处理必须可解释：为什么这条更权威，另一条为什么降级，时间是否重叠，scope 是否相同。没有解释的自动选择，会变成另一个黑箱。

第五个失败模式是人工修复无法回流。用户指出 FAQ 过期，owner 修改了文档，但冲突簇没有重新计算，缓存答案仍然引用旧冲突。断言仓库、向量索引、冲突簇和答案缓存之间要有失效机制。文档或配置变更后，相关 subject-predicate-scope 的簇应该重新评估。

第六个失败模式是把所有冲突都推给用户。系统动不动回答“存在冲突，无法确认”，会让用户失去效率。对于有明确来源优先级的场景，系统应该给出当前最佳答案，同时暴露冲突；只有低置信或 scope 不明时才拒答或追问。冲突处理的目标是提高决策质量，不是让系统逃避回答。

上线 checklist

  • 高价值谓词有明确字典和抽取 schema，未知关系不进入强事实断言。
  • assertion 保存 subject、predicate、object、scope、valid_from、valid_to、source、evidence、confidence。
  • scope 有层级和包含关系，至少覆盖环境、版本、用户群、地域、产品线。
  • 来源权威度可配置，并能在 trace 和答案中解释。
  • 冲突检测区分当前冲突、历史演进、范围差异和抽取不确定。
  • 在线检索把冲突簇作为上下文结构传给生成层，而不是只传原文片段。
  • 答案对冲突来源使用时间、范围、状态限定，避免把互斥值合并成单一叙述。
  • 评测集包含真正冲突样本和非冲突负例，覆盖配置、文档、会议、FAQ。
  • 文档或配置更新后，相关断言、冲突簇和答案缓存能失效重算。
  • 用户反馈有结构化原因，可路由到文档 owner、配置 owner 或 RAG 工程。

总结

冲突知识处理的核心，是把“文本里说了什么”提升为“在什么范围、什么时间、由什么来源支持的事实断言”。RAG 系统不能指望模型在一堆互相矛盾的片段里自动判断真相。它需要断言仓库、范围解析、时间有效性、来源权威度、冲突簇和可解释的选择策略。这样系统才能在当前事实、历史演进和不确定场景之间切换。

落地时不要试图一次性解决所有冲突。先挑最影响用户决策的谓词和来源，例如限额、状态、owner、provider、依赖、权限。把这些断言结构化，建立黄金评测，记录冲突 trace，让答案明确展示来源和范围。冲突不可怕，未识别的冲突才可怕。一个成熟的知识系统不是永远没有矛盾，而是能把矛盾讲清楚，并把用户带到当前最可信的证据上。
