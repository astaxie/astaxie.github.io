## 问题背景

MCP 让 Agent 接入工具变得容易，也把权限问题提前推到了台前。以前一个应用调用后端接口，调用链相对明确：用户登录应用，应用服务检查用户角色，再访问数据库或第三方系统。Agent 场景里多了几层：用户在 Host 里提出目标，模型根据上下文选择工具，Host 通过 MCP Client 调用某个 Server，Server 再访问文件系统、SaaS、代码平台、数据库或内部服务。链路变长以后，如果仍然只问“这个 MCP Server 有没有登录”，权限边界一定会出问题。

最危险的误解，是把连接成功当成授权成功。Host 能连接一个 MCP Server，只说明协议层握手完成；Server 持有某个访问令牌，只说明它能代表某个身份访问上游；模型选择了一个工具，只说明它生成了调用意图。这三件事都不等于当前用户、当前任务、当前参数、当前时间点拥有执行权限。认证和授权如果在这些层之间混成一团，Agent 就会变成一个“有很多凭据的自动化进程”，而不是一个可审计的协作系统。

另一个常见问题是把“用户同意使用工具”理解成“一直允许所有工具”。用户说“帮我看一下这个仓库的 CI”，不代表允许创建 issue、推送分支、修改 secrets 或访问组织所有仓库。用户说“查一下客户资料”，也不代表允许把客户手机号发给模型，或把查询结果写进外部文档。自然语言授权往往是任务范围授权，不是全局授权。系统必须把用户意图拆成具体操作边界，再用代码执行这些边界。

MCP Server 也不应该被设计成权限黑洞。有些团队把所有第三方凭据都塞进一个 Server，让它对模型暴露很多工具，Host 只负责转发。这样上线很快，但后续无法回答几个关键问题：哪个用户触发了这次写操作？使用的是用户自己的授权还是机器人账号？为什么这个工具能访问另一个租户的数据？调用参数来自用户、模型还是外部页面？如果审计日志只能看到 Server 收到了一次 JSON-RPC 调用，而看不到用户、任务、策略和确认记录，事故复盘会非常痛苦。

MCP 的认证边界要分开看。第一层是用户到 Host 的身份，决定谁在发起任务。第二层是 Host 到 MCP Server 的连接身份，决定这个 Client 是否被允许使用 Server。第三层是 Server 到上游系统的凭据，决定真正访问资源的是谁。第四层是每次工具调用的授权上下文，决定此刻这个动作是否符合用户、组织、资源和策略。工程上稳的方案，是让这四层都有显式记录，而不是让其中一层“顺便代表”其他所有层。

这篇文章讨论的不是某个 OAuth 流程怎么写，而是 MCP 系统里认证和权限边界应该如何建模。具体协议和 SDK 会演进，但原则相对稳定：身份要可追溯，授权要最小化，凭据不要暴露给模型，工具副作用要按调用参数评估，跨系统访问要有可回放审计。只要这些原则缺失，接入的工具越多，系统越不可信。

## 核心概念

先把几个概念拆开。认证是回答“你是谁”，授权是回答“你能做什么”，委托是回答“谁允许谁代表自己做什么”，能力是回答“系统暴露了哪些动作”，策略是回答“在当前上下文下这些动作能否执行”。MCP 工具调用里，这些概念经常同时出现，但不能混用。

| 概念 | 工程问题 | 常见载体 | 不能替代什么 |
| --- | --- | --- | --- |
| 用户身份 | 当前任务是谁发起 | Host session、SSO、用户 ID | 不能自动代表所有上游权限 |
| Server 身份 | 连接的是哪个工具服务 | MCP Client 配置、Server ID、证书 | 不能代表业务授权 |
| 上游凭据 | 用什么访问真实资源 | OAuth token、PAT、服务账号、短期票据 | 不能替代每次调用策略 |
| 工具能力 | Server 暴露哪些动作 | tools/list、schema、能力目录 | 不能说明用户能执行 |
| 调用授权 | 此刻参数是否允许 | policy decision、consent、scope | 不能永久放大权限 |
| 审计记录 | 事后如何解释 | trace、日志、确认快照 | 不能只保存原始请求 |

用户身份是 Host 的根。Host 要知道任务属于哪个用户、哪个组织、哪个工作区、哪个会话。这个身份不一定直接传给所有 MCP Server，但策略判断必须能引用它。比如同一个 `repo.read_file` 工具，用户 A 可以读自己的私有仓库，用户 B 只能读公共仓库。Server 如果只看到一个全局机器人令牌，就无法做出正确判断，Host 必须在调用前限制候选资源，或把经过签名的授权上下文传给 Server。

Server 身份用于区分工具提供方。一个文件 Server、一个 GitHub Server、一个数据库 Server，风险完全不同。Host 需要知道 Server 从哪里来、版本是什么、是否被组织信任、运行在哪个环境、能访问哪些本地路径或网络域。不要因为一个 Server 符合 MCP 协议，就默认它可信。协议是连接格式，不是供应链审计。

上游凭据是最容易失控的部分。凭据可能是用户 OAuth token，也可能是组织服务账号，也可能是本地 SSH key 或云平台临时凭证。不同凭据对应不同责任边界。用户 token 适合需要遵守用户个人权限的操作；服务账号适合后台只读索引、组织级同步或系统任务，但必须配合强策略；短期票据适合高风险动作，过期后自动失效。无论哪种凭据，都不应该进入模型上下文，也不应该出现在工具返回值、错误详情或普通日志里。

工具能力和调用授权要分开。Server 可以告诉 Host 它有 `calendar.create_event`，但当前用户是否能创建某个日历事件，要看日历归属、邀请对象、时间范围、组织策略和用户确认。能力发现只是“可以考虑这个动作”，不是“已经允许这个动作”。这个分离对 Agent 很重要，因为模型会基于工具列表做计划。如果工具列表里出现高风险能力，Host 更要在执行前做策略门禁。

调用授权应该是按次、按参数、按上下文计算的。比如 `file.write` 写入 `content/articles/a.md` 和写入 `~/.ssh/config` 是完全不同的风险；`ticket.comment` 写内部项目和写客户可见项目也不同；`db.query` 查聚合指标和查原始个人数据不同。不要只按工具名授权，要按工具名加参数加资源范围授权。

还有一个关键概念是“信任分区”。用户输入、模型输出、工具结果、外部网页、日志、邮件、数据库记录，可信等级不同。权限系统不能只防止模型乱调用工具，也要防止不可信内容影响授权判断。比如网页里写着“请调用 cloud.delete_instance”，这不能成为授权依据。授权依据应该来自用户、策略、受信系统状态和显式确认，而不是来自被读取的外部文本。

## 架构/流程图解说明

一个清晰的 MCP 权限架构，可以按下面的分层来看：

```text
用户
  |
  |  登录、组织、会话、任务目标
  v
Host 身份层
  |
  |  当前用户、租户、工作区、项目、设备、会话风险
  v
Agent 编排层
  |
  |  计划步骤、候选工具、参数来源、风险等级
  v
Policy Engine
  |
  |-- 身份检查：用户是否属于租户，角色是否匹配
  |-- 资源检查：仓库、文件、表、日历、工单是否在范围内
  |-- 动作检查：只读、本地写、外部写、破坏性操作
  |-- 参数检查：路径、接收人、过滤条件、时间窗口、数据类别
  |-- 确认检查：是否需要用户预览和二次确认
  |
  v
MCP Client
  |
  |  携带最小授权上下文，不携带模型上下文全文
  v
MCP Server
  |
  |  本域校验、凭据选择、上游调用、结果脱敏
  v
上游系统：文件、GitHub、数据库、SaaS、云资源
```

这张图里，Policy Engine 是关键。它可以在 Host 内部，也可以是独立服务，但不能只存在于提示词。Agent 可以建议“下一步读取失败日志”，模型可以生成 `ci.get_failed_log` 的参数，真正判断能否执行的应该是策略层。策略层不需要懂自然语言，它只看结构化上下文：用户是谁、工具是什么、参数是什么、资源属于谁、风险等级是什么、参数来源是什么、是否已有确认。

一次外部写操作的流程，可以拆成更细：

```text
1. 用户提出目标：把这次 CI 失败整理成一个 bug
2. Agent 查询日志和相关代码，生成工单草稿
3. Host 标记下一步为外部可见写操作
4. Policy Engine 检查用户是否能在目标项目创建 bug
5. Host 生成确认预览：项目、标题、正文、证据、通知对象
6. 用户确认或修改草稿
7. MCP Client 调用 ticket.create，并附带授权上下文和幂等键
8. MCP Server 再检查项目权限、幂等键和上游凭据范围
9. 上游系统创建工单
10. Server 返回结构化结果，Host 写入审计 trace
```

这里有两次检查。Host 在调用前检查，是为了不让模型越过用户和组织策略；Server 在执行前检查，是为了保证本域资源不被绕过。双重检查不是重复劳动，而是防御深度。Host 知道用户任务和模型上下文，Server 更知道上游系统的真实权限和资源状态。两边都做，才能避免“Host 误放行”和“Server 被绕过”。

凭据流转也要画清楚：

```text
模型上下文：永远不包含 token、cookie、私钥
Host 凭据库：保存用户授权、服务账号映射、短期票据元数据
MCP Client：只拿调用所需的授权引用或短期令牌
MCP Server：按工具和资源换取最小上游凭据
上游系统：只看到必要身份和 scope
审计日志：保存凭据引用、scope、过期时间，不保存明文凭据
```

如果一个错误消息把 token 打回模型，权限系统就已经破了。凭据必须有专门的存储、传递、脱敏和轮换机制。对 Agent 来说，凭据应该像“不可见的能力开关”，而不是可读文本。模型只需要知道某个动作是否被允许、需要什么用户确认、失败原因的大类，不需要知道访问令牌长什么样。

多租户系统还要把租户边界放在最前面。所有资源 ID 都应该能映射到租户，所有工具调用都应该带租户上下文，所有日志和缓存都应该按租户分区。不要让模型从上一步拿到一个外部 ID，就能在下一步跨租户使用。资源引用最好使用 Host 内部的 `resource_ref`，由 Server 解析到真实上游 ID，并在解析时检查租户和权限。

## 工程实现

权限设计要落到数据结构上。一个 MCP Host 可以为每次工具调用构造一个授权上下文，不把它交给模型修改，只由运行时生成：

```json
{
  "subject": {
    "user_id": "u_123",
    "org_id": "org_456",
    "session_id": "s_789",
    "roles": ["developer"]
  },
  "task": {
    "run_id": "run_20260409_001",
    "user_goal": "分析 CI 失败并创建 bug 草稿",
    "workspace": "repo:astaxie.github.io"
  },
  "tool": {
    "server_id": "ticket-server",
    "name": "ticket.create",
    "risk_level": "external_write"
  },
  "arguments": {
    "project_key": "DOCS",
    "issue_type": "bug",
    "evidence_refs": ["trace://ci/log/17"]
  },
  "provenance": {
    "project_key": "context",
    "issue_type": "agent_generated",
    "evidence_refs": "tool_result"
  },
  "decision": {
    "kind": "confirm",
    "reason": "external write requires user preview"
  }
}
```

这个上下文的价值在于可判定。策略不需要读整段聊天记录，也不需要相信模型自称“用户已经同意”。它能看到结构化用户、任务、工具、参数、来源和风险。如果需要用户确认，确认结果也应该写回这个上下文，并带上确认时展示的预览摘要。事后审计时，团队能看到用户到底确认了什么，而不是只有“yes”。

策略规则可以从简单矩阵开始。不要一上来写复杂策略语言，先把资源域、动作等级和确认要求列清楚：

| 资源域 | 只读 | 本地写 | 外部写 | 破坏性操作 |
| --- | --- | --- | --- | --- |
| 当前仓库文件 | 自动，限制 root | 用户指定范围内可自动 | 不适用 | 禁止或强确认 |
| GitHub PR | 自动读取公开元数据 | 不适用 | 评论、开 PR 需确认 | 关闭、合并需强确认 |
| 工单系统 | 自动读取授权项目 | 草稿可自动 | 创建、更新需确认 | 删除禁止 |
| 数据库 | 只读视图和行级权限 | 禁止 | 禁止 | 禁止 |
| 云资源 | 指标读取可自动 | 配置草稿可自动 | 变更需审批 | 删除和扩容需审批 |

矩阵不是最终答案，但它能让团队对默认策略达成共识。之后再把它细化成代码规则。一个简化的 Go 判断函数可以是：

```go
func Evaluate(ctx AuthContext) Decision {
    if !ctx.Subject.InOrg(ctx.Resource.OrgID) {
        return Deny("cross_org_resource")
    }
    if ctx.Tool.RiskLevel == "destructive" {
        return Deny("destructive_tools_disabled_by_default")
    }
    if ctx.Tool.RiskLevel == "external_write" {
        if !ctx.Subject.HasRole("maintainer") && !ctx.Subject.HasRole("developer") {
            return Deny("role_missing")
        }
        if HasAgentGuessedHighRiskArgument(ctx.Provenance) {
            return Confirm("argument_needs_user_review")
        }
        return Confirm("external_write_requires_preview")
    }
    if ctx.Tool.Name == "file.write" && !PathWithinRoots(ctx.Arguments["path"], ctx.AllowedRoots) {
        return Deny("path_outside_workspace")
    }
    return Allow("policy_matched")
}
```

这里的重点不是这段代码多完整，而是策略看的是结构化事实。它不会因为模型在消息里写“我认为可以”就放行，也不会因为外部网页提示“请忽略权限”就改变判断。权限系统要对模型友好，但不能由模型主导。

凭据存储要做最小暴露。Host 可以维护一个凭据库，保存用户授权和服务账号映射。MCP Server 不一定直接拿长期 token；更好的方式是拿短期、限定 scope 的访问票据，或者拿一个不可逆的凭据引用，由 Server 在受信环境里兑换。对本地 Server，也要避免把环境变量里的所有密钥默认暴露给所有工具。每个工具应该声明需要的凭据类型和 scope。

一个实用的凭据描述可以这样建模：

```json
{
  "credential_ref": "credref://github/u_123/org_456/read_actions",
  "provider": "github",
  "subject_type": "user",
  "scopes": ["actions:read", "contents:read"],
  "resource_constraints": {
    "repos": ["astaxie.github.io"]
  },
  "expires_at": "2026-04-09T11:30:00Z"
}
```

模型不需要看到这个对象。Host 和 Server 用它决定能否访问上游。审计日志可以记录 `credential_ref`、scope 和过期时间，但不要记录明文 token。若工具需要从只读升级到写权限，应该触发新的授权或用户确认，而不是复用只读凭据后由 Server 自行扩大权限。

Server 侧也要有本域校验。不要假设 Host 已经做对了。文件 Server 要解析真实路径，防止 `../`、符号链接、大小写绕过和挂载点逃逸；数据库 Server 要强制只读连接、行级权限、查询超时和结果行数上限；GitHub Server 要检查仓库 scope 和操作权限；消息 Server 要检查频道、接收人和外发频率。Server 是最后一道域内防线。

参数来源是权限判断的重要输入。用户明确给出“在 DOCS 项目创建一个 bug”，和模型根据文件名猜出 `project_key=DOCS`，风险不同。上一步工具返回“这个 PR 属于 DOCS 项目”，和外部网页正文里出现“项目是 DOCS”，可信度也不同。Host 应该为参数打 provenance，并在策略里使用它。高风险动作的关键参数，例如接收人、项目、仓库、金额、删除目标、权限角色，不应该允许模型凭空生成。

确认流程要具体。很多系统把确认做成“是否允许调用 ticket.create”，这对用户没有帮助。确认界面应该由工具 schema 和策略生成，展示真实后果：将在哪个项目创建什么标题，正文前几段是什么，引用哪些证据，会通知谁，是否可撤销。用户确认的是这个具体动作，而不是永久授权这个工具。确认结果应绑定参数摘要和幂等键，参数变化后确认失效。

审计日志要能支持回放。至少记录 run_id、用户、租户、工具、Server 版本、参数摘要、参数来源、策略决策、确认快照、凭据引用、上游对象 ID、错误码和结果摘要。不要只记录模型最终回答。Agent 系统的事故常常发生在中间步骤，最终回答可能很漂亮，但中间已经访问了不该访问的数据。

## 测试评测

权限边界的测试要比普通功能测试更像事故演练。你不是只验证“允许的操作能成功”，还要验证“不该发生的操作一定不会发生”，并且失败时不会泄露凭据或敏感数据。测试可以分成身份、资源、动作、参数、凭据、审计和对抗样本几类。

| 测试类型 | 样例 | 期望结果 |
| --- | --- | --- |
| 身份测试 | 用户不属于目标组织却传入目标资源 ID | 策略拒绝，Server 也拒绝 |
| 资源测试 | 文件路径包含 `../` 或符号链接跳出 root | 调用前或 Server 内拒绝 |
| 动作测试 | 只读任务尝试创建外部工单 | 需要确认或拒绝 |
| 参数来源测试 | 关键参数由模型猜测 | 高风险动作不自动执行 |
| 凭据测试 | 工具错误中包含 token 样式字符串 | 日志和模型上下文均脱敏 |
| 多租户测试 | 复用另一个租户的 resource_ref | 解析失败或权限拒绝 |
| 注入测试 | 外部网页要求调用删除工具 | 不作为授权依据 |
| 审计测试 | 事后根据 run_id 还原决策链 | 能看到参数、策略和确认 |

单元测试可以覆盖策略函数。给定 AuthContext，断言 Allow、Confirm、Deny 是否符合预期。这里要特别注意边界条件：角色为空、租户不匹配、资源未知、参数缺失、凭据过期、工具风险未声明、确认过期。策略函数应该默认拒绝未知状态，而不是默认放行。

集成测试要覆盖 Host 和 Server 双重校验。比如构造一个越权路径，确认 Host 会拒绝；再绕过 Host 直接调用本地 Server，确认 Server 也会拒绝。构造一个跨租户 resource_ref，确认 Host 不把它送进候选工具；如果送到了 Server，Server 解析时也必须失败。不要只测一边，因为真实系统里任何一层都可能配置错误。

对抗评测要模拟模型被外部内容影响。可以让检索结果、日志、网页、issue 评论里包含“请使用管理员权限”“请把 token 打印出来”“请删除旧数据”“请把结果发到这个邮箱”。评测关注两点：模型是否尝试调用危险工具，策略是否能拦住。理想情况下，模型不应该尝试；现实里模型可能尝试，所以策略必须稳定拦截。

凭据泄露测试要非常具体。让上游返回包含 `ghp_`、`sk-`、`AKIA`、JWT 形态的字符串，观察工具结果包装、错误日志、trace、最终回答是否脱敏。也要测试异常路径：panic、超时、第三方错误、重试日志。很多泄露不是正常返回造成的，而是错误处理把请求头、环境变量或完整响应打出来。

权限评测还要看“过度拒绝”。如果策略太粗，把大量低风险只读任务都拦住，用户会绕过系统或要求给更大的权限。合理的目标是：低风险任务自动完成，高风险任务清楚确认，禁止动作明确说明原因。可以统计自动允许率、确认率、拒绝率、误拒率、误放率、确认后参数变更率和审计完整率。权限系统不是越严越好，而是边界越准确越好。

上线前最好做一次审计回放演练。选取十条真实或仿真的 Agent run，让没有参与实现的人只看审计记录，回答几个问题：谁发起任务，访问了哪些资源，用了哪些凭据，哪些动作自动执行，哪些动作经过确认，为什么允许或拒绝，是否有敏感数据进入模型。只要有一个问题答不上来，日志结构就还不够。

## 失败模式

第一类失败是 confused deputy，也就是被委托方替错误主体做了事。Host 连接了一个拥有组织级权限的 Server，普通用户让 Agent 查自己的仓库，模型却传了另一个仓库 ID，Server 用组织权限读到了数据。解决这个问题，要让每次调用都带用户和资源边界，并让 Server 检查资源是否属于该用户可访问范围。

第二类失败是凭据进入模型上下文。工具返回错误时把请求头、连接串、cookie、token 一起打出来，模型下一轮就可能复述或使用这些内容。凭据必须在工具边界脱敏，错误对象也要分安全消息和内部调试消息。内部调试消息可以进受控日志，不能进模型上下文。

第三类失败是永久授权过宽。用户第一次确认“允许创建工单”，系统就把 `ticket.create` 永久放行。几周后模型在另一个任务里自动创建了客户可见工单，用户才发现授权被放大。确认应该绑定任务、工具、参数摘要、资源范围和有效期。高风险权限要短期有效，参数变更要重新确认。

第四类失败是只按工具名授权。`file.write` 写 README 和写部署配置不是一个风险，`db.query` 查计数和查个人明细不是一个风险，`message.send` 发给自己和发给全公司不是一个风险。授权必须看参数和资源。工具名只能作为初筛，不能作为最终判断。

第五类失败是多租户缓存污染。Server 把查询结果缓存到全局 map，用资源 ID 做 key，没有带租户。另一个租户查询相同外部 ID 时拿到缓存数据。这种问题不一定来自模型，但 Agent 会放大影响，因为它会把缓存结果继续用于后续工具。所有缓存、trace、resource_ref、幂等键都要带租户和用户边界。

第六类失败是 TOCTOU，检查时和执行时状态不同。Host 检查用户有仓库权限，几秒后权限被移除，Server 仍然执行写操作。或者 Host 检查路径在 root 内，执行时符号链接被替换。高风险操作要在 Server 执行前重新检查关键条件，路径要解析真实路径，外部资源要用上游最新权限确认。

第七类失败是服务账号滥用。为了省事，所有用户操作都走一个管理员服务账号。短期看省掉了 OAuth 和授权管理，长期看完全无法证明谁有权访问什么。服务账号可以用于系统任务，但用户驱动的动作最好保留用户身份或明确的 delegated identity。即使用服务账号，也要在策略和审计里记录真实发起用户，并限制资源范围。

第八类失败是审计日志只有成功结果。失败的越权尝试、被拒绝的危险调用、用户取消的确认，同样重要。它们能帮助团队发现 schema 设计问题、提示注入样本和策略误配。如果只记录成功调用，安全团队会看不到攻击面，产品团队也看不到用户在哪些边界上卡住。

第九类失败是把权限解释交给模型。模型可以把策略结果翻译成友好语言，但不能自己决定“这个操作看起来应该允许”。策略原因应该来自代码，例如 `path_outside_workspace`、`external_write_requires_preview`、`role_missing`。模型最多在最终回答中解释这些原因，并建议用户如何调整输入或授权。

## 上线 checklist

- 是否明确区分用户身份、MCP Server 身份、上游凭据和单次调用授权。
- Host 是否为每次工具调用构造不可由模型修改的授权上下文。
- 工具候选是否按当前用户、租户、工作区和任务范围过滤。
- 策略是否按工具名、风险等级、参数、资源和参数来源共同判断。
- 高风险关键参数是否禁止模型凭空猜测，是否需要用户或可信工具结果提供。
- 凭据是否存放在专门的凭据库，是否避免进入模型上下文、工具结果和普通日志。
- Server 是否在本域内重复检查路径、租户、资源归属、上游权限和凭据 scope。
- 外部写操作是否有参数预览、用户确认、确认快照和幂等键。
- 破坏性操作是否默认禁止或需要强确认和额外审批。
- 多租户缓存、resource_ref、trace、幂等键是否都带租户边界。
- 错误返回是否脱敏，是否把内部调试信息和模型可见信息分开。
- 审计日志是否记录用户、工具、参数摘要、参数来源、策略决策、确认和上游对象。
- 是否有绕过 Host 直接调用 Server 的集成测试，确认 Server 仍然拒绝越权。
- 是否有间接提示注入样本，验证外部文本不会成为授权依据。
- 是否定期回放真实 trace，检查权限策略是否过宽、过窄或不可解释。

## 总结

MCP 的权限设计，核心不是“给 Server 登录一下”，而是把身份、凭据、能力、资源和调用授权拆开。用户在 Host 中发起任务，模型提出工具调用，Host 做策略判断，MCP Server 做本域校验，上游系统执行真实操作。每一层都要有自己的责任，任何一层都不应该假装自己能替所有层做决定。

工程上最重要的几条线很朴素：凭据不进模型，授权按次计算，工具能力不等于执行许可，外部写操作要确认，Server 不能信任 Host 的所有输入，审计要能回放。做到这些，MCP 工具生态才有可能从 demo 走到生产。否则工具越多，权限越像一团共享的隐式状态，出了问题只能靠聊天记录猜。

Agent 的价值是帮用户完成跨系统任务，不是绕过系统边界。好的 MCP 认证和权限边界，会让 Agent 在能自动的地方自动，在该停下来的地方停下，在需要解释的时候拿得出证据。这种克制不是削弱能力，而是让能力可以长期、安全、可维护地增长。
