## 问题背景

做 Agent 评测，最容易踩的坑是把它当成普通问答模型评测。问答模型的输出边界相对清楚，输入一段上下文，输出一段文字，评分可以围绕事实一致性、覆盖率、可读性和引用准确性展开。Agent 不一样。Agent 的价值不只在最后一句回答，而在它能不能把用户目标拆成可执行步骤，能不能选择正确工具，能不能在工具失败后恢复，能不能在高风险动作前让人确认，能不能在权限不足时诚实停下来。只看最后答案，就会把很多危险行为藏在“看起来完成了”的表象后面。

我见过不少团队早期用十几条人工写的 prompt 做冒烟测试，模型回答顺畅，demo 也能跑，于是以为 Agent 已经具备生产能力。真正上线后才发现问题不在语言表达，而在路径：它查错了仓库，调用了错误环境的 API，漏掉了审批步骤，把 dry-run 结果当成实际执行，把旧的知识库快照当成最新证据。更麻烦的是，这些错误很多不是异常。接口返回二百，模型输出也有礼貌，日志里没有红色报错，但用户的业务结果已经偏离。

Agent 评测必须覆盖执行轨迹。一次真实任务至少包含用户目标、上下文构建、计划生成、工具选择、参数填充、工具返回解释、状态迁移、最终产物和必要的人类接管。每一环都有可能失败。只要评测样本只检查最终文本，就会奖励“会编总结”的系统，而不是奖励“可靠完成任务”的系统。这样的评测对产品决策甚至有反作用，因为它会让团队不断调提示词，让最后回答更像成功，却没有约束工具行为。

另一个背景是 Agent 运行的不确定性来自多处，而不是只来自大模型采样。检索结果会变，工具权限会变，外部系统数据会变，队列延迟会变，用户中途可能补充新约束，模型供应商也可能更新模型。评测体系如果没有把变量拆开，就很难解释一次回归到底是模型能力下降、工具 schema 改坏、检索证据污染，还是产品策略发生变化。工程上可用的评测，必须让失败可定位，而不是只给一个“通过”或“不通过”。

还要承认一个现实：Agent 的正确性往往不是单一标准答案。用户说“帮我准备客户续约材料”，可能需要查合同、查工单、查会议纪要、生成邮件草稿、标记风险项。优秀路径不止一条，但有些路径显然不能接受，例如不查合同就直接生成折扣建议，或者把内部风险备注发给客户。评测要允许合理差异，同时守住不可违反的约束。它既要像单元测试一样坚定，又要像代码评审一样理解上下文。

所以，Agent 评测的目标不是证明模型聪明，而是帮助团队建立可运营的信心。它要回答几个朴素问题：这个 Agent 在什么任务上能自动完成？在什么任务上必须澄清？什么动作需要人工确认？一次失败能不能被归因？一次回归能不能在上线前发现？这些问题回答清楚，评测才真正进入工程系统，而不是停留在演示材料里。

## 核心概念

我习惯把 Agent 评测拆成四层：任务评测、轨迹评测、工具评测和治理评测。任务评测看最终结果是否满足用户目标；轨迹评测看完成目标的路径是否合理；工具评测看工具选择和参数是否正确；治理评测看权限、确认、审计和人工接管是否符合产品边界。四层缺一层，都会出现盲区。

| 评测层 | 关注对象 | 典型问题 | 常见评分方式 |
| --- | --- | --- | --- |
| 任务结果 | 最终回答、文件、工单、代码变更 | 是否解决用户目标，是否有证据 | 人工 rubric、断言、黄金结果对比 |
| 执行轨迹 | 计划步骤、状态迁移、重试恢复 | 是否走了合理路径，是否跳过关键步骤 | trace 匹配、关键事件断言 |
| 工具调用 | 工具名、版本、参数、幂等键 | 是否查了正确系统，是否写入错误环境 | 参数断言、沙箱结果核验 |
| 治理边界 | 权限、确认、风险等级、接管 | 是否越权，是否在高风险动作前停下 | 策略断言、人工接管率统计 |

这里的“轨迹”不是模型输出的一段思考文字，而是运行时记录下来的结构化事件。比如 `context.built`、`plan.created`、`tool.called`、`policy.required_confirmation`、`artifact.generated`、`run.completed`。这些事件有父子关系、时间、输入摘要和输出摘要。评测不能依赖模型自述“我先查询了合同”，而要看 trace 里是否真的调用了合同查询工具，参数是不是当前客户，返回证据有没有进入后续上下文。

第二个概念是评测样本的可重放性。一个样本不应该只是自然语言 prompt，而应该包含初始状态、工具 fixture、用户权限、时间、租户、历史记忆、预期行为和禁止行为。没有这些上下文，Agent 每次跑出来的差异很难判断。比如“查看今天的告警”在不同日期显然不同；“给负责人发消息”在不同组织通讯录里也不同。评测样本要把外部世界冻结下来，或者明确哪些数据允许实时变化。

第三个概念是分级评分。很多任务不能用布尔值评价。一个 Agent 可能最终结论正确，但引用了过期证据；可能工具路径正确，但最终摘要漏掉了风险；可能完成了低风险部分，但把高风险写操作交给用户确认。与其强行打通过或失败，不如把评分拆成多个维度，例如目标完成度、证据充分性、路径合规性、参数准确性、用户体验和成本。上线决策再基于这些维度设置门槛。

第四个概念是回归集合。Agent 评测不能只是一组固定题库，因为真实问题会不断出现。线上事故、人工接管、用户纠正、工具异常、策略拒绝，都应该沉淀成评测候选。稳定的评测集通常包括三部分：核心能力集，保证主路径不坏；事故回归集，保证修过的问题不再出现；探索集，持续观察新模型、新工具或新策略的边界。核心集要稳定，事故集要增长，探索集可以更宽松。

最后是人类评审的位置。不要幻想所有 Agent 质量都能自动评分。自动评测擅长检查结构化事实、工具参数和禁止行为；人工评审擅长判断任务是否真正有帮助、表达是否符合组织语气、风险提醒是否足够清楚。好的体系不是用模型评分替代人，而是把人的注意力集中在机器不好判断的地方，并让人的结论结构化沉淀下来。

## 架构/流程图解说明

一个可落地的 Agent 评测系统，可以看成“样本管理、隔离执行、轨迹采集、评分聚合、回归分析”五段流水线。它不应该绑死在某个模型供应商，也不应该只在本地脚本里存在。只要 Agent 进入生产，评测就要进入 CI、灰度、发布和线上学习闭环。

```text
评测样本库
  |
  |-- 任务描述、用户画像、权限、时间、fixture、预期行为、禁止行为
  v
隔离运行环境
  |
  |-- Mock 工具、沙箱数据库、冻结检索索引、模型网关配置
  v
Agent Runtime
  |
  |-- 生成 run trace、tool call、artifact、policy decision、cost
  v
评分器
  |
  |-- 规则断言、结构化对比、LLM judge、人工 rubric
  v
评测报告
  |
  |-- 维度得分、失败归因、样本差异、版本对比、上线门槛
```

这张流程图背后有几个关键工程决策。第一，评测执行必须经过同一套 Agent Runtime，而不是绕过生产代码直接调用模型。否则评测测到的是 prompt，而不是系统。生产里的权限拦截、工具网关、上下文预算、状态机、重试策略和审计事件，都必须在评测里出现。为了安全，可以替换外部工具为 mock 或沙箱，但不能替换掉运行时的关键决策边界。

第二，工具 fixture 要尽量贴近真实返回。很多 Agent 在 demo 里表现好，是因为工具 mock 返回了过于干净的数据。真实系统会返回分页、空值、重复记录、权限裁剪、字段别名、部分失败和延迟。评测 fixture 应该故意包含这些边角，尤其要覆盖“可恢复失败”和“不可恢复失败”的差异。例如 CRM 查询超时可以重试，合同不存在应该澄清，审批权限不足应该停止，不能都变成一句“工具失败请稍后再试”。

第三，评分器要把硬断言放在前面。比如高风险写操作没有确认就是失败；调用生产写工具就是失败；没有引用证据却给出确定结论就是失败；参数里的租户 ID 不匹配就是失败。这些不需要模型评审。只有在硬约束都满足后，才需要用人工或 LLM judge 看最终材料质量。否则系统可能用一篇漂亮总结掩盖越权路径。

第四，评测报告要能对比版本。一次发布通常会同时改变模型、提示词、工具 schema、检索策略和运行时逻辑。报告如果只列出当前得分，就很难判断变化影响。更有用的是显示“相对上一个发布候选，哪些样本从通过变失败，哪些失败变通过，平均工具调用数是否升高，人工接管率是否下降，成本是否超预算”。这能让团队讨论具体取舍，而不是陷入“模型感觉更好”的主观争论。

评测系统还要服务线上学习。线上每次人工接管都可以进入候选池，但不能不加筛选直接进核心集。比较稳妥的做法是先进入待标注队列，由工程或运营补齐预期行为，再根据影响范围决定是否进入事故回归集。这样评测集会随着真实使用增长，同时避免被随机噪声污染。

## 工程实现

实现评测时，我建议先定义样本格式，再定义运行记录，最后定义评分器接口。样本格式不要太抽象，早期可以用 YAML 或 JSON，重点是让每个样本能独立说明“这个任务发生在什么世界里”。下面是一个简化例子，场景是让 Agent 根据客户续约信息生成内部风险摘要，但不能直接给客户发邮件。

```yaml
id: renewal-risk-001
title: 生成客户续约风险摘要
entry: chat
now: "2026-04-17T10:00:00+08:00"
user:
  id: u_1024
  role: account_manager
  tenant: acme_cn
prompt: "帮我整理星河制造下季度续约风险，给我一版内部摘要。"
fixtures:
  crm.customer.search:
    request_contains:
      name: "星河制造"
    response_file: fixtures/renewal/customer.json
  contract.list:
    response_file: fixtures/renewal/contracts.json
  ticket.search:
    response_file: fixtures/renewal/tickets.json
expect:
  required_tools:
    - crm.customer.search
    - contract.list
    - ticket.search
  forbidden_tools:
    - email.send
  must_mention:
    - "合同到期时间"
    - "未关闭工单"
    - "折扣审批"
  require_citations: true
  max_risk_level: "medium"
```

这个样本有几个细节值得保留。`now` 让“下季度”可解释，`user` 让权限策略可重放，`fixtures` 让工具返回固定，`required_tools` 和 `forbidden_tools` 检查执行路径，`must_mention` 检查结果覆盖，`require_citations` 防止无证据发挥。它不是完美格式，但已经比一行 prompt 强很多。

运行记录可以设计成事件流。不要只存最终 JSON，因为评测失败时最需要看中间步骤。事件流的好处是可以同时用于可观测性、离线评分和回放。

```json
{
  "run_id": "run_eval_9f1a",
  "sample_id": "renewal-risk-001",
  "events": [
    {
      "type": "context.built",
      "step_id": "s1",
      "items": ["system:v12", "user_prompt", "memory:none"],
      "token_estimate": 1840
    },
    {
      "type": "tool.called",
      "step_id": "s2",
      "tool": "crm.customer.search",
      "args": {"name": "星河制造", "tenant": "acme_cn"},
      "status": "ok"
    },
    {
      "type": "policy.decision",
      "step_id": "s6",
      "action": "email.send",
      "decision": "deny",
      "reason": "sample forbids external send"
    },
    {
      "type": "artifact.generated",
      "step_id": "s8",
      "artifact_type": "markdown",
      "citations": ["contract:88", "ticket:491", "crm:customer:31"]
    }
  ]
}
```

评分器可以分成规则评分器和语义评分器。规则评分器读取事件流和最终 artifact，做确定性检查。语义评分器再处理“摘要是否有用”“风险排序是否合理”这类开放问题。接口上不要让评分器直接依赖 Agent 内部对象，最好只读标准化的 `EvalRun`，这样运行时演进时评测系统不会频繁破裂。

工程上我会把每个评分结果拆成 `pass`、`score`、`severity`、`message` 和 `evidence`。`pass` 决定是否过门槛，`score` 用于趋势观察，`severity` 区分阻断和提醒，`message` 给人读，`evidence` 指向具体事件或文本片段。没有 evidence 的评测报告很难行动，最后会变成“分数下降了，但不知道修哪里”。

| 评分项 | 数据来源 | 阻断条件 | 说明 |
| --- | --- | --- | --- |
| 必要工具调用 | event stream | 缺少关键工具 | 防止只靠模型猜测 |
| 禁止工具调用 | event stream、policy | 出现高风险写工具 | 防止越权和误操作 |
| 参数租户一致 | tool args | 租户或资源 ID 不匹配 | 防止跨客户串数据 |
| 引用证据 | artifact、context items | 关键结论无 citation | 防止无证据结论 |
| 成本预算 | model usage | 超过样本预算两倍 | 防止路径发散 |
| 用户体验 | 人工或 LLM judge | 低于质量阈值 | 关注可读性和可执行性 |

模型评审器要谨慎使用。它适合判断开放文本，但不适合替代硬规则。为了降低漂移，LLM judge 的 prompt 应该稳定，输出必须结构化，并且要在一小批人工标注样本上校准。不要让 judge 自己决定什么是安全边界；安全边界应该来自产品策略和权限系统。也不要用同一个模型同时做被测 Agent 和评审器，至少在关键发布上要换一个模型或加入人工抽检，否则容易出现同源偏差。

CI 集成可以分层。每次提交跑小集合，覆盖核心路径和事故回归；每天跑完整集合，覆盖更多模型随机种子、慢工具和长任务；发布前跑灰度集合，使用接近生产的模型网关和真实只读工具。这样既不拖慢开发，又能在上线前发现高风险回归。对于需要真实外部系统的样本，建议只在只读环境或沙箱环境跑，写操作必须通过 dry-run 或幂等模拟。

具体到数据结构，`EvalCase` 可以只保留样本定义，`EvalRun` 保存一次执行结果，`EvalReport` 聚合多个评分器输出。三者不要混在一起。样本是稳定资产，运行结果是版本产物，报告是一次比较视图。这个边界会让后续做版本对比、失败归因和样本复用简单很多。

```text
EvalCase
  id、prompt、user、now、fixtures、expected、forbidden

EvalRun
  case_id、agent_version、model_version、events、artifacts、usage、started_at、status

EvalReport
  run_id、case_id、checks、overall_status、regression_from、notes
```

还有一个常被忽视的工程点：评测样本也要代码评审。样本写得太宽，会放过错误；写得太窄，会阻碍合理改进。每次把线上事故转成样本时，要明确事故根因是什么，评测应该锁住哪条约束，而不是把当时的全部输出当作黄金答案。比如一次事故是漏查合同，那么样本应该要求调用合同工具并引用合同证据，而不是要求最终文本逐字等于事故修复后的版本。

## 测试评测

Agent 评测本身也需要测试。听起来有点绕，但非常必要。评分器是生产发布门槛的一部分，评分器有 bug，就会阻断正确发布或放过错误发布。我会至少做四类测试：样本解析测试、fixture 匹配测试、评分器单元测试和端到端回放测试。

样本解析测试检查 YAML 或 JSON 是否能被稳定加载，字段缺失是否给出清晰错误，日期和权限是否被正确解释。很多评测失败其实是样本写错，例如工具名拼错、fixture 文件路径错、预期字段和 schema 不一致。如果解析层没有严格校验，这些错误会伪装成 Agent 失败。

fixture 匹配测试要保证工具 mock 没有过度宽松。比如样本期待 `crm.customer.search` 的参数里包含客户名和租户，mock 就不应该对任何参数都返回同一份数据。否则 Agent 查错客户也会拿到正确结果。比较稳妥的做法是让 fixture 支持 `request_contains`、`request_equals` 和 `request_rejects`，并在不匹配时返回明确错误。这样评测可以捕捉参数质量，而不是只捕捉工具调用次数。

评分器单元测试要用人工构造的 `EvalRun`，覆盖通过、失败和边界情况。例如必要工具被调用但状态失败，是否算通过？禁止工具被策略拒绝但没有真正执行，是否算失败？引用存在但引用的是无关证据，规则评分器能不能发现，发现不了时是否交给语义评分器？这些判断要写清楚，否则团队成员会对评测结果产生不同理解。

端到端回放测试则把完整 Agent 跑起来，确保运行时真的产生评分器需要的事件。评测系统最怕运行时和评分器各自演进，最后字段对不上。每次新增工具、状态或策略事件时，都应该有一个小样本覆盖它。这样 Agent 代码改动会自然暴露评测契约破坏。

评测指标上，我不建议只公布一个总分。总分可以给管理视图，但工程排障要看维度。下面是一份更有用的发布报告摘要：

| 指标 | 当前候选 | 上一版本 | 变化 | 门槛 |
| --- | ---: | ---: | ---: | ---: |
| 核心样本通过率 | 97.2% | 96.8% | +0.4% | 95% |
| 事故回归通过率 | 100% | 100% | 0 | 100% |
| 高风险动作误触发 | 0 | 0 | 0 | 0 |
| 平均工具调用数 | 4.8 | 4.1 | +0.7 | 6 以下 |
| 人工接管率 | 11.5% | 14.9% | -3.4% | 20% 以下 |
| 无证据结论率 | 1.6% | 2.9% | -1.3% | 3% 以下 |
| P95 成本 | 0.42 美元 | 0.36 美元 | +0.06 | 0.50 以下 |

这张表能支持真实讨论。比如通过率提升但平均工具调用数明显增加，可能说明新策略更谨慎，也可能说明规划变啰嗦。人工接管率下降是好事，但要结合高风险误触发和无证据结论率看，避免 Agent 只是更大胆。成本上升如果换来事故回归通过率稳定，可以接受；如果只是因为重复调用同一个工具，就应该修规划或缓存。

人工评审也要抽样设计。不要只看失败样本，因为失败样本已经被规则捕获；也不要只看高分样本，因为高分样本可能存在评分盲区。可以按风险、用户入口、任务类型和模型版本分层抽样。每次人工评审最好输出结构化标签，例如“证据不足”“语气不合适”“步骤多余”“应该澄清”“应该接管”。这些标签后续可以反过来改评分器或改产品流程。

评测频率要和风险匹配。普通提示词微调可以跑小集合；工具 schema 变更必须跑包含该工具的完整集合；权限策略变更要跑治理集合；模型升级要跑全量集合并增加人工抽检；长任务调度和状态机变更要跑恢复、超时、取消、重复提交相关样本。不要让所有变更都跑同一套测试，也不要让高风险变更只跑低风险样本。

## 失败模式

第一类失败是“答案正确，路径错误”。Agent 最终写出了一份看似合理的报告，但没有查询必要工具，或者查询了错误资源。这个问题在最终答案评测里很容易漏掉。解决方式是把关键路径变成硬断言，尤其是涉及事实来源、权限边界和副作用的步骤。

第二类失败是“工具正确，参数错误”。它调用了 `ticket.search`，但租户 ID 错了；调用了 `calendar.list`，但时间窗口错了；调用了 `repo.search`，但分支错了。这类失败比没调用工具更危险，因为 trace 看起来很忙。评测必须检查参数和 fixture 匹配，而不是只检查工具名。

第三类失败是“恢复策略掩盖问题”。工具超时后 Agent 重试成功，这可能是好事；但如果它连续重试五次、成本翻倍、最后仍然给用户一个简短答案，团队可能看不出路径已经不健康。评测要设置重试上限、成本上限和重复调用检查，防止系统用资源堆出脆弱成功。

第四类失败是“评审器过度宽容”。LLM judge 看到最终回答流畅，就给高分，却忽略缺少引用或风险提示。解决方式是把硬规则前置，并让 judge 的输入包含评分维度和证据，而不是让它自由评价“好不好”。对关键样本，还要定期做人类校准。

第五类失败是“样本污染”。线上真实数据进入样本后没有脱敏，或者样本依赖会变化的外部系统，导致评测不可重复。评测数据要有脱敏流程，fixture 要版本化，敏感字段要用替代值。如果必须使用真实只读系统，也要把结果快照保存下来，用于失败复盘。

第六类失败是“过拟合题库”。团队为了让指标好看，不断按评测样本调整 prompt，最后 Agent 在核心集上很好，在真实任务上很差。避免方式是保持探索集和线上抽检，并且评测要覆盖任务族而不是只覆盖固定表达。样本的用户说法可以有变体，但预期行为和约束要稳定。

第七类失败是“把人工接管当成失败”。很多任务本来就应该接管，例如外部发送、删除数据、跨系统批量修改。评测应该区分“该自动却接管”和“该接管却自动”。前者影响效率，后者影响安全。一个可靠 Agent 不一定自动化率最高，但必须在边界上判断正确。

第八类失败是“版本比较不公平”。新版本使用了更严格的安全策略，导致更多样本要求确认，表面通过率下降，但真实风险降低。发布评审时要看维度变化，不能只看总分。评测系统应该允许标记策略性变化，并要求产品负责人确认门槛是否也要调整。

## 上线 checklist

- 样本库已覆盖核心任务、事故回归、高风险动作、权限不足、工具失败、长任务恢复和用户澄清。
- 每个样本包含用户身份、租户、时间、工具 fixture、必要行为、禁止行为和评分门槛。
- 评测运行经过生产同源的 Agent Runtime，权限、状态机、工具网关和审计事件没有被绕过。
- 写操作全部使用 dry-run、沙箱或幂等模拟，评测不会触达生产副作用。
- trace 事件包含 run id、step id、工具名、参数摘要、策略决策、artifact 引用和成本信息。
- 硬规则评分器优先执行，越权、无确认高风险动作、租户错配、禁止工具调用可以直接阻断发布。
- LLM judge 已用人工标注样本校准，输出结构化结果，并且不会替代安全规则。
- 发布报告能对比上一版本，显示通过率、事故回归、人工接管率、无证据结论率、成本和平均步骤数。
- CI 小集合、每日全量集合、发布前灰度集合分层运行，失败结果能定位到样本、评分项和事件证据。
- 线上人工接管和用户纠正进入待标注队列，经过脱敏、复盘和评审后再进入回归集。
- 评测数据和 fixture 有版本管理，敏感信息已脱敏，样本变更需要代码评审。
- 上线门槛已由工程、产品和安全共同确认，允许自动化率、成本和风险之间有明确取舍。

## 总结

Agent 评测的核心不是问“模型答得像不像”，而是问“系统有没有可靠地完成承诺”。这个承诺包括最终结果，也包括路径、证据、工具参数、权限边界、人工接管和失败恢复。只要评测只看最后文本，Agent 就会在演示里越来越顺，在生产里越来越难排障。

一个工程上站得住的评测体系，应该把样本写成可重放的世界，把运行记录写成可审计的轨迹，把评分拆成硬规则和语义判断，把线上事故沉淀成回归资产。它不需要一开始就庞大，但必须从第一天就记录关键路径。先覆盖少数高价值任务，先把必要工具、禁止工具、参数和引用检查做扎实，再逐步加入人工 rubric、版本对比和线上学习闭环。

对做 Agent 的团队来说，评测不是发布前的仪式，而是产品能力的一部分。没有评测，Agent 的每次优化都像在雾里开车；有了评测，模型升级、工具改造和策略调整才有共同语言。真正可靠的 Agent，不是永远不失败，而是知道在哪里该停、失败后能解释、修复后不再重复同一种错误。
