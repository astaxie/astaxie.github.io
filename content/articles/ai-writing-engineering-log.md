## 问题背景

工程团队每天都在做大量判断，但真正留下来的信息很少。代码仓库里有最终 diff，issue 里有任务标题，聊天工具里有零散讨论，监控系统里有报警曲线，PR 评论里有局部争论。等到三周后有人问“当时为什么这样改”“这个参数为什么这么保守”“为什么没有选另一个方案”，大家往往只能凭记忆回答。记忆会过期，聊天记录难以检索，commit message 太短，设计文档又经常只写最终结论。工程日志要补的就是这个空档：把问题、判断、实验、结果和后续影响记录成未来能用的资产。

AI 写作进入团队后，这件事变得更急迫。过去不写日志，最多是知识流失；现在如果把 AI 接进编码、调研、测试和文档流程，却不记录它读了什么、建议了什么、哪些建议被采纳、哪些被拒绝，团队很快会失去可解释性。一个代理改了十几个文件，测试通过了，但中间做过哪些假设？一个模型帮忙总结事故，引用了哪些 trace？一个写作助手把复盘改得很顺，是否把关键不确定性淡化了？这些问题不解决，AI 只是在加速产出，不是在提升工程质量。

工程日志不是个人日记，也不是流水账。它不是“今天修了 bug，明天开会，后天上线”。有用的工程日志应该围绕决策和证据组织。它至少回答：遇到了什么问题，影响范围是什么，尝试过哪些路径，关键证据来自哪里，最终做了什么取舍，结果如何验证，留下了什么风险，下次遇到类似问题应该从哪里开始。日志的读者不是今天的自己，而是未来要维护这段系统的人，其中也包括未来接手上下文的 AI 代理。

我更愿意把工程日志看成一条轻量的事件流。每个日志条目不是文学作品，而是一个带结构的工程事件。它可以很短，但要有定位能力。比如“修复检索失败”太弱；“2026-02-15，修复知识库增量索引漏删旧 chunk 的问题。根因是 document_version 更新后删除任务只按 document_id 扫描，未过滤 stale embedding。加入 tombstone 表并补充回归样本。风险是历史索引需要后台清理。”这段信息不长，却能让后来者知道问题、根因、修复、测试和风险。

AI 写作在工程日志里的角色应该是“整理和补全”，不是“编造和美化”。人负责事实、判断和取舍，AI 负责把散乱材料整理成可读结构，提醒缺失字段，生成初稿，合并重复记录，抽取时间线，转成周报或复盘。边界一旦反过来，让 AI 在没有证据时补齐原因和结论，日志就会变成漂亮但危险的噪声。工程日志的第一原则是可追溯，第二原则是可检索，第三原则才是好读。

在个人知识库里，工程日志还有一个长期价值：它把“过程知识”从聊天和脑子里拿出来。很多经验无法直接写成教程，比如一次模型升级为什么失败，一段 RAG 检索为什么误召回，一次权限改造为什么选择保守方案，一次前端重构为什么没有拆组件。这些判断只有放在当时的上下文里才成立。日志保存上下文，未来做类似事情时，AI 可以把它们作为案例召回，而不是只根据最终代码猜意图。

## 核心概念

工程日志可以分为五类：工作日志、决策日志、实验日志、事故日志和交接日志。工作日志记录日常推进，适合短周期回顾；决策日志记录取舍，适合长期追溯；实验日志记录假设和结果，适合避免重复试错；事故日志记录问题、影响和恢复过程，适合提高系统韧性；交接日志记录当前状态、未完成事项和风险，适合让人或 AI 继续工作。它们可以存在同一个目录，但字段和写法不能完全一样。

| 日志类型 | 核心问题 | 必填信息 | 最容易缺失的内容 |
| --- | --- | --- | --- |
| 工作日志 | 今天推进了什么 | 任务、变更、验证、下一步 | 为什么这样推进 |
| 决策日志 | 为什么选这个方案 | 备选项、证据、取舍、影响 | 被放弃方案的理由 |
| 实验日志 | 假设是否成立 | 假设、方法、样本、结果 | 负结果和环境参数 |
| 事故日志 | 如何发生和恢复 | 时间线、影响、根因、修复 | 检测缺口和预防措施 |
| 交接日志 | 后续怎么接手 | 当前状态、阻塞、风险、命令 | 真实未完成事项 |

第二个概念是“原子日志”。一条日志应该围绕一个可命名的工程事件，而不是按自然日无限堆叠。自然日适合归档，但不适合检索。比如一天里同时做了检索评测、修了样式、讨论了权限模型，最好拆成三条有标题的日志，再在当天索引中引用。这样未来搜索“权限模型”时，不需要读完整日报。

第三个概念是“证据优先”。日志里每个关键判断都应该尽量指向证据：命令输出、测试结果、监控截图、trace id、PR、issue、用户反馈、代码位置、模型输入输出。证据不一定都要内嵌在正文里，但要能找到。没有证据的内容可以写，但要标记为观察、猜测或待验证。工程日志不怕不确定，怕的是把不确定写成确定。

第四个概念是“AI 可消费”。如果日志只是给人读，可以写得很随意；如果希望未来的 AI 代理能用它，就需要稳定结构。标题要包含对象和动作，字段名要固定，时间要明确，项目名和模块名要统一，代码路径要准确，命令要可复制，结论要和证据分开。AI 很擅长从文本中抽取信息，但前提是文本里真的有信息。结构化不是为了形式，而是为了后续召回、摘要、聚合和迁移。

第五个概念是“日志不是结论仓库，而是推理轨迹”。很多文档只保留最终答案，工程日志要保留从问题到答案的关键路径。比如一次缓存策略调整，最终结论是“保留请求级缓存，暂不加全局语义缓存”。如果只看结论，后来者可能觉得保守。日志应该补充：全局缓存命中率在样本中只有百分之十八，错误复用风险较高，失效策略复杂，当前主要瓶颈在检索重排而不是模型调用，因此先做请求级缓存。这样的推理轨迹能帮助未来判断是否要重新打开这个决策。

## 架构/流程图解说明

一个实用的 AI 工程日志系统，可以按下面的流程运行：

```text
工程活动
  |
  |-- 编码、调研、调试、评审、上线、事故处理
  v
原始材料捕获
  |
  |-- 命令、diff、测试、trace、聊天摘录、会议纪要
  v
人工标记关键事实
  |
  |-- 问题、判断、风险、结果、下一步
  v
AI 整理草稿
  |
  |-- 结构化字段、时间线、摘要、缺失项提醒
  v
人工确认
  |
  |-- 删除推测、补证据、确认结论
  v
日志入库
  |
  |-- Markdown、JSON 索引、全文索引、向量索引
  v
后续消费
  |
  |-- 周报、复盘、交接、RAG、评测样本、决策回顾
```

这条链路里，人类确认是硬边界。AI 可以从 `git diff`、测试输出和聊天摘录中整理日志，但不能自动把“可能是根因”写成“根因已经确认”。如果没有人工确认，日志最多进入草稿状态。这个设计看起来慢一点，但长期会省很多时间，因为错误日志一旦进入知识库，未来检索和 RAG 都会反复引用它。

系统架构上可以分成四个组件：捕获器、整理器、存储层和消费层。

```text
┌──────────────┐       ┌──────────────┐
│ Capture      │ ----> │ Draft Builder│
│ 命令/diff/trace │       │ AI 整理与缺项检查 │
└──────────────┘       └──────┬───────┘
                              |
                              v
                       ┌──────────────┐
                       │ Human Review │
                       │ 人工确认与修订 │
                       └──────┬───────┘
                              |
                              v
┌──────────────┐       ┌──────────────┐       ┌──────────────┐
│ Markdown Log │ <---- │ Log Store    │ ----> │ Consumers    │
│ 可读正文       │       │ 索引与元数据     │       │ 周报/RAG/交接  │
└──────────────┘       └──────────────┘       └──────────────┘
```

捕获器不一定复杂。最简单的版本就是手工复制关键命令和输出，配合 Git 状态、PR 链接、测试结果。更自动的版本可以从本地代理会话、CI、issue 和监控系统收集材料。但无论自动化程度多高，都要避免收集过量噪声。工程日志不是黑盒录屏，过量材料会让人不愿整理，也会让 AI 摘要时淹没重点。

整理器的职责是把材料变成日志草稿。它可以做三件事：抽取事实、组织结构、提示缺失。比如它发现日志里有“修复了问题”，但没有测试命令，就提示补充验证；发现有“根因是缓存”，但没有证据，就标记为待确认；发现提到“后续清理”，就把它放进 follow-up。整理器不要替人下结论，它应该像一个严格的编辑，逼迫日志从含糊变清楚。

存储层最好同时保存 Markdown 和机器可读索引。Markdown 适合人读，也适合在仓库里版本化；JSON 索引适合做搜索、统计和 RAG。二者不要割裂，索引应该从日志 front matter 和结构化段落生成，或者由同一个提交一起更新。这样既保留写作自由，又能让工具稳定消费。

消费层决定日志有没有生命力。日志如果只写不看，很快会变成负担。它应该能被用于周报生成、发布复盘、架构决策回顾、事故预防、AI 代理交接、评测样本挖掘和新人 onboarding。每一种消费场景都会反过来要求日志字段更清楚。比如交接需要当前状态和下一步；事故复盘需要时间线；评测样本挖掘需要输入、期望和失败模式；架构回顾需要备选方案和取舍理由。

## 工程实现

工程日志的实现不必从大系统开始。一个仓库内的 `logs/engineering/` 目录，加上一套约定和几个检查脚本，就能产生价值。关键是格式要稳定，字段要足够表达工程事件。

我会先定义一个 Markdown 模板：

```markdown
---
id: 2026-02-15-rag-index-stale-chunk
date: 2026-02-15
project: knowledge-rag
area: indexing
type: debugging
status: confirmed
related:
  - issue: KB-184
  - pr: 42
tags:
  - rag
  - indexing
  - regression
---

# 修复增量索引遗留 chunk

## 背景
发生了什么，影响谁，为什么现在处理。

## 证据
- 命令、trace、日志、截图、用户反馈。

## 判断
我们如何解释证据，哪些仍然不确定。

## 变更
改了哪些模块，为什么这样改。

## 验证
运行了什么测试，结果如何，还有哪些没有覆盖。

## 风险和后续
剩余风险、清理任务、监控项。
```

这不是为了把每篇日志写成大作文，而是为了让关键字段不要丢。短日志也可以每节只写两三句。真正重要的是：背景、证据、判断、变更、验证、风险都在。未来 AI 读到这篇日志时，可以明确地区分事实和解释。

如果要让工具消费，可以再定义一个 Go 结构：

```go
type EngineeringLog struct {
    ID        string
    Date      string
    Project   string
    Area      string
    Type      string // debugging, decision, experiment, incident, handoff
    Status    string // draft, confirmed, superseded
    Title     string
    Related   []Reference
    Tags      []string
    Sections  LogSections
    Evidence  []EvidenceRef
}

type LogSections struct {
    Background string
    Evidence   []string
    Reasoning  string
    Changes    []ChangeRef
    Validation []ValidationRef
    Risks      []string
    FollowUps  []string
}

type EvidenceRef struct {
    Kind   string // command, trace, test, metric, diff, user_feedback
    Ref    string
    Note   string
}
```

这个结构支持几个实用能力。第一，可以按 `Project` 和 `Area` 过滤，找出某个模块的历史问题。第二，可以按 `Type` 聚合，比如列出过去一个月所有实验日志。第三，可以让 RAG 检索时优先召回 `confirmed` 状态的日志，降低草稿污染。第四，可以抽取 `ValidationRef`，检查某类变更是否长期缺少测试。第五，可以从 `FollowUps` 自动生成待办列表。

AI 整理日志时，可以使用一个明确的工作流，而不是一句“帮我写日志”。例如：

1. 收集材料：当前分支 diff、最近提交、相关 issue、测试输出、人工补充的上下文。
2. 抽取事实：列出已经发生的动作、观察到的错误、运行过的命令和可引用证据。
3. 标记推断：把“可能原因”“怀疑风险”“后续猜测”和已确认事实分开。
4. 生成草稿：填入固定章节，保持具体路径、命令和版本。
5. 缺项检查：检查是否缺验证、风险、后续事项和关联链接。
6. 人工确认：工程师删除不准确内容，补充判断，并把状态改为 `confirmed`。

一个具体实现例子可以这样。假设 AI 代理刚完成一次修复，材料如下：

```text
diff: 修改 indexer/delete.go 和 tests/incremental_index_test.go
test: go test ./internal/indexer -run TestIncrementalDelete
issue: 删除文档后，旧 chunk 仍然能被检索到
observation: vector_store 删除按 document_id 执行，但 chunk 表里 version 已更新
decision: 增加 tombstone 记录，后台 worker 根据 chunk_id 删除向量
```

整理后的日志不应该写成“修复了增量索引 bug”。更有价值的是：

```text
背景：删除文档后，用户仍能在知识库检索到旧片段，影响增量索引一致性。
证据：复现用例中 document_id 相同但 chunk version 已变化，旧向量未被删除。
判断：根因不是检索重排，而是删除路径只覆盖当前版本 chunk，漏掉历史向量。
变更：增加 tombstone 表记录待删除 chunk_id，删除 worker 按 chunk_id 清理向量存储。
验证：新增 TestIncrementalDelete，覆盖删除后无法召回旧 chunk。
风险：历史遗留向量仍需一次性清理，后台 worker 失败需要报警。
```

这段日志能直接帮助未来排查类似问题。它还可以转成评测样本：输入是删除文档后的检索请求，期望是旧 chunk 不可召回；失败模式是索引删除路径不一致。日志不再只是记录，它变成了测试和知识库的来源。

为了防止日志失控，还要给写作设定边界。不是每一次小改动都需要完整日志。我的判断标准是：只要一个事件满足以下任意条件，就值得写日志：改变了架构方向，修复了非平凡 bug，做了有结果的实验，发生了用户可感知事故，做了上线取舍，引入了新的风险，或者需要别人接手。纯格式调整、简单依赖升级、无争议的小改动，可以只靠 commit message。

## 测试评测

工程日志也需要测试。很多人觉得文档无法测试，其实可以测试的是结构完整性、证据覆盖、可检索性和复用价值。日志质量如果完全靠主观感觉，最后一定会回到流水账。

最基础的是格式检查。front matter 必须有 `id`、`date`、`project`、`area`、`type`、`status`，正文必须有背景、证据、判断、变更、验证、风险。日志 ID 要唯一，日期格式要统一，状态只能是允许枚举，关联链接要能解析。这个检查可以在提交前运行，成本很低。

第二层是内容检查。可以让 AI 做一次“审稿”，但审稿规则要明确。它不评价文笔，而是找缺口：是否有结论无证据，是否有“已修复”但没有验证，是否有风险但没有后续，是否把猜测写成事实，是否缺少代码路径或命令，是否存在未来无法理解的代词。AI 审稿适合发现“这段话以后看不懂”，但最终仍然由人确认。

第三层是检索评测。选一组真实问题，测试能否从日志库里找回相关记录。例如“为什么当时没有做全局语义缓存”“增量索引旧 chunk 怎么清理”“模型升级失败的原因是什么”。如果搜索结果召回的是周报而不是具体决策日志，说明标题、标签或正文结构需要调整。日志系统的价值很大程度取决于未来能不能找得到。

第四层是复用评测。拿一条旧日志，让没有参与当时工作的人根据它复现问题、理解决策或继续下一步。如果对方仍然要回头问大量问题，说明日志没有承担交接职责。这个评测不需要频繁做，但对关键模块很有效。

| 评测项 | 自动化程度 | 失败信号 | 修复方式 |
| --- | --- | --- | --- |
| 字段完整 | 高 | 缺日期、类型、状态 | 提交检查 |
| 证据覆盖 | 中 | 判断没有引用 | AI 审稿加人工确认 |
| 验证记录 | 高 | 没有测试命令或结果 | 模板强制 |
| 可检索性 | 中 | 搜不到相关历史 | 调整标题和标签 |
| 可交接性 | 低 | 接手者仍需大量追问 | 补背景和下一步 |
| 防污染 | 中 | 草稿被 RAG 引用 | 状态过滤 |

还有一个指标很实用：日志到行动的转化率。比如一周内日志里出现了十个 follow-up，其中几个变成 issue，几个被明确关闭，几个一直悬空。如果 follow-up 长期悬空，说明日志只是记录焦虑，没有进入工程流程。工程日志要和 issue、测试、监控形成连接，否则它会变成另一个信息孤岛。

AI 写作质量也要评估。可以抽样比较人工原始材料、AI 草稿和确认后的最终日志，记录三类问题：AI 添加了不存在的事实，AI 删除了关键不确定性，AI 把结论写得过于宽泛。每次发现问题，都应该回到提示词和检查规则。比如要求 AI 在草稿里用“已确认”“推测”“待验证”标记判断状态，要求没有证据的句子进入待确认列表，而不是直接进入正文。

## 失败模式

第一种失败是“日志变成日报”。每天写一长段做了什么，但没有问题、证据和判断。这样的日志短期看很勤奋，长期几乎不可检索。解决办法是按事件拆分，标题写对象和动作，例如“修复检索重排超时”而不是“周二工作记录”。

第二种失败是“AI 过度润色”。模型会把粗糙的工程记录改成完整流畅的段落，同时删掉不确定性、命令细节和失败尝试。读起来舒服，复盘时没用。解决办法是把写作目标从“润色”改成“结构化整理”，并要求保留命令、路径、版本、失败结果和开放风险。

第三种失败是“证据污染”。草稿状态的日志被放进知识库，RAG 检索时把未经确认的猜测当成事实。解决办法是状态字段强制过滤，只有 `confirmed` 或明确允许的 `superseded` 日志能进入默认检索；草稿只用于个人继续编辑。

第四种失败是“记录太重”。模板太复杂，大家懒得写，最后只有重大事故才有日志。解决办法是允许短日志。只要六个字段清楚：问题、证据、判断、变更、验证、风险，就可以入库。复杂事件写长文，小事件写短条，保持节奏比追求完美重要。

第五种失败是“只写成功”。实验失败、方案被否、回滚原因都不记录。结果半年后团队重新踩同一个坑。工程日志必须欢迎负结果。负结果的价值不低于成功记录，尤其在模型、检索、性能和产品假设上，失败样本能节省大量重复试错。

第六种失败是“和代码版本断开”。日志说改了某模块，但没有 commit、PR 或文件路径。未来代码变化后，日志无法定位。解决办法是要求日志关联具体引用，至少包含 PR、commit、issue 或关键文件路径。引用不需要很长，但要能把人带回现场。

第七种失败是“没有淘汰机制”。旧日志里的结论已经失效，但仍然被搜索结果排在前面。解决办法是引入 `status: superseded`，并在新日志里引用旧日志，说明哪些结论被替代。知识库不可能永远只增不改，工程日志也需要生命周期管理。

## 上线 checklist

如果要在团队里上线 AI 写作辅助的工程日志，我会按下面清单推进：

1. 定义日志类型枚举：工作、决策、实验、事故、交接。
2. 定义最小模板：背景、证据、判断、变更、验证、风险。
3. 约定状态流转：草稿、已确认、已取代。
4. 明确 AI 边界：AI 可以整理和提示缺项，不能自动确认事实。
5. 建立证据引用规范：命令、trace、测试、PR、issue、监控和用户反馈。
6. 加提交前检查，保证 front matter、标题、必填章节和状态合法。
7. 加 AI 审稿规则，检查无证据结论、缺验证、缺风险和代词不明。
8. 设计目录结构，按项目和月份归档，但保持事件级文件。
9. 建立索引，支持按项目、模块、类型、状态和标签检索。
10. 默认 RAG 只索引已确认日志，草稿必须排除。
11. 把 follow-up 接到 issue 或待办，避免后续事项沉没。
12. 在周会或发布复盘中消费日志，让写日志的人看到回报。
13. 为关键模块建立检索评测，定期检查历史记录能否找回。
14. 设定轻量标准，小事件允许短日志，重大事件要求完整复盘。
15. 建立过期和取代机制，旧结论失效时更新状态并链接新记录。

上线时不要一次要求所有人写长文。可以先从三个场景开始：非平凡 bug 修复、架构决策和 AI 代理交接。这三个场景最容易体现价值，也最容易产生后续复用。一旦团队发现日志能减少重复解释、减少上下文丢失，习惯会自然扩展到实验和事故复盘。

## 总结

AI 写作让工程日志更容易写，也让错误日志更容易大量产生。工程团队需要的不是更多漂亮文字，而是更可靠的过程记忆。好的工程日志把问题、证据、判断、变更、验证和风险连接起来，让未来的人和 AI 都能接住当时的上下文。它不是替代代码、issue 或监控，而是把这些碎片组织成可解释的工程事件。

真正可用的做法很朴素：用稳定模板降低写作成本，用结构化字段支持检索，用 AI 提醒缺项和整理草稿，用人工确认保护事实边界，用状态过滤防止草稿污染知识库，用复盘和交接不断消费日志。长期看，工程日志会变成团队的第二套测试资产。测试告诉我们系统现在能不能跑，日志告诉我们系统为什么变成现在这样。AI 时代的工程效率，不只取决于生成了多少代码，也取决于我们能否把每一次判断留下来，供下一次判断复用。
