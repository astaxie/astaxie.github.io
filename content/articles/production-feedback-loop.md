# 线上反馈闭环

## 问题背景

AI 应用上线后，质量问题不会整齐地出现在测试报告里。它会出现在用户的一句“这个答案不对”、客服群里的一张截图、运营同学的一次人工修正、日志里的一个低置信事件、销售团队的一条抱怨、某个用户没有点击采纳按钮的沉默里。很多团队收集了大量反馈，却没有形成闭环：问题被记录在聊天窗口里，工程师临时看一下 trace，改一版 prompt，几天后类似问题又出现。反馈如果不能变成样本、问题、修复和发布记录，它就只是噪声。

线上反馈闭环的本质是把生产环境里的质量信号转成工程资产。一个反馈不能只停留在“用户不满意”。它要绑定具体 trace，知道用户看到的输入、上下文、模型版本、工具返回和最终输出；它要被归类，知道是事实错误、引用错误、权限错误、工具失败、体验不好还是用户目标不清；它要有负责人，知道应该改检索、改 prompt、改工具、改产品交互还是改数据；它要能进入评测集，确保修过的问题不再回归；它还要进入发布记录，告诉团队这次发布解决了哪些真实问题。

很多 AI 产品早期会把反馈入口做得很轻，只放一个点赞或点踩按钮。这当然比没有强，但远远不够。一个点踩没有上下文，工程师很难行动。用户可能因为答案太长点踩，可能因为引用错点踩，可能因为界面卡顿点踩，可能因为权限不够导致系统拒绝而点踩，也可能只是问题本身不可回答。没有结构化信息，后续只能靠人工猜测。

另一个常见误区是把线上反馈直接拿去训练或直接加入 golden dataset。这样很危险。线上反馈里有隐私数据，有误操作，有用户情绪，有重复问题，有无法复现的临时状态，也有产品策略本来就应该拒绝的请求。反馈进入工程系统之前必须经过清洗、脱敏、归因和标注。闭环不是“把线上数据全量倒进评测”，而是“把有代表性、可复现、能定义期望的反馈沉淀成资产”。

反馈闭环还要和发布节奏对齐。如果一个问题被修复了，但没有对应评测样本，下一次模型升级可能又退回去。如果一个问题进入了评测集，但没有关联 issue 和发布记录，产品和运营不知道修复范围。如果一个反馈被关闭了，但没有说明原因，用户支持团队下次遇到同类问题仍然要重新问工程。工程上真正有价值的闭环，是从反馈产生到发布验证再到知识沉淀的完整链路。

## 核心概念

我通常把线上反馈拆成五层：信号、证据、归因、行动和记忆。信号是用户或系统发出的质量提示；证据是能复盘当时情况的 trace、输入、输出、工具返回和版本信息；归因是对问题类型和根因的判断；行动是 issue、代码修复、配置调整、数据修补或产品改动；记忆是评测样本、运行手册、发布记录和经验文档。

| 层级 | 内容 | 没做好会怎样 | 需要的系统能力 |
| --- | --- | --- | --- |
| 信号 | 点踩、纠正、人工接管、异常、低采纳 | 反馈多但不可解释 | 统一反馈入口和事件采集 |
| 证据 | trace、版本、上下文、工具结果、用户权限 | 无法复现，靠截图猜 | trace 绑定和受控 artifact |
| 归因 | 错误类型、根因、影响范围、严重级别 | 所有问题都被当成 prompt 问题 | 标签体系和 triage 流程 |
| 行动 | issue、owner、修复方案、验证计划 | 改了但不知道改对没有 | 工作流和发布门禁 |
| 记忆 | regression case、runbook、release note | 同类问题反复出现 | 样本库和知识库沉淀 |

反馈对象应该是结构化的。最小字段包括 `feedback_id`、`trace_id`、`user_scope`、`surface`、`rating`、`reason_code`、`free_text`、`created_at`、`model_version`、`prompt_version` 和 `artifact_refs`。其中 `trace_id` 最重要。没有 trace，反馈就是孤岛；有了 trace，工程师可以看到模型输入、检索证据、工具调用、策略判断和最终输出。

归因标签要适度细。太粗会不可行动，例如只有“答案不好”；太细会没人维护，例如几十种相近标签。比较实用的一组标签包括：事实错误、证据缺失、引用不匹配、上下文遗漏、工具参数错误、工具不可用、权限策略错误、输出格式错误、语气不合适、拒绝策略不当、延迟过高、成本异常、用户需求不明确。每个标签都应该有 owner 倾向和默认处理方式。

闭环还需要状态机。反馈不是创建后就结束，它应该经过待分流、已确认、需补证据、已转样本、已建 issue、修复中、已验证、已发布、已关闭几个状态。不同状态需要不同责任人。用户支持负责补充业务现象，工程负责复现和归因，产品负责判断预期行为，质量负责人负责决定是否进入回归集。状态机不是为了流程好看，而是为了避免反馈在聊天记录里丢失。

还有一个核心概念是“反馈的代表性”。线上每个点踩都值得尊重，但不是每个点踩都应该进入核心评测集。评测集要有代表性、可复现、能定义期望、能阻止回归。重复的反馈可以合并成一个问题簇；一次性环境故障可以进入运维复盘，不一定进入模型评测；用户目标不清的问题可能更适合改交互，引导用户补充信息，而不是把模糊请求写成黄金样本。

## 架构/流程图解说明

一套完整的线上反馈闭环，可以从产品入口开始，经过事件归一化、证据绑定、分流标注、样本构造、修复验证和发布记录。它不一定一开始就需要复杂平台，但链路上的对象关系要设计清楚：反馈绑定 trace，trace 绑定版本，反馈簇绑定 issue，issue 绑定修复，修复绑定评测结果，发布绑定用户可读的变更说明。

```text
用户反馈入口
  |
  |-- 点踩、纠正、采纳失败、人工接管、客服备注
  v
反馈事件总线
  |
  |-- 补齐 trace_id、版本、租户、入口、风险等级
  v
反馈仓库
  |
  |-- 原始反馈、脱敏摘要、artifact 引用、状态机
  v
分流与归因
  |
  |-- 标签、严重级别、owner、重复聚类、是否可复现
  v
样本与 issue
  |
  |-- regression case、修复任务、验收标准、影响范围
  v
修复与评测
  |
  |-- 代码、prompt、数据、工具、交互改动，跑回归集
  v
发布与记忆
  |
  |-- release note、runbook、dashboard、关闭反馈簇
```

这里的反馈事件总线可以很轻，早期就是后端接收反馈后写数据库并发一条队列消息。关键是要在写入时补齐上下文，而不是等工程师排查时再找。比如前端点踩时只知道当前消息 ID，后端应该根据消息 ID 查到 trace_id、run_id、model_version、prompt_version、retrieval_index_version、tool_schema_version 和用户权限范围。补齐越早，后面越不容易丢证据。

反馈仓库要把原始内容和可流转摘要分开。原始输入输出可能包含敏感数据，只允许少数人查看；摘要、标签、状态、影响范围和修复记录可以给更多团队协作。不要为了协作便利把完整 prompt、客户数据和工具返回复制到 issue 描述里。更好的方式是 issue 里放脱敏摘要和受控 artifact 链接。

分流与归因最好不要完全自动化。模型可以帮助聚类、推荐标签、生成摘要，但最终归因需要人确认。尤其是权限、合规、业务规则和客户影响，不能只依赖自动分类。一个务实做法是让系统先按规则和模型给出候选：类似反馈已有多少条、涉及哪些版本、是否可复现、可能 owner 是谁；然后由值班工程师或质量负责人确认。

样本与 issue 的分离也很重要。不是每个 issue 都有评测样本，也不是每个样本都对应代码 bug。有些反馈需要改文档，有些需要补数据，有些需要改产品提示，有些只是需要客服解释。闭环系统要允许多种行动类型，但对于真正修复质量缺陷的行动，最好都能留下至少一个回归样本或验收脚本。

## 工程实现

先从数据模型说起。反馈对象不要只是一张 `feedback` 表。它至少需要反馈表、反馈事件表、归因表、聚类表、样本链接表和修复链接表。早期可以都放在一个数据库里，但逻辑边界要清楚。

```sql
create table ai_feedback (
  feedback_id text primary key,
  trace_id text not null,
  run_id text not null,
  surface text not null,
  rating text not null,
  reason_code text,
  free_text text,
  user_scope text not null,
  model_version text,
  prompt_version text,
  index_version text,
  severity text default 'unknown',
  status text default 'new',
  created_at timestamptz not null
);

create table ai_feedback_triage (
  feedback_id text references ai_feedback(feedback_id),
  category text not null,
  root_cause text,
  owner_team text,
  reproducible boolean,
  impact_note text,
  decided_by text,
  decided_at timestamptz
);

create table ai_feedback_links (
  feedback_id text references ai_feedback(feedback_id),
  link_type text not null,
  link_target text not null,
  note text
);
```

这只是骨架。真实系统还要考虑租户隔离、数据保留、权限审计和脱敏策略。`free_text` 不一定能给所有人看，`trace_id` 指向的 artifact 也要做权限控制。反馈系统如果忽略隐私，很快会变成新的数据泄漏面。

反馈入口的设计要让用户少负担，但让系统多拿证据。前端可以提供几个原因按钮，例如“事实不对”“引用不对”“没有解决问题”“太慢”“格式不对”“不该拒绝”。用户可以补充文字，但不是必须。后端收到反馈后，不要只保存按钮值，还要立即关联 trace，并把当前输出、引用列表、工具摘要和版本号写成只读快照。因为后续索引和数据会变，如果不保存当时证据，几天后很难复盘。

一个反馈事件可以长这样：

```json
{
  "feedback_id": "fb_20260303_8ac1",
  "trace_id": "tr_81f2",
  "run_id": "run_91b0",
  "surface": "knowledge_answer",
  "rating": "negative",
  "reason_code": "citation_mismatch",
  "free_text": "引用的制度不是最新版本，答案里说的报销额度也不对。",
  "snapshot": {
    "answer_artifact": "art_answer_773",
    "citation_artifacts": ["doc_policy_2024", "doc_policy_2026"],
    "model_version": "model-a-2026-03",
    "prompt_version": "expense_answer_v14",
    "index_version": "kb_20260303_0900"
  }
}
```

收到这条反馈后，系统可以自动做三件事。第一，检查 trace 中是否同时出现旧制度和新制度，如果出现，标记为可能的检索排序或上下文组装问题。第二，查找最近七天是否有相同 reason_code 和相同知识库领域的反馈，如果数量增加，提升严重级别。第三，生成一个脱敏摘要，进入分流队列。自动化只做辅助，不直接关闭问题。

分流队列需要支持聚类。AI 应用的反馈往往重复度很高，同一个根因可能产生几十条点踩。如果每条都建一个 issue，工程团队会被淹没。可以用规则和向量相似度做初步聚类：同一入口、同一 reason_code、相近 trace 阶段、相同文档或工具、相近用户问题。聚类后人工确认，形成一个 feedback cluster。Cluster 才是通常意义上的待修问题。

```yaml
cluster_id: fc_expense_policy_20260303
title: 报销制度问答引用旧版本
signals:
  count_7d: 37
  negative_rate: 0.42
  affected_surfaces:
    - knowledge_answer
suspected_root_cause:
  stage: retrieval.rerank
  detail: 新制度文档召回但排序低于旧制度
owner_team: knowledge-platform
candidate_actions:
  - 调整制度文档 freshness 特征
  - 将旧制度标记为 superseded
  - 增加冲突文档评测样本
```

从 cluster 转成 issue 时，要写清楚验收标准。不要只写“修复报销制度回答不准”。更好的 issue 描述包括：影响范围、复现 trace、根因假设、预期行为、禁止行为、回归样本路径和发布验证方式。这样工程师接手时不会重新做一遍考古。

把反馈转成评测样本时，最关键的是补齐期望。线上反馈的原始问题可能是“我能报销多少钱”，但评测样本要说明用户所在地区、适用制度版本、时间、候选文档和正确答案约束。否则样本无法长期稳定。样本里也要保留“这个样本来自哪些 feedback_id”，方便将来追溯它的业务价值。

```yaml
id: gd_expense_policy_latest_001
origin:
  feedback_cluster: fc_expense_policy_20260303
  feedback_ids:
    - fb_20260303_8ac1
now: "2026-03-03T15:00:00+08:00"
prompt: "上海员工出差餐补一天能报销多少？"
fixtures:
  retrieval.index:
    documents:
      - doc_policy_2024
      - doc_policy_2026
expect:
  must_use_document: doc_policy_2026
  must_not_use_as_primary: doc_policy_2024
  must_mention:
    - "上海"
    - "出差餐补"
    - "2026 版制度"
  require_citation: true
```

这个样本不是复制用户原话后就结束，而是把可复现条件和正确性约束写清楚。它要求使用新版制度，不允许把旧制度作为主要证据。未来如果索引、重排、prompt 或模型升级导致旧制度重新占优，评测就会失败。这就是反馈变成资产的过程。

修复动作要分类记录。AI 应用质量问题不都是 prompt 问题。事实错误可能来自知识库版本，引用错误可能来自上下文裁剪，工具错误可能来自 schema，拒绝不当可能来自策略，格式错误可能来自后处理，体验差可能来自前端交互。反馈闭环系统应该统计根因分布，避免团队把所有问题都丢给 prompt 工程。

| 根因类型 | 常见修复 | 验证方式 |
| --- | --- | --- |
| 检索召回遗漏 | 更新索引、补元数据、调整查询改写 | 召回集包含正确文档 |
| 重排错误 | 加 freshness 特征、调整权重、增加冲突样本 | top evidence 符合预期 |
| 工具参数错误 | 收紧 schema、增加参数校验、改 planner | trace 中参数断言通过 |
| 权限策略错误 | 修正策略规则、补充确认流程 | 高风险样本触发确认 |
| 输出结构错误 | 加 schema、修后处理、增加格式测试 | artifact schema 通过 |
| 产品交互不清 | 增加澄清问题、调整入口文案 | 用户补充信息率提升 |

上线前要把修复和反馈 cluster 重新关联。发布候选跑评测时，不只看总通过率，还要看本次修复对应的 regression cases 是否通过，相关线上 trace 是否能复现为通过，是否引入新失败。发布后，系统要观察同类反馈是否下降。如果反馈没有下降，说明修复没有打到根因，或者用户入口还有别的问题。

Dashboard 需要分角色。工程师看失败阶段、trace、样本、版本差异；产品看问题类型、影响用户、功能入口、关闭周期；运营看用户可解释信息和 workaround；管理者看趋势、严重问题和发布质量。不要试图用一个大表满足所有人。反馈闭环的难点不是画图，而是让不同角色看到能行动的信息。

还要设计反馈优先级。不是所有负反馈都应该立刻打断开发节奏。一个合理的排序可以综合严重级别、影响用户数、客户等级、是否涉及安全合规、是否有稳定复现、是否已有 workaround、是否阻断关键业务。排序结果要可解释，不能只是一个黑盒分数。比如“高价值客户、合同生成入口、事实错误、已复现、无 workaround”的反馈簇，优先级一定高于“单个用户、语气不喜欢、不可复现”的反馈。

优先级还要和响应承诺绑定。严重问题应该有明确响应时间，例如当天确认、两天内给出修复计划、下一次热修或小版本发布验证。低优先级问题可以进入观察队列，但不能永久沉没。观察队列需要定期回顾，如果同类反馈数量上升，或者影响入口变成核心路径，就自动提升优先级。这样反馈系统既不会被噪声拖垮，也不会错过慢慢扩散的问题。

闭环里还需要保留“符合预期”的处理路径。AI 产品经常会收到用户希望系统越权、绕过审批、访问不可见数据、生成没有依据结论的请求。系统拒绝后用户可能点踩，但这不代表产品应该放开限制。Triage 时要能把这类反馈标成策略符合预期，并生成面向客服或产品的解释。必要时改进拒绝文案或澄清交互，而不是降低安全边界。这个分类很重要，否则负反馈会持续把系统推向更危险的行为。

对于企业内部应用，还要把反馈闭环接入权限和审计。谁查看了原始反馈，谁打开了包含客户信息的 trace，谁把反馈转成样本，谁关闭了高严重问题，都应该有记录。质量系统通常会聚合很多敏感上下文，它本身也需要被当成生产系统治理。不要因为它是“内部工具”就绕过权限设计。

## 测试评测

反馈闭环也要被测试。第一类是采集测试：每个反馈入口都必须带上 trace_id 和消息 ID；后端能补齐版本信息；失败时不会丢反馈。可以在端到端测试里模拟用户点踩，断言数据库里出现反馈记录、snapshot 和 artifact 引用。

第二类是隐私测试。反馈自由文本和 artifact 可能包含客户信息，系统必须验证脱敏策略。比如 issue 摘要里不应该出现原始邮箱、电话号码、访问令牌或完整客户合同。测试可以构造带敏感字段的反馈，确认公开摘要只包含 hash、占位符或受控链接。

第三类是分流测试。给定一组反馈，系统应该按 reason_code、文档、工具、trace 阶段聚类，并允许人工拆分或合并。聚类不需要百分百自动正确，但要保证不会因为字段缺失而完全失效。人工操作也要留下审计记录，方便之后复盘为什么某些反馈被归为同一根因。

第四类是样本生成测试。把一个已确认反馈转成 regression case 后，样本必须通过校验：有时间，有输入，有 fixture，有预期，有来源链接，有禁止行为或必要行为。缺任何关键字段都不能进入核心评测集。否则评测集会慢慢变成一堆不可维护的历史问题。

第五类是闭环验证测试。选择一个反馈 cluster，从创建到建 issue、生成样本、修复、评测通过、发布、关闭，跑一条完整的流程演练。这个测试不一定每天跑，但每次改反馈系统状态机或发布流程时都应该跑。它能发现最实际的问题，例如状态无法流转、权限看不到 artifact、issue 链接丢失、发布记录没有带上反馈 cluster。

衡量反馈闭环的指标不能只看反馈数量。反馈数量上升可能是用户变多，也可能是质量变差，也可能是入口更明显。更有用的指标包括确认率、可复现率、转样本率、平均关闭时间、重复问题回归率、修复后同类反馈下降幅度、严重反馈响应时间。

| 指标 | 含义 | 目标方向 |
| --- | --- | --- |
| 确认率 | 反馈中被确认真实有效的比例 | 稳定，过低说明入口噪声大 |
| 可复现率 | 能通过 trace 或样本复现的问题比例 | 越高越好 |
| 转样本率 | 进入评测集的问题比例 | 不追求过高，追求代表性 |
| 重复回归率 | 已修问题再次出现的比例 | 越低越好 |
| 平均关闭时间 | 从反馈到关闭的周期 | 随严重级别设目标 |
| 修复有效率 | 发布后同类反馈下降比例 | 越高越好 |

还要做反向评测：随机抽取已关闭反馈，检查关闭理由是否充分、是否关联修复或解释、是否有必要沉淀样本。很多闭环系统刚上线时关闭很快，但关闭质量不高。过一段时间后，同类问题又出现，才发现当初只是把反馈标成“已处理”。关闭质量要被抽检，否则状态机会变成形式。

## 失败模式

第一个失败模式是反馈没有 trace。用户点踩了，但系统只保存 rating 和文字，工程师无法知道当时模型看到了什么、检索到了什么、工具返回什么。解决办法是反馈入口必须绑定消息和 trace，后端立即保存快照。

第二个失败模式是所有问题都归因到 prompt。Prompt 确实常改，但很多线上问题来自数据、检索、权限、工具、产品交互和后处理。解决办法是建立根因标签，并在 triage 时要求选择 owner 和验证方式。

第三个失败模式是反馈直接进入评测集。原始反馈没有脱敏，没有稳定 fixture，没有明确期望，进入评测后会造成隐私风险和 flaky。解决办法是设置候选样本状态，只有补齐上下文和预期后才能进入回归集。

第四个失败模式是重复反馈淹没团队。同一根因产生很多点踩，每条都单独处理，工程师忙于整理而不是修复。解决办法是做 feedback cluster，把处理对象从单条反馈提升到问题簇。

第五个失败模式是修复没有发布记忆。工程改了配置，评测也通过，但发布记录没有写明解决了哪个反馈簇，客服和产品不知道如何对用户解释。解决办法是发布模板强制关联 feedback cluster、issue 和 regression case。

第六个失败模式是关闭状态不可信。很多反馈被标成已关闭，但没有验证结果或关闭原因。解决办法是关闭必须包含处理类型：已修复、无法复现、符合预期、重复问题、需要产品决策，并且关键类型要有证据链接。

第七个失败模式是只看负反馈。负反馈重要，但采纳、复制、继续追问、人工覆盖、用户停留时间也能说明质量。解决办法是把显式反馈和隐式行为都纳入信号层，但在进入工程闭环时区分权重。

## 上线 checklist

- 每个反馈入口都能绑定 `trace_id`、消息 ID、模型版本、prompt 版本和入口 surface。
- 反馈写入时会创建当时输出、引用、工具摘要和版本信息的快照。
- 原始内容、脱敏摘要和 issue 可见内容分层存储，敏感 artifact 有权限控制。
- reason_code 和 root_cause 标签有明确说明，且能映射到 owner 团队。
- 反馈状态机包含待分流、已确认、需补证据、已转样本、已建 issue、修复中、已验证、已发布、已关闭。
- 支持重复反馈聚类，并允许人工拆分、合并和保留审计。
- 从反馈转样本必须补齐时间、输入、fixture、预期、来源链接和禁止行为。
- 修复 issue 必须写明复现 trace、根因假设、验收标准和回归样本。
- 发布候选必须跑本次关联的 regression cases，并在发布记录里列出反馈簇。
- 发布后观察同类反馈趋势，未下降的问题重新打开或重新归因。
- Dashboard 按工程、产品、运营分视图，不用一个总分替代行动信息。
- 已关闭反馈定期抽检，检查关闭理由、证据和样本沉淀质量。

## 总结

线上反馈闭环不是在产品里加一个点踩按钮，也不是把用户抱怨堆进表格。它是一条工程链路：信号要绑定证据，证据要支持归因，归因要变成行动，行动要经过评测验证，验证结果要进入发布和长期记忆。链路上任何一段断掉，反馈都会重新变成噪声。

AI 应用的质量改进依赖真实使用，但真实使用的数据必须被谨慎处理。我们要保护隐私，去重聚类，补齐上下文，明确预期，再把有代表性的问题沉淀成 regression case。这样每一次线上不满意都可能变成下一次发布的防线。长期看，反馈闭环会让团队从“被动救火”转向“用生产信号持续校准系统”，这是 AI 产品走向稳定运营的关键能力。
