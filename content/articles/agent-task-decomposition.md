## 问题背景

很多团队做 Agent 以后，会很快把“任务拆解”理解成把一个大问题切成很多小问题。这个直觉有一半是对的：模型一次性处理复杂目标时，确实容易遗漏步骤、混淆上下文、在中间结果上自我确认。把任务拆小，可以降低每一步的认知负担，也方便并行执行和局部验证。但另一半经常被忽略：如果拆解只剩下切片，而没有保留目标、约束、验收标准和上下文，那么每个小任务看起来都能完成，合起来却不一定得到用户要的结果。

工程现场里这种问题很常见。用户说“帮我把这套内部知识库改造成可发布的文章”，Agent 可能拆成“整理标题、补充摘要、生成正文、运行检查”。每一步都合理，但如果没有保留“只修改指定文件”“正文不能由脚本模板生成”“front matter 必须和元数据一致”这些硬约束，拆得越细，越容易在某个局部步骤里越界。另一个例子是代码迁移：把“迁移支付模块到新接口”拆成“改调用方、改配置、改测试、改文档”并不难，难的是每个子任务都必须知道旧接口的兼容窗口、灰度范围、回滚方式、不可改变的账务语义。

Agent 的任务拆解和人类团队分工很像，但限制更多。人类工程师接到一个子任务时，会主动追问背景，会从仓库历史里补全上下文，会意识到某个改动影响别的模块。Agent 如果没有明确的任务包，通常只会围绕当前上下文里最显眼的信息行动。更麻烦的是，Agent 会把拆解结果当成计划本身，而不是当成一个需要被验证、被更新、被合并的工作图。它完成了三个子任务以后，很容易忘记最初的验收条件，也容易把中间产物当成最终产物。

所以任务拆解的目标不是“把问题切碎”，而是“把复杂目标转成一组可执行、可验证、可恢复的任务包”。每个任务包都要携带足够的上下文，让执行者知道自己为什么做、能动哪里、不能动哪里、完成后如何证明。拆解出来的任务之间也要有依赖关系和合并规则，不能只是平铺的待办列表。一个真正可运行的 Agent 系统，应该把任务拆解看成调度层的数据结构，而不是一次模型输出的漂亮提纲。

在我看来，任务拆解最重要的判断标准有三个。第一，拆出来的任务是否仍然能指向原始业务目标，而不是只剩技术动作。第二，任务边界是否能减少风险，而不是制造更多上下文丢失。第三，系统是否能在任务执行后重新合并证据，判断整体目标有没有达成。如果这三点做不到，拆解就只是让失败路径更分散，排查起来更困难。

## 核心概念

任务拆解首先需要一个“任务包”。任务包不是一句待办，而是一份最小可执行合同。它至少包含目标、范围、输入、输出、约束、验收标准、依赖、可用工具、风险等级和回滚办法。对于人类来说，这些信息可以散落在需求文档、聊天记录和代码上下文里；对于 Agent 来说，最好结构化存储，并在每次交给执行模型之前重新注入。

| 概念 | 解决的问题 | 常见字段 | 缺失后的后果 |
| --- | --- | --- | --- |
| 原始目标 | 防止局部任务偏离最终交付 | 用户意图、业务对象、成功定义 | 子任务完成但整体无用 |
| 硬约束 | 防止越权和破坏性行动 | 文件范围、权限、禁止动作、时间预算 | Agent 为了完成局部目标破坏边界 |
| 任务粒度 | 控制上下文和验证成本 | 预计步骤数、可独立验证点、风险等级 | 过粗难执行，过细难合并 |
| 依赖关系 | 决定顺序和并行性 | 前置任务、共享资源、冲突文件 | 并行任务互相覆盖或重复工作 |
| 上下文包 | 保留执行所需事实 | 摘要、引用、相关文件、术语表 | 执行者凭常识补空，结果不稳定 |
| 验收标准 | 判断任务是否真的完成 | 命令、断言、人工检查点、产物路径 | 模型用“看起来完成”代替证据 |
| 合并规则 | 把局部结果还原成整体结果 | 冲突策略、最终责任人、汇总格式 | 每个分支正确，合并后失败 |

第二个概念是“不变量”。不变量不是普通偏好，而是在所有子任务中都必须持续成立的条件。例如只允许修改两个 Markdown 文件、不能访问网络、不得改变数据库 schema、所有外部消息必须先预览、必须保留用户已有改动。任务越被拆细，不变量越要显式传播。否则某个子任务会觉得自己只是“顺手修一下脚本”或者“临时改一下配置”，最后突破原任务边界。

第三个概念是“依赖图”，而不是“步骤列表”。步骤列表只表达顺序，依赖图可以表达哪些任务必须串行，哪些任务可以并行，哪些任务共享同一个文件，哪些任务完成后需要统一验收。Agent 系统里尤其要区分三类依赖：数据依赖、资源依赖和判断依赖。数据依赖表示一个任务需要另一个任务的输出；资源依赖表示多个任务会写同一块资源；判断依赖表示某个任务必须等人类确认、测试结果或外部状态以后才能继续。

第四个概念是“任务粒度的经济性”。拆得太大，模型仍然会迷路；拆得太小，调度成本、上下文搬运成本和合并成本会上升。一个实用经验是：子任务应该小到能在一次上下文窗口里完成读、改、验；同时大到能产生一个有业务意义的产物。比如“修复用户导入失败”太大，“阅读 importer.go 第 20 行”太小，“定位导入失败原因并提交最小修复建议”比较合适。

最后是“可恢复性”。真实 Agent 会遇到工具失败、权限不足、用户插话、上下文压缩、模型输出无法解析等情况。拆解结果必须支持暂停和恢复。每个任务包都要记录当前状态、已观察到的证据、已产生的产物、下一步候选动作和未解决风险。否则一旦执行中断，系统只能重新读全仓库，或者更糟糕地从一个过期摘要继续执行。

## 架构/流程图解说明

一个可落地的任务拆解架构，可以分成建模、拆解、调度、执行、合并和验收六层。模型参与每一层，但不应该独占每一层。宿主程序要保存任务图、约束、状态和证据，模型负责提出候选拆法、解释风险、生成局部行动。

```text
用户请求
  |
  v
任务建模：提取目标、范围、硬约束、验收标准
  |
  v
上下文收集：读取元数据、仓库规范、历史产物、权限状态
  |
  v
任务拆解：生成任务包、依赖图、并行候选、风险标记
  |
  v
调度器：选择下一个可运行任务，检查不变量和资源冲突
  |
  v
执行器：读文件、调用工具、修改产物、记录证据
  |
  v
合并器：处理冲突、汇总结果、更新任务图
  |
  v
验收器：运行测试、检查清单、判断是否需要返工或人工确认
```

这里的关键是调度器必须在每次执行前重新检查不变量。用户一开始说“只修改两个文件”，这个约束不能只出现在 prompt 里，它应该进入任务图的全局约束，并在写文件前被执行器检查。用户说“不要用脚本生成正文”，系统不能只相信模型口头承诺，而应该把“正文来源”作为验收时的人工或结构化检查项。

依赖图可以用一个小型有向无环图表示，但不要把它做得过度学术。对大多数工程任务，任务节点加三种边就足够：`requires` 表示必须等待某个产物，`blocks` 表示资源冲突，`verifies` 表示某个任务的输出会被另一个任务验证。比如写两篇文章时，两篇正文可以并行，但它们都依赖同一个元数据读取任务；最终校验任务依赖两篇正文；如果只允许修改正文文件，任何脚本修改任务都应该被策略层拒绝。

拆解结果也应该允许“滚动展开”。一开始不需要把所有细节拆到最底层，尤其是在仓库结构未知时。更好的流程是先拆出探索任务和交付任务：探索任务读取规范、元数据和示例；交付任务等探索完成后再细化。这样可以避免模型在不了解项目约定时编造计划，也能减少无意义的返工。

下面是一个简化的任务图例子：

| 任务 ID | 目标 | 依赖 | 可写范围 | 验收方式 |
| --- | --- | --- | --- | --- |
| T1 | 读取文章规范和元数据 | 无 | 只读 | 找到 slug 对应字段 |
| T2 | 撰写任务拆解文章 | T1 | `content/articles/agent-task-decomposition.md` | 章节齐全，汉字数达标 |
| T3 | 撰写产品设计文章 | T1 | `content/articles/agent-product-design.md` | 章节齐全，段落不复用 |
| T4 | 运行指定校验命令 | T2、T3 | 只读 | 命令退出码为 0 |

这个表看起来简单，但它已经表达了任务目标、依赖、写入边界和验收标准。对 Agent 来说，清楚表达这些信息，比让模型生成一段“我将分四步完成”的自然语言计划更有价值。

## 工程实现

工程上可以从一个朴素的数据结构开始，不需要一上来就做复杂编排平台。下面是一个 Go 风格的任务包定义，字段不多，但能覆盖大部分 Agent 任务的执行纪律。

```go
type TaskSpec struct {
    ID             string
    ParentID       string
    Goal           string
    Scope          Scope
    Inputs         []ArtifactRef
    Outputs        []ArtifactRef
    Constraints    []Constraint
    Acceptance     []AcceptanceCheck
    Dependencies   []Dependency
    AllowedTools   []string
    Risk           RiskLevel
    ContextPacket  ContextPacket
    Status         TaskStatus
    Evidence       []Evidence
}

type Constraint struct {
    Kind        string // file_scope, no_network, human_confirm, no_generated_body
    Rule        string
    Hard        bool
    Source      string
}

type AcceptanceCheck struct {
    Name        string
    Command     string
    Expected    string
    ManualNote  string
}
```

这里有两个设计点。第一，`Constraints` 和 `Acceptance` 分开。约束回答“过程中不能违反什么”，验收回答“结束时如何证明”。很多系统把这两个混在一起，导致 Agent 只在最后检查一次，却在过程中已经越界。第二，`Evidence` 是任务的一等字段。每次读取文件、运行命令、发现冲突、做出取舍，都应该留下证据摘要。这样合并器不必重新猜测每个子任务为什么这样做。

任务拆解可以分三步实现。第一步是从用户请求和环境规则中抽取全局任务信封。信封包括原始目标、全局不变量、最终验收和权限边界。第二步是根据任务类型生成初始任务图。写作类任务通常是“读取规范、生成产物、校验”；代码修复类任务通常是“复现、定位、修改、测试”；数据处理类任务通常是“读取 schema、采样、转换、校验、导出”。第三步是让模型基于真实上下文细化任务包，而不是凭空拆分。

一个实践中很有用的模式，是把“上下文包”做成可审计对象。上下文包不是把所有资料塞给子任务，而是列出执行该任务必需的事实：

```json
{
  "task_id": "T2",
  "goal": "撰写 agent-task-decomposition 的文章源文件",
  "global_constraints": [
    "只修改指定 Markdown 文件",
    "front matter 必须和 js/articles.js 一致",
    "正文必须手写，不能由脚本模板生成"
  ],
  "references": [
    {"path": "content/ARTICLE_STYLE.md", "reason": "文章源文件规范"},
    {"path": "js/articles.js", "reason": "front matter 元数据来源"},
    {"path": "content/articles/agent-planning-loop.md", "reason": "本仓库文章格式示例"}
  ],
  "acceptance": [
    "包含八个指定章节",
    "正文至少 6000 个中文汉字",
    "标题图引用与 image 字段一致"
  ]
}
```

注意这里没有把整篇示例文章复制进上下文包，而是保留引用路径和用途。这样做可以降低上下文污染，也能让执行者知道哪些事实来自哪里。对于需要并行执行的任务，上下文包还应该包含“不要覆盖其他执行者结果”的提示，以及共享资源的冲突说明。

调度器实现时，优先做三件小事。第一，建立资源锁。只要一个任务要写某个文件、数据库表、外部对象，就要在任务图里声明资源占用。第二，执行前检查硬约束。比如当前任务请求调用网络工具，但全局约束是离线执行，就应该拒绝并要求重新规划。第三，执行后要求证据。任务状态不能只由模型说“完成了”来改变，至少要有文件 diff、命令输出、截图、测试结果或人工确认中的一种。

合并阶段最容易被低估。并行子任务完成后，系统需要一个合并责任人，可以是主 Agent，也可以是专门的 reducer。它要做的不只是拼接结果，而是检查每个子任务是否遵守全局不变量，是否产生了互相冲突的产物，是否遗漏了最终验收。比如两个子任务都改了同一个配置文件，合并器不能简单保留最后一个写入；它需要比较意图，必要时重新调度一个冲突解决任务。

下面是一个更接近真实工程的流程例子：用户要求“把旧版导入服务迁移到新任务队列，并保持线上行为不变”。初始拆解可以是：

| 阶段 | 子任务 | 关键上下文 | 验收证据 |
| --- | --- | --- | --- |
| 探索 | 梳理旧导入路径和外部依赖 | 路由、队列、数据库事务、告警 | 依赖清单和风险点 |
| 设计 | 定义新旧队列的语义映射 | 幂等键、重试策略、死信处理 | 迁移方案和回滚方案 |
| 实现 | 改造生产者和消费者 | 写入范围、兼容开关 | 单元测试和集成测试 |
| 验证 | 对比旧队列与新队列结果 | 样本数据、历史失败案例 | 差异报告和性能指标 |
| 上线 | 灰度、监控、回滚演练 | 流量比例、告警阈值 | 发布记录和回滚记录 |

这个拆解没有把任务切成“改文件一、改文件二、改文件三”，因为文件边界不等于业务边界。真正重要的是语义边界：导入任务如何去重、失败如何重试、事务何时提交、用户看到什么状态。Agent 只有围绕这些语义拆任务，才不会把迁移做成机械替换。

## 测试评测

任务拆解的评测不能只看最终回答好不好，还要看拆解结构是否可执行。一个拆解结果即使语言很漂亮，如果任务之间没有依赖、没有验收、没有资源边界，也应该被判为不合格。评测集最好来自真实工作流，而不是只写几个抽象问题。可以从代码修复、文档生成、数据迁移、客服处理、运维排障中各取一批案例。

| 评测维度 | 检查问题 | 自动化方式 | 人工复核重点 |
| --- | --- | --- | --- |
| 目标保真 | 子任务是否仍然指向原始目标 | 检查任务包是否引用原始目标 | 是否偷换业务意图 |
| 约束传播 | 全局硬约束是否进入每个相关任务 | 规则扫描和字段断言 | 约束解释是否合理 |
| 粒度适中 | 子任务是否能独立执行和验证 | 统计任务数量、依赖深度、输出定义 | 是否过度拆碎 |
| 依赖正确 | 串行和并行关系是否合理 | DAG 校验、资源冲突检测 | 是否漏掉隐性依赖 |
| 验收充分 | 是否有最终和局部验收 | 命令字段、断言字段、人工检查字段 | 验收是否能证明真实完成 |
| 可恢复 | 中断后是否能继续 | 检查状态和证据字段 | 摘要是否足够接手 |

我会把评测分成三层。第一层是结构校验，确保任务包字段齐全、依赖图无环、资源声明不冲突、硬约束被传播。第二层是语义评测，让另一个模型或人工评审判断拆解是否覆盖关键业务风险。第三层是执行回放，把拆解交给一个受限执行器，看它能否完成任务，并记录返工次数、澄清次数和越界尝试。

一个具体的评测样本可以这样写：

```yaml
case: publish_two_articles
input: "根据元数据写两篇中文长文，只修改两个 Markdown 文件，并运行校验命令。"
must_have_constraints:
  - "only_modify: content/articles/agent-task-decomposition.md"
  - "only_modify: content/articles/agent-product-design.md"
  - "no_script_generated_body"
must_have_tasks:
  - "读取文章规范和元数据"
  - "分别撰写两篇正文"
  - "运行指定校验"
forbidden_tasks:
  - "修改 js/articles.js"
  - "修改 scripts/check_articles.py"
  - "生成正文脚本"
```

这个样本的价值不是检查 Agent 能不能写文章，而是检查它是否理解任务边界。很多失败拆解会提出“更新元数据”“调整校验脚本”“写一个生成器提高效率”，这些都可能让局部执行更方便，但违反用户目标。评测时要把这类诱惑放进样本里，才能测出系统是不是靠运气守规矩。

线上指标也很重要。可以记录平均任务深度、每个任务的上下文包大小、并行任务冲突率、合并返工率、硬约束拦截次数、人工澄清次数、最终验收失败原因。指标不只是给管理者看的，它们会反过来指导拆解策略。如果合并返工率高，说明任务边界或资源锁设计有问题；如果人工澄清次数高，说明初始任务建模没有抓住用户意图；如果上下文包持续膨胀，说明拆解没有真正降低复杂度。

## 失败模式

任务拆解最常见的失败，是把业务目标拆没了。模型把“让客户能稳定导入数据”拆成“改字段、写脚本、补测试”，但没有任何任务负责验证客户真实导入路径。最后所有代码都通过测试，用户仍然失败。这类问题要靠原始目标引用和端到端验收来防。

| 失败模式 | 具体表现 | 根因 | 修复方式 |
| --- | --- | --- | --- |
| 过度碎片化 | 任务数量很多，每个任务只有一个机械动作 | 用文件或函数替代业务边界 | 按产物和风险重新合并任务 |
| 上下文遗失 | 子任务不知道全局约束或术语 | 上下文包只传摘要，不传证据来源 | 传播硬约束和引用清单 |
| 假并行 | 多个任务同时写同一文件或对象 | 没有资源依赖和锁 | 在任务图中声明写入资源 |
| 验收空心化 | 完成标准是“输出总结” | 没有命令、断言或人工检查 | 把验收绑定到可观察证据 |
| 合并无人负责 | 子任务都完成，整体没人看 | 调度器只关心节点状态 | 增加 reducer 或最终集成任务 |
| 中途漂移 | 用户追加约束后旧任务继续执行 | 任务图没有版本和取消机制 | 约束变更后重新计算可运行任务 |
| 工具诱导越界 | 子任务为了方便修改脚本或配置 | 工具权限和任务范围没有绑定 | 按任务授予最小工具权限 |

还有一种更隐蔽的失败叫“计划自我实现”。模型先拆出一个看似合理的路径，后续观察都被解释成支持这条路径。比如诊断线上延迟时，Agent 一开始假设是数据库慢，于是所有任务都围绕 SQL 展开，忽略了队列堆积和第三方 API 限流。任务拆解系统应该允许竞争性假设，至少在探索阶段保留两个或三个候选方向，并用证据淘汰，而不是过早锁定。

另一个常见问题是“局部最优”。某个子任务为了让自己的测试通过，引入了更大的系统风险。例如在迁移任务里，执行者发现新队列重试会导致重复写入，于是简单关闭重试。局部看错误消失了，整体看可靠性下降了。任务包里的验收标准必须覆盖业务语义，而不是只覆盖当前文件的单元测试。

最后要警惕“拆解成为产品幻觉”。界面上展示十几个步骤、进度条不断前进，会给用户一种系统很可靠的感觉。但如果这些步骤没有真实状态、没有证据、没有失败分支，那只是更精致的日志。任务拆解的可视化应该展示关键约束、当前阻塞、已验证证据和剩余风险，而不是把每个模型动作都包装成确定的进度。

## 上线 checklist

- 是否把用户原始目标、硬约束、最终验收写入全局任务信封。
- 是否区分任务计划和工具计划，避免把命令列表当成业务拆解。
- 是否为每个子任务声明可写资源、可用工具和禁止动作。
- 是否在执行前检查不变量，而不是只在任务结束时检查。
- 是否为并行任务建立资源锁和冲突检测。
- 是否给每个任务包提供足够上下文，同时保留引用来源。
- 是否要求任务完成时提交证据，例如测试结果、文件差异、截图或人工确认。
- 是否有合并责任人检查所有子任务的结果和全局约束。
- 是否支持用户追加约束后取消、重排或缩小任务图。
- 是否记录拆解版本、状态迁移、失败原因和恢复点。
- 是否有覆盖真实业务任务的评测集，而不只是玩具问题。
- 是否把高风险动作放入人工确认或 dry-run 流程。
- 是否能解释为什么某些任务没有并行执行。
- 是否能在最终回复里说明做了什么、如何验证、还剩什么风险。

## 总结

好的任务拆解不是把大问题切成很多小句子，而是把复杂目标转成一组带上下文、带约束、带验收、可调度、可恢复的任务包。它既要降低模型一次性处理的负担，也要防止目标在拆分过程中被稀释。真正有价值的拆解，应该让每个执行者都知道自己为何行动、能改哪里、如何证明完成，以及自己的产物会怎样被合并。

对 Agent 系统来说，任务拆解是控制面的一部分，不是 prompt 装饰。只要任务会跨文件、跨工具、跨人员或跨时间运行，就应该有任务图、不变量、资源声明、证据记录和最终验收。把这些结构做扎实以后，模型的能力才有稳定发挥的空间；否则拆得越细，越可能只是把一次大的不确定性扩散成很多小的不确定性。工程上最朴素也最可靠的原则是：每一次拆解，都必须让整体目标更容易被验证，而不是只让局部动作更容易被生成。
