## 问题背景

很多团队第一次把 LLM 功能接进产品时，Prompt 往往只是代码里的一个字符串。它可能写在 handler 里，可能放在配置表里，也可能散落在运营后台。上线初期大家改得很快：今天把“请简洁回答”改成“请分点回答”，明天加一句“不要编造”，后天又因为一个客户投诉临时补一段边界说明。每一次修改看起来都很小，最后却会变成一个没有变更记录、没有责任人、没有回归样本的隐形生产系统。

Prompt 的特殊性在于，它既像代码，又不像代码。说它像代码，是因为它改变系统行为，能引入回归，能造成安全问题，能影响成本和延迟；说它不像代码，是因为它的结果不是确定的，模型版本、上下文内容、检索结果、工具返回和温度参数都会放大 Prompt 变化。只看一两个例子就合并 Prompt 修改，等于在没有测试的情况下改业务规则。

我见过最常见的事故是“善意修复造成横向回归”。例如客服总结功能为了减少道歉话术，在系统提示词里加入“不要使用过度客套语言”。这个变化解决了某一类中文客服对话的问题，却让英文场景里的品牌语气变得生硬，还让高风险投诉总结少了必要的安抚标记。另一个例子是代码生成助手为了避免长输出，加了“只返回最终代码”，结果调试任务里模型不再解释推理路径，用户无法判断补丁是否可信。Prompt 没有版本和评测，这些回归通常要等线上用户反馈才暴露。

Prompt 版本管理不是把提示词塞进 Git 就结束。真正要管理的是一组运行时契约：这个 Prompt 服务哪个任务，依赖哪些输入变量，绑定什么模型和参数，允许哪些工具，输出必须符合什么结构，评测样本覆盖哪些业务风险，变更时谁来评审，线上如何灰度，出问题如何回滚。Prompt 文本只是其中最显眼的一部分。

还有一个容易忽略的问题：Prompt 的消费者不一定只有模型。产品、客服、风控、法务、数据同学都可能关心同一段 Prompt。Prompt 里的业务口径如果只存在代码仓库，非工程团队很难参与审阅；如果只存在协作文档，工程系统又很难保证运行时真的使用了被批准的版本。版本管理要把“可读”和“可执行”合在一起，避免出现文档里是一套、线上是一套、评测又是另一套。

Prompt 也会形成依赖关系。一个分类 Prompt 的标签集合会影响后续检索策略；一个摘要 Prompt 的字段命名会影响下游结构化解析；一个工具调用 Prompt 的语义边界会影响权限控制。如果上游 Prompt 变了，下游流程不一定立即失败，反而可能悄悄改变数据分布。没有版本和血缘关系，团队很难回答“这次转化率下降是不是某个 Prompt 变更造成的”。

因此，Prompt 版本管理的目标不是制造流程负担，而是把 Prompt 从“谁都能改的文案”升级为“可审计、可评测、可回滚的行为配置”。它要让工程团队在快速迭代和稳定上线之间找到平衡：小改动可以快速验证，大改动必须有样本和灰度；实验可以灵活，生产必须有账本；人可以写自然语言，但系统要保存结构化元数据。

## 核心概念

第一个核心概念是 Prompt Artifact。不要把 Prompt 当成孤立文本，而要把它包装成一个带元数据的构件。一个构件至少包含：唯一标识、语义版本、适用任务、模板正文、变量 Schema、输出 Schema、模型配置、工具权限、评测集引用、变更说明和发布状态。这样系统才能知道这段 Prompt 在什么条件下可用，也能在回放历史请求时准确还原当时的运行环境。

第二个概念是语义版本。Prompt 的 `1.2.3` 不必完全照搬库版本，但应该表达行为兼容性。我的经验是：修正错别字、改善措辞但不改变输出契约，可以升 patch；改变少量决策边界、增加限制、优化字段描述，可以升 minor；改变任务定义、标签集合、输出结构、工具使用规则，必须升 major。版本号不是仪式，它让调用方知道是否需要重新评测和适配。

第三个概念是变更意图。很多 Prompt diff 都很难用纯文本判断风险，因为自然语言里的一个词可能影响很大。每次提交都应该写清楚变更意图，例如“降低幻觉引用”“提高召回覆盖”“减少输出长度”“适配新标签”“修复日文场景”。评审时先看意图，再看 diff，最后看评测结果。没有意图的 Prompt 修改不应该直接进入生产。

第四个概念是回归样本。Prompt 评测样本不是随机聊天记录的堆积，而是对业务风险的编码。样本要覆盖高频路径、边界输入、历史事故、对抗输入、多语言、长上下文、缺失字段、噪声检索、工具失败等场景。每条样本都要有判定方式：可以是精确 JSON 断言，可以是规则校验，可以是人工打分，也可以是模型辅助评审，但必须写明为什么这条样本重要。

第五个概念是发布通道。开发版本、候选版本和生产版本要分开。开发版本可以频繁改，候选版本必须跑评测，生产版本必须可回滚。不要让业务后台的“保存”按钮直接改变所有线上流量。更稳妥的方式是让后台生成候选版本，经过评测和审批后绑定到路由规则或灰度策略。

第六个概念是运行时指纹。一次 LLM 调用应该记录 Prompt 版本、模型版本、参数、输入摘要、检索批次、工具版本和输出校验结果。出现线上问题时，指纹能帮助我们复现当时的行为。没有指纹，团队只能在最新 Prompt 上猜老问题，最后经常得出错误结论。

| 管理对象 | 常见坏做法 | 推荐做法 | 风险收益 |
| --- | --- | --- | --- |
| Prompt 正文 | 写在代码字符串里，热修时直接改 | 独立构件，提交 diff，保留历史 | 可审计，可回滚 |
| 输入变量 | 模板里临时拼接 | 用 Schema 描述字段、类型和空值策略 | 降低注入和缺字段风险 |
| 输出格式 | 靠一句“请返回 JSON” | 绑定输出 Schema 和解析器 | 减少下游解析事故 |
| 评测样本 | 上线前手测三条 | 维护分层样本集和阈值 | 发现横向回归 |
| 发布 | 保存即生效 | 候选、灰度、生产三态 | 控制影响范围 |
| 观测 | 只看成功率 | 记录版本指纹、错误类型和质量指标 | 支持追溯和诊断 |

Prompt 版本管理还要处理“文本”和“策略”的边界。系统提示词里不应该无限塞业务逻辑。有些规则更适合写成前置校验、后置过滤、工具权限或路由策略。例如“用户未登录不能查询订单”应该由权限系统控制，而不是只在 Prompt 里提醒模型；“输出金额必须保留两位小数”可以由格式化函数保证，而不是指望模型永远听话。版本管理的价值之一，就是逼我们把自然语言规则拆到合适的工程层。

## 架构/流程图解说明

一个可落地的 Prompt 版本管理系统，可以拆成五层：仓库层、评测层、发布层、运行时层和观测层。仓库层负责保存构件，评测层负责判断候选版本是否可用，发布层负责把版本绑定到环境和流量，运行时层负责解析模板并调用模型，观测层负责把结果和版本指纹写回分析系统。

```text
作者提交 Prompt 变更
  |
  v
Prompt 仓库
  |-- 模板正文
  |-- 变量 Schema
  |-- 输出 Schema
  |-- 模型参数
  |-- 变更说明
  |
  v
候选版本生成
  |
  v
评测流水线
  |-- 静态检查
  |-- 样本回放
  |-- 结构校验
  |-- 成本延迟统计
  |-- 人工抽检
  |
  v
发布决策
  |-- 拒绝
  |-- 灰度
  |-- 全量
  |-- 回滚
  |
  v
运行时解析与调用
  |
  v
日志、指标、问题追踪
```

这里最容易被低估的是静态检查。静态检查不需要调用模型，却能拦下一批低级事故。例如模板变量是否都在 Schema 里声明，输出 Schema 是否和示例一致，是否引入了未批准的工具名称，是否包含内部密钥或调试文本，是否把用户输入放进了系统级指令位置，是否删除了必须出现的安全边界。静态检查成本低，应该放在候选版本生成之前。

评测流水线则要分层。第一层是确定性检查：模板能否渲染，输出能否解析，字段是否齐全，长度是否超过上限。第二层是规则评估：分类标签是否在枚举里，引用是否来自给定材料，敏感动作是否被拒绝。第三层是模型辅助评估：摘要是否覆盖重点，回答是否有帮助，语气是否合规。第四层是人工抽检：高风险 Prompt 每次 major 变更都应该有人看样本输出，而不是完全交给自动分。

发布层要支持环境绑定。开发环境可以使用最新候选版本，预发环境使用即将上线版本，生产环境使用已批准版本。更细一点，生产环境还可以按租户、语言、任务类型、用户分桶绑定不同版本。比如 `support_summary@2.4.1` 只给中文客服灰度 10%，英文客服仍然使用 `2.3.7`。当质量指标下降时，发布系统要能把绑定切回旧版本，而不是让工程师临时改配置。

运行时层要避免“最后一公里漂移”。如果运行时从数据库读取 Prompt，再在代码里追加一段系统指令，日志里却只记录数据库版本，那么版本指纹就是假的。所有会影响模型行为的内容都要进入构件或者进入调用指纹。包括默认系统规则、上下文压缩策略、工具说明、输出修复提示、重试 Prompt，都要有版本。否则回放时会发现同一个 Prompt 版本在不同服务里行为不一致。

一个简单的状态图可以这样描述：

```text
draft
  -> candidate
  -> evaluated_passed
  -> staged
  -> canary
  -> production
  -> deprecated

candidate -> evaluated_failed -> draft
canary -> rolled_back -> staged
production -> hotfix_candidate -> evaluated_passed
```

这个状态图的重点不是状态名，而是每条边都应该有条件。`candidate -> evaluated_passed` 需要评测报告，`staged -> canary` 需要发布审批，`canary -> production` 需要观察窗口，`production -> hotfix_candidate` 需要事故编号或明确原因。Prompt 系统一旦进入多人协作，没有状态机就会靠聊天记录传递事实，迟早丢上下文。

## 工程实现

工程实现可以从一个轻量的数据结构开始，不必一开始就做复杂平台。下面是一个面向 Go 服务和通用 Web 后台的 Prompt 构件示例。重点是把会影响行为的内容收敛到一个对象里，而不是分散在代码、配置和文档中。

```yaml
slug: support_summary
version: 2.4.1
status: candidate
owner: ai-platform
task: summarize_support_ticket
locale:
  - zh-CN
model:
  provider: openai
  name: gpt-5.4-mini
  temperature: 0.2
  max_output_tokens: 900
input_schema:
  type: object
  required:
    - ticket_title
    - messages
    - customer_tier
  properties:
    ticket_title:
      type: string
    messages:
      type: array
      items:
        type: object
        required:
          - role
          - content
    customer_tier:
      type: string
      enum:
        - free
        - pro
        - enterprise
output_schema:
  type: object
  required:
    - summary
    - risk_level
    - next_actions
  properties:
    summary:
      type: string
      maxLength: 500
    risk_level:
      type: string
      enum:
        - low
        - medium
        - high
    next_actions:
      type: array
      items:
        type: string
template: |
  你是客服工单总结助手。请只依据给定消息生成总结，不要补充未出现的事实。
  如果客户明确要求退款、投诉升级、法律处理或媒体曝光，将 risk_level 标为 high。
  工单标题：{{ticket_title}}
  客户等级：{{customer_tier}}
  对话消息：
  {{messages}}
eval_sets:
  - support_summary_regression_v3
  - safety_refusal_cn_v2
change_note: "减少低风险工单里不必要的升级动作，补充企业客户高风险判断。"
```

这个对象进入仓库后，要经过四个工程步骤。第一步是规范化。系统把模板去掉无意义空白，把模型参数补齐默认值，把 Schema 排序，把标签标准化，然后生成内容哈希。内容哈希用于判断两个版本是否实际一致，也用于幂等发布。第二步是静态检查。检查变量是否完整、模板是否引用未声明字段、输出字段是否和解析器兼容、标签是否符合站点约定。第三步是评测回放。系统从样本库读取输入，使用候选 Prompt 调用模型，保存输出和评分。第四步是产出发布记录。发布记录不只写“通过”，还要写评测集、样本数量、质量指标、成本变化、延迟变化和审批人。

在服务端，我通常会把 Prompt 运行时封装成一个显式接口，而不是让业务代码自己拼接字符串：

```go
type PromptVersion struct {
    Slug          string
    Version       string
    Status        string
    Template      string
    InputSchema   json.RawMessage
    OutputSchema  json.RawMessage
    Model         ModelConfig
    Tools         []ToolPolicy
    ContentHash   string
}

type RenderedPrompt struct {
    Slug        string
    Version     string
    Messages    []Message
    Model       ModelConfig
    Fingerprint string
}

type PromptStore interface {
    Resolve(ctx context.Context, slug string, env string, attrs map[string]string) (PromptVersion, error)
}

type PromptRenderer interface {
    Render(ctx context.Context, version PromptVersion, input any) (RenderedPrompt, error)
}
```

业务代码只传任务名和结构化输入，不直接知道模板细节：

```go
version, err := promptStore.Resolve(ctx, "support_summary", "prod", map[string]string{
    "locale": "zh-CN",
    "tenant": tenantID,
})
if err != nil {
    return err
}

rendered, err := renderer.Render(ctx, version, SupportSummaryInput{
    TicketTitle: ticket.Title,
    Messages: messages,
    CustomerTier: tier,
})
if err != nil {
    return err
}

resp, err := llm.Invoke(ctx, rendered.Messages, rendered.Model)
```

这个封装带来一个好处：所有调用都会天然带上版本指纹。日志里可以记录 `prompt_slug`、`prompt_version`、`content_hash`、`model`、`eval_gate`、`release_channel` 和 `output_schema_version`。当线上报警出现时，不需要翻代码判断哪段 Prompt 生效。

样本库的设计也很关键。不要只保存输入和理想输出，还要保存样本意图和判定器。比如：

```json
{
  "id": "support_cn_refund_017",
  "task": "summarize_support_ticket",
  "risk": "refund_escalation",
  "input": {
    "ticket_title": "连续扣费后要求退款",
    "customer_tier": "pro",
    "messages": [
      {"role": "customer", "content": "我已经取消订阅还被扣费，请今天退款，否则我要投诉。"},
      {"role": "agent", "content": "我们会帮你核实账单。"}
    ]
  },
  "assertions": [
    {"type": "json_field_equals", "path": "$.risk_level", "value": "high"},
    {"type": "contains_any", "path": "$.next_actions", "values": ["退款", "账单", "升级"]},
    {"type": "not_contains", "path": "$.summary", "value": "已经退款"}
  ]
}
```

这个例子说明，评测不一定要追求唯一答案。对于摘要任务，完全一致的金标反而会让评测脆弱。更实用的方式是检查关键事实、风险标签、禁止内容和结构约束。模型输出可以有表达差异，但不能漏掉退款风险，也不能把“核实账单”编造成“已经退款”。

版本差异展示也要做得适合 Prompt。普通文本 diff 只能显示增删行，不一定能解释行为影响。更好的评审页应该同时展示：模板 diff、变量 Schema diff、输出 Schema diff、模型参数 diff、工具权限 diff、评测指标 diff 和失败样本列表。评审者不需要从一大段自然语言里猜风险，而是直接看到“新增了 high 风险条件”“输出字段 `confidence` 被删除”“平均输出 token 增加 18%”“日文样本失败 4 条”。

对于存储，早期可以直接把 Prompt 文件放在仓库中，用 CI 跑检查和评测；当业务团队参与变多后，可以增加后台，但后台最终仍然生成可审计的版本对象。不要让数据库成为没有历史的黑盒。即使使用数据库，也要有不可变版本表和发布绑定表。一个版本一旦进入候选或生产，就不要原地修改；修复应该生成新版本。

发布绑定表可以长这样：

| 字段 | 含义 |
| --- | --- |
| binding_id | 发布绑定唯一标识 |
| prompt_slug | 任务名 |
| version | 绑定版本 |
| environment | dev、staging、prod |
| selector | 租户、语言、用户分桶等条件 |
| traffic_percent | 灰度比例 |
| started_at | 生效时间 |
| ended_at | 结束时间 |
| operator | 操作人 |
| reason | 发布原因 |

有了绑定表，回滚就是新增一条指向旧版本的绑定，而不是删除历史。这样事后复盘可以看清楚某个时间窗口到底哪个版本服务了哪些流量。

## 测试评测

Prompt 测试要承认一个现实：模型输出有波动，测试目标不是让每个字固定，而是让关键行为稳定。测试体系可以分成五类。

第一类是渲染测试。它检查模板在各种输入下能否正确生成消息。字段缺失时应该返回明确错误，数组为空时应该有空值策略，长文本应该触发截断或压缩，而不是把模板变量原样发给模型。渲染测试通常是确定性的，适合放进普通单元测试。

第二类是结构测试。它检查模型输出是否能被解析，是否符合 JSON Schema，是否包含必填字段，枚举值是否有效，字符串长度是否越界。结构测试对生产系统很重要，因为很多事故不是回答差一点，而是下游解析失败导致整个流程中断。

第三类是行为回归测试。它用历史样本检查候选 Prompt 是否保持关键行为。例如客服摘要不能编造退款，合同审查不能把“建议咨询律师”改成“必须签署”，代码助手不能在只读任务里建议执行破坏性命令。这类测试要覆盖历史事故，每发生一次线上问题，就应该沉淀至少一条回归样本。

第四类是质量评测。它关注摘要覆盖、回答有用性、语气、信息密度、引用准确性等较软指标。质量评测可以用人工打分，也可以用模型评审，但模型评审要有固定 rubric，并且要抽样校准。不要把一个评审模型的分数当绝对真理，只把它当趋势信号。

第五类是运行指标评测。Prompt 变化会影响成本和延迟。一个看似更清晰的提示词可能让输出 token 增加一倍，也可能让模型更频繁触发工具调用。评测报告必须列出输入 token、输出 token、平均延迟、重试率、解析失败率和拒答率。质量提升如果伴随成本暴涨，发布决策就不能只看准确率。

| 测试类型 | 自动化程度 | 主要指标 | 失败处理 |
| --- | --- | --- | --- |
| 渲染测试 | 高 | 模板变量、消息顺序、长度 | 阻塞合并 |
| 结构测试 | 高 | JSON 合法率、字段完整率 | 阻塞候选 |
| 行为回归 | 中高 | 关键断言通过率 | 低于阈值阻塞 |
| 质量评测 | 中 | 覆盖度、有用性、语气分 | 人工复核 |
| 运行指标 | 高 | token、延迟、重试、成本 | 触发发布评审 |

评测阈值要按任务风险设置。低风险的营销文案润色可以允许更多波动，高风险的合规分类必须非常严格。不要为所有 Prompt 设置同一个 80 分线。更合理的是为每个评测集定义门槛：结构合法率必须 100%，高风险拒答样本必须全部通过，普通摘要覆盖率不低于当前生产版本，平均成本不能上涨超过 15%，延迟 P95 不能超过某个服务预算。

还要做对照评测。候选版本不要只和金标比，也要和当前生产版本比。因为很多任务没有完美金标，生产版本的表现是一个重要基线。报告里应该显示“候选赢、生产赢、打平、都失败”的样本分布。候选版本如果在总分上略高，但输掉几个关键高风险样本，就不应该上线。

人工抽检要有节制但不能省。我的做法是：major 变更必须抽检，minor 变更按风险抽检，patch 变更如果只修 typo 可以免抽检。抽检界面要把输入、生产输出、候选输出、评分理由和 diff 放在一起，让评审者能快速判断。评审意见也要回写样本库，形成新的断言或注释，而不是停留在聊天工具里。

最后，评测结果本身也要版本化。样本集会变，评审规则会变，模型也会变。一次发布报告必须能说明：使用了哪个样本集版本，候选 Prompt 是哪个版本，评审模型是什么，运行时间是什么，失败样本有哪些。否则半年后复盘时只看到一个“通过”，却不知道当时通过的标准是什么。

## 失败模式

第一种失败模式是 Prompt 漂移。团队以为生产使用的是 `2.4.1`，但某个服务在运行时又追加了本地系统指令，另一个服务使用了缓存里的旧模板，灰度租户还绑定着实验版本。解决办法是统一运行时入口，并在每次调用日志里记录完整指纹。只记录版本号不够，最好记录内容哈希。

第二种失败模式是评测集过窄。候选版本在十几条样本上表现很好，上线后却在长对话、多语言、缺字段或噪声检索场景里失败。解决办法是按风险建样本分层，并把线上失败持续沉淀为回归样本。样本库不是一次性资产，而是生产经验的压缩。

第三种失败模式是输出契约被 Prompt 隐式修改。比如在提示词里把 `risk_level` 的含义从“客户风险”改成“工单处理风险”，字段名没变，下游指标却完全变味。解决办法是把字段语义写进输出 Schema 和文档，并要求语义变化升 major。字段没改不代表兼容。

第四种失败模式是热修绕过流程。线上出问题时，有人直接在后台改 Prompt，全量生效，事故暂时消失，但没有评测、没有版本、没有回滚点。几天后另一个问题出现，团队已经不知道差异从哪里来。热修可以存在，但必须生成 hotfix 版本，至少跑最小回归集，并写清楚事故编号。

第五种失败模式是把所有规则都塞进 Prompt。提示词越写越长，模型越难抓重点，成本越来越高，规则之间还会互相冲突。例如同一段 Prompt 同时要求“极度简洁”和“详细解释每个步骤”，最后输出取决于模型偏好。解决办法是拆分责任：权限交给系统，格式交给 Schema，确定性变换交给代码，Prompt 只处理需要语言理解的部分。

第六种失败模式是忽视模型升级。Prompt 版本没变，但底层模型变了，行为仍然会变。模型升级应该触发 Prompt 回归评测，尤其是工具调用、拒答、安全和结构输出任务。运行时指纹里必须记录模型版本，否则会把模型变化误判为 Prompt 变化。

第七种失败模式是过度依赖模型评审。模型评审便宜快速，但它也会偏好某些表达风格，甚至被候选输出说服。高风险任务不能只看模型评审分数，要结合规则断言、人工抽检和线上指标。评审模型适合做筛选，不适合做最终责任人。

第八种失败模式是版本太碎。每个小文案都发一个版本，评测报告泛滥，评审者疲劳，最后大家开始跳过流程。解决办法是定义变更级别和批量发布窗口。真正影响行为的变化严格管理，纯注释或无行为变化的修订走轻流程。治理的目的不是让所有人填表，而是让风险变化被看见。

## 上线 checklist

上线前可以用下面这份清单做最后确认：

- Prompt 构件是否包含 slug、version、owner、任务描述、模板正文、输入 Schema、输出 Schema、模型参数和变更说明。
- front matter 或后台元数据是否能追溯到同一个不可变版本，是否禁止原地修改已发布版本。
- 模板变量是否全部声明，用户输入是否只进入允许的位置，是否避免把不可信内容放进系统指令。
- 输出结构是否有机器可校验的 Schema，下游解析器是否和候选版本一起通过测试。
- 变更级别是否正确，输出字段语义、标签集合、工具权限或任务定义变化是否升 major。
- 静态检查是否通过，包括未声明变量、敏感信息、工具权限、长度预算和禁用词。
- 回归样本是否覆盖历史事故、高频路径、边界输入、多语言和安全场景。
- 候选版本是否和当前生产版本做了对照评测，失败样本是否有人看过。
- 质量指标是否达到任务阈值，成本、延迟、解析失败率和拒答率是否在预算内。
- 发布绑定是否支持灰度，是否能按租户、语言或用户分桶控制流量。
- 运行时日志是否记录 prompt_slug、prompt_version、content_hash、model、参数和输出校验结果。
- 回滚方案是否明确，旧版本是否仍然可用，缓存是否会在回滚后及时刷新。
- 相关业务、客服或合规人员是否已审阅高风险变更。
- 上线后观察窗口是否定义了报警指标和负责人。

这份 checklist 不需要每次都用同样重量执行。一个内部低风险润色 Prompt 可以轻一些，一个会触发外部动作或影响合规判断的 Prompt 必须严格一些。关键是把“风险”说清楚，而不是让每次修改都靠个人经验。

## 总结

Prompt 版本管理的本质，是把自然语言指令纳入工程系统。Prompt 会改变产品行为，所以它需要版本、评审、测试、发布、观测和回滚。把 Prompt 放进 Git 只是第一步，更重要的是把输入输出契约、模型参数、评测样本、发布绑定和运行时指纹一起管理起来。

对团队来说，最小可行的起点并不复杂：为每个生产 Prompt 建立不可变版本文件，补齐变量和输出 Schema，沉淀一批历史回归样本，CI 里跑静态检查和样本回放，运行时记录版本指纹。做到这几件事后，Prompt 修改就不再是凭感觉上线，而是有证据、有边界、有回滚点的工程变更。

长期看，Prompt 版本管理会促使团队重新划分责任。能用代码确定处理的规则，不要塞进 Prompt；需要模型判断的语言任务，要给足样本和评测；涉及高风险业务的变更，要让业务方能读懂并参与审阅。这样的系统不一定最花哨，但它能在产品持续迭代时保持可控。LLM 应用真正进入生产后，稳定性往往不是来自某一句神奇提示词，而是来自这套朴素但扎实的版本治理。
