## 问题背景

本地知识库听起来像一个很直接的能力：把文档、代码、会议记录、产品方案、PDF、网页剪藏放进一个目录，DeepChat 负责检索，模型负责回答。实际用下来，最影响效果的不是向量库品牌，也不是模型参数，而是三个更朴素的问题：哪些文件应该被纳入知识库，检索出来的片段如何压缩进上下文，回答里的引用能不能回跳到原文位置。只要这三件事没有处理好，本地知识库就会从“个人第二大脑”变成“一个看起来懂很多但经常答偏的文件堆”。

本地知识和公网搜索不一样。公网搜索的目标通常是找到足够权威、足够新的材料；本地知识的目标是还原你自己的工作语境。比如同一个词“Agent 评测”，在一个团队里可能指自动化测试框架，在另一个团队里可能指人工打分表。一个文件名叫 `final.md` 的文档，在本地可能是最重要的发布方案，也可能是废弃草稿。模型不知道这些隐含语境，DeepChat 也不能只凭文件内容判断权重。知识库需要把目录结构、文件时间、文档状态、标签、引用关系和用户最近任务一起考虑。

我用 DeepChat 做本地知识库时，最先遇到的问题是“检索命中但回答不准”。向量检索确实能找到语义接近的段落，但段落不一定是当前问题需要的证据。比如问“这个项目为什么不用某个框架”，检索可能命中一篇比较该框架优缺点的文章，却漏掉真正的 ADR；问“上次讨论的上线风险是什么”，检索可能命中会议纪要里的风险列表，却漏掉后续修订的发布清单。答案看起来合理，但不是基于最新、最决策性的材料。

第二个问题是“上下文过载”。本地资料越多，越容易把相关但不关键的内容塞给模型。模型上下文不是仓库，不应该承担全部存储责任。一个好的本地知识流程应该先召回候选，再过滤，再聚合，再只把回答问题所需的证据放进上下文。否则模型会在十几个片段之间做平均，最后给出没有立场的概括。尤其是技术决策类问题，真正重要的通常不是所有观点，而是最后采用的决策、约束条件和放弃方案。

第三个问题是“引用不可回跳”。如果回答说“根据本地文档，某某模块使用增量索引”，但不能跳回原文，用户就很难信任。引用不只是为了漂亮，它是本地知识库的调试接口。你需要知道答案来自哪个文件、哪个标题、哪个段落、这个段落是不是过期、有没有相邻上下文。没有可回跳引用，错误答案很难纠正，知识库也很难维护。引用一旦可回跳，用户看到错误就能立刻判断：是文档过期、切片错误、召回错了，还是模型综合错了。

所以我不把 DeepChat 本地知识库当成“把文件喂给模型”的功能，而是当成一个小型知识工程系统。它要处理采集、清洗、切片、索引、召回、重排、压缩、回答、引用和反馈。每一步都不一定复杂，但每一步都需要明确策略。对个人和小团队来说，不需要上来就做企业级知识平台；只要把文件选择、上下文压缩和引用回跳这三件事做好，本地知识库的可用性就会明显提升。

## 核心概念

本地知识库的第一个核心概念是“知识边界”。不是所有本地文件都应该进索引。临时下载、构建产物、依赖目录、重复导出、过期草稿、包含密钥的配置、自动生成的大文件，都会污染检索。知识边界要回答：哪些目录是权威来源，哪些文件只做附件，哪些内容永远不索引，哪些内容需要脱敏后索引。这个边界最好写成规则，而不是靠每次手选。

第二个概念是“文档身份”。同一份知识可能有多个版本：草稿、评审稿、发布稿、归档稿；也可能同时存在 Markdown、PDF、飞书导出、网页剪藏。向量库如果只把它们当作片段，会让重复内容互相竞争。文档身份应该包含稳定 ID、来源路径、版本、时间、状态和父子关系。回答时应该优先使用状态更权威、时间更新、路径更接近当前项目的文档。

第三个概念是“切片策略”。切片不是把文本每 800 字切一刀。好的切片要保留标题层级、列表结构、表格上下文、代码块边界和引用位置。技术文档里，一个段落单独看可能没有意义，必须带上所在标题和前后约束。会议纪要里，一个行动项要带上会议日期和参与人。ADR 里，一个决策必须带上状态和后果。切片策略决定了检索能不能命中真正有解释力的证据。

第四个概念是“召回和重排分离”。召回阶段追求别漏，重排阶段追求别乱。召回可以用向量相似度、关键词、路径权重、最近打开文件、标签过滤等多路组合；重排再根据问题类型、文档权威性、时间、标题匹配、片段完整度来排序。只用向量相似度，经常会把“语义相似的讨论稿”排在“最终决策文档”前面。

第五个概念是“上下文压缩”。DeepChat 最终要把候选片段放进模型上下文，但上下文预算有限。压缩不是简单摘要，而是把证据按问题重新组织：保留结论、约束、数据、引用位置，去掉重复解释和无关铺垫。对于代码和配置类知识，压缩还要保留符号名、路径和版本；对于会议类知识，要保留日期和责任人；对于产品方案，要保留决策状态和未决问题。

| 概念 | 解决的问题 | 推荐做法 | 没做好时的症状 |
| --- | --- | --- | --- |
| 知识边界 | 防止垃圾文件污染索引 | 白名单目录加黑名单模式 | 命中构建产物、旧导出、重复草稿 |
| 文档身份 | 区分版本和权威来源 | 记录路径、状态、时间、来源 | 答案引用过期版本 |
| 切片策略 | 保留可解释上下文 | 按标题、列表、代码块、表格切 | 片段命中但缺少前提 |
| 多路召回 | 避免只靠语义相似 | 向量、关键词、路径、最近任务组合 | 找到相似文章，漏掉决策文件 |
| 重排 | 让更可信材料靠前 | 权威性、时间、标题匹配加权 | 草稿压过正式方案 |
| 引用回跳 | 让答案可验证 | 保存文件、行号、标题、片段哈希 | 无法判断模型是否编造 |

这里还有一个容易被忽略的概念：负知识。很多本地知识问题不是“我们采用了什么”，而是“我们为什么没采用什么”。负知识通常存在于 ADR、评审意见、复盘文档和会议纪要里。如果切片和重排只偏爱结论段，就会漏掉这些放弃原因。对工程团队来说，负知识非常有价值，因为它能避免重复踩坑。

## 架构/流程图解说明

一个实用的 DeepChat 本地知识流程，可以拆成索引侧和问答侧。索引侧负责把文件变成可检索、可回跳、可更新的知识单元；问答侧负责把用户问题变成检索计划，再把证据压缩给模型。

```text
本地文件目录
  |
  v
文件选择：白名单、黑名单、大小限制、敏感规则
  |
  v
解析清洗：Markdown/PDF/代码/表格，保留标题和位置
  |
  v
切片：按语义块生成 chunk，带标题路径和行号
  |
  v
索引：向量索引 + 关键词索引 + 元数据索引
  |
  v
用户问题
  |
  v
查询规划：识别问题类型、时间范围、项目范围、需要的证据
  |
  v
召回与重排：多路候选合并，按权威性和相关性排序
  |
  v
上下文压缩：去重、合并、保留引用、构造回答材料
  |
  v
模型回答：给出结论、依据、限制和可回跳引用
```

文件选择是第一道闸门。我的规则通常是白名单优先：只索引 `docs/`、`notes/`、`adr/`、`meetings/`、`content/` 这类明确知识目录；代码仓库如果要索引，也只索引源码、测试和设计文档，不索引 `node_modules`、`dist`、`vendor`、日志、缓存、图片二进制和自动生成锁文件。PDF 要谨慎，很多 PDF 是导出副本，可能和 Markdown 源重复。对重复来源，要给源文件更高权重，导出文件只作为附件。

解析清洗要尽量保留结构。Markdown 的标题层级、表格、代码块、任务列表都应该进入元数据；PDF 至少要保留页码和标题；代码文件要保留符号名、注释块和导入关系；表格要保留表头。不要把所有东西压成纯文本后再切片，那会损失很多判断权威性和位置的信息。

切片阶段的目标是生成“可以独立解释，又能回到原文”的知识单元。下面是一个 chunk 结构例子：

```json
{
  "chunk_id": "adr-0023#section-4#p3",
  "doc_id": "adr-0023-local-rag",
  "path": "docs/adr/0023-local-rag.md",
  "title_path": ["本地知识库方案", "决策", "为什么不用全文塞上下文"],
  "line_start": 78,
  "line_end": 96,
  "status": "accepted",
  "updated_at": "2026-01-18",
  "content": "我们不把全文直接塞入模型上下文，而是采用多路召回和重排...",
  "anchors": ["上下文预算", "引用回跳", "增量索引"],
  "hash": "sha256:..."
}
```

这份结构比单纯的文本片段有用得多。`title_path` 能帮助模型理解上下文，`line_start` 和 `line_end` 支持回跳，`status` 和 `updated_at` 支持重排，`hash` 支持增量更新和引用校验。DeepChat 如果能把这类元数据带到回答链路里，本地知识库的可靠性会明显上升。

问答侧要先做查询规划。用户问“这个方案的风险是什么”，系统要判断“这个方案”可能指当前会话、最近打开文件、某个目录里的方案，还是用户上一轮提到的文档。用户问“我们之前为什么没有用 GraphRAG”，系统要优先找 ADR、评审记录和会议纪要，而不是普通教程。用户问“DeepChat 的本地知识怎么调参”，系统要找使用记录、配置说明和评测结果。不同问题类型对应不同检索策略。

召回与重排可以用一个简单分数开始：

```text
score = 0.45 * vector_similarity
      + 0.20 * keyword_match
      + 0.15 * authority_weight
      + 0.10 * recency_weight
      + 0.10 * path_affinity
```

这个公式不需要神秘。`authority_weight` 可以根据目录和文档状态计算，比如 `docs/adr` 高于 `meetings/raw`，`accepted` 高于 `draft`；`recency_weight` 不能盲目偏新，因为老 ADR 可能仍然有效；`path_affinity` 表示用户当前任务所在项目和文档路径是否接近。公式可以逐步调整，关键是把排序理由显式化，方便调试。

上下文压缩阶段要把候选片段变成一份“证据包”。证据包可以按结论组织，而不是按检索排名堆叠。比如问题是“本地知识库为什么需要引用回跳”，证据包应包含：决策文档里的结论、失败案例里的错误答案、实现文档里的回跳字段、评测记录里的命中率差异。每条证据都带来源。模型回答时就能给出清晰结构，而不是把十段材料揉成一段模糊总结。

## 工程实现

工程实现上，我会先从文件清单和增量索引做起。不要一开始就追求复杂 UI。一个可靠的本地知识库，首先要知道自己索引了什么、为什么索引、什么时候更新、能不能排除。文件清单可以是一个本地 SQLite 表，也可以是轻量 JSONL，但字段要够用。

```sql
CREATE TABLE documents (
  doc_id TEXT PRIMARY KEY,
  path TEXT NOT NULL,
  source_type TEXT NOT NULL,
  status TEXT,
  title TEXT,
  mtime INTEGER NOT NULL,
  size INTEGER NOT NULL,
  content_hash TEXT NOT NULL,
  indexed_at INTEGER NOT NULL,
  authority INTEGER NOT NULL DEFAULT 50
);

CREATE TABLE chunks (
  chunk_id TEXT PRIMARY KEY,
  doc_id TEXT NOT NULL,
  title_path TEXT NOT NULL,
  line_start INTEGER,
  line_end INTEGER,
  content TEXT NOT NULL,
  content_hash TEXT NOT NULL,
  embedding_id TEXT,
  FOREIGN KEY(doc_id) REFERENCES documents(doc_id)
);
```

有了 `content_hash`，增量索引就简单很多：文件未变不重建，文件变了只更新相关 chunks。对大目录来说，这比每次全量重扫更重要。DeepChat 本地知识如果每次启动都重新索引，用户会很快失去耐心；如果索引状态不可见，用户又不知道答案是不是基于最新文件。

文件过滤规则建议明确写成配置：

```yaml
knowledge:
  include:
    - docs/**
    - notes/**
    - content/articles/**
    - meetings/**/*.md
  exclude:
    - "**/node_modules/**"
    - "**/dist/**"
    - "**/.git/**"
    - "**/*.log"
    - "**/*secret*"
    - "**/drafts/abandoned/**"
  max_file_size_mb: 8
  prefer_source_over_export: true
```

这份配置的价值在于可解释。用户问“为什么没查到那篇 PDF”，可以回到规则看它是不是被排除了；用户发现过期草稿总是命中，也可以把目录加入黑名单或降低权重。知识库维护应该像代码维护一样，有规则、有 diff、有复盘。

切片实现要避免固定长度暴力切。对于 Markdown，我会按标题树解析，再在标题下按段落、列表、表格和代码块组合。每个 chunk 最好不跨越两个二级标题；如果某个表格很长，可以按行组切，但每组都带表头；代码块不要拆在中间，除非超过上限。对于会议纪要，日期、会议主题、参与人应该作为每个行动项的元数据继承下去。

一个具体例子：下面这段会议纪要不能只切出“上线前需要补引用回跳”这一句。

```markdown
## 2026-02-10 DeepChat 知识库评审

参与人：工程、产品、内容

### 结论

- 第一版只支持 Markdown 和 PDF，不接入网盘。
- 回答必须带来源文件和段落位置。

### 风险

- PDF 抽取可能丢标题。
- 过期草稿命中会误导用户。
```

更好的 chunk 内容应该携带会议日期、主题、参与人、标题路径和列表项。否则模型只知道“必须带来源”，不知道这是评审结论，也不知道适用范围是第一版。

问答时，DeepChat 可以把查询先改写成检索计划：

```json
{
  "question": "本地知识库为什么必须支持引用回跳？",
  "intent": "decision_reason",
  "preferred_sources": ["docs/adr", "meetings", "notes"],
  "required_evidence": ["decision", "failure_case", "implementation_field"],
  "time_range": "all",
  "must_return_citations": true
}
```

这个检索计划能帮助系统选择召回路径。`decision_reason` 类问题优先找 ADR 和评审记录；`how_to` 类问题优先找操作说明；`bug` 类问题优先找复盘、issue 和日志。计划不一定要展示给用户，但应该能在调试模式里看到。

上下文压缩可以用“证据卡片”表示：

```json
{
  "claim": "引用回跳是本地知识库可信度的基础",
  "evidence": [
    {
      "path": "docs/adr/0023-local-rag.md",
      "lines": "78-96",
      "summary": "决策要求回答保留文件和段落位置，便于核查与纠错"
    },
    {
      "path": "meetings/2026-02-10-deepchat-review.md",
      "lines": "12-18",
      "summary": "评审结论要求第一版回答必须带来源文件和段落位置"
    }
  ],
  "counterpoints": [
    "引用不能替代原文阅读，长文仍需打开上下文确认"
  ]
}
```

模型拿到证据卡片后，回答会更稳。它可以先给结论，再列依据，再说明限制。如果检索不到足够证据，应该明确说“本地知识库里没有找到直接决策记录”，而不是用常识补一段看似合理的解释。

引用回跳的实现要注意位置稳定性。只存行号在 Markdown 里通常够用，但文件修改后行号会漂移。可以同时存 `content_hash`、标题路径和段落前几个字符作为锚点。打开引用时，先按行号定位，再用 hash 或文本锚点校验；如果失败，就在同一标题下搜索相近段落，并提示“引用位置可能因文件更新发生偏移”。这种细节决定了用户是否愿意长期依赖知识库。

## 测试评测

本地知识库的评测不能只看“模型回答好不好”。模型回答受检索、重排、压缩、提示词和模型本身共同影响。如果不拆开评测，很难知道问题出在哪里。我会建立一组固定问题，每个问题都有期望命中文档、期望引用、必要结论和禁止编造项。

| 评测维度 | 目标 | 指标 | 示例 |
| --- | --- | --- | --- |
| 文件选择 | 该进的进，不该进的不进 | 索引文件数、排除原因覆盖率 | 草稿目录是否被排除 |
| 召回 | 找到足够候选证据 | Recall@k | ADR 是否出现在前 10 |
| 重排 | 权威来源靠前 | MRR、首条命中类型 | accepted ADR 是否高于会议草稿 |
| 压缩 | 上下文保留关键事实 | 证据覆盖率、重复率 | 是否保留状态、日期、路径 |
| 回答 | 结论正确且有引用 | 人工打分、引用准确率 | 引用是否能打开到原段落 |
| 稳定性 | 更新后不退化 | 增量索引耗时、旧引用漂移率 | 修改一篇文档后是否只重建相关 chunk |

评测集不需要很大，二三十个高质量问题就能暴露很多问题。问题要覆盖不同类型：事实查询、决策原因、操作步骤、失败案例、跨文档综合、时间敏感问题、否定问题。每个问题都写清楚期望证据。例如：

```yaml
- id: q_local_citation
  question: "为什么本地知识库回答必须带引用回跳？"
  expected_sources:
    - docs/adr/0023-local-rag.md
    - meetings/2026-02-10-deepchat-review.md
  must_mention:
    - 可核查
    - 纠错
    - 文件和段落位置
  forbidden:
    - 声称已经接入网盘
```

有了这样的评测集，每次调整切片大小、权重公式、排除规则、模型提示词，都可以跑一遍。不要只凭当天几个问题感觉更好。知识库系统很容易出现局部优化：某类问题变好，另一类问题变差；近期文档命中提高，历史决策被压下去；召回更多，回答却更啰嗦。

召回评测要看候选列表，不只看最终答案。如果期望 ADR 没进入前 20，说明召回或过滤有问题；如果进了前 20 但排在很后，说明重排有问题；如果排在前 3 但答案没用，说明上下文压缩或提示词有问题；如果答案用了但引用错了，说明回跳元数据有问题。分层定位比反复改 prompt 有效。

还要测试过期和重复文档。可以故意放一份旧方案和一份新方案，问题问“当前采用什么方案”。好的系统应该引用新方案，并说明旧方案已经归档或状态较低。也可以放两份内容相同但路径不同的导出文件，观察系统是否去重。重复内容如果不处理，会让模型误以为某个观点被多次独立确认，从而放大权重。

性能评测同样重要。个人知识库可能有几千到几万份文件，索引和查询不能让用户等待太久。需要关注初次索引耗时、增量索引耗时、单次查询召回耗时、重排耗时、最终上下文大小。更实际的指标是“用户提问到开始生成答案的等待时间”。如果前置检索超过几秒，体验就会明显下降。可以先返回“找到这些来源，正在整理答案”的中间状态，让用户知道系统没有卡住。

## 失败模式

第一个失败模式是把本地目录当成真理。文件在本地不代表它是最新、正确或权威的。很多团队的知识库里同时存在草稿、备份、导出、截图、旧会议纪要。解决办法是给文档状态和路径权重，不要让所有片段平等竞争。

第二个失败模式是切片丢前提。比如只召回一条“我们不采用某方案”，却没有召回上一段的约束“在第一版离线模式下”。模型回答时就会把临时限制说成永久原则。切片时要继承标题路径，必要时带上前导段落或决策状态。

第三个失败模式是引用错位。文件更新后，行号变了，回答里的引用打开到另一个段落。用户看到一次这种错误，就会怀疑所有引用。解决办法是多锚点校验：行号、标题路径、内容哈希、文本片段一起用。

第四个失败模式是隐私泄漏。本地知识库可能包含密钥、客户数据、个人信息、合同、内部财务。即使模型在本地运行，也要考虑哪些内容可以索引、哪些内容可以出现在回答里、哪些内容需要脱敏。对云端模型尤其要谨慎，敏感目录默认不进索引，或者只在明确授权时使用。

第五个失败模式是“越检索越糊”。召回片段太多，模型为了综合所有材料，给出没有明确结论的平均答案。解决办法不是盲目增加上下文，而是重排后只保留最有证据力的材料，并把冲突来源显式列出来。

第六个失败模式是反馈没有回流。用户发现答案引用了旧文档，如果只是手动纠正一次，系统下次还会犯。应该把反馈转成维护动作：降低某目录权重、标记文档归档、修正标题、补充 ADR、调整评测集。知识库质量来自持续维护，不是一次索引。

## 上线 checklist

- 已经定义知识边界，白名单目录和黑名单模式清楚。
- 构建产物、依赖目录、日志、缓存、密钥和敏感文件默认不索引。
- 每个文档有稳定身份，包括路径、状态、更新时间、内容哈希。
- Markdown、PDF、代码、表格的解析策略已经区分，不全部压成无结构文本。
- 切片保留标题路径、行号、表头、代码块边界和必要前提。
- 召回至少结合向量和关键词，重排考虑权威性、时间和路径亲和度。
- 回答必须带可回跳引用，引用打开时能定位到原文件和段落。
- 引用位置有哈希或文本锚点校验，能处理文件更新后的漂移。
- 有一组固定评测问题，覆盖事实、决策、步骤、失败案例和跨文档综合。
- 调整切片、权重或提示词后，会跑评测集而不是只凭感觉判断。
- 敏感目录和云端模型使用策略已经明确，用户能知道哪些内容会被发送。
- 增量索引可观察，用户能看到最近索引时间、失败文件和排除原因。
- 错误答案有反馈入口，反馈能转成文档维护或排序规则调整。

上线不是指把功能放进界面就结束。本地知识库一旦进入日常工作流，用户会把它当成记忆的一部分。它答错一次，比普通聊天答错更伤信任，因为用户以为答案来自自己的资料。因此上线前要优先保证可解释和可纠错，再追求回答风格漂亮。

## 总结

DeepChat 本地知识库的关键，不是把更多文件塞进模型，而是把本地资料变成可选择、可检索、可压缩、可回跳、可维护的证据系统。文件选择决定知识边界，切片决定证据质量，召回和重排决定材料顺序，上下文压缩决定模型能否抓住重点，引用回跳决定用户能否信任和纠错。

对个人和小团队来说，最实用的路线是从小范围权威目录开始，先做好 Markdown 和少量 PDF，保留标题、行号和状态，建立二三十个固定评测问题。等回答和引用稳定以后，再扩展更多来源。不要一开始追求全量接入，也不要把所有问题交给模型总结。本地知识库的价值在于贴近自己的工作现场，而工作现场的知识往往有版本、有语境、有取舍。DeepChat 如果能尊重这些结构，就会从一个问答入口变成可靠的工程记忆层。
