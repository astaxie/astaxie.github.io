## 问题背景

在桌面端使用 AI 助手时，多模型不是一个炫技功能，而是一个很实际的生产力问题。写一段产品说明、审一段 Go 代码、总结一份本地 PDF、拆一个重构计划、对比两个接口日志、把英文资料翻成中文，这些任务看起来都可以交给“一个大模型”，但真正长期使用下来，会发现不同模型的优势、成本、延迟、上下文长度、工具调用稳定性和风格偏好差异很大。如果桌面助手只能把用户锁在一个模型上，最后要么浪费昂贵模型处理简单任务，要么让便宜模型承担不适合它的复杂推理，用户只能靠复制粘贴在不同客户端之间切换。

DeepChat 这类桌面助手的价值，不应该只是把多个模型供应商的 API 填在一个界面里。真正难的是把“切换模型”变成低成本、可解释、可回退的工作流。用户在思考问题时，不希望被模型选择打断；模型失败时，也不希望从头重新描述上下文；同一个会话需要在草稿、审查、验证、压缩、翻译、搜索、代码生成之间流动。多模型工作流的核心目标，是让模型成为可调度的执行单元，而不是让用户记住一堆模型名称和参数表。

我在使用桌面 AI 工具时，最常见的痛点有三个。第一是任务和模型之间的匹配靠记忆，今天写代码用哪个、做总结用哪个、需要长上下文时换哪个，全部放在脑子里。第二是切换后上下文丢失，用户刚刚解释过项目背景、约束、语气和目标，换模型又要重新铺垫。第三是失败不可诊断，答案差的时候不知道是模型能力不够、提示词不清、上下文过长、工具结果污染，还是供应商限流。只要这些问题存在，多模型就会从“自由选择”变成“额外负担”。

多模型工作流还涉及一个成本问题。桌面助手往往被个人开发者、独立创作者和小团队高频使用，调用成本不是企业后台那种按月统一预算，而是直接体现在个人账户和团队配额里。一个好的多模型系统应该能把任务分层：便宜模型负责改写、分类、摘要、格式化；强模型负责复杂推理、架构取舍、失败诊断；长上下文模型负责整包资料阅读；本地模型负责隐私敏感和离线草稿。这样做不是为了省小钱，而是让用户愿意把 AI 融入每一天的工作流，而不是每次调用前都心里算账。

还有一个容易忽略的地方：多模型工作流不是“自动路由”就能解决。完全自动路由在演示里很漂亮，但生产使用中会遇到信任问题。用户需要知道系统为什么选这个模型，是否可以覆盖选择，失败后是否能把同一上下文交给另一个模型重试。DeepChat 更适合做“半自动调度”：系统给出默认建议，用户保留显式控制；常用任务有快捷模板，复杂任务允许临时指定；路由结果写入会话元数据，方便后续复盘。这样既降低选择成本，也保留可解释性。

因此，本文讨论的多模型工作流，不是简单列一个模型榜单，而是把桌面端 AI 助手当作一个本地调度系统来设计。它需要模型画像、任务分类、上下文打包、路由策略、结果对比、失败回退、成本观测和权限边界。这里面没有单点银弹，只有一套足够朴素、可落地、能长期维护的工程结构。

## 核心概念

多模型工作流的第一个核心概念是“模型画像”。模型画像不是营销页面上的参数，而是桌面助手在本地维护的一组可执行属性：上下文窗口、输入输出价格、平均首 token 延迟、流式稳定性、是否支持工具调用、结构化输出可靠性、适合任务、禁用任务、隐私等级、供应商可用状态。模型画像要能被路由器读取，也要能被界面展示给用户。没有画像，所谓多模型只是下拉框。

第二个概念是“任务意图”。用户输入的一句话可能是“帮我看下这段代码”，系统要进一步判断它是解释代码、找 bug、写测试、做重构建议，还是生成补丁。不同意图对模型能力要求不同。解释代码可以用中等模型，跨文件重构建议需要强推理模型，生成最终补丁还需要工具上下文和更严格的输出约束。任务意图不需要一开始就非常复杂，但至少要区分写作、总结、翻译、代码、规划、评审、检索问答、结构化抽取和故障诊断。

第三个概念是“上下文包”。多模型切换时，不能把整个会话历史无脑塞过去。上下文包应该是一个有结构的对象，里面包含用户目标、当前任务阶段、关键约束、相关文件、工具结果、已确认事实、未解决问题和前一个模型的输出摘要。上下文包是可压缩、可审计、可复用的，它决定了换模型时是不是能保留思路。如果上下文只是一串聊天记录，模型切换越多，噪声越大。

第四个概念是“路由策略”。路由策略负责从任务意图、上下文包、用户偏好、成本预算和供应商状态中选择模型。策略可以从简单规则开始，不必上来做复杂学习系统。比如“长文档总结优先长上下文模型”“代码评审优先强推理模型”“翻译和润色优先低延迟模型”“涉及本地敏感文件优先本地模型或用户确认”。关键是策略要可解释、可覆盖、可记录。

第五个概念是“结果编排”。多模型并不总是单次选择一个模型。有些任务适合串行：便宜模型先抽取事实，强模型再做判断；有些任务适合并行：两个模型分别给方案，再由用户或评审模型比较；有些任务适合回退：第一个模型超时、拒答或结构错误时，自动换备用模型。结果编排需要明确每一步的输入输出契约，否则会话很快变成不可维护的混乱链条。

| 概念 | 工程含义 | 桌面端表现 | 风险点 |
| --- | --- | --- | --- |
| 模型画像 | 记录能力、价格、延迟、上下文、工具支持 | 模型选择器显示适合任务和成本提示 | 画像过期导致错误路由 |
| 任务意图 | 把用户请求归类成可执行任务 | 快捷入口、自动建议模型 | 误判意图会浪费上下文 |
| 上下文包 | 可迁移的会话状态摘要 | 换模型后保留目标和约束 | 压缩过度会丢关键事实 |
| 路由策略 | 根据约束选择模型 | 默认推荐、可手动覆盖 | 自动化过强会失去信任 |
| 结果编排 | 串行、并行、回退、对比 | 一键重试、双模型对照 | 输出契约不清会难以调试 |
| 成本观测 | 记录 token、价格、耗时 | 会话账单和预算提醒 | 没有观测就无法优化 |

这些概念背后的原则很简单：模型是资源，不是人格；会话是状态机，不是聊天气泡堆；用户要的是任务完成，不是参与模型供应商评测。只要把这三点想清楚，DeepChat 的多模型能力就会从功能列表变成真正可用的工程系统。

## 架构/流程图解说明

一个可落地的多模型工作流，可以把 DeepChat 分成界面层、会话层、路由层、执行层和观测层。界面层负责让用户选择任务、查看建议、覆盖模型；会话层负责维护上下文包；路由层负责模型选择；执行层负责调用供应商或本地模型；观测层负责记录延迟、成本、错误和用户反馈。

```text
用户输入
  |
  v
任务意图识别：写作 / 代码 / 总结 / 评审 / 诊断 / 抽取
  |
  v
上下文包构建：目标、约束、文件、历史摘要、工具结果、预算
  |
  v
路由策略
  |-- 模型画像：能力、价格、上下文、工具支持
  |-- 用户偏好：默认供应商、隐私模式、预算上限
  |-- 当前状态：供应商健康度、限流、离线可用性
  v
执行计划
  |-- 单模型直接执行
  |-- 多模型并行对比
  |-- 低成本预处理 + 强模型决策
  |-- 失败回退
  v
模型调用适配器
  |
  v
结果归一化：正文、结构化字段、引用、耗时、token、错误
  |
  v
会话呈现 + 观测记录 + 可复用上下文摘要
```

这张流程图里最值得注意的是“执行计划”不是模型名称，而是一段可执行的策略。例如用户把一个大型项目目录拖进 DeepChat，问“帮我判断这次迁移风险”，路由层不应该直接把全部文件交给最贵模型。更稳的执行计划是：先用快速模型对文件清单和变更摘要做分类，再用代码能力强的模型阅读关键模块，再用强推理模型生成风险矩阵，最后把引用和不确定项展示给用户。这个流程虽然多一步，但成本可控，结果也更可解释。

再看一个并行对比场景。用户写了一份技术方案，想要“从架构评审角度挑问题”。系统可以同时调用两个风格不同的模型：一个偏保守，关注安全、兼容和运维；另一个偏产品工程，关注交付路径和用户影响。DeepChat 不需要假装能自动判定哪个更正确，而是把两个结果按统一结构展示：阻塞问题、非阻塞风险、需要补充的数据、建议修改。用户可以保留其中一份，也可以让第三步模型合并。并行对比的关键不是让模型互相辩论，而是让输出结构可比较。

回退流程也要在架构里一开始就设计。模型失败不只有 HTTP 错误，还包括空回复、流式中断、结构化 JSON 不合法、工具调用循环、回答偏题、上下文超限、内容策略拒答等。执行层应该把这些错误归一化成可判断的类型，然后交给路由策略决定是重试同一模型、降低上下文、换备用模型、请求用户确认，还是停止并暴露错误。否则用户看到的只是“失败了”，没有办法继续推进任务。

## 工程实现

多模型工作流可以从一个很朴素的数据结构开始。下面这个 TypeScript 结构并不绑定具体框架，它表达的是本地桌面助手需要保存的最小状态。重点不是字段多，而是每个字段都能服务于路由、观测和回退。

```ts
type TaskIntent =
  | "write"
  | "summarize"
  | "translate"
  | "code_review"
  | "debug"
  | "plan"
  | "extract"
  | "rag_qa";

type ModelProfile = {
  id: string;
  provider: string;
  displayName: string;
  contextWindow: number;
  inputCostPerMTok: number;
  outputCostPerMTok: number;
  medianFirstTokenMs: number;
  supportsTools: boolean;
  supportsJsonSchema: boolean;
  privacyTier: "local" | "configured_api" | "external_api";
  strengths: TaskIntent[];
  avoidFor: TaskIntent[];
  fallbackIds: string[];
  updatedAt: string;
};

type ContextPackage = {
  conversationId: string;
  userGoal: string;
  intent: TaskIntent;
  constraints: string[];
  selectedFiles: Array<{ path: string; digest: string; summary?: string }>;
  confirmedFacts: string[];
  openQuestions: string[];
  previousAnswerSummary?: string;
  tokenBudget: number;
  privacyMode: "normal" | "sensitive" | "offline";
};

type RouteDecision = {
  modelId: string;
  reason: string;
  executionMode: "direct" | "parallel" | "cascade" | "fallback";
  estimatedCost: number;
  maxInputTokens: number;
  maxOutputTokens: number;
  requiresUserConfirm: boolean;
};
```

有了这些结构，路由器就可以先写成透明的规则函数。桌面端工程最忌讳一开始就把路由藏到黑盒里，因为用户遇到坏结果时无法判断系统做了什么。一个可解释规则的价值，是它能被日志记录、被界面展示、被用户覆盖。

```ts
function routeTask(
  profiles: ModelProfile[],
  context: ContextPackage,
  userPreferredModel?: string
): RouteDecision {
  if (userPreferredModel) {
    const profile = profiles.find((item) => item.id === userPreferredModel);
    if (!profile) throw new Error("preferred model not found");
    return {
      modelId: profile.id,
      reason: "使用用户手动指定的模型",
      executionMode: "direct",
      estimatedCost: estimateCost(profile, context.tokenBudget),
      maxInputTokens: Math.min(context.tokenBudget, profile.contextWindow * 0.7),
      maxOutputTokens: 4096,
      requiresUserConfirm: context.privacyMode === "sensitive" && profile.privacyTier === "external_api",
    };
  }

  const candidates = profiles
    .filter((profile) => !profile.avoidFor.includes(context.intent))
    .filter((profile) => context.privacyMode !== "offline" || profile.privacyTier === "local")
    .filter((profile) => profile.contextWindow >= context.tokenBudget);

  const scored = candidates.map((profile) => {
    let score = 0;
    if (profile.strengths.includes(context.intent)) score += 40;
    if (profile.supportsTools && ["debug", "code_review", "rag_qa"].includes(context.intent)) score += 15;
    if (profile.supportsJsonSchema && context.intent === "extract") score += 15;
    score -= profile.inputCostPerMTok + profile.outputCostPerMTok;
    score -= Math.min(profile.medianFirstTokenMs / 1000, 20);
    return { profile, score };
  });

  scored.sort((a, b) => b.score - a.score);
  const selected = scored[0]?.profile;
  if (!selected) throw new Error("no available model for current context");

  return {
    modelId: selected.id,
    reason: `根据任务类型 ${context.intent}、上下文预算和隐私模式选择`,
    executionMode: context.intent === "code_review" ? "cascade" : "direct",
    estimatedCost: estimateCost(selected, context.tokenBudget),
    maxInputTokens: Math.min(context.tokenBudget, selected.contextWindow * 0.75),
    maxOutputTokens: context.intent === "write" ? 8192 : 4096,
    requiresUserConfirm: context.privacyMode === "sensitive" && selected.privacyTier === "external_api",
  };
}
```

这段代码故意保留了很多朴素判断，因为真实桌面产品里，可维护比聪明更重要。后续可以逐步把规则拆到配置文件里，允许用户为特定任务固定模型，也可以记录用户每次覆盖行为，用来调整默认策略。但第一版不要追求完美自动化。只要它能解释“为什么选这个模型”，并且能稳定退回用户手动选择，就已经比纯下拉框进了一步。

上下文包的构建同样关键。多模型切换时，DeepChat 应该把历史消息分成三类：必须保留、可以摘要、可以丢弃。必须保留的是用户目标、硬性约束、被引用文件、用户明确纠正过的事实；可以摘要的是长讨论、探索过程、模型草稿；可以丢弃的是失败的中间输出、重复解释、无关闲聊和已经被新决策覆盖的内容。这个分类可以先靠规则，也可以在上下文过长时调用便宜模型做压缩，但压缩结果必须以结构化形式写回会话。

一个实际的上下文包例子如下：

```json
{
  "conversationId": "conv_20260220_0915",
  "userGoal": "评审 DeepChat 多模型路由方案，找出上线前风险",
  "intent": "code_review",
  "constraints": [
    "不能把敏感本地文件发送给外部模型",
    "输出必须包含阻塞风险和非阻塞建议",
    "优先考虑桌面端离线和限流场景"
  ],
  "selectedFiles": [
    { "path": "src/router/model-router.ts", "digest": "sha256:9ab..." },
    { "path": "src/stores/conversation.ts", "digest": "sha256:19f..." }
  ],
  "confirmedFacts": [
    "当前版本已经支持多个供应商配置",
    "会话记录保存在本地数据库",
    "用户可以手动选择模型"
  ],
  "openQuestions": [
    "模型画像是否有自动过期机制",
    "失败回退是否会重复扣费且无提示"
  ],
  "tokenBudget": 32000,
  "privacyMode": "sensitive"
}
```

工程上还要做模型适配器的归一化。不同供应商的流式协议、错误码、工具调用格式、用量统计字段都不一样。DeepChat 的上层不应该直接依赖供应商响应，而应该统一成内部事件流。

```ts
type ModelEvent =
  | { type: "start"; requestId: string; modelId: string }
  | { type: "delta"; text: string }
  | { type: "tool_call"; name: string; arguments: unknown }
  | { type: "usage"; inputTokens: number; outputTokens: number; cost: number }
  | { type: "finish"; reason: "stop" | "length" | "tool" }
  | { type: "error"; code: string; retryable: boolean; message: string };
```

只要事件流统一，界面层就能统一显示进度，观测层就能统一统计成本，回退逻辑也能统一判断错误。比如遇到 `retryable=true` 的网络错误，可以重试同一模型；遇到 `finish=length`，可以触发上下文压缩后继续；遇到结构化输出解析失败，可以用同一模型做修复，也可以换一个 JSON 稳定性更好的模型。适配器越薄越好，但边界一定要清楚。

多模型工作流的存储也需要早设计。最少要有三张逻辑表：模型画像表、路由记录表、调用记录表。模型画像表保存配置和能力；路由记录表保存每次选择的理由；调用记录表保存用量、耗时、错误和用户反馈。不要只把最终答案存在聊天记录里，否则后面无法回答“为什么这个月成本变高”“哪个模型经常超时”“用户在哪些任务上经常手动覆盖默认模型”。

| 表/集合 | 关键字段 | 用途 |
| --- | --- | --- |
| `model_profiles` | `id`、`provider`、`context_window`、`cost`、`strengths`、`updated_at` | 给路由器提供可执行画像 |
| `route_decisions` | `conversation_id`、`intent`、`model_id`、`reason`、`estimated_cost` | 解释每次模型选择 |
| `model_calls` | `request_id`、`latency_ms`、`tokens`、`cost`、`error_code` | 做成本和稳定性分析 |
| `context_packages` | `conversation_id`、`summary`、`constraints`、`files`、`privacy_mode` | 支持跨模型迁移上下文 |
| `user_overrides` | `intent`、`from_model`、`to_model`、`accepted` | 调整默认策略 |

界面实现上，我会避免把所有复杂度暴露给用户。默认界面只需要三个层次：当前模型、为什么选它、是否有更合适的替代。高级设置里再放任务路由规则、成本预算、隐私偏好和供应商健康状态。用户手动覆盖后，系统应该记录这次覆盖，但不要立刻学习成永久规则；连续多次在同一任务上覆盖，才提示用户是否更新默认策略。这个交互细节很重要，因为一次覆盖可能只是临时需求，不代表长期偏好。

## 测试评测

多模型工作流的评测不能只看最终回答好不好。最终回答受模型能力影响太大，容易把路由、上下文、适配器、供应商波动混在一起。更可控的做法是把评测拆成五层：意图识别准确率、路由选择合理性、上下文包完整性、执行稳定性、用户可接受度。

第一层是意图识别。准备一组真实桌面使用场景，例如“把这份会议纪要整理成行动项”“检查这段 SQL 有没有性能问题”“基于这些文件写一个迁移计划”“把刚才的回答改成更短的中文”。每条样本标注期望意图、隐私等级、是否需要工具、是否需要长上下文。评测时不调用大模型生成正文，只检查分类是否正确，成本低且稳定。

第二层是路由合理性。这里不要求唯一答案，而是定义可接受集合。比如代码评审可以选择强代码模型或强推理模型，但不应该选择只适合翻译的低成本模型；敏感离线模式下不应该选择外部 API；上下文预算超过模型窗口时不应该硬塞。路由评测要输出原因，方便发现规则冲突。

第三层是上下文包完整性。可以构造几轮会话：用户提出目标，模型给出草稿，用户纠正一个关键事实，再要求换模型继续。测试点是换模型后是否保留了用户纠正、文件引用和硬性约束。很多多模型工具的问题不是模型差，而是换模型时丢了“用户刚刚说过不能这样做”。

第四层是执行稳定性。用模拟适配器注入错误：限流、超时、流式中断、输出超长、JSON 解析失败、供应商返回空内容。每个错误都应该有明确行为。能重试的重试，能回退的回退，不能自动处理的给用户可理解的错误。这里建议写成确定性测试，不要依赖真实 API。

第五层是用户可接受度。桌面助手最终要服务真实工作，不可能完全靠离线指标。可以记录用户是否复制答案、是否继续追问、是否手动换模型、是否点击“结果不好”、是否回退到前一个模型。注意这些信号只是弱指标，不能简单等同于质量，但可以帮助发现工作流摩擦。

| 评测层 | 样本来源 | 指标 | 失败例子 |
| --- | --- | --- | --- |
| 意图识别 | 手工标注任务句 | intent 准确率、隐私等级准确率 | 把代码评审识别成普通问答 |
| 路由选择 | 任务和模型画像组合 | 可接受路由占比、违规路由数 | 敏感模式选外部模型 |
| 上下文包 | 多轮会话回放 | 约束保留率、事实保留率 | 换模型后忘记用户纠正 |
| 执行稳定性 | 模拟供应商错误 | 成功回退率、错误展示质量 | JSON 错误直接崩溃 |
| 用户反馈 | 本地匿名统计或显式反馈 | 覆盖率、重试率、复制率 | 用户频繁手动换同一模型 |

我会特别关注两个指标：手动覆盖率和失败后继续率。手动覆盖率过高，说明默认路由不符合用户直觉；失败后继续率过低，说明错误恢复没有给用户下一步。多模型工作流的目标不是永远选对，而是在选错和失败时仍然让用户能顺着系统继续完成任务。

## 失败模式

第一个失败模式是“模型画像过期”。供应商升级、降价、调整限流、改变工具调用能力后，本地画像仍然用旧数据，路由就会越来越不准。解决办法是给画像加 `updatedAt` 和来源，过期后提醒用户刷新；对关键能力不要只靠配置声明，启动时可以做轻量探测，例如结构化输出、最大上下文和流式响应是否可用。

第二个失败模式是“自动路由过度自信”。系统觉得自己知道该用哪个模型，但用户的真实任务有隐含约束。比如用户说“帮我整理这段客户反馈”，系统选择外部强模型，但反馈里有未脱敏的客户信息。路由器必须把隐私模式、文件来源和用户配置放在能力评分之前。涉及敏感文件、密钥、客户数据、内部设计时，宁可多一次确认，也不要自动发送。

第三个失败模式是“上下文压缩丢约束”。为了适配另一个模型窗口，系统把历史对话摘要成几句话，结果丢掉了用户明确说过的限制。后续模型给出违反约束的方案，看起来像模型不听话，其实是上下文包构建失败。解决办法是把约束、确认事实和开放问题作为结构化字段单独保存，不和普通摘要混在一起。

第四个失败模式是“并行对比制造噪声”。两个模型同时给方案，如果输出结构不同，用户反而要花更多时间阅读。并行工作流必须先约定输出格式，最好限制为同一张风险表、同一组评审维度或同一套评分项。多模型不是让用户看更多文本，而是让用户更快看到差异。

第五个失败模式是“回退重复扣费但没有提示”。调用失败后自动重试或换模型，可能产生额外成本。桌面端应该明确显示重试次数、已产生用量和预计新增成本。对于长上下文任务，回退前最好请求用户确认，或者默认只回退一次。用户可以接受失败，但很难接受自己不知道的消耗。

第六个失败模式是“供应商错误泄漏到产品体验”。不同 API 的错误信息可能很长、很技术、甚至包含无关字段。适配器应该把它们归一化成用户能理解的错误：网络不可用、鉴权失败、余额不足、限流、上下文过长、内容被拒、服务端异常。详细原始错误可以进日志，但界面上要给出下一步建议。

第七个失败模式是“本地模型被错误期待”。本地模型在隐私和离线场景很有价值，但不一定适合复杂推理或长文生成。如果界面只强调本地模型安全，用户可能把所有任务都交给它，然后觉得 DeepChat 效果差。更好的做法是明确标注本地模型适合草稿、分类、简单总结和敏感预处理，不适合的任务给出建议。

## 上线 checklist

- 模型画像包含上下文、成本、延迟、工具调用、结构化输出、隐私等级和更新时间。
- 每个任务意图都有默认路由规则，并且规则能在界面或日志中解释。
- 用户手动指定模型时，系统尊重选择，并记录覆盖行为。
- 敏感模式和离线模式优先级高于能力评分，不会自动调用外部模型。
- 上下文包把目标、约束、确认事实、文件引用、开放问题分开保存。
- 换模型继续会话时，用户纠正过的事实必须保留。
- 模型适配器输出统一事件流，不把供应商响应直接暴露给上层。
- 限流、超时、流式中断、上下文超限、结构化解析失败都有确定行为。
- 自动重试和回退有次数限制，并显示可能产生的额外成本。
- 并行对比任务使用统一输出结构，避免让用户阅读两份散文。
- 调用记录保存 request id、模型、token、成本、耗时、错误码和完成状态。
- 路由记录保存选择原因，便于排查“为什么用了这个模型”。
- 预算提醒按会话和自然周期展示，不只在设置页展示。
- 模型画像过期时有提示，关键能力可以被轻量探测。
- 测试集覆盖写作、总结、翻译、代码评审、调试、抽取、检索问答和规划。
- 发布前用模拟适配器跑失败注入，不依赖真实供应商偶然报错。
- 对用户反馈只做渐进学习，避免一次覆盖改变长期默认策略。
- 日志中不保存敏感正文，必要时只保存摘要、哈希和统计字段。

## 总结

DeepChat 的多模型工作流，真正要解决的是“在不同任务之间低成本地选择合适能力，并在失败时继续推进”。这件事看起来像产品交互，底层其实是工程系统：模型画像提供事实，任务意图提供分类，上下文包提供可迁移状态，路由策略提供选择理由，适配器提供统一执行边界，观测数据提供长期优化依据。

我更倾向从简单、透明、可覆盖的规则开始，而不是一开始就追求完全自动的智能路由。桌面助手服务的是高频个人工作流，信任感比自动化程度更重要。用户需要知道系统为什么选这个模型，也需要随时能换成自己想要的模型；系统需要记住这次选择，但不要过度解读一次临时覆盖。

多模型的最终形态不是一个越来越长的模型列表，而是一套围绕任务流动的调度能力。写作可以先低成本起草再强模型审稿，代码评审可以先检索上下文再强推理判断，敏感资料可以先本地处理再选择性外发，失败时可以压缩上下文或切换备用模型。只要这些路径稳定、可解释、可观测，DeepChat 就能把多模型从“供应商聚合”推进到“工程化工作台”。
