## 问题背景

开发者使用 AI 助手时，最浪费时间的往往不是模型思考，而是上下文搬运。一个问题从代码仓库里出现，开发者先复制报错、复制相关文件、说明运行命令、解释分支状态，再把模型给出的建议搬回编辑器和终端里验证。验证失败后，又要把新的日志复制回聊天窗口。这个循环如果每天发生十几次，AI 助手就会从加速器变成另一个需要照顾的窗口。

DeepChat 的开发者快捷流，应该围绕“把聊天、文件、终端和项目上下文连起来”来设计。开发者不是在空白对话框里提问，而是在一个具体仓库、一个具体分支、一个具体失败、一个具体文件范围里工作。快捷流的价值，是让这些上下文以可控方式进入会话，让模型输出能被快速落地验证，同时不越过用户对文件系统、命令执行和隐私边界的控制。

我理解的快捷流不是多放几个按钮，也不是把所有 IDE 能力搬进聊天工具。开发者真正需要的是低摩擦的动作链：选中一段代码后问“这里的边界条件是什么”；终端命令失败后问“根据这段错误定位原因”；打开一个项目后问“先读哪些文件”；准备提交前问“帮我做一次变更审查”；写完测试后问“失败样本说明了什么”。这些动作都很短，但每个动作都依赖当前工作现场。

如果没有快捷流，开发者会用很粗糙的方式补齐上下文：把整个文件贴进去，把错误日志全部贴进去，把无关目录也塞进去。模型收到的信息越多，越不一定更准。它可能被旧日志误导，被无关文件稀释注意力，或者在没有运行结果的情况下给出看似合理但无法验证的建议。快捷流的目标不是把更多内容给模型，而是把正确的上下文、正确的权限和正确的下一步连接起来。

这里还有一个产品边界问题。开发者工具必须尊重用户的主控权。AI 可以建议命令，但不应该默认执行危险命令；可以生成补丁，但应该让用户查看差异；可以读取文件，但要遵守项目范围和忽略规则；可以总结终端日志，但不要把密钥、token、客户数据自动发送到外部模型。快捷不是越权，自动化也不是跳过确认。一个让开发者长期信任的 DeepChat，应该在效率和控制之间留出清晰的接口。

因此，本文讨论的开发者快捷流，会把重点放在工程结构，而不是单个交互技巧。我们需要定义项目上下文、快捷动作、输入采集、权限检查、会话注入、结果落地、命令验证和失败恢复。只有这些环节闭合，DeepChat 才能真正成为开发者工作台，而不是一个带代码高亮的聊天窗口。

## 核心概念

第一个核心概念是“项目上下文”。项目上下文包括仓库根目录、当前分支、未提交变更、语言栈、包管理器、测试命令、最近打开文件、忽略规则和用户当前关注的文件范围。它不是一次性读取全仓库，而是一个动态快照。开发者问同一句“这个失败怎么修”，在不同项目、不同分支、不同测试输出下，答案完全不同。

第二个概念是“快捷动作”。快捷动作不是普通 prompt，而是带有输入契约、权限要求和输出形态的操作。例如“解释选中代码”需要文件路径和行号，输出是解释和潜在风险；“根据终端失败诊断”需要命令、退出码和日志尾部，输出是假设、证据和下一步命令；“生成提交前审查”需要 diff，输出是问题列表和测试建议。快捷动作应该少而精，覆盖高频场景。

第三个概念是“上下文注入”。注入不是把所有内容拼成一个大 prompt，而是按角色组织信息：用户目标、项目摘要、文件片段、终端结果、版本状态、限制条件。每段信息都应该带来源和截断说明。比如终端日志只注入最后 200 行，就要明确说明“日志已截断”；文件片段来自哪一行，也要标出路径和范围。这样模型才能知道证据边界，用户也能追踪答案来源。

第四个概念是“权限分层”。开发者快捷流至少有四种权限：读取项目文件、读取终端输出、写入文件、执行命令。读取也不应该无限制，默认要遵守 `.gitignore` 和用户配置的排除目录；写入要显示 diff；执行命令要区分安全命令、需要确认命令和禁止命令。权限分层不是为了让产品显得保守，而是为了避免一次方便破坏用户仓库。

第五个概念是“验证闭环”。模型建议如果不能验证，就只是建议。快捷流应该尽量把“下一步验证”变成显式动作：运行哪个测试、检查哪个文件、比较哪个输出、回滚哪段修改。DeepChat 可以把建议和验证命令放在一起，但执行前让用户确认。验证结果再回到会话，形成诊断闭环。

| 概念 | 输入 | 输出 | 关键约束 |
| --- | --- | --- | --- |
| 项目上下文 | 根目录、分支、diff、语言栈、文件范围 | 可注入的项目快照 | 避免全仓库无差别读取 |
| 快捷动作 | 当前选择、终端日志、diff、用户目标 | 结构化任务请求 | 每个动作有明确输入契约 |
| 上下文注入 | 文件片段、命令结果、限制条件 | 带来源的 prompt 块 | 标注路径、行号和截断 |
| 权限分层 | 用户配置、动作类型、风险等级 | 允许、确认、拒绝 | 写入和执行要可审计 |
| 验证闭环 | 建议、补丁、测试命令 | 运行结果和下一步判断 | 不把生成等同于完成 |

这些概念的共同点，是把“聊天”拆成可执行的开发动作。开发者不需要每次重新描述工作现场，DeepChat 也不需要猜测上下文。快捷流越清晰，模型越容易给出可验证的答案，用户越容易把它纳入日常开发节奏。

## 架构/流程图解说明

开发者快捷流可以拆成四条链路：上下文采集链路、动作构建链路、模型执行链路、落地验证链路。四条链路之间要通过结构化对象传递，而不是靠拼接字符串隐式约定。

```text
开发现场
  |-- 编辑器选区：文件路径、起止行、选中文本
  |-- 终端会话：命令、退出码、日志尾部、工作目录
  |-- Git 状态：分支、diff、未跟踪文件
  |-- 项目探测：语言、依赖、测试入口、忽略规则
  v
快捷动作选择
  |-- 解释选区
  |-- 诊断失败
  |-- 生成测试
  |-- 审查 diff
  |-- 准备提交说明
  v
权限与范围检查
  |-- 读取允许
  |-- 写入需确认
  |-- 命令风险分级
  v
上下文包生成
  |-- 目标
  |-- 证据块
  |-- 项目约束
  |-- 输出格式
  v
模型响应
  |-- 解释
  |-- 补丁建议
  |-- 命令建议
  |-- 风险说明
  v
落地验证
  |-- 查看 diff
  |-- 运行测试
  |-- 采集新日志
  |-- 继续追问或结束
```

这个流程里，最重要的是“开发现场”不是由模型自己探索，而是由 DeepChat 的本地层采集。模型可以请求更多上下文，但默认不能直接越权读取。比如用户选择 `router.ts` 的 40 行代码问“这里有没有并发问题”，上下文采集只需要当前文件片段、相邻类型定义和可能的调用点，不应该把整个仓库塞进去。后续如果模型判断需要调用方，再明确提出需要读取哪些文件，由用户或策略确认。

另一个关键点是命令链路。终端失败诊断最容易做成“复制日志问模型”，但更好的做法是把命令本身、工作目录、退出码、环境摘要和日志尾部一起构造成证据。很多错误只看日志最后几行会误判，例如测试失败前面已经提示数据库连接地址不对；也有一些日志很长但关键在退出码和命令参数。快捷流应该帮助模型看到结构化事实，而不是只看一段文本。

落地验证也要从架构上闭合。模型给出修复建议后，DeepChat 可以生成补丁预览、列出推荐测试命令，并在用户确认后把运行结果接回会话。这样模型下一轮不是凭空继续猜，而是基于真实反馈调整假设。对开发者来说，这个闭环比一次性回答更重要，因为真实问题经常需要两到三轮定位。

## 工程实现

我会从一个 `ShortcutAction` 数据结构开始实现开发者快捷流。每个快捷动作都声明自己需要哪些输入、需要什么权限、如何构造上下文、期望模型返回什么结构。这样后续新增动作不会散落在界面事件和 prompt 字符串里。

```ts
type Permission = "read_files" | "read_terminal" | "write_files" | "run_commands";

type ShortcutAction = {
  id: string;
  title: string;
  requiredInputs: Array<"selection" | "terminal" | "git_diff" | "project_summary" | "user_text">;
  permissions: Permission[];
  riskLevel: "low" | "medium" | "high";
  buildContext: (input: ShortcutInput) => DeveloperContextPackage;
  expectedOutput: "explanation" | "review" | "patch_plan" | "test_plan" | "commit_message";
};

type ShortcutInput = {
  projectRoot: string;
  userText?: string;
  selection?: {
    path: string;
    startLine: number;
    endLine: number;
    text: string;
  };
  terminal?: {
    cwd: string;
    command: string;
    exitCode: number;
    outputTail: string;
    truncated: boolean;
  };
  git?: {
    branch: string;
    diffStat: string;
    diffPatch?: string;
  };
};

type EvidenceBlock = {
  kind: "file" | "terminal" | "git" | "project";
  source: string;
  content: string;
  truncated: boolean;
};

type DeveloperContextPackage = {
  actionId: string;
  goal: string;
  projectRootName: string;
  constraints: string[];
  evidence: EvidenceBlock[];
  outputContract: string;
};
```

以“诊断终端失败”为例，动作实现可以很明确。它不需要读取全仓库，默认只需要终端输出和项目摘要；如果用户允许，再加上相关配置文件。它的输出也不应该是一篇泛泛解释，而应该是“最可能原因、证据、验证命令、是否需要读取更多文件”。

```ts
const diagnoseTerminalFailure: ShortcutAction = {
  id: "diagnose_terminal_failure",
  title: "诊断终端失败",
  requiredInputs: ["terminal", "project_summary"],
  permissions: ["read_terminal"],
  riskLevel: "low",
  expectedOutput: "test_plan",
  buildContext(input) {
    if (!input.terminal) throw new Error("terminal input required");
    return {
      actionId: "diagnose_terminal_failure",
      goal: "根据命令、退出码和日志尾部定位失败原因，并给出下一步验证动作。",
      projectRootName: input.projectRoot.split("/").pop() ?? "project",
      constraints: [
        "不要假设未提供的文件内容",
        "如果证据不足，明确说明需要补充的文件或命令",
        "命令建议必须区分只读检查和可能修改状态的操作"
      ],
      evidence: [
        {
          kind: "terminal",
          source: `${input.terminal.cwd} $ ${input.terminal.command}`,
          content: `exit code: ${input.terminal.exitCode}\n${input.terminal.outputTail}`,
          truncated: input.terminal.truncated
        }
      ],
      outputContract: [
        "用表格列出最多三个假设",
        "每个假设必须写明来自日志的证据",
        "给出下一步验证命令，不要直接要求用户大面积重装依赖",
        "如果日志被截断，说明结论置信度"
      ].join("\n")
    };
  }
};
```

这个动作的关键是把证据和输出契约固定下来。模型不会因为看到长日志就开始自由发挥，而是被要求输出假设、证据、验证命令和置信度。用户也能快速判断下一步要不要执行。对于桌面端来说，这种结构化约束比写一个很长的提示词更稳定。

“审查 diff”是另一个高频动作。它需要读取 Git diff，但不应该默认读取未跟踪的大文件，也不应该把密钥文件发给外部模型。实现上可以先生成 diff stat，再根据文件类型和大小决定是否包含 patch。对超大 diff，可以按文件分组摘要，再让用户选择重点。

```ts
type DiffFileSummary = {
  path: string;
  status: "added" | "modified" | "deleted" | "renamed";
  additions: number;
  deletions: number;
  language: string;
  includedPatch: boolean;
  reason?: string;
};

type DiffReviewRequest = {
  branch: string;
  baseRef: string;
  files: DiffFileSummary[];
  patches: Array<{ path: string; patch: string }>;
  reviewFocus: Array<"correctness" | "tests" | "security" | "maintainability" | "docs">;
};
```

有了这个结构，DeepChat 可以做更细的策略：小 diff 直接审查；大 diff 先按文件列出摘要，让用户选择关注范围；包含 `.env`、证书、数据库导出等敏感路径时直接阻止外发；生成文件和锁文件默认只做统计，不进入模型上下文。这样既减少 token，也降低泄漏风险。

文件读取也需要上下文窗口策略。开发者经常问“这个函数有没有问题”，模型真正需要的可能包括函数定义、调用方、相关类型、测试文件和最近修改。DeepChat 可以做一个轻量的邻域收集器：从选区开始，向上找到函数或类边界，向下包含必要的错误处理分支，再用语言服务或简单文本搜索找调用点。第一版不一定要做完整语义索引，但至少要避免只给选中的三行代码。

```text
选区上下文收集策略：
1. 保留用户选中的精确行号，作为主要证据。
2. 向上扩展到最近的函数、方法、类型或导出声明。
3. 向下扩展到当前语义块结束，避免截断 defer、catch、return。
4. 查找同文件内相关类型、常量和私有 helper。
5. 对调用点只取文件路径、函数名和少量片段，不展开全文件。
6. 如果总 token 超过预算，优先保留选区、签名、错误处理和测试。
```

写入文件的快捷流要更谨慎。模型可以给补丁计划，也可以生成 unified diff，但落地前必须显示变更。桌面端最好把“生成建议”和“应用修改”拆开：先让模型输出修改意图和候选 patch，再由 DeepChat 校验 patch 是否能应用、是否越过允许路径、是否修改了用户没有授权的文件。应用后还要能展示 diff，并把验证命令放到下一步。

命令执行同样要分级。只读命令如 `git status`、`go test ./...`、`npm test` 通常风险较低，但也可能耗时或访问网络；修改命令如 `rm`、`git clean`、数据库迁移、包管理器安装，需要明确确认；包含管道下载执行、强制删除、重置分支的命令应该默认禁止或要求用户手动复制到终端。快捷流的目标是减少重复劳动，不是替用户承担不可逆操作。

| 命令类型 | 例子 | 默认处理 | 说明 |
| --- | --- | --- | --- |
| 只读检查 | `git status`、`go test ./...`、`npm run lint` | 可建议，执行前轻确认 | 仍要显示工作目录和预计影响 |
| 生成产物 | `npm run build`、`go generate` | 需要确认 | 可能修改文件或耗时较长 |
| 依赖安装 | `npm install`、`go get` | 需要确认 | 会改 lockfile，可能访问网络 |
| 数据迁移 | `rails db:migrate`、自定义迁移脚本 | 高风险确认 | 可能影响本地或远程数据 |
| 破坏性操作 | `rm -rf`、`git reset --hard`、`git clean -fd` | 默认禁止 | 用户明确要求也要二次确认 |

最后，快捷流需要和会话系统结合。每次动作都应该在会话里留下可读记录：使用了哪个动作、读取了哪些证据、是否截断、模型建议了什么、用户是否应用、验证结果如何。这样过几天回看时，不会只看到一段回答，而能看到完整的工程过程。对于团队协作，这些记录也可以转成 issue、PR 描述或复盘材料。

## 测试评测

开发者快捷流的测试重点不在“模型是否永远答对”，而在上下文是否准确、权限是否守住、输出是否可执行、失败后是否能继续。可以从单元测试、集成测试、回放评测和人工走查四层展开。

单元测试覆盖动作构建。给定选区、终端日志、diff 和项目摘要，检查 `DeveloperContextPackage` 是否包含正确证据、是否标记截断、是否保留路径和行号、是否设置输出契约。这里不需要调用模型，应该是确定性测试。比如终端日志超过限制时，测试要确认 `truncated=true`，并且上下文里写明日志只包含尾部。

集成测试覆盖权限和范围。准备一个临时仓库，里面放普通源码、`.env`、大文件、生成目录和未跟踪文件。运行“审查 diff”动作，确认敏感文件不会进入模型上下文，生成目录被排除，大 diff 会被摘要而不是全量注入。再测试写入动作，确认模型建议的 patch 不能修改允许范围之外的文件。

回放评测覆盖真实开发场景。可以收集一批脱敏样本：测试失败、类型错误、构建失败、代码审查、提交说明、迁移计划。每个样本包括输入现场、期望动作、可接受输出结构和验证标准。回放时可以替换不同模型，但固定上下文包，观察模型输出差异。这样能把快捷流质量和模型能力部分分离。

人工走查关注体验摩擦。让开发者完成几个任务：根据失败日志定位问题、审查一个小 diff、为一个函数补测试、把修改整理成提交说明。记录他们在哪一步停顿、是否知道 DeepChat 读取了什么、是否信任建议命令、是否能找到 diff 预览、失败后是否知道怎么继续。快捷流是高频工具，少一次不必要的点击都值得优化，但不能用隐藏风险换点击数。

| 测试对象 | 检查点 | 自动化方式 | 通过标准 |
| --- | --- | --- | --- |
| 动作构建 | 输入契约、证据块、输出契约 | 单元测试 | 缺输入时报错，证据来源完整 |
| 上下文截断 | 日志、文件、大 diff | 快照测试 | 明确标记截断和保留范围 |
| 权限分层 | 读文件、写文件、执行命令 | 集成测试 | 敏感路径不外发，高风险命令阻止 |
| 补丁应用 | unified diff、路径检查 | 临时仓库测试 | 只修改授权文件，失败可解释 |
| 诊断闭环 | 命令失败到验证命令 | 场景回放 | 输出包含假设、证据和下一步 |
| 用户体验 | 点击路径、确认文案、错误恢复 | 人工走查 | 用户知道系统做了什么 |

我会把“错误拒绝率”和“危险放行率”分开看。错误拒绝率太高，快捷流会显得碍手碍脚；危险放行率只要出现，就会损害信任。开发者工具宁可在破坏性操作上多一步确认，也不要为了流畅感默认执行不可逆命令。对低风险只读动作，可以通过记住用户偏好降低确认成本；对高风险动作，偏好也不应该完全跳过确认。

## 失败模式

第一个失败模式是“上下文过宽”。用户只是问一个函数，系统把整个文件、调用图、测试输出和历史聊天都塞给模型。结果模型回答变长、变慢，还可能被无关内容误导。解决办法是给每个快捷动作设置上下文预算和证据优先级。选区永远优先，相关定义其次，调用点摘要再次，历史会话只有在用户要求延续时才进入。

第二个失败模式是“上下文过窄”。只给选中的几行代码，模型看不到类型定义、错误处理、调用约束，于是提出不适用的修改。这个问题要靠邻域收集器和“证据不足”协议解决。模型如果缺少信息，应该明确请求更多文件，而不是编造项目结构。DeepChat 也可以在界面上提供“一键加入相关文件”。

第三个失败模式是“终端日志泄漏敏感信息”。构建日志、测试输出、环境变量打印里可能包含 token、内网地址、客户数据。终端快捷流需要做基础脱敏：匹配常见密钥格式、隐藏 Authorization 头、遮盖连接串密码、对长随机字符串做提示。脱敏不是百分百安全，所以敏感项目还要配合本地模型或显式确认。

第四个失败模式是“命令建议看似安全但实际有副作用”。例如 `npm install` 会改 lockfile，`go test` 可能启动依赖服务，某些项目的 `make test` 会清理数据库。DeepChat 不能只按命令名称判断风险，还要允许项目配置覆盖。仓库可以声明哪些命令是安全的，哪些必须确认，哪些禁止由助手建议执行。

第五个失败模式是“补丁应用越界”。模型生成的 diff 可能修改用户没有授权的文件，或者 patch 上下文过期导致应用到错误位置。应用前要检查路径白名单、文件哈希、上下文匹配和当前工作区是否有新变化。应用后必须展示 diff。失败时不要尝试强行修补，而是把冲突反馈给模型或用户。

第六个失败模式是“快捷动作泛化成万能入口”。如果每个按钮最后都只是把一堆上下文拼给模型，并要求“帮我看看”，动作就失去了价值。每个快捷动作必须有明确输出形态。诊断输出假设表，审查输出问题列表，测试生成输出用例矩阵，提交说明输出摘要和风险。输出越明确，用户越容易执行下一步。

第七个失败模式是“会话记录不可复盘”。开发者连续用了几个快捷动作后，只剩一串模型回答，看不出读了哪些文件、运行了哪些命令、应用了哪些修改。解决办法是把动作元数据作为会话事件保存，并且能展开查看证据。这个设计对调试模型也有帮助，因为很多错误来自输入证据，而不是模型本身。

## 上线 checklist

- 项目上下文能识别根目录、分支、diff、语言栈、测试入口和忽略规则。
- 快捷动作使用结构化定义，不把 prompt 散落在 UI 事件里。
- 每个动作声明必需输入、权限、风险等级和输出契约。
- 文件读取默认遵守 `.gitignore`、用户排除规则和大小限制。
- 终端日志注入时保留命令、工作目录、退出码、尾部输出和截断标记。
- 常见密钥、连接串、Authorization 头和长随机 token 会被脱敏或拦截。
- 选区上下文包含路径、行号、语义块边界和必要的相邻定义。
- 大 diff 先摘要，敏感路径和生成文件不默认进入模型上下文。
- 写入文件前校验路径、文件哈希和 patch 上下文，应用后展示 diff。
- 命令建议区分只读、生成产物、依赖安装、数据迁移和破坏性操作。
- 高风险命令默认禁止或二次确认，用户偏好不能完全绕过破坏性确认。
- 模型输出包含下一步验证动作，而不只是解释。
- 验证命令的运行结果能回到同一会话，形成诊断闭环。
- 会话事件记录动作、证据、截断、权限、应用结果和验证结果。
- 权限错误、patch 冲突、命令失败都有用户可理解的提示。
- 测试覆盖动作构建、上下文截断、敏感文件排除、补丁应用和命令分级。
- 发布前用真实脱敏仓库做人工走查，记录每个快捷动作的停顿点。
- 默认配置保守，允许高级用户按项目逐步放宽低风险确认。

## 总结

DeepChat 的开发者快捷流，核心不是让模型多懂一点代码，而是减少开发者在聊天、文件、终端和验证之间来回搬运上下文。一个好的快捷流应该知道当前项目是什么、用户选中了什么、刚刚哪个命令失败、工作区有哪些变更，也应该知道自己不能随便读什么、写什么、执行什么。

工程上，我会把快捷动作做成有输入契约、权限声明和输出形态的结构化操作。这样每个动作都能被测试、被审计、被逐步优化。上下文注入要带来源和截断说明，补丁落地要经过路径和哈希校验，命令执行要分风险等级，验证结果要回到会话。只有这样，AI 助手才会从“给建议”走向“帮助完成工程循环”。

开发者工具的效率来自短路径，但信任来自清晰边界。DeepChat 如果能把低风险动作做得足够顺，把高风险动作挡得足够稳，再把每次读取、建议、修改和验证都留在可复盘的会话里，就能形成一种很实用的开发体验：开发者仍然掌控仓库，模型负责承担重复分析和初步推理，终端和测试负责给出事实反馈。快捷流的终点不是更会聊天，而是让一次问题定位、一次 diff 审查、一次测试补齐都更快闭环。
