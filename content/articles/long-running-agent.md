## 问题背景

短任务 Agent 的工程难点通常在“能不能做对”，长任务 Agent 的工程难点还要加上“中断之后能不能接着做”。真实任务很少是一次模型调用结束。修一个复杂 bug 可能要读几十个文件、跑多轮测试、回滚错误尝试；整理一个知识库可能要处理上百篇文档、去重、抽取实体、生成索引；帮团队做发布准备可能要查 issue、跑 CI、写变更说明、生成检查清单、等人工确认。只要任务超过几分钟，系统就必须面对进程重启、网络抖动、工具超时、上下文压缩、权限过期、用户改口和模型中途跑偏。

很多 demo 把 Agent 运行看成一段内存里的 while 循环：模型计划一步，工具执行一步，再把结果拼回上下文。这个模型适合演示，不适合生产。因为 while 循环没有持久状态，进程一挂就不知道已经做了什么；工具调用没有幂等键，重试可能重复发送邮件或重复改数据；中间判断没有记录，恢复后模型只能重新猜；用户确认没有绑定具体版本，恢复后可能拿旧确认执行新对象。长任务不是“让模型多跑几轮”，而是要有工作流系统的持久性、可恢复性和可审计性。

可恢复设计的目标不是永不失败，而是失败后知道停在哪里、为什么停、哪些副作用已经发生、哪些步骤可以重放、哪些步骤必须人工确认。Agent 由于有模型推理，恢复比传统任务队列更复杂。传统队列的步骤通常是确定性函数，输入输出清楚；Agent 的下一步可能依赖上下文摘要、模型采样、工具返回和用户反馈。如果恢复时重新让模型“想一想”，可能会走到另一条路线。生产系统要把“计划、证据、决策、工具调用、输出物”都记录下来，让恢复尽量从明确状态继续，而不是从模糊对话重演。

长任务还有一个容易被忽略的问题：用户不会一直盯着屏幕。任务运行中可能需要等待测试、等待审批、等待外部 API、等待另一个人回复。Agent 必须能把自己挂起，保存当前意图和待办事项，等条件满足后继续。这个能力看起来像任务调度，实际牵涉权限和上下文。比如用户授权 Agent 修改某个仓库，授权是否持续到明天；昨天读取的文件今天是否变化；昨天通过的测试今天是否仍然有效；昨天的人工确认是否还能用于今天执行。

我更倾向把长任务 Agent 设计成“有状态的执行器”，而不是“很长的聊天”。聊天是交互界面，执行器才是可靠边界。执行器维护任务状态机、checkpoint、工具调用日志、输出物索引和恢复策略。模型是执行器里的规划和解释组件，不是唯一状态源。这样做会牺牲一点原型速度，但能换来生产环境最需要的东西：任务可暂停、可恢复、可回放、可追责。

## 核心概念

长任务可恢复设计有五个核心概念：任务状态、checkpoint、可重放日志、幂等工具和恢复策略。任务状态描述当前处于哪一阶段，下一步期待什么。checkpoint 是某个稳定点的完整快照，包含计划、已完成步骤、关键上下文和输出物引用。可重放日志记录每次模型调用、工具调用、人工确认和状态转换。幂等工具保证重复执行不会产生不可控副作用。恢复策略决定中断后从哪里继续、哪些证据需要刷新、哪些步骤不能自动重试。

| 概念 | 解决的问题 | 典型内容 | 常见错误 |
| --- | --- | --- | --- |
| 任务状态 | 当前任务在哪里 | 阶段、目标、阻塞点、下一动作 | 只保存自然语言摘要 |
| checkpoint | 稳定恢复点是什么 | 计划、证据、文件版本、输出物 | 每一步都保存大上下文但无语义 |
| 可重放日志 | 发生过什么 | 模型输入摘要、工具参数、结果、耗时 | 只记录最终答案 |
| 幂等工具 | 重试是否安全 | 幂等键、执行哈希、去重表 | 超时后盲目重试写操作 |
| 恢复策略 | 中断后怎么继续 | 重放、刷新、跳过、等待确认 | 恢复时重新让模型自由规划 |

任务状态要比“进行中、成功、失败”更细。至少需要 `created`、`planning`、`running`、`waiting_user`、`waiting_external`、`verifying`、`suspended`、`completed`、`failed`、`cancelled` 这些状态。状态越粗，恢复越危险。比如同样是 `running`，正在读文件和正在执行支付退款的风险完全不同；同样是 `waiting`，等待用户确认和等待 CI 结果的恢复方式也不同。

checkpoint 的边界很重要。不是每次模型输出都适合作为 checkpoint。一个好的 checkpoint 应该出现在语义稳定点：计划已确认、关键文件已读取、dry-run 已完成、补丁已写入、测试已通过、用户已确认、外部任务已提交。checkpoint 要能回答三个问题：恢复后是否知道目标，是否知道已经完成的副作用，是否知道下一步的最小安全动作。如果回答不了，只是把一段上下文存下来，恢复时仍然会混乱。

可重放日志不是为了让系统把所有事情再执行一遍，而是为了让系统能重建事实。模型调用通常不需要完全重放，但要保存模型当时看到的关键上下文版本、输出的计划、选择工具的理由。工具调用要保存参数、幂等键、调用开始和结束时间、返回摘要、错误分类和副作用结果。人工确认要保存确认人、确认内容、确认对象集合、确认时间和绑定哈希。恢复时先读日志，不要先问模型。

幂等工具是长任务的生命线。只读工具天然容易重试，写操作必须设计幂等。幂等键可以来自 `task_id + step_id + normalized_arguments_hash`。如果同一步因为网络超时被重试，工具网关应该返回上一次执行结果，而不是再次执行。对于不能幂等的外部系统，要在 Agent 层限制自动重试，改成人工确认或补偿流程。

恢复策略要显式保存。中断后有几种选择：从最近 checkpoint 继续；重新刷新外部证据后继续；回滚到上一个稳定点；等待用户确认；终止并生成恢复报告。不要让模型在恢复时自由解释“我应该继续做什么”。模型可以帮忙总结恢复上下文，但执行器必须先给出安全边界。

checkpoint 还要区分“事实快照”和“推理快照”。事实快照来自工具和控制器，例如文件哈希、外部对象版本、测试命令结果、已创建的工单 ID、用户确认记录；推理快照来自模型，例如当前假设、下一步建议、风险解释。恢复时事实快照优先级更高，推理快照只能作为辅助。这样设计可以防止模型在摘要里把猜测写成事实，也能让执行器在关键前提变化时果断暂停，而不是沿着过期假设继续执行。

任务队列是长任务可靠性的另一半。一个长任务不应该一直占着同一个进程等待外部事件，而应该被拆成可调度的执行片段。每个片段拿到任务租约，读取最新 checkpoint，推进一个有限步骤，然后续租、挂起或释放。队列消息里只放 `task_id`、唤醒原因和尝试次数，不放完整上下文；完整状态从持久化存储读取。这样即使消息重复投递、worker 重启、调度器延迟，系统也能靠状态机和租约判断是否真的应该继续。

## 架构/流程图解说明

长任务 Agent 的核心架构可以分为控制面和执行面。控制面负责任务状态、调度、权限、恢复和人工交互；执行面负责模型规划、工具调用、文件修改和验证。两者通过持久化日志连接。

```text
用户提交目标
   |
   v
任务控制器：创建 task_id、权限上下文、初始状态
   |
   v
计划阶段：模型生成计划 -> 策略层检查 -> 可选人工确认
   |
   v
执行循环
   |-- 读取 checkpoint 和当前状态
   |-- 组装本轮上下文
   |-- 模型选择下一步
   |-- 工具网关执行并记录幂等结果
   |-- 状态转换和 checkpoint 判断
   |-- 验证、等待、继续或暂停
   |
   v
完成报告：输出物、证据、测试、剩余风险
```

这套流程里，任务控制器是持久化系统，不应该依赖模型上下文。它保存任务目标、权限、状态、当前阶段、创建人、最后活动时间、取消信号和恢复策略。模型可以建议状态转换，但最终转换由控制器执行。例如模型说“测试已经通过，可以完成”，控制器要检查是否真的有最近的测试通过记录，且测试时间晚于最后一次文件修改。

工具网关在长任务里要承担更多职责。短任务里工具失败可以直接把错误返回模型，长任务里还要判断错误是否可重试、是否污染 checkpoint、是否需要暂停。网络超时、速率限制、权限过期、参数错误、外部系统业务拒绝，是不同类型的失败。工具网关应该返回结构化错误，而不是一段自然语言。执行器根据错误分类决定重试、退避、刷新权限、要求用户输入或标记任务失败。

checkpoint 触发可以用规则加人工标注。比如每次完成高风险写操作后必存；每次用户确认后必存；每次生成重要输出物后必存；运行超过一定时间或步骤数后保存轻量 checkpoint。对于代码任务，写文件前、写文件后、测试通过后是三个重要点。写文件前保存读到的证据和计划，写文件后保存 diff 和理由，测试通过后保存命令结果和最终风险。

恢复流程要先“验证世界是否还一样”。如果任务中断前读取了某个文件，恢复时要检查文件 mtime 或内容哈希；如果中断前 dry-run 影响了 120 条记录，恢复时要重新确认集合是否变化；如果中断前拿到用户授权，恢复时要检查授权是否过期。恢复不是简单从日志末尾接着跑，而是确认关键前提仍然成立。

恢复动作最好设计成小而确定的第一步。比如任务在测试前中断，恢复后的第一步不是“继续修 bug”，而是“确认工作区 diff 仍是 checkpoint 中的 diff，并运行上次尚未完成的测试命令”。任务在批量写入前中断，第一步不是“执行写入”，而是“重新计算 dry-run 集合并比较确认哈希”。把恢复第一步收窄，能显著降低模型重新规划带来的漂移，也能让用户和审计系统看清恢复到底改变了什么。

## 工程实现

一个可落地的长任务存储模型可以从几张表开始：`agent_tasks`、`agent_steps`、`tool_calls`、`checkpoints`、`artifacts`、`approvals`。不一定一开始就上复杂工作流引擎，但必须把这些概念分清。

```sql
create table agent_tasks (
  id text primary key,
  user_id text not null,
  workspace_id text not null,
  goal text not null,
  status text not null,
  phase text not null,
  recovery_policy text not null,
  permission_snapshot jsonb not null,
  created_at timestamptz not null,
  updated_at timestamptz not null,
  completed_at timestamptz
);

create table agent_steps (
  id text primary key,
  task_id text not null references agent_tasks(id),
  step_index integer not null,
  kind text not null,
  intent text not null,
  status text not null,
  input_refs jsonb not null,
  output_refs jsonb not null,
  decision_summary text,
  created_at timestamptz not null,
  finished_at timestamptz
);

create table tool_calls (
  id text primary key,
  task_id text not null,
  step_id text not null,
  tool_name text not null,
  tool_version text not null,
  idempotency_key text not null,
  arguments_hash text not null,
  arguments_redacted jsonb not null,
  status text not null,
  result_summary jsonb,
  side_effects jsonb,
  error_class text,
  started_at timestamptz not null,
  finished_at timestamptz
);
```

这里有几个细节。`permission_snapshot` 保存任务开始时的权限上下文，但恢复时不能只信它，还要重新校验当前权限。`input_refs` 和 `output_refs` 不直接存大文本，而是引用对象存储或日志仓库里的证据。`arguments_hash` 用于判断幂等请求是否真的是同一操作。`side_effects` 要保存外部系统确认的结果，例如创建的 issue ID、发送的消息 ID、修改的文件路径或数据库事务 ID。

checkpoint 可以用更偏文档的结构：

```json
{
  "checkpoint_id": "ckpt_018",
  "task_id": "task_123",
  "phase": "verifying",
  "goal": "修复订单导出测试失败并保持接口兼容",
  "completed_steps": [
    "读取失败日志",
    "定位 exportOrders 的时间格式处理",
    "修改 formatter.go",
    "运行 go test ./internal/export"
  ],
  "current_state": {
    "changed_files": ["internal/export/formatter.go"],
    "last_successful_command": "go test ./internal/export",
    "pending_checks": ["go test ./..."],
    "blocked_on": null
  },
  "evidence": [
    {"type": "file_hash", "path": "internal/export/formatter.go", "sha256": "abc..."},
    {"type": "command", "ref": "cmd_992", "status": "passed"}
  ],
  "next_safe_action": "运行全量测试，若失败只分析新失败，不回滚已通过的局部修复"
}
```

这个 checkpoint 不是聊天摘要，而是恢复契约。恢复后的 Agent 不需要重新猜已经做过什么；它知道改了哪些文件、哪个命令通过、还缺什么检查、下一步安全动作是什么。`next_safe_action` 尤其重要，它把恢复后的第一步收窄，避免模型一醒来就重新规划整个任务。

执行循环可以按以下伪代码组织：

```go
for {
    task := store.LoadTask(taskID)
    if task.Status.IsTerminal() {
        return
    }

    if task.HasCancelSignal() {
        suspendWithCheckpoint(task, "cancelled_by_user")
        return
    }

    ctx := contextBuilder.Build(task)
    decision := model.PlanNextStep(ctx)
    decision = policy.Check(decision, task)

    result := executor.Execute(decision, WithIdempotency(task.ID, decision.StepID))
    store.AppendStep(task.ID, decision, result)

    transition := stateMachine.Next(task, decision, result)
    store.UpdateTaskState(task.ID, transition)

    if checkpointPolicy.ShouldSave(task, decision, result) {
        store.SaveCheckpoint(buildCheckpoint(task, result))
    }

    if transition.RequiresWait() {
        scheduler.Park(task.ID, transition.WakeupCondition)
        return
    }
}
```

这个循环的重点不是代码多复杂，而是每个动作都落盘。即使进程在 `Execute` 后、`AppendStep` 前崩溃，也要靠工具层幂等和事务边界恢复。更严谨的做法是先创建 `tool_call` 记录为 `started`，再执行工具，工具完成后更新结果。恢复时发现有 `started` 但无结果的调用，就根据幂等键查询外部系统或标记为不确定，不能直接再次执行。

人工确认也要有版本绑定。例如一个 Agent 准备批量关闭 37 个 issue，确认记录不能只写“用户同意关闭 issue”。它必须保存 issue 列表、每个 issue 的版本或更新时间、关闭理由、确认人和确认时间。恢复时如果 issue 列表变化，原确认失效。没有版本绑定的确认，在长任务里很容易变成安全漏洞。

幂等实现不能只停留在客户端传一个字符串。工具网关需要维护去重记录，至少包含幂等键、规范化参数哈希、调用状态、外部结果引用和副作用摘要。同一个幂等键但参数哈希不同，应当直接拒绝，因为这说明恢复逻辑或调用方已经混乱；同一个幂等键且上一轮状态为成功，就返回旧结果；上一轮状态不确定，则先查询外部系统或进入人工处理。对于邮件、支付、权限变更、删除数据这类不可轻易补偿的动作，默认策略应该是“不确定就暂停”，而不是自动再试一次。

副作用治理要把工具分级。只读工具可以自动重试，临时文件和草稿类写入可以自动清理或覆盖，外部可见写入需要 dry-run 和确认，破坏性操作需要更强的双人确认或明确禁止自动执行。每次副作用都要有所有权：谁发起、哪个任务、哪个步骤、基于哪份证据、如何回滚或补偿。长任务 Agent 一旦跨越多个系统，最危险的不是模型答错一句话，而是在用户已经离开页面后悄悄完成了一组不可逆操作。

## 测试评测

长任务 Agent 的测试要故意制造中断。只跑 happy path 没意义。最基本的测试矩阵包括：模型调用前崩溃、模型调用后崩溃、工具执行前崩溃、工具执行后崩溃、写操作超时、checkpoint 写入失败、用户在等待期间修改目标、权限过期、外部系统返回重复结果、调度器重复唤醒。每一种都要验证系统不会丢状态、不会重复副作用、不会把不确定状态当成功。

| 测试场景 | 注入方式 | 期望结果 |
| --- | --- | --- |
| 工具执行后进程崩溃 | 在工具返回后杀进程 | 恢复时用幂等键查询结果，不重复执行 |
| 用户确认后对象变化 | 修改 dry-run 结果集合 | 原确认失效，任务回到等待确认 |
| 文件被外部修改 | 改变目标文件哈希 | 恢复时刷新证据，避免覆盖他人改动 |
| 权限过期 | 移除 token 或 ACL | 任务暂停并要求重新授权 |
| 调度重复唤醒 | 同一 task 收到两次 wakeup | 状态机只允许一个执行器持有租约 |
| 摘要损坏 | 删除最新上下文摘要 | 系统可从 checkpoint 和日志重建关键状态 |

可恢复性评测还要看恢复质量，而不只是任务是否最终成功。指标可以包括恢复耗时、重复工具调用次数、重复写操作次数、不确定状态数量、需要人工介入次数、恢复后偏离原计划的比例。对于工程 Agent，我会特别关注“恢复后是否会重新修改已经通过测试的代码”。如果系统经常从头开始探索，说明 checkpoint 没有提供足够约束。

长任务还需要并发测试。一个任务可能被多个 worker 竞争执行，或者用户在界面上点击取消时，后台 worker 正在跑工具。任务表要有租约字段，例如 `locked_by`、`lock_expires_at`。worker 续租失败就必须停止，不允许两个执行器同时推进同一任务。对于高风险写操作，锁应该覆盖确认到执行的关键区间，避免重复提交。

评测中不要忽略人机交互。恢复报告应该能让用户看懂：任务为何暂停，已经完成什么，哪些外部副作用已经发生，下一步需要用户做什么。用户说“继续”时，系统要能把这句话绑定到具体恢复动作，而不是泛泛追加到聊天历史。长任务的用户体验不是持续输出大量过程日志，而是在关键节点提供可理解、可确认、可撤销的信息。

## 失败模式

第一种失败是“恢复即重跑”。任务中断后，系统把历史聊天重新喂给模型，让模型重新规划，结果重复调用工具、重复修改文件，甚至走到另一条路线。根因通常是没有语义 checkpoint，只有对话记录。修复方式是把已完成步骤、副作用和下一安全动作结构化保存。

第二种失败是“超时状态不确定”。写操作请求发出后网络断开，系统不知道外部是否执行成功，于是自动重试。这个问题必须靠幂等键、外部查询和不确定状态处理。遇到不确定状态时，宁可暂停并查询，也不要盲目再次执行。

第三种失败是“旧确认执行新对象”。用户确认时看到的是一组对象，恢复时对象集合已变化，但系统仍然使用旧确认。解决办法是确认绑定对象哈希和版本，执行前重新比较。凡是批量操作、外发、删除、权限变更，都必须这样做。

第四种失败是“摘要漂移”。长任务运行一小时后，任务摘要逐渐偏离原始目标，把临时猜测写成事实。恢复时模型依据错误摘要继续推进。解决办法是摘要分层：硬约束保留原文，证据引用可回源，模型推断标记为低置信，重大结论需要工具或用户证据。

第五种失败是“等待没有语义”。任务状态只是 `paused`，不知道等待什么。一天后恢复，系统无法判断是等用户、等 CI、等权限还是等外部回调。等待状态必须保存唤醒条件、过期时间、提醒策略和恢复动作。等待用户确认和等待定时重试是两种完全不同的状态。

第六种失败是“日志不可审计”。系统保存了大量原始 prompt，却没有步骤、工具、证据和副作用索引。出了事故后很难回答谁确认了什么、哪个工具改了哪个对象、哪次测试覆盖了改动。审计日志要服务排查，不是为了堆存储。

## 上线 checklist

- 为任务建立明确状态机，区分运行、等待用户、等待外部、验证、暂停、失败、取消和完成。
- 每个任务有稳定 `task_id`，每个步骤有 `step_id`，每次工具调用有 `tool_call_id` 和幂等键。
- 所有工具调用先落盘为开始状态，再执行，再写入结构化结果和副作用摘要。
- 写操作必须支持幂等或显式禁止自动重试，超时后进入不确定处理流程。
- checkpoint 出现在语义稳定点，至少保存目标、阶段、已完成步骤、证据、输出物和下一安全动作。
- 恢复前检查文件哈希、对象版本、权限、授权有效期和外部系统状态。
- 人工确认绑定对象集合、参数哈希、影响范围、确认人和确认时间。
- 调度器使用租约控制，避免多个 worker 同时推进同一任务。
- 等待状态保存唤醒条件、过期时间、提醒策略和恢复动作。
- 模型生成的计划和总结不能作为唯一事实源，关键状态必须由工具结果和控制器记录支撑。
- 对恢复路径做故障注入测试，覆盖崩溃、超时、重复唤醒、权限过期和对象变化。
- 最终报告列出完成步骤、输出物、验证命令、未覆盖风险和可回滚信息。

## 总结

长任务 Agent 的可恢复设计，本质上是把模型驱动的开放式工作放进一个可靠执行框架。模型负责理解目标、制定局部计划、解释结果；执行器负责保存状态、控制副作用、处理恢复、管理确认和记录审计。只要任务会跨越分钟级时间、会调用有副作用工具、会等待外部条件，就不能只靠内存循环和聊天历史。

工程上最值得先做的三件事是：持久化任务状态和步骤日志，为所有写工具接入幂等键，在关键稳定点保存可恢复 checkpoint。做到这三点，系统即使失败也能清楚地停住，而不是把风险留给下一次模型调用。长任务 Agent 的质量，不体现在它永远不中断，而体现在它中断之后仍然知道自己是谁、做过什么、还能安全地做什么。
