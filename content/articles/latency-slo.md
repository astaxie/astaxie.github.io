## 问题背景

AI 应用的延迟问题很容易被一个平均值掩盖。仪表盘上显示平均响应三秒，看起来还能接受，但用户体感可能很差：有人半秒看到首字，有人等了十二秒才看到任何反馈；有人答案流式输出很快，最后引用检查又卡住；有人第一次调用正常，重试后排队到超时；有人在移动网络下被代理缓冲，界面像死了一样。平均延迟不能解释这些体验差异，甚至会把最需要优化的尾部请求藏起来。

传统接口常用一个请求响应时间来描述延迟。AI 应用不够用。一次智能请求可能包含输入规范化、权限检查、检索、重排、上下文压缩、模型排队、首 token、持续生成、工具调用、结构化校验、引用检查和结果持久化。用户关心的也不是同一个时间点。对聊天助手来说，首字出现得早，用户会觉得系统在工作；对结构化生成来说，最终 JSON 可用才算完成；对自动化代理来说，中间每个工具步骤都可能影响信任；对后台批处理来说，排队时间比首字更重要。延迟 SLO 必须按照体验边界拆开定义。

很多团队会在上线后才讨论“模型太慢”。这时往往已经把长上下文、串行工具、同步后处理和无限重试写进主流程，改起来牵一发而动全身。更稳妥的做法是在功能设计阶段就分配延迟预算：用户最多多久看到反馈，多久拿到可操作结果，哪些步骤可以并行，哪些步骤可以后台完成，哪些异常要快速降级，哪些场景允许排队。延迟预算不是运维附属品，它会反过来决定产品交互和系统架构。

延迟还和质量、成本、可靠性相互牵制。更强模型通常更慢，更长上下文通常更慢，更多工具和评测通常更慢。为了压低延迟直接砍上下文，可能导致答案质量下降；为了保证质量不断重试，可能让用户等到放弃；为了省成本使用批处理或低优先级队列，可能让交互体验不可接受。SLO 的价值就在于把这些取舍显式化。团队需要知道某个功能的目标不是“越快越好”，而是“首字 P95 小于两秒、可用答案 P95 小于十秒、超时后在十二秒内给出可理解降级”。

AI 应用还有一个特殊现象：用户对等待的忍耐取决于界面是否诚实。一个没有任何反馈的六秒等待，比一个一秒内开始流式输出、十秒完成的回答更让人焦躁。一个代理如果能清楚展示正在检索、正在读取文件、正在执行工具，用户愿意等更久；如果界面只有转圈，三秒就会让人怀疑系统卡住。延迟 SLO 不能只埋在后端指标里，还要映射到前端状态、文案、取消按钮和降级选项。

所以，延迟 SLO 的工程目标不是追求某个漂亮数字，而是建立一套从用户体验到后端执行计划的契约。它要定义可观测的 SLI，分配阶段预算，约束超时和重试，指导降级策略，并在发布前用评测和压测验证。没有这套契约，系统会在流量增长、模型抖动和工具变慢时随机牺牲体验。

## 核心概念

第一个概念是 SLI。SLI 是实际测量的指标，不是愿望。AI 应用常见的 SLI 包括请求排队时间、首字时间、首个有意义片段时间、完成时间、工具阶段耗时、重试后完成时间、取消响应时间、流式中断率和超时降级率。不同功能应该选择不同 SLI。聊天场景必须看首字时间，文档生成必须看完成时间，代理任务必须看每个步骤的进展时间，后台任务则更关注任务排队和总完成。

第二个概念是 SLO。SLO 是对 SLI 的目标承诺，通常用百分位表达。比如“交互式问答首字 P95 小于 1800ms，完整答案 P95 小于 9000ms”。不要只写 P50，也不要只写平均值。用户投诉通常来自尾部延迟，P95 和 P99 才能暴露排队、冷启动、慢工具和大上下文问题。SLO 也要有时间窗口，比如按天、按周或按发布批次统计，否则单次尖峰会被长期平均稀释。

第三个概念是阶段预算。总延迟目标要拆给各阶段，否则每个模块都觉得自己慢一点没关系，最后整体不可用。一个九秒完成目标可以拆成：鉴权和策略 100ms，检索 700ms，重排 500ms，上下文组装 200ms，模型首字 1800ms，生成 5000ms，后处理 700ms。阶段预算不是死板限制，而是定位工具。某天完整答案变慢时，团队能看到是哪一段吃掉了预算。

第四个概念是体验边界。不是所有超时都等价。首字超时意味着用户看不到系统开始工作；完成超时意味着用户拿不到最终结果；工具超时意味着某个外部依赖不可靠；后处理超时意味着模型已经生成但产品还不能展示。每种超时对应不同用户体验。首字超时要尽快展示状态或降级，工具超时可以跳过非关键工具，后处理超时可以先展示草稿再补充引用。SLO 要把这些边界写清楚。

第五个概念是错误预算。延迟 SLO 不是要求百分之百请求都达标，而是允许小比例请求失败或超时。错误预算可以用来指导发布节奏。如果本周首字 SLO 已经消耗过多，就不应该继续发布会增加上下文和工具调用的新功能；如果错误预算充足，可以更大胆地测试高质量但略慢的路径。错误预算让性能讨论从主观感受变成明确决策。

第六个概念是降级。降级不是失败后随便返回一句“请稍后再试”。好的降级要在体验边界内发生，并尽量保留用户价值。检索慢时可以减少候选片段，工具慢时可以基于已有信息回答并标注缺失，模型首字慢时可以切换更快模型，完整生成慢时可以先返回摘要再继续扩展。降级策略必须提前设计，不能靠异常处理临时拼凑。

| 场景 | 首要 SLI | 推荐 SLO | 体验边界 | 常见降级 |
| --- | --- | --- | --- | --- |
| 知识库问答 | 首字时间、完整答案时间 | 首字 P95 < 2s，完成 P95 < 10s | 用户需要看到系统在工作 | 缓存答案、减少片段、快速模型 |
| 合同审查 | 可用报告完成时间 | 完成 P95 < 45s | 用户接受等待但需要进度 | 分章节输出、后台完成、人工复核 |
| 代码助手 | 首个建议时间 | 首建议 P95 < 1.5s | 编辑器不能被阻塞 | 本地规则先返回、远端建议补齐 |
| 数据分析代理 | 步骤进展间隔 | 每步进展 P95 < 5s | 用户要理解代理在做什么 | 跳过非关键图表、返回中间表 |
| 批量标注 | 队列等待、任务吞吐 | 每小时完成固定批次 | 不需要交互式反馈 | 低优先级队列、批处理模型 |

这些目标不是通用答案，而是提醒团队按产品语义定义指标。一个系统可以同时存在多个 SLO。免费用户和企业用户可以不同，交互请求和后台任务可以不同，低风险建议和高风险操作可以不同。重要的是每个 SLO 都能落到真实测量和真实策略上。

## 架构/流程图解说明

延迟 SLO 要通过架构表达出来。一个没有 trace、没有阶段超时、没有取消传播的系统，很难靠事后优化满足 SLO。推荐把每次 AI 请求拆成可观测阶段，并在入口创建统一的延迟预算对象。每个阶段领取自己的预算，超时后按策略降级或中止。

```text
用户操作
  |
  v
入口网关：创建 trace 和 latency budget
  |
  v
请求准备
  |-- 鉴权
  |-- 会话裁剪
  |-- 意图识别
  |
  v
并行上下文阶段
  |-- 检索
  |-- 缓存查找
  |-- 工具预取
  |
  v
执行计划选择
  |-- 模型路由
  |-- 片段数量
  |-- 是否需要后置评测
  |
  v
流式生成
  |-- 首字
  |-- 中间片段
  |-- 完整答案
  |
  v
后处理与持久化
  |-- 引用校验
  |-- 结构化校验
  |-- 日志和指标
```

这里最重要的是预算对象要贯穿调用链，而不是每个模块各自设置 timeout。入口如果承诺十秒内完成，那么检索不能自己等待五秒，模型调用不能自己等待二十秒，后处理不能无限等待。所有外部调用都应该从剩余预算里扣时间。这样当上游已经花掉太多时间时，下游会自动选择更保守路径，而不是继续执行必然超时的计划。

一个 trace 视图应该能回答这些问题：请求排队花了多久，首字前哪些阶段耗时最多，模型服务是排队慢还是生成慢，工具是否串行阻塞，用户是否在完成前取消，超时发生时剩余预算是多少，降级是否按预期触发。没有这些信息，团队只能猜测“模型慢”，但模型慢可能只是上下文太长、网络代理缓冲、工具阻塞或后处理卡住。

```text
trace tr_21c8 latency budget 10000ms
  gateway.admission          24ms
  auth.policy                41ms
  context.trim               18ms
  retrieval.vector_search   230ms
  retrieval.rerank          410ms
  cache.lookup               35ms
  prompt.assemble            64ms
  model.queue               520ms
  model.first_token        1120ms
  model.stream             4820ms
  citation.check            340ms
  response.persist           55ms
  total                    7677ms
```

流程上还要区分同步路径和异步路径。用户必须等待的步骤应该少而明确，非关键步骤要尽量放到后台。比如答案生成后，详细评测可以异步抽样；长报告可以先返回任务 ID 和首段摘要，再后台生成完整版本；会话标题生成、推荐追问、使用量统计都不应该阻塞主响应。很多延迟优化不是把代码写得更快，而是把不该阻塞的事情移出用户等待路径。

## 工程实现

实现延迟 SLO 的第一步是定义预算对象。预算对象既包含总期限，也包含阶段上限和降级策略。它应该从产品配置生成，而不是散落在代码常量里。

```go
type LatencyBudget struct {
	TraceID              string
	Profile              string
	StartedAt            time.Time
	Deadline             time.Time
	FirstTokenTargetMs   int
	CompleteTargetMs     int
	StageLimits          map[string]time.Duration
	AllowDegrade         bool
	AllowBackgroundWork  bool
	DegradeReason        string
}

func (b LatencyBudget) Remaining(now time.Time) time.Duration {
	if now.After(b.Deadline) {
		return 0
	}
	return b.Deadline.Sub(now)
}
```

在 Go 服务里，预算应该和 `context.Context` 绑定。入口根据 SLO 创建总 deadline，阶段调用再用 `context.WithTimeout` 派生更短的上下文。注意不要让阶段 timeout 大于剩余预算，也不要吞掉取消信号。用户关闭页面、前端取消请求、网关断开连接，都应该传播到检索、工具和模型调用。否则用户已经走了，后端还在生成，延迟指标看起来只是慢，成本和资源却继续消耗。

```go
func runRetrieval(ctx context.Context, b LatencyBudget, query string) ([]Chunk, error) {
	limit := b.StageLimits["retrieval"]
	if remaining := b.Remaining(time.Now()); remaining < limit {
		limit = remaining
	}
	if limit <= 0 {
		return nil, context.DeadlineExceeded
	}
	stageCtx, cancel := context.WithTimeout(ctx, limit)
	defer cancel()
	return searchChunks(stageCtx, query)
}
```

第二步是把串行链路改成可控并行。很多 AI 请求首字慢，是因为缓存查找、检索、用户画像、工具预取和策略查询按顺序做。它们中有些可以并行，有些可以先返回粗结果再补充。并行不是无脑开 goroutine，而是要有 fanout 上限、取消传播和最慢任务截断。比如知识库问答可以同时查精确缓存和向量检索；缓存命中且新鲜度足够时取消检索；检索慢但缓存有旧答案时，在允许 stale 的场景先返回旧答案并后台刷新。

第三步是控制上下文大小。模型首字延迟通常和输入 token、模型排队和供应商状态有关。输入越长，首字越慢，生成越慢。延迟预算应该反过来影响上下文组装：剩余预算充足时可以带更多片段和更强引用检查；预算紧张时减少片段、优先高分证据、关闭低价值工具。上下文裁剪不能只按字符数，要按信息价值排序。高相关、权限明确、更新时间新的片段优先，重复片段和长模板应该被压缩。

一个实际的执行计划可以这样描述：

```yaml
profiles:
  interactive_qa:
    first_token_target_ms: 1800
    complete_target_ms: 9000
    stages:
      admission: 100ms
      retrieval: 800ms
      rerank: 500ms
      prompt_assemble: 200ms
      model_first_token: 2200ms
      model_stream: 5500ms
      postprocess: 700ms
    degradation:
      retrieval_timeout: use_top_keyword_results
      rerank_timeout: use_vector_order
      model_first_token_timeout: switch_fast_model
      postprocess_timeout: return_answer_without_extra_suggestions
```

第四步是流式输出的真实可用性。很多系统后端已经拿到 token，却因为代理、框架缓冲或前端渲染策略，用户看不到首字。SSE 和 chunked response 要逐层验证：应用服务是否 flush，反向代理是否关闭缓冲，CDN 是否支持流式，前端是否按块渲染，移动端是否有最小缓冲阈值。首字 SLI 应该同时记录服务端首 token 时间和浏览器可见首字时间。两者差异大时，问题不在模型，而在传输链路。

第五步是后处理前移或异步化。结构化输出校验、引用检查和安全检查很重要，但不能全部阻塞用户看到任何内容。对于低风险问答，可以先流式展示答案主体，引用检查完成后再显示引用状态；对于高风险操作，必须等校验通过才允许提交。不同风险等级要有不同阻塞边界。不要用一个全局后处理流程处理所有请求。

第六步是把超时设计成产品状态。后端返回 `deadline_exceeded` 只是技术事实，前端需要知道该展示什么。响应可以包含 `degraded=true`、`degrade_reason=retrieval_timeout`、`partial=true`、`retry_after_ms`、`background_task_id`。这样界面可以展示部分答案、继续生成按钮、后台完成状态或重新尝试入口。没有结构化降级信息，前端只能显示通用错误，用户会认为系统失败。

第七步是做入口准入，而不是让所有请求进入同一条慢队列。AI 服务最怕把交互请求、后台批处理、评测任务和重试请求混在一起。队列一旦拥塞，原本可以快速完成的小请求也会排在长任务后面，形成头阻塞。入口应该根据请求画像分配优先级和最大等待时间：编辑器补全、客服对话这类短交互请求走低等待队列；长文档分析和批量总结走可排队队列；离线评测和预热任务只能使用剩余容量。准入控制拒绝或延后请求，比让请求在队列里沉默等待二十秒更诚实。

可以用一个简单的调度结构表达这种策略：

```go
type AdmissionClass struct {
	Name             string
	MaxQueueWait     time.Duration
	MaxConcurrency   int
	ReservePercent   int
	CanUseBatchModel bool
	DropPolicy        string
}

var classes = []AdmissionClass{
	{Name: "interactive", MaxQueueWait: 300 * time.Millisecond, MaxConcurrency: 200, ReservePercent: 60, DropPolicy: "fast_degrade"},
	{Name: "analysis", MaxQueueWait: 3 * time.Second, MaxConcurrency: 80, ReservePercent: 30, DropPolicy: "background_task"},
	{Name: "eval", MaxQueueWait: 30 * time.Second, MaxConcurrency: 20, ReservePercent: 10, CanUseBatchModel: true, DropPolicy: "delay"},
}
```

这里的重点不是调度算法多复杂，而是资源隔离。交互请求需要稳定首字，不能被离线评测挤掉连接池；后台分析可以等待，但需要明确进度；评测任务应该能暂停，不能在业务高峰抢占容量。队列指标也要进入 SLO：如果请求还没进入模型调用就已经等待过久，后面再快也救不回体感延迟。很多系统只优化模型阶段，却忽略入口排队，最后在流量高峰表现为随机慢。

第八步是把模型输出速度纳入计划。首字快不代表完整答案快。不同模型的 token 生成速度、长上下文退化、并发限流策略都不同。对于长报告，系统应该估算输出长度，如果完整生成一定超过交互 SLO，就不要假装能同步完成。更好的方式是先生成概要和目录，把详细章节放入后台任务，或者分块生成并让用户明确看到进度。对于表格、JSON 和代码这类必须完整才可用的输出，则应该减少范围、拆分任务或提前提示用户需要更长时间。

第九步是把延迟和质量评测关联。一个请求很快返回空洞答案，不应该被算作成功；一个请求慢但完整解决高价值问题，也不应该和闲聊问题使用同一目标。评测数据可以给延迟 SLO 增加质量门槛，例如“完成 P95 小于十秒，且引用正确率不低于九成”“首字 P95 小于两秒，且首段不能是无意义占位”。这能防止团队为了达标而输出没有价值的快速文本。SLO 衡量的是可用体验，不只是网络时间。

## 测试评测

延迟测试要从用户路径开始，而不是只压测单个模型调用。一个完整评测集应该包含短问题、长问题、多文档问题、缓存命中、缓存未命中、工具慢响应、模型限流、用户取消和移动网络。每个样本都要记录首字、完成、阶段耗时和降级结果。只测一条理想路径，会让 SLO 在真实流量下一碰就碎。

| 测试类型 | 目的 | 关键指标 | 注意点 |
| --- | --- | --- | --- |
| 离线回放 | 比较不同执行计划 | 阶段 P50/P95、完成 P95 | 使用真实输入长度分布 |
| 负载压测 | 暴露排队和资源竞争 | 排队时间、超时率、CPU、连接数 | 区分模型慢和本服务排队 |
| 故障注入 | 验证降级 | 降级触发率、用户可见错误 | 模拟检索慢、工具慢、供应商 429 |
| 浏览器实测 | 验证首字可见 | 服务端首 token 与前端首字差 | 检查代理缓冲和渲染节流 |
| Canary | 控制发布风险 | 新旧版本 SLO 差异 | 按租户、功能和模型路由拆分 |

评测要看分布。P50 说明普通体验，P95 说明大多数用户是否可接受，P99 说明尾部风险和容量问题。还要看分桶：输入 token 分桶、租户分桶、模型分桶、功能分桶、网络地区分桶。一个总体 P95 达标，可能掩盖了长文档用户全部不达标；一个模型整体稳定，可能在某个供应商区域排队严重。延迟分析如果不分桶，只能得到模糊结论。

回归测试要绑定发布流程。每次修改 prompt、增加工具、调整检索片段数量、切换模型或改变后处理，都应该跑延迟回放。很多性能回退不是代码慢，而是 prompt 多了一段说明、工具 schema 膨胀、引用检查从抽样变成全量。发布说明里应该同时写质量变化和延迟变化。如果一次改动让答案质量提升一点，却让首字 P95 翻倍，产品必须明确接受这个取舍。

线上观测要记录用户取消。用户在首字前取消，通常说明等待不可接受；用户在生成中取消，可能说明答案方向不对或输出太慢；用户在完成后立即重试，可能说明质量不够。取消不是噪音，而是体验指标。模型调用层也要接收取消信号，否则取消只会减少前端等待，不会减少后端压力。

浏览器端 RUM 很重要。服务端记录完成八秒，不代表用户八秒看到结果。代理缓冲、弱网、前端主线程阻塞、Markdown 渲染过重、代码高亮过慢，都会扩大体感延迟。对于富文本、表格和代码块很多的 AI 输出，前端渲染本身可能成为瓶颈。RUM 指标至少要包含用户点击时间、请求发出时间、首个可见字符、首个可交互结果、完成渲染和取消时间。

## 失败模式

第一类失败是只设置总 timeout。总 timeout 看似简单，但阶段之间会互相挤占。检索用了八秒，模型还被允许跑十秒，最后用户等到十八秒才失败。解决办法是阶段预算加剩余时间检查，前面超支后后面必须降级或中止。

第二类失败是队列隐藏在供应商或内部池里。应用看到模型调用慢，却不知道慢在排队还是生成。需要记录请求发出时间、供应商首字时间、输出 token 速度和限流响应。如果供应商不提供细分指标，至少在内部记录连接池等待、请求发送和首字返回。

第三类失败是重试把尾部延迟拖垮。一次超时后立即重试，可能让用户等待更久，也加重拥塞。交互请求的重试要有严格预算，首字前重试尤其谨慎。很多时候快速降级比完整重试更好。后台任务可以重试，但要从交互队列隔离出去。

第四类失败是流式被缓冲。服务端代码使用流式 API，但 Nginx、CDN、网关或前端框架把内容攒到一定大小才发送。仪表盘显示模型首 token 很快，用户却看不到。解决办法是端到端测试流式，检查响应头、flush 行为、代理配置和浏览器渲染。首字 SLO 必须以用户可见时间为准。

第五类失败是后处理阻塞首字。模型生成前必须做的安全检查和权限检查当然要阻塞，但会话标题、推荐问题、完整日志写入、低风险评测不应该阻塞首字。把所有事情放在返回前做，会让用户为后台便利买单。

第六类失败是上下文无边界增长。历史消息、工具描述、检索片段和系统提示词一点点变长，首字延迟慢慢恶化，没有单次明显故障。需要为 prompt 各部分建立 token 预算，并在 trace 中记录。每次发布都看上下文长度分布，而不是只看模型耗时。

第七类失败是超时不一致。前端十五秒超时，网关三十秒，服务端六十秒，模型客户端没有超时。用户看到失败后，后端还在跑，下一次请求又进入拥堵队列。所有层的 deadline 要有明确关系：前端可以略大于服务端可用结果时间，服务端必须小于网关连接时间，外部调用必须小于剩余预算。

第八类失败是降级没有观测。系统可能已经大量降级，SLO 看起来达标，但质量和信任在下降。延迟指标必须和降级率、降级原因、质量分、用户重试率一起看。否则团队会误以为优化成功，实际上只是把慢请求变成了低质量结果。

## 上线 checklist

- 每个交互功能都有明确 SLI：首字、完整答案、步骤进展、取消响应或队列等待。
- SLO 使用 P95/P99 和固定时间窗口，不用平均值替代用户体验。
- 入口创建统一 latency budget，并通过 `context` 或等价机制传递到检索、工具和模型调用。
- 每个阶段有上限、剩余预算检查和结构化降级原因。
- trace 能展示排队、检索、重排、模型首字、生成、后处理和持久化耗时。
- 流式输出经过端到端验证，服务端首 token 和浏览器首字都有指标。
- 用户取消能传播到后端，已取消请求不会继续消耗模型和工具资源。
- 重试策略区分交互请求和后台任务，并受剩余预算约束。
- 上下文组装有 token 预算，发布时检查输入长度分布变化。
- 非关键后处理异步化，不阻塞首字或可用答案。
- 降级结果带结构化字段，前端能展示部分结果、后台任务或重试入口。
- 每次 prompt、模型、检索和工具变更都跑延迟回放和 canary 对比。

## 总结

延迟 SLO 的核心是把“快一点”变成可执行的工程契约。AI 应用不能只看平均响应时间，因为用户体验由首字、进展、完成、取消、重试和降级共同决定。一个好的 SLO 会定义真实 SLI，拆分阶段预算，绑定 trace，约束 timeout，并指导产品在异常时如何保留价值。

从落地顺序看，先把 trace 补齐，再定义每类功能的首字和完成目标，然后把总 deadline 传进检索、工具和模型调用。接着做流式端到端验证、取消传播、上下文预算和降级策略。最后把延迟回放接入发布流程，让每次模型或 prompt 变更都能看到体验成本。成熟的 AI 系统不会承诺所有请求都瞬间完成，而是清楚地知道用户什么时候需要反馈、什么时候需要完整结果、什么时候应该降级，以及每一段等待到底花在了哪里。
