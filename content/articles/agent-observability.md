## 问题背景

做 Agent 最容易低估的一件事，是它不像普通接口那样只有一次请求和一次响应。一个真实任务里，模型会读用户目标，拆步骤，选择工具，等待工具结果，修正计划，可能再调用另一个工具，最后把多个中间事实压成一个回答。用户看到的是一句“我已经处理好了”，工程团队面对的却是一条由模型推理、系统策略、外部工具、状态机和人工确认共同组成的执行链路。只要其中一段没有记录，出了问题就只能猜。

传统后端排障通常看三类东西：日志、指标和 trace。Agent 也需要这些，但不能简单照搬。普通服务的 trace span 多半对应一次 RPC、一次 SQL、一次缓存访问；Agent 的关键节点还包括模型为什么选择这个工具、参数从哪里来、哪些上下文被放进 prompt、哪些候选动作被策略层拒绝、模型输出是否经过结构化解析、执行器是否使用了上一次 dry-run 的哈希、用户确认时看到的影响范围是什么。这些信息如果只存在模型上下文里，任务结束后就消失了。

还有一个现实问题：Agent 错误经常是“软错误”。它没有抛异常，工具也返回成功，但结果不符合用户真实意图。例如用户说“帮我整理这周客户反馈”，Agent 只查了一个产品线；用户说“更新文档”，Agent 修改了生成产物而不是源文件；用户说“把会议纪要发给参会人”，Agent 漏掉了临时加入会议的人。这类问题很难靠错误率指标发现，必须把执行路径、证据来源和决策理由记录下来，才能复盘“为什么它会这么做”。

可观测性不是上线后加几个日志字段。它应该从 Agent 架构开始就进入设计。Agent 的每一步都要有可追踪的 run id，每次模型调用都要有 prompt 摘要和上下文清单，每次工具调用都要有关联的计划步骤和权限决策，每次状态变化都要能重放到当时的输入。只有这样，团队才敢把 Agent 放进更长、更有副作用的工作流。

我更倾向把 Agent 可观测性看成三个目标。第一，调试：开发者能定位一次失败是规划问题、检索问题、工具问题、权限问题还是生成问题。第二，治理：产品和安全团队能审计 Agent 做了什么、为什么做、谁批准了。第三，学习：团队能从大量真实任务中发现高频失败路径，把它们转成评测样本、产品约束或工具改造。缺了第三点，可观测性就只是事故后查日志，不会让系统真正变好。

## 核心概念

Agent 可观测性的核心对象不是单条日志，而是一次完整运行。一次运行可以叫 `run`，里面包含多个 `step`，每个 step 可能包含一次模型调用、一次检索、一次工具调用、一次人工确认或一次策略判断。每个对象都应该有稳定 ID，并通过父子关系串成 trace。这样你既能从用户会话进入，也能从某个工具异常反查到所有受影响的 Agent 运行。

| 对象 | 记录重点 | 典型用途 | 不记录的后果 |
| --- | --- | --- | --- |
| Run | 用户目标、入口、租户、模型版本、最终状态 | 复盘整次任务 | 无法还原任务边界 |
| Step | 步骤类型、输入摘要、输出摘要、耗时、状态 | 定位失败位置 | 只知道失败，不知道在哪失败 |
| Model Call | 上下文来源、输出结构、token、温度、解析结果 | 判断模型是否误判 | 把规划问题误认为工具问题 |
| Tool Call | 工具名、版本、参数、权限、幂等键、结果 | 审计副作用 | 无法追责和补偿 |
| Policy Decision | 允许、拒绝、要求确认、风险等级 | 治理和安全 | 策略形同虚设 |
| Human Action | 确认人、确认内容、差异快照、时间 | 高风险操作审计 | 用户无法承担真实责任 |
| Artifact | 生成文件、消息、工单、代码变更、引用 | 验证最终产物 | 只能相信模型总结 |

这里最容易混淆的是“模型理由”和“系统证据”。模型可以输出它为什么这样做，但这只是一个可观察信号，不是事实来源。真正的证据是：哪些文档被检索出来、哪些工具返回了什么、策略引擎做了什么判断、用户确认页面展示了什么。可观测系统要保存模型理由，但不能只保存模型理由。

另一个关键概念是上下文清单。很多 Agent 出错并不是模型太差，而是上下文组装错了。比如 prompt 里缺了最新约束，或者把旧任务的结果带进来，或者检索结果排序不合理。每次模型调用都要记录“进入上下文的材料列表”，包括系统指令版本、开发者指令版本、用户消息、长期记忆、检索证据、工具返回、压缩摘要。正文可以脱敏或只保存哈希，但清单必须存在。

指标也要按 Agent 特性重做。普通服务看 P95 延迟、错误率、QPS；Agent 还要看平均步骤数、工具调用成功率、澄清率、人工接管率、自动执行率、计划回滚率、无证据回答率、重复调用率、上下文预算使用率、每个成功任务成本。一个 Agent 如果延迟很低但人工接管率很高，它不是性能好，而是能力边界没设计清楚。

最后是回放能力。回放不等于重新调用真实工具，因为很多工具有副作用。回放应该能在隔离环境里重建当时的输入、模型输出、策略判断和工具返回快照，让开发者验证“如果换一个提示词、换一个重排策略、换一个工具 schema，路径会不会变化”。没有回放，Agent 评测就很难贴近真实事故。

trace 设计要尽量贴近 Agent 的真实决策边界。一次 `run` 不只是 HTTP 请求的延长线，而是一个业务承诺：用户希望系统完成某个目标。`step` 表示这个承诺被拆成的可解释动作，`span` 表示动作内部消耗资源或产生副作用的技术边界。这样的层次能避免两个极端：一是把所有模型、检索和工具调用都压成一个大 span，复盘时看不见分叉；二是把底层函数调用全部暴露出来，时间线被实现细节淹没。我的经验是，凡是会影响用户结果、权限判断、成本、等待状态或外部副作用的动作，都应该有独立 span；纯粹的本地字符串处理和 UI 拼装则可以只留在调试日志里。

指标也要能回答“Agent 为什么值得信任”。除了成功率和耗时，最好把指标拆成路径指标、质量指标和治理指标。路径指标关注平均计划深度、每个任务的工具调用分布、重试次数、等待时长和恢复次数；质量指标关注有证据回答比例、引用命中率、用户修正率、评测回归失败率；治理指标关注高风险动作确认率、策略拒绝率、越权尝试、脱敏命中和审计访问次数。这样看板不会只告诉你系统忙不忙，而会告诉你 Agent 是否在按预期方式完成任务。

## 架构/流程图解说明

一个可落地的 Agent 可观测性架构，可以放在运行时旁边，而不是散落在业务代码里。所有入口统一创建 run，运行时在每个关键边界发事件，事件进入队列后同时写入 trace store、日志存储、指标系统和审计仓库。

```text
用户请求
  |
  v
Run Manager 创建 run_id、租户、入口、目标摘要
  |
  v
Agent Runtime
  |-- Context Builder: 记录上下文清单、token 预算、证据来源
  |-- Model Gateway: 记录模型版本、请求摘要、结构化输出、解析错误
  |-- Planner: 记录候选步骤、选择理由、废弃路径
  |-- Policy Engine: 记录风险判断、权限过滤、确认要求
  |-- Tool Gateway: 记录参数、幂等键、dry-run、结果摘要
  |-- State Store: 记录状态迁移、检查点、恢复原因
  |
  v
Event Bus
  |-- Trace Store: run/step/span 父子关系
  |-- Metrics: 成功率、耗时、成本、接管率
  |-- Audit Log: 高风险操作、人工确认、副作用
  |-- Replay Store: 脱敏输入、工具快照、模型输出
  |
  v
Debug UI / Eval Builder / Alerting
```

这张图里最重要的边界是 Model Gateway 和 Tool Gateway。模型调用必须从网关走，否则 prompt、上下文和输出就无法统一记录。工具调用也必须从网关走，否则权限、幂等和审计会被不同团队各写一套。Agent Runtime 可以很灵活，但观测事件的契约要稳定。

事件不应该只在失败时记录。Agent 的很多失败需要和成功样本对比才看得出来。例如同样的问题，成功样本调用了检索工具再回答，失败样本直接回答；成功样本先 dry-run 再确认，失败样本跳过预览；成功样本生成了澄清问题，失败样本猜了一个项目名。只记录失败，会丢掉这种差异。

Debug UI 也不该只展示流水账。有效的视图应该按层次展开：先看整次任务状态，再看步骤时间线，再点开某个步骤看输入、输出、证据、策略、工具参数和错误。对于长任务，还要支持按“等待用户”“等待工具”“模型生成”“重试”“被策略拒绝”过滤。否则 trace 很完整，但人看不懂。

告警要关注业务语义。模型调用失败率升高当然要告警，但更值得告警的是：高风险工具自动执行比例异常升高；某个工具的重复幂等命中突然增多；无引用回答比例升高；某个租户的人工接管率持续超过阈值；平均步骤数突然变长；同一类任务出现新的失败模式。这些信号比单纯的 500 错误更早暴露问题。

## 工程实现

实现时可以先定义统一事件结构。不要一开始就追求完美的 OpenTelemetry 语义映射，先保证所有 Agent 关键动作都能落到同一个模型里。下面是一个简化的事件对象：

```json
{
  "event_id": "evt_01",
  "run_id": "run_20260419_0001",
  "step_id": "step_03",
  "parent_span_id": "span_02",
  "span_id": "span_03",
  "tenant_id": "team_astaxie",
  "user_id": "u_123",
  "event_type": "tool_call.completed",
  "component": "tool_gateway",
  "name": "repo.read_file",
  "status": "ok",
  "started_at": "2026-04-19T10:02:03+08:00",
  "duration_ms": 128,
  "attributes": {
    "tool_version": "v3",
    "risk": "read_only",
    "argument_hash": "sha256:...",
    "result_hash": "sha256:...",
    "policy_decision": "allow"
  }
}
```

正文内容和敏感参数不一定全部进事件。更稳妥的做法是事件保存摘要、哈希和引用，敏感原文进入加密的 replay store，并按租户、场景和保留期控制访问。开发者排障时先看摘要，需要进一步查看时走审计授权。否则可观测系统会变成新的数据泄露面。

Go 后端可以把观测写成运行时依赖，而不是到处手写日志。一个简单接口如下：

```go
type Recorder interface {
    StartSpan(ctx context.Context, kind string, name string, attrs map[string]any) (context.Context, Span)
    Event(ctx context.Context, typ string, attrs map[string]any)
    Error(ctx context.Context, err error, attrs map[string]any)
}

type Span interface {
    End(status string, attrs map[string]any)
}
```

模型网关使用它时，不直接记录完整 prompt，而是先生成上下文清单：

```go
type ContextItem struct {
    ID        string
    Kind      string // system, user, memory, evidence, tool_result
    Source    string
    Tokens    int
    Hash      string
    Redacted  bool
    Trust     string // trusted, user_input, external, generated
}

type ModelCallRecord struct {
    Model        string
    Purpose      string // plan, decide_tool, summarize, final_answer
    ContextItems []ContextItem
    PromptHash   string
    OutputHash   string
    ParsedAs     string
    TokensIn     int
    TokensOut    int
}
```

这个结构能回答一个关键问题：模型到底看到了什么。很多团队只保存最终 prompt 文本，排障时反而难用，因为 prompt 太长、太敏感、缺少来源。上下文清单更适合聚合分析。例如可以统计失败任务里 `memory` 类型上下文是否过多，或者某类工具返回是否经常占满预算。

工具网关要记录参数，但不能把所有参数明文写进普通日志。做法是每个工具声明参数字段的敏感等级：

```yaml
name: email.send_draft
version: "2026-04-19"
risk: external_write
arguments:
  to:
    type: string_array
    sensitivity: pii
  subject:
    type: string
    sensitivity: business
  body:
    type: markdown
    sensitivity: confidential
  idempotency_key:
    type: string
    sensitivity: public
```

网关根据敏感等级决定日志里保存明文、脱敏、哈希还是只保存长度。对排障来说，知道收件人数、域名分布、正文哈希和工具版本，通常已经足够；真正需要看正文时再走审计流程。

状态机也要可观测。Agent 长任务最好不要只存在内存里，而是每走一步写 checkpoint。checkpoint 里保存当前阶段、待执行步骤、已完成工具调用、等待事项、可恢复原因。服务重启后可以恢复，失败后也能解释。状态迁移应当是显式事件，例如 `run.waiting_for_user`、`run.resumed`、`step.retry_scheduled`、`step.compensating`。

为了让可观测数据可用于评测，运行记录还要能被抽样转成 eval case。一次失败运行可以自动提取：用户目标、相关上下文清单、预期工具路径、实际工具路径、最终结果、人工修正、失败标签。人工只需要补充“理想行为是什么”。这样真实事故能变成回归测试，而不是停留在群里讨论。

一个实用的存储分层是：热 trace 存 7 到 14 天，用于排障；聚合指标保留更久，用于趋势分析；审计日志按合规要求保存；脱敏 replay 样本进入评测仓库；包含敏感原文的快照设置短保留期和严格访问。不要把所有数据都永久保存，成本和风险都会失控。

日志采样要按风险和信息价值设计，而不是简单随机抽样。所有 run 的结构化骨架、状态迁移、错误标签和副作用摘要应该全量保存；模型原始输入、工具原始返回和大体量 artifact 可以分层采样。失败运行、高风险工具、策略拒绝、人工接管、用户差评、恢复执行和异常高成本运行，应当自动提升采样等级。低风险成功运行可以只保存摘要和哈希，但要保留足够的聚合字段，方便之后发现某个版本的成功路径突然变短、某类工具调用突然消失、某个租户的澄清率异常升高。

隐私脱敏不能只靠正则扫描。Agent 的上下文来源复杂，同一句文本可能来自用户输入、内部知识库、外部网页、工具结果或模型生成，敏感等级并不相同。更稳的做法是让数据在进入上下文时就带上来源、租户、字段级敏感度和可保留策略，日志系统再根据这些标签决定展示方式。比如邮箱地址可以显示域名和数量，合同条款只保留段落哈希，密钥样式字符串直接删除，客户名称在跨租户视图中只显示稳定匿名 ID。脱敏后的数据仍要能排障，否则工程师会绕过系统去查原文，反而扩大风险。

## 测试评测

可观测性本身也要测试。很多团队只测试 Agent 能不能完成任务，却不测试“完成任务时是否留下足够证据”。结果上线后发现 trace 断裂、run id 丢失、工具参数没记录、确认页面没有进入审计。这些问题必须在集成测试里覆盖。

第一类测试是链路完整性。构造一个包含模型规划、只读工具、写工具 dry-run、人工确认、最终执行的任务，验证所有 span 都挂在同一个 run 下，父子关系正确，步骤顺序可还原，最终状态和产物一致。这个测试不关心模型智能，可以用固定模型响应或 mock gateway。

第二类测试是脱敏策略。给工具传入邮箱、手机号、客户备注、合同金额、密钥样式字符串，验证普通日志里没有明文敏感信息，replay store 中的加密对象能按权限读取，摘要字段仍然保留排障价值。脱敏测试要使用真实边界样本，不能只测一个 `password` 字段。

第三类测试是错误归因。模拟检索无结果、模型输出 JSON 解析失败、工具超时、策略拒绝、用户取消确认、执行部分成功，检查每类错误是否产生清晰标签。好的错误标签应该能驱动下一步：是重试、澄清、转人工、降级还是停止。

可以把 Agent 运行质量指标分成四层：

| 层级 | 指标 | 说明 |
| --- | --- | --- |
| 运行层 | run 成功率、P95 耗时、平均步骤数、恢复次数 | 看系统是否稳定 |
| 模型层 | 解析失败率、无工具直答率、上下文预算占用 | 看模型调用是否可靠 |
| 工具层 | 工具成功率、超时率、幂等命中、策略拒绝 | 看外部行动是否健康 |
| 产品层 | 澄清率、人工接管率、用户修正率、重复任务率 | 看用户是否真的得到结果 |

评测时不要只看最终答案。Agent 的路径同样重要。一个答案看起来正确，但它调用了不该调用的工具、跳过了确认、使用了越权材料，这在生产里就是失败。评测样本里应包含预期路径约束，例如“必须先检索再回答”“不得调用写工具”“必须在影响超过十个对象时要求确认”“必须引用两个独立证据”。

告警验证也要做演练。比如人为把某个工具延迟提高，把某个模型解析失败率调高，把策略配置改成拒绝所有写操作，检查告警是否触发、看板是否能定位、值班人员是否知道下一步。可观测性不是有数据就行，还要有人能在压力下使用这些数据。

故障复盘最好直接从 trace 生成初稿。一次复盘至少要包含用户目标、实际结果、首次异常信号、关键决策点、工具副作用、人工确认、未触发的防线和后续修复项。这里不要急着把责任推给“模型幻觉”。很多事故表面上是模型选错工具，底层原因可能是上下文清单缺失、检索证据过期、策略只在 UI 层校验、日志采样漏掉了成功对照组，或者确认页没有展示真实影响范围。复盘的产物也不该停在文档里，应该沉淀成新的评测样本、告警规则、策略约束或工具 schema 变更。

## 失败模式

第一个失败模式是 trace 断裂。最常见原因是异步队列、后台任务、回调和人工确认没有传递 run id。用户确认发生在另一个 Web 请求里，如果确认对象没有绑定原始 run，审计链就断了。解决办法是所有待确认对象、队列消息和外部回调都必须携带 run id、step id 和幂等键。

第二个失败模式是日志很多但没有语义。工程师在每个函数里打印一行，最后得到几百条文本，却看不出模型为什么进入这个分支。Agent 需要结构化事件和统一字段，尤其是 `purpose`、`risk`、`policy_decision`、`context_items`、`tool_version`、`artifact_id`。没有这些字段，日志搜索只能靠运气。

第三个失败模式是可观测数据污染模型上下文。有人为了调试，把完整 prompt、工具返回和错误堆栈直接喂回模型，让模型自己分析。这样很容易把敏感信息、内部策略和不可信外部文本混在一起。调试助手可以读取脱敏摘要，但不能无控制地接触全部原文。

第四个失败模式是只观测成功率，不观测接管率。Agent 可能把大量不确定任务都交给人工，看起来没有失败，实际没有自动化价值。也可能相反，自动执行率很高但用户频繁撤销。产品层指标必须和技术层指标一起看。

第五个失败模式是采样过早。为了省成本，只采样一小部分运行，结果事故发生时没有完整记录。我的建议是事件摘要全量保存，敏感原文和 replay 快照按风险采样；高风险工具、失败运行、用户差评运行、策略拒绝运行必须提高保留等级。

第六个失败模式是模型版本和工具版本没有进入记录。Agent 行为变化经常来自模型升级、提示词调整、工具 schema 改动、检索参数变化。如果运行记录里没有版本，回归分析会非常痛苦。版本字段不是可选项。

## 上线 checklist

- 每次用户任务都有唯一 `run_id`，并能贯穿模型、工具、队列、确认和最终产物。
- 每个 step 有类型、状态、耗时、父子关系、输入摘要和输出摘要。
- 模型调用记录模型版本、提示词版本、上下文清单、token、结构化解析结果和错误标签。
- 工具调用统一经过网关，记录工具版本、风险等级、参数摘要、权限决策、幂等键和结果摘要。
- 高风险操作保存确认对象，包括展示给用户的差异快照、确认人、确认时间和执行哈希。
- 日志脱敏策略经过测试，普通日志不包含密钥、个人信息、合同正文或未授权业务内容。
- 可从失败 run 一键进入时间线视图，并能看到每一步的证据和状态迁移。
- 指标覆盖成功率、耗时、成本、步骤数、工具失败、解析失败、澄清率、人工接管率和无证据回答率。
- 告警规则覆盖高风险工具异常、策略拒绝异常、上下文预算异常、重复调用异常和用户修正异常。
- replay store 能在无副作用环境中重建运行输入，并能把事故样本转成评测用例。
- 数据保留期和访问权限明确，审计日志、排障日志、评测样本分层存储。
- 模型、提示词、工具 schema、策略配置、索引版本都进入运行记录。

## 总结

Agent 可观测性的重点，不是把更多文本塞进日志，而是把一次任务拆成可解释、可审计、可回放的运行对象。模型判断、上下文组装、策略决策、工具调用、人工确认和状态迁移，都应该在同一条 trace 里。这样开发者能定位失败，产品能看到真实使用成本，安全团队能追踪副作用，评测系统能从事故里长出新样本。

真正有用的可观测性要在架构早期建立。等 Agent 已经接入多个工具、多个队列和多种确认入口以后再补，通常只能得到一堆局部日志。我的经验是先把 run、step、model call、tool call、policy decision 这几个对象定义清楚，再让所有组件围绕它们发事件。系统可以慢慢变复杂，但观测骨架不能含糊。Agent 越像一个能行动的同事，就越需要留下清楚的工作记录。
