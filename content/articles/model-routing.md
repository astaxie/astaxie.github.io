## 问题背景

LLM 应用刚起步时，最容易采用的方案是“所有请求都打到同一个模型”。这样做简单，研发成本低，问题定位也方便。但只要流量、任务类型和用户预期稍微复杂一点，单模型策略就会暴露明显缺陷：简单任务花了大模型的钱，复杂任务又因为上下文或推理能力不够而失败；高优先级用户和批处理任务抢同一条队列；某个模型服务降级时，整个产品一起抖；成本账单上涨后，团队才发现大量请求其实只是分类、改写或格式化。

模型路由的目标不是追逐最多模型，也不是把每个请求都交给一个“聪明调度器”黑箱判断。它的工程目标很直接：把任务需求、质量要求、成本预算、延迟预算、上下文长度、隐私边界和供应商可用性统一到一个可解释的决策层里。不同请求应该进入不同的模型、参数和执行路径；同一个请求在不同状态下也可能需要升级、降级或回退。

举个常见场景：一个知识库问答系统收到用户问题。问题可能只是“怎么重置密码”，也可能是“比较过去三份合同里付款条款的差异，并指出风险”。前者不需要最强推理模型，检索到正确文档后用小模型就能回答；后者需要更长上下文、更强结构化能力和更严格引用检查。如果所有请求都走同一个高端模型，成本会很难控制；如果所有请求都走便宜模型，复杂问题质量会崩。模型路由就是在这两者之间做工程化分配。

模型路由还要面对模型生态的变化。模型会升级、降价、限流、下线，供应商会改变上下文窗口和工具调用能力，企业内部也可能新增本地模型。把模型名写死在业务代码里，会让每一次变更都变成跨服务改造。更稳妥的方式是把业务任务声明成能力需求，再由路由层选择当前最合适的模型。业务代码说“我需要低延迟分类、最多一千输入 token、输出 JSON”，而不是说“我必须调用某个具体模型”。

很多团队做模型路由时会走向两种极端。一种是完全静态：按接口写死模型，无法适应输入长度、质量反馈和服务状态。另一种是完全动态：让一个模型先读请求再决定用哪个模型，结果路由本身变贵、变慢，还难解释。生产系统更适合混合策略：先用确定性规则覆盖明确场景，再用轻量分类器处理模糊场景，最后用运行时指标和质量反馈调参。路由层应该像负载均衡和策略引擎的结合，而不是一段随手写的 if else。

模型路由也直接影响组织协作。产品经理关心体验，财务关心成本，安全团队关心数据流向，工程团队关心延迟和可靠性，算法团队关心评测分数。没有统一路由层时，每个业务线会各自配置模型，最后出现重复采购、重复踩坑和指标不可比。统一路由不是为了集中控制所有创新，而是为了让模型选择有共同语言：任务类型、能力要求、预算、风险等级、回退策略和观测指标。

## 核心概念

第一个概念是任务画像。一次调用在进入模型之前，系统应该知道它大致属于哪类任务：分类、抽取、改写、摘要、问答、代码生成、工具规划、复杂推理、评审打分、对话闲聊。不同任务对模型能力的要求不同。分类任务通常要求稳定和便宜，复杂推理要求能力和上下文，代码任务要求语法和工具理解，评审任务要求一致性和抗偏见。任务画像越清楚，路由越容易解释。

第二个概念是能力维度。模型不是简单分成“大”和“小”。一个路由策略至少要考虑：推理能力、上下文长度、结构化输出能力、工具调用能力、多语言能力、代码能力、视觉或音频能力、吞吐、延迟、价格、数据驻留、可用区和服务稳定性。某个模型可能便宜但 JSON 稳定性差，另一个模型可能推理强但延迟高。路由表需要记录这些维度，而不是只保存模型名称。

第三个概念是服务等级。并不是所有请求都应该享受同样资源。交互式用户请求通常有低延迟预算，后台批处理可以排队；企业客户可能有更高质量要求，免费用户可能有更严格成本上限；高风险任务需要更保守模型和评测，低风险任务可以用便宜路径。模型路由要把服务等级显式纳入决策，否则系统会在高峰期用随机方式牺牲用户体验。

第四个概念是升级和降级。路由不是一次性选择。一个请求可以先走低成本模型，如果结构校验失败、置信度不足或检测到复杂度超限，再升级到更强模型；反过来，当主模型限流或超时，可以降级到备用模型，但要改变输出策略，例如减少上下文、关闭非必要工具或提示用户结果可能更粗略。升级和降级要有规则，不能在失败后靠模型自己猜。

第五个概念是预算。预算包括金钱、延迟、token、并发和错误预算。路由层应该在调用前估算输入 token 和成本，在调用后记录真实消耗。没有预算意识的路由，最后会变成“质量优先”的口号系统，把账单和排队问题留给运维。实际生产里，预算约束越早进入决策，系统越稳定。

第六个概念是可观测性。模型路由如果不可观测，就会成为质量问题的黑洞。每次调用要记录为什么选择某个模型、有哪些候选、命中了哪条规则、是否升级或降级、输出是否通过校验、用户是否重试、成本和延迟是多少。路由决策需要像数据库查询计划一样可解释，至少工程团队能复盘。

| 维度 | 路由问题 | 典型信号 | 决策影响 |
| --- | --- | --- | --- |
| 任务类型 | 这是分类还是复杂推理 | API 名称、Prompt slug、轻量分类器 | 选择能力档位 |
| 输入规模 | 上下文是否过长 | token 估算、文件数量、检索片段数 | 选择上下文窗口 |
| 风险等级 | 错误代价有多高 | 业务标签、用户等级、操作权限 | 是否用强模型和复核 |
| 延迟预算 | 用户能等多久 | 同步请求、批处理、SLA | 是否走快速模型 |
| 成本预算 | 单次调用可花多少钱 | 租户配额、功能套餐、月度预算 | 是否降级或截断 |
| 数据边界 | 数据能否出域 | 隐私标签、地区、合规策略 | 选择本地或指定供应商 |
| 供应状态 | 模型是否健康 | 错误率、限流、P95 延迟 | 切换备用模型 |

模型路由还需要区分“选择模型”和“选择执行计划”。执行计划可能包括检索、压缩、分块、并行调用、二次评审和工具执行。模型只是计划中的一部分。例如长文档问答可以先用小模型抽取候选段落，再用强模型综合回答；代码审查可以先用规则扫描，再让模型关注高风险 diff；客服摘要可以先用便宜模型生成，再对高风险样本用强模型复核。好的路由策略选择的是整条路径。

## 架构/流程图解说明

一个模型路由系统可以放在业务服务和模型供应商之间，承担策略决策、预算控制、调用执行和结果记录。它不一定是独立微服务，也可以先做成内部库，但边界要清楚：业务层描述任务和约束，路由层选择计划并执行，观测层记录结果。

```text
业务请求
  |
  v
任务画像构建
  |-- prompt_slug
  |-- input_token_estimate
  |-- risk_level
  |-- latency_budget
  |-- tenant_policy
  |
  v
路由决策器
  |-- 静态规则
  |-- 能力匹配
  |-- 预算检查
  |-- 健康状态过滤
  |-- 复杂度分类
  |
  v
执行计划
  |-- 模型选择
  |-- 参数选择
  |-- 上下文压缩
  |-- 工具开关
  |-- 升级降级策略
  |
  v
模型调用网关
  |
  v
输出校验与质量信号
  |
  v
日志、指标、成本账本
```

路由决策器的输入不能只是一段用户文本。它应该接收结构化画像。画像可以由业务代码提供一部分，由平台计算一部分。例如业务代码知道当前是 `contract_review`，平台可以估算 token、检测语言、读取租户预算、查询模型健康状态，再决定候选模型。这样路由层既不盲目，也不会过度侵入业务。

静态规则适合处理确定场景。比如“包含客户个人身份信息并且租户要求数据不出境，只能走本地模型”“批处理任务默认使用低成本队列”“输出必须严格 JSON 且失败不能重试时，使用结构化输出稳定的模型”。这些规则应该显式配置、可审计、可测试。不要把合规和预算规则藏在分类器里。

能力匹配负责从候选模型中筛选。假设任务需要 80k 上下文、工具调用和中文能力，那么不满足上下文窗口或工具能力的模型直接排除。剩下的模型再根据成本、延迟和质量分排序。这个过程类似数据库查询优化器：先过滤不可行计划，再给可行计划打分。

复杂度分类适合处理不确定输入。例如同样是问答，有的问题只需要检索一个片段，有的问题需要跨文档综合。可以用轻量模型、规则或嵌入特征判断复杂度。关键是分类器本身要便宜，并且允许保守策略。分类器不确定时宁可选择中档模型，也不要为了省钱把复杂任务打到低能力路径。

执行计划要包含回退路径。一个典型状态流如下：

```text
planned
  -> primary_call
  -> validated
  -> completed

primary_call -> timeout -> fallback_call -> degraded_completed
primary_call -> schema_failed -> repair_call -> validated
primary_call -> low_confidence -> upgrade_call -> validated
fallback_call -> failed -> user_visible_error
```

这个状态流说明，路由策略不是只在调用前发生。输出校验失败后，系统可以选择修复、升级或返回错误；主模型超时后，可以切换备用模型；低置信度结果可以触发二次评审。每一步都要记录原因，否则后续成本分析和质量分析会失真。

## 工程实现

模型路由可以从一张模型能力表和一个路由请求结构开始。能力表描述模型，路由请求描述任务。下面这个结构足够支撑早期系统，后续可以增加供应商权重、区域策略和实时价格。

```yaml
models:
  - id: fast-json-cn
    provider: vendor_a
    model: small-structured-2026-02
    capabilities:
      reasoning: 2
      json: 5
      code: 2
      tool_calling: false
      max_context_tokens: 32000
      languages:
        - zh
        - en
    cost:
      input_per_million: 0.2
      output_per_million: 0.8
    latency:
      p50_ms: 600
      p95_ms: 1800
    data_regions:
      - cn
    status: healthy
  - id: deep-reasoning-global
    provider: vendor_b
    model: reasoning-large-2026-03
    capabilities:
      reasoning: 5
      json: 4
      code: 5
      tool_calling: true
      max_context_tokens: 200000
      languages:
        - zh
        - en
        - ja
    cost:
      input_per_million: 2.5
      output_per_million: 10.0
    latency:
      p50_ms: 2500
      p95_ms: 9000
    data_regions:
      - us
      - eu
    status: healthy
```

路由请求则像这样：

```json
{
  "task": "knowledge_qa",
  "prompt_slug": "answer_with_citations",
  "tenant": "enterprise_42",
  "locale": "zh-CN",
  "input_tokens_estimate": 18000,
  "requires": {
    "json": true,
    "tool_calling": false,
    "min_reasoning": 3,
    "min_context_tokens": 50000
  },
  "budget": {
    "max_latency_ms": 6000,
    "max_cost_usd": 0.08
  },
  "risk": "medium",
  "data_policy": {
    "allowed_regions": ["cn", "us"],
    "allow_external_provider": true
  }
}
```

在 Go 服务里可以把决策过程拆成几个纯函数，方便测试：

```go
type RouteRequest struct {
    Task                string
    PromptSlug          string
    Tenant              string
    Locale              string
    InputTokensEstimate int
    Requires            CapabilityRequirement
    Budget              Budget
    Risk                string
    DataPolicy          DataPolicy
}

type RouteDecision struct {
    PlanID          string
    PrimaryModel    string
    FallbackModels  []string
    Parameters      ModelParams
    ContextStrategy string
    ReasonCodes     []string
    EstimatedCost   float64
}

func Route(req RouteRequest, catalog []ModelSpec, health HealthSnapshot, tenant TenantPolicy) (RouteDecision, error) {
    candidates := FilterByPolicy(catalog, req.DataPolicy, tenant)
    candidates = FilterByHealth(candidates, health)
    candidates = FilterByCapabilities(candidates, req.Requires, req.InputTokensEstimate)
    ranked := RankByScore(candidates, req)
    if len(ranked) == 0 {
        return RouteDecision{}, ErrNoRoute
    }
    return BuildPlan(req, ranked), nil
}
```

`ReasonCodes` 很重要。它让路由决策可解释，例如：

```text
policy.region.allowed: cn/us
capability.context.required: 50000
capability.reasoning.min: 3
budget.latency.max: 6000
rank.cost.selected_lower_cost_within_quality_band
fallback.added_for_provider_rate_limit
```

有了这些原因码，排查问题会简单很多。用户投诉“为什么这个请求慢”，可以看到它被判为中风险长上下文，因此选了强模型；财务问“为什么某租户成本升高”，可以看到最近大量请求触发了复杂度升级；工程师调整策略时，也能写单元测试断言原因码。

打分函数不要一开始做得太聪明。一个可维护的版本可以这样：

```text
score = quality_weight * quality_score
      - cost_weight * normalized_cost
      - latency_weight * normalized_latency
      + reliability_weight * health_score
      + policy_bonus
```

其中 `quality_score` 可以来自离线评测，按任务类型分开维护；`normalized_cost` 根据本次 token 估算；`health_score` 来自最近几分钟错误率和限流率；`policy_bonus` 用于表达企业偏好，例如优先使用本地模型。打分函数要输出分项，不要只输出总分。总分方便排序，分项方便争论和调参。

执行层还要处理上下文策略。模型路由不只是选模型，还要决定如何喂上下文。输入超长时有几种选择：截断低价值内容、先摘要再回答、按文档分块并行处理、要求用户缩小范围、升级到长上下文模型。不同任务适合不同策略。合同审查不应该随意截断尾部，因为关键条款可能在附件末尾；聊天摘要可以按时间窗口压缩；代码问答可以优先保留相关文件和错误日志。

一条实际执行计划可能如下：

| 步骤 | 动作 | 模型 | 触发条件 |
| --- | --- | --- | --- |
| 1 | 估算 token 和语言 | 本地 tokenizer | 所有请求 |
| 2 | 检索候选文档 | 向量检索和关键词检索 | knowledge_qa |
| 3 | 复杂度判断 | fast-json-cn | 检索片段超过 8 个 |
| 4 | 主回答 | mid-reasoning-cn | 复杂度低或中 |
| 5 | 升级回答 | deep-reasoning-global | 复杂度高或校验失败 |
| 6 | 引用校验 | 规则和小模型 | 必须带 citations |
| 7 | 记录账本 | 成本服务 | 所有请求 |

这里第 3 步看似增加了一次调用，但可能节省整体成本，因为它避免把所有问题都送进最贵模型。是否值得，需要通过线上指标验证，而不是凭直觉。

预算账本要做成硬约束和软约束结合。硬约束包括数据区域、合规限制、最大上下文和用户套餐上限，违反就不能调用。软约束包括单次成本目标、月度预算进度和延迟偏好，可以影响排序和降级。比如某租户本月预算已用 90%，低风险任务可以自动降级，高风险任务仍然走强模型但记录预算告警。这样不会因为省钱牺牲关键业务，也不会让低价值流量吞掉全部预算。

缓存也是路由的一部分。分类、摘要、评审等任务如果输入稳定，可以缓存模型输出或中间结果。缓存命中时相当于选择了“无模型计划”。但缓存必须带版本指纹：Prompt 版本、模型版本、路由策略版本、输入哈希和上下文哈希。否则模型升级或 Prompt 修复后，旧缓存会继续污染结果。

## 测试评测

模型路由的测试不能只测“返回了一个模型名”。它要验证策略正确性、质量收益、成本收益和故障行为。测试可以分四层。

第一层是策略单元测试。给定固定请求、模型目录、健康状态和租户策略，断言路由结果和原因码。例如：当数据政策只允许中国区域时，外部区域模型必须被排除；当输入估算为 80k token 时，32k 上下文模型不能入选；当主模型状态为限流时，备用模型应该被选择。这类测试不调用模型，速度快，应该覆盖大部分规则。

第二层是回放测试。用历史请求日志重放路由策略，比较新策略和旧策略的差异。回放报告应该显示多少请求改变了模型、预计成本变化、预计延迟变化、哪些租户受影响、哪些任务被升级或降级。没有回放就改路由策略，风险很高，因为一个小权重变化可能让大量流量转向昂贵模型。

第三层是离线质量评测。对代表性样本分别运行不同计划，比较答案质量、结构合法率、引用准确率、工具成功率和人工评分。路由策略不是单个模型评测，而是计划评测。比如“小模型回答 + 强模型复核”可能质量接近强模型直接回答，但成本更低；“先摘要再回答”可能降低上下文成本，却损失细节。评测要看整条链路。

第四层是线上实验。离线评测无法完全代表真实用户。灰度时要监控用户追问率、重新生成率、人工接管率、投诉率、任务完成率、P95 延迟和单请求成本。路由策略如果让成本下降 30%，但用户追问率上升 20%，可能并不划算。线上指标要和业务目标绑定。

| 测试层 | 输入 | 输出 | 关键问题 |
| --- | --- | --- | --- |
| 策略单测 | 构造请求和模型表 | 决策与原因码 | 规则有没有错 |
| 日志回放 | 历史请求 | 差异和成本估算 | 影响面多大 |
| 离线评测 | 标注样本 | 质量和链路指标 | 计划是否有效 |
| 线上灰度 | 真实流量 | 体验和成本指标 | 用户是否受益 |

评测样本要按任务类型拆分。分类任务看准确率、稳定性和 JSON 合法率；问答任务看引用准确、覆盖和幻觉率；代码任务看编译、测试和补丁最小性；工具规划任务看工具选择、参数正确性和副作用控制。不要用一个通用分数评价所有路由。模型路由的目标是“让每类任务匹配合适计划”，所以评测也必须按类观察。

还要测失败路径。把主模型设置为超时，确认备用模型是否接管；让输出 Schema 校验失败，确认是否触发修复或升级；让预算不足，确认低风险任务是否降级，高风险任务是否保留质量并报警；让所有候选模型都不可用，确认系统是否返回可解释错误，而不是无限重试。很多路由系统平时表现不错，真正事故时才发现回退路径没人测过。

质量评测要结合成本曲线，而不是只看最高分。可以把每种计划画成“质量分-成本-延迟”的三维比较。理想计划不一定是质量最高，而是在业务阈值以上成本最低、延迟可接受的那一个。例如一个中档模型在客服摘要上达到 96% 结构合法率和 4.3 人工分，强模型达到 4.4 分但成本高四倍，那么中档模型可能更适合默认路径；强模型保留给高风险升级。

路由策略本身也要版本化。每次修改规则、权重、模型目录或预算参数，都应该生成策略版本。线上日志记录 `routing_policy_version`，回放系统可以比较版本差异。没有策略版本，质量问题会被误归因到模型或 Prompt，实际上可能只是路由权重改变。

## 失败模式

第一种失败模式是成本倒挂。团队引入路由本来想省钱，结果因为复杂度分类器太敏感，大量请求被升级到强模型，总成本反而上升。解决办法是做日志回放和线上预算监控，给升级设置明确触发条件，并抽样检查升级请求是否真的需要强模型。

第二种失败模式是低估上下文风险。路由层只看输入 token 数，不看内容结构，把长合同随意压缩后交给便宜模型，导致关键条款遗漏。解决办法是让上下文策略按任务定制，并对高风险文档禁止无依据截断。路由要知道“长”不只是数字，也代表信息分布和错误代价。

第三种失败模式是策略不可解释。业务方只看到某请求被某模型处理，却不知道原因。质量下降时，大家争论模型差、Prompt 差还是检索差。解决办法是记录原因码、候选列表、过滤步骤和打分分项。路由决策越自动化，解释日志越重要。

第四种失败模式是供应商健康信号滞后。主模型已经限流，路由还继续发送请求，造成排队和级联超时。解决办法是健康状态要短窗口更新，包括错误率、429、超时、P95 延迟和队列长度；同时给恢复设置冷却时间，避免模型刚恢复就被全部流量打满。

第五种失败模式是降级破坏契约。主模型不可用时，系统切到备用模型，但备用模型不支持工具调用或结构化输出，结果下游解析失败。解决办法是降级不是简单换模型，而是换执行计划。备用计划必须满足最低输出契约；如果不能满足，就应该返回受控错误或进入人工路径。

第六种失败模式是模型目录过期。新模型已经上线，旧模型价格改变或能力退化，路由表却没人维护。解决办法是把模型目录当配置资产管理，有 owner、有评测、有更新流程。模型目录变化也要跑回放和评测，不能静默替换。

第七种失败模式是忽视租户和数据政策。为了统一成本，所有请求都走同一供应商，后来发现某些租户的数据不能出特定区域。解决办法是把数据政策作为硬过滤条件放在路由最前面，并在日志里记录被过滤的原因。合规约束不应该和成本权重混在一起。

第八种失败模式是优化单点指标。只优化成本会牺牲质量，只优化质量会拖垮预算，只优化延迟会降低复杂任务成功率。路由策略必须承认多目标权衡，并把权衡显式化。对不同任务和用户等级，权重可以不同，但不能没有说明。

## 上线 checklist

上线模型路由前，可以按下面清单逐项确认：

- 是否定义了任务画像字段，包括 task、prompt_slug、输入 token 估算、风险等级、延迟预算和租户策略。
- 模型目录是否记录能力、上下文窗口、结构化输出、工具调用、多语言、价格、延迟、数据区域和健康状态。
- 合规和数据驻留规则是否作为硬过滤条件执行，并有单元测试覆盖。
- 成本预算和延迟预算是否进入决策，是否区分硬约束和软偏好。
- 路由决策是否输出原因码、候选模型、过滤原因、打分分项和策略版本。
- 是否为每类任务定义质量基线，而不是用一个全局分数判断所有模型。
- 是否完成历史日志回放，确认新策略影响的请求比例、租户范围、成本变化和延迟变化。
- 是否测试主模型超时、限流、输出校验失败、预算不足和无可用模型等失败路径。
- 升级策略是否有触发条件，例如低置信度、结构失败、复杂度高或高风险任务。
- 降级策略是否仍然满足最低输出契约，不能满足时是否返回可解释错误。
- 上下文策略是否按任务定义，是否避免对高风险长文档做无依据截断。
- 缓存键是否包含 Prompt 版本、模型版本、路由策略版本和输入上下文哈希。
- 线上指标是否覆盖成本、延迟、错误率、重试率、升级率、降级率、用户追问率和人工接管率。
- 灰度是否可以按租户、任务、用户分桶和流量比例控制，是否有快速回滚办法。
- 模型目录和路由策略是否有 owner，更新是否会触发评测和审批。

这份 checklist 的重点是让模型选择从个人经验变成可验证策略。早期系统可以很简单，但不要省掉原因码、回放和失败路径测试。这三件事决定了路由系统能不能在真实生产问题里站得住。

## 总结

模型路由是 LLM 应用从 demo 走向生产后的必需层。它解决的不是“哪个模型最好”这个抽象问题，而是“在当前任务、预算、风险和服务状态下，哪条执行计划最合适”。同一个产品里，分类、摘要、问答、代码、评审和工具规划本来就不该共享同一套模型假设。

可落地的模型路由不需要一开始就复杂。先建立模型能力表，定义结构化路由请求，写清楚硬过滤规则，用简单打分函数排序，记录原因码和策略版本，再通过历史回放、离线评测和线上灰度不断校准。随着流量增长，再加入复杂度分类、自动升级、降级计划、预算账本和多供应商健康控制。

最重要的是保持可解释。路由层一旦变成黑箱，成本问题、质量问题和合规问题都会被混在一起。一个好的路由系统应该能回答：为什么选择这个模型，为什么没有选择另一个模型，本次调用花了多少钱，是否命中预算限制，失败时会怎么退，质量变化和哪条策略有关。把这些问题回答清楚，模型路由就不只是省钱工具，而是 LLM 应用的运行时控制面。
