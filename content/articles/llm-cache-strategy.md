# LLM 缓存策略

LLM 应用做大以后，缓存一定会被提上桌面。原因很现实：模型调用贵，长上下文慢，工具查询重复，用户问题有大量相似表达，热门文档会被反复问。缓存看起来像一个简单的成本优化，但在 LLM 场景里，它更像一个产品正确性问题。缓存命中错了，不只是返回旧数据那么简单，它可能越过权限边界，复用错误上下文，把已经撤回的政策重新说出来，或者在用户明确要求最新信息时给出昨天的答案。

传统 Web 缓存主要围绕 URL、参数、响应头和过期时间展开。LLM 缓存面对的是自然语言、上下文、权限、工具结果、模型版本和提示词策略。两个问题字面不同，可能语义相同；两个问题字面相同，也可能因为用户角色、会话历史、时间窗口、语言偏好和可访问文档不同而不能复用。缓存策略如果只看 prompt 字符串，会漏掉大量机会，也会埋下权限事故。策略如果只看向量相似度，又可能把相近但不等价的问题混在一起。

我更倾向于把 LLM 缓存设计为“可证明边界内的复用系统”。它的目标不是尽可能多命中，而是在明确的输入、权限、时效和质量边界内复用计算成果。命中率只是结果指标，不是最高目标。对于面向企业知识库、代码助手、客服助手、运营分析助手这类应用，缓存策略必须回答四个问题：复用的到底是什么，谁有资格复用，复用多久，复用后如何证明答案仍然可接受。

这篇文章讨论应用层 LLM 缓存，不讨论模型供应商内部的上下文缓存计费机制。供应商缓存可以降低 token 成本，但它解决不了业务语义、租户权限、工具结果和答案质量问题。应用层缓存要把提示词、检索、工具、模型输出和最终答案拆开看，针对不同层次定义不同键、不同失效策略和不同观测指标。最重要的是，缓存要保守。宁可少命中，也不要把不该复用的结果送到用户面前。

## 问题背景

很多 LLM 应用早期没有缓存，因为流量不大，先追求可用性是正确选择。但当使用量增长后，系统会出现三个明显痛点。第一个是成本上升，尤其是长上下文、多轮会话和高价模型路由。第二个是延迟不稳定，同一个热门问题反复触发检索、重排、工具调用和生成，用户仍然要等待完整链路。第三个是外部依赖压力，数据库、搜索引擎、工单系统、CRM 和第三方 API 会被重复查询。缓存能缓解这些问题，但前提是命中结果真的等价。

LLM 缓存难在“等价”很难定义。用户问“上季度增长慢的原因是什么”和“Q1 为什么没达标”，在某个销售分析场景里可能等价；但如果公司按自然季度和财务季度分别统计，这两个问题就不等价。用户问“帮我写一封道歉邮件”可以复用模板，但如果会话前文提到客户姓名和具体事故，答案必须带上当前语境。用户问“报销制度是什么”看似常见，但不同地区、职级、合同类型和生效日期会影响答案。缓存不能只比较自然语言，还要比较业务约束。

权限边界是另一个核心问题。假设企业知识库里有公开文档和部门私有文档。A 用户问了一个问题，答案引用了私有文档。B 用户提出相似问题，如果系统只按语义命中缓存，就可能把 A 的答案返回给 B。即使答案正文没有直接暴露文档标题，也可能泄露结论。更隐蔽的是聚合答案，比如“本周裁撤计划涉及三个团队”。这个结论本身就是敏感信息，不能因为没有引用原文就认为安全。缓存必须把权限快照纳入键或校验流程。

时效性也会破坏复用。LLM 答案经常依赖外部世界或内部数据状态。价格、库存、工单状态、版本发布、故障进展、政策生效日期都会变化。如果缓存没有时效模型，用户会在需要最新信息时拿到旧结论。简单 TTL 可以解决一部分问题，但不够。不同数据源变化频率不同，不同问题对新鲜度要求不同，不同用户对答案可追溯性要求也不同。缓存策略要能表达“这类答案可缓存一天”“这类工具结果只缓存五分钟”“涉及运行中事故时禁止缓存”。

还有一个常见误区是把缓存当作纯后端优化，前端和产品不参与。实际上缓存会改变用户体验。命中缓存后答案更快，但用户可能需要知道答案基于哪个时间点的数据。未命中时系统可能慢很多，产品需要解释为什么。缓存命中低质量答案时，用户需要反馈通道触发失效。对于严肃场景，界面应该能显示“基于缓存结果，数据更新于某时间”，或者在用户要求最新时强制刷新。缓存不是隐形魔法，它是用户信任的一部分。

## 核心概念

LLM 缓存首先要区分缓存对象。常见对象至少包括提示词片段、检索结果、工具结果、模型中间结果、最终答案和评测结论。它们的复用条件完全不同。提示词片段可以按版本缓存，检索结果要按查询、索引版本和权限缓存，工具结果要按工具参数和数据版本缓存，最终答案要按用户意图、上下文、权限、模型版本和生成策略缓存。把所有东西塞进一个“prompt 到 answer”的缓存，会让策略无法精细化，也难以排查错误命中。

第二个概念是缓存键。缓存键不是简单字符串，而是一组边界声明。一个安全的键通常包含用户意图表示、租户、权限范围、数据源版本、提示词版本、模型路由、工具参数、新鲜度要求和输出格式。键越完整，误命中越少，但命中率也可能下降。工程上要在可复用性和正确性之间取舍。我的经验是，权限、数据版本、提示词版本和输出格式不能省；模型版本是否纳入键取决于产品是否允许不同模型生成风格混用；会话历史是否纳入键取决于答案是否依赖上下文。

第三个概念是语义缓存。语义缓存不是把向量相似度超过阈值的答案直接返回。它应该是一个候选召回步骤：先用向量或归一化意图找到可能复用的历史项，再用规则和轻量模型做等价性判定，最后检查权限、时效和质量。相似不等于可替代。比如“如何删除账号”和“如何注销企业账号”相似度很高，但操作流程和风险不同。语义缓存必须有拒绝能力，不能把阈值当成真理。

第四个概念是失效。传统缓存经常靠 TTL，LLM 缓存还需要事件驱动失效。文档重新索引、权限变更、提示词升级、模型路由调整、工具 schema 变化、用户负反馈、质量评测失败，都应该触发相关缓存失效或降权。失效不一定总是删除，也可以把条目标记为 stale，需要重新验证后才能返回。对于高成本答案，stale-while-revalidate 是一个实用策略：先返回带时间标记的旧答案，同时后台刷新；但只有在产品允许旧答案暂时可用时才适合。

下面这张表列出不同缓存层的典型边界：

| 缓存层 | 缓存内容 | 关键键字段 | 失效触发 | 风险 |
| --- | --- | --- | --- | --- |
| 提示词缓存 | system prompt、工具描述、模板展开 | prompt_version、locale、feature_flag | 提示词发布、功能开关变化 | 旧策略继续生效 |
| 检索缓存 | query 的候选文档和分数 | normalized_query、index_version、permission_hash | 文档更新、索引重建、权限变化 | 越权或漏召回 |
| 工具缓存 | API 查询结果、聚合数据 | tool_name、args_hash、data_version、ttl | 数据源变更、参数变化 | 返回过期业务数据 |
| 答案缓存 | 最终 Markdown、引用、结构化输出 | intent_key、context_hash、policy_hash、model_key | 反馈差、提示词升级、引用失效 | 错误答案被放大 |
| 评测缓存 | 对答案的质量判断和安全判断 | answer_hash、rubric_version | 评测标准变化 | 旧质量结论误导路由 |

缓存还要区分“私有命中”和“共享命中”。私有缓存只在同一用户或同一会话内复用，风险低，命中粒度可以放宽，适合多轮追问、刷新页面和失败重试。共享缓存跨用户复用，必须更严格，至少要限制在同租户、同权限集合或同公共资料范围内。很多系统可以先做私有缓存和公共问题缓存，不必一开始就做全局语义缓存。保守路线通常更容易上线。

## 架构/流程图解说明

一个可控的 LLM 缓存架构应该在生成链路中有多个检查点，而不是只在最外层包一层缓存。请求进入后，系统先解析意图和约束，再检查是否可以使用缓存。如果可以，先走精确缓存；精确未命中，再走语义候选；候选通过等价性、权限和时效校验后，才能返回。未命中则执行正常生成链路，并把可复用的中间结果按各自策略写入缓存。

流程可以表达为：

```text
用户请求
  |
  v
请求规范化：语言、时间窗口、输出格式、会话依赖
  |
  v
构造 cache policy：租户、权限、新鲜度、是否允许共享
  |
  v
精确缓存查找
  |
  +-- 命中 -> 权限和时效校验 -> 返回或后台刷新
  |
  v
语义候选召回
  |
  v
等价性判定 -> 权限校验 -> 引用有效性校验 -> 质量阈值校验
  |
  +-- 通过 -> 返回缓存答案并记录命中原因
  |
  v
正常生成：检索、工具、模型、评测
  |
  v
按层写入缓存并记录版本、来源、失效依赖
```

这里的 `cache policy` 是关键。它不是配置文件里一个全局 TTL，而是每次请求动态生成的策略对象。策略应该回答：这次请求是否允许缓存，允许哪些层命中，是否允许跨用户共享，最大可接受数据年龄是多少，是否必须重新校验引用，是否允许返回 stale，用户是否显式要求刷新。这样缓存才能服从产品语义，而不是产品被缓存牵着走。

一个策略对象可以长这样：

```ts
type CachePolicy = {
  tenantId: string
  subjectId: string
  permissionHash: string
  allowShared: boolean
  allowSemantic: boolean
  maxAgeSeconds: number
  requireFreshSources: boolean
  promptVersion: string
  modelFamily: string
  outputSchema: string
  dataScopes: Array<{ source: string; version: string; freshness: string }>
}
```

缓存条目也要保存足够证据，不能只保存答案文本。至少需要保存创建时间、键字段、权限摘要、依赖的数据源版本、引用列表、模型信息、提示词版本、质量评分、命中次数、负反馈次数和失效状态。这样当一个答案被投诉时，工程团队能知道它为什么命中、基于什么材料生成、是否跨用户复用、是否已经过期。

架构上可以把缓存分成三类存储。第一类是低延迟 KV，比如 Redis，用于精确键命中、短 TTL 工具结果和私有会话缓存。第二类是向量库或支持向量索引的数据库，用于语义候选召回。第三类是持久化元数据存储，比如 PostgreSQL，用于缓存条目的证据、依赖关系、反馈和审计。不要把所有信息都塞进 Redis。缓存可以快，但缓存决策需要可解释。

多层缓存还要注意写入时机。检索结果可以在重排后写入，因为它们不依赖模型生成是否成功。工具结果要在工具返回并通过基础校验后写入。最终答案最好在评测和安全检查通过后写入，不能一生成完就缓存。否则一个有明显错误或违规风险的答案会被后续请求快速放大。写入失败不应该影响用户当前请求，但要记录告警，因为长期写入失败会让成本和延迟回升。

## 工程实现

工程实现的第一步是做请求规范化。自然语言请求里有很多表面差异，比如空格、标点、大小写、称呼、语言混用、时间表达。规范化不是改写用户问题，而是抽取缓存所需的稳定字段。比如把“这周”“本周”“最近一周”解析成明确时间窗口，把“用表格给我”解析成输出格式，把“继续上面的内容”标记为强会话依赖。强会话依赖的请求通常不适合共享缓存，最多使用同会话私有缓存。

可以从下面这个数据结构开始：

```go
type CacheKey struct {
	TenantID        string
	IntentHash      string
	ContextHash     string
	PermissionHash  string
	PromptVersion   string
	OutputSchema     string
	DataVersionHash  string
	FreshnessBucket  string
	ModelFamily      string
}

type CacheEntry struct {
	Key             CacheKey
	AnswerMarkdown  string
	Citations       []Citation
	CreatedAt       time.Time
	ExpiresAt       time.Time
	QualityScore    float64
	SourceVersions  map[string]string
	AccessMode      string
	Status          string
}
```

`IntentHash` 不应该直接用原始问题哈希。更好的做法是用一个规范化后的意图表示，例如意图类别、实体、时间范围、输出格式和约束集合。早期可以用规则和轻量模型生成 JSON，再对 JSON 做稳定序列化和哈希。需要注意的是，意图抽取本身也有版本。抽取提示词或规则变化后，旧的 `IntentHash` 可能不可比，所以要把抽取器版本纳入 `PromptVersion` 或单独字段。

`PermissionHash` 是安全核心。它不能只写用户 ID，因为共享缓存需要比较权限集合。也不能把完整权限列表放入键里，既大又可能泄露。常见做法是把用户在本次数据范围内可访问的资源集合、角色、策略版本做稳定排序后哈希。对于大型知识库，不一定能列出所有文档，可以按 ACL 版本、部门、项目、数据域和策略版本组合成摘要。关键是，当权限变化时，摘要必须变化，旧缓存不能继续跨权限复用。

`DataVersionHash` 用于表达答案依赖的数据状态。知识库可以使用索引版本和文档更新时间水位，业务数据库可以使用表级版本或查询结果版本，外部 API 可以使用响应里的 `etag`、更新时间或短 TTL。对于无法提供版本的数据源，只能保守地缩短 TTL，或者禁止最终答案共享缓存。不要假装没有版本的数据可以长期缓存。

精确缓存流程比较直接。服务端构造键后查 KV，命中后加载元数据，检查状态、过期时间、权限摘要、引用有效性和质量分。如果全部通过，返回缓存答案，并在响应元数据里记录 `cache_hit=exact`、`cache_age`、`entry_id`。如果过期但允许 stale，可以返回旧答案并发起后台刷新；如果用户请求里带有“最新”“重新查询”“不要用缓存”等意图，则跳过命中。这个跳过也要记录，否则命中率分析会误判。

语义缓存要更谨慎。一个实用流程是先生成 query embedding，召回 top N 历史意图，然后做三道门。第一道门是规则门，要求租户、权限模式、输出格式和数据范围一致。第二道门是语义等价门，用小模型或规则判断两个问题是否在当前业务语境下可互换，并输出理由。第三道门是证据门，检查缓存答案引用的数据源是否仍然有效，质量分是否达标，负反馈是否超过阈值。只有三道门都通过，才返回共享语义缓存。

伪代码如下：

```ts
async function lookupAnswerCache(req: NormalizedRequest, policy: CachePolicy) {
  const exactKey = buildExactKey(req, policy)
  const exact = await kv.get(exactKey)
  if (exact && await validateEntry(exact, policy)) {
    return { hit: true, mode: "exact", entry: exact }
  }

  if (!policy.allowSemantic) return { hit: false }

  const candidates = await vector.search(req.intentEmbedding, { tenantId: policy.tenantId, limit: 8 })
  for (const item of candidates) {
    if (!ruleGate(item, req, policy)) continue
    if (!await equivalenceGate(item.intent, req.intent)) continue
    if (!await validateEntry(item.entry, policy)) continue
    return { hit: true, mode: "semantic", entry: item.entry }
  }
  return { hit: false }
}
```

写入缓存时也要分层。检索缓存的 value 不一定是完整文档内容，可以是文档 ID、分块 ID、分数和索引版本，读取时再做权限确认和内容加载。工具缓存要保存原始参数和响应摘要，避免不同默认参数混淆。答案缓存要保存引用和证据，不能只保存最终 Markdown。评测缓存要绑定 rubric 版本，因为评测标准会变化。

失效机制可以从事件表开始。每个缓存条目记录依赖项，例如 `doc:123@v8`、`index:kb-main@2026-03-23-01`、`prompt:answer-v17`、`tool:sales_summary@schema3`。当文档更新、索引发布、提示词发布或工具 schema 变化时，写入一条 invalidation event。后台任务扫描依赖关系，把受影响条目标记为 stale 或 deleted。这样比全量清空更温和，也能解释为什么某类缓存突然命中率下降。

质量反馈要进入缓存策略。用户点踩、人工审核发现问题、自动评测低分，都应该降低缓存条目权重，严重时直接失效。反过来，高频命中且反馈良好的公共答案，可以延长 TTL 或升级到预热缓存。这里要小心幸存者偏差，没人反馈不等于答案好。关键场景可以抽样复评，尤其是在模型版本、提示词版本和知识库大规模更新后。

隐私和合规实现也要前置。缓存正文可能包含用户输入、内部数据和模型生成的敏感结论。需要明确哪些字段加密、哪些字段脱敏、哪些字段不落盘、哪些场景只缓存中间结果不缓存最终答案。多租户系统要避免共享底层 key 空间导致扫描风险。日志里可以记录缓存命中模式和 entry id，但不要把完整问题和答案散落在普通日志里。缓存系统越成功，存的数据越有价值，也越需要被当成一等数据资产管理。

## 测试评测

LLM 缓存测试不能只测命中率。命中率高但错命中多，是最危险的状态。评测应该同时覆盖正确性、安全性、时效性、成本和延迟。上线前可以建立一个离线请求集，包含真实用户问题的脱敏样本、人工标注的等价问题、相似但不等价的问题、不同权限用户的问题、不同时间窗口的问题和强制刷新问题。每次调整缓存策略，都在这套集合上跑回归。

测试矩阵可以这样组织：

| 测试类型 | 样例 | 期望 | 失败信号 |
| --- | --- | --- | --- |
| 精确命中 | 同用户同上下文刷新 | 返回同一答案并标记命中 | 重复调用模型 |
| 权限隔离 | 不同部门问相似私有问题 | 不共享答案 | 跨权限命中 |
| 语义拒绝 | 删除账号 vs 注销企业账号 | 拒绝复用 | 相似度阈值误放行 |
| 时效控制 | 查询运行中工单状态 | 短 TTL 或不缓存 | 返回旧状态 |
| 提示词升级 | answer-v17 升到 v18 | 旧答案失效或降级 | 旧格式继续出现 |
| 负反馈失效 | 缓存答案被点踩多次 | 降权或删除 | 继续高频命中 |
| 强制刷新 | 用户要求重新查询 | 跳过答案缓存 | 仍返回缓存 |

离线评测之外，还要做在线观测。关键指标包括精确命中率、语义候选召回率、语义放行率、缓存节省 token、节省延迟、错命中投诉率、权限拒绝次数、stale 返回次数、后台刷新成功率、条目失效率和写入失败率。命中率要按场景拆分，不能只看全局。客服问答、知识库总结、代码解释、数据分析的缓存边界完全不同。一个场景命中率低可能是正常保守，另一个场景命中率高可能是风险。

质量评测要引入“缓存答案是否仍然回答当前问题”的判断。可以用小模型做初筛，但关键样本需要人工审查。评审时不要只看答案语言流畅度，要看引用是否仍然有效、结论是否和当前权限内证据一致、时间表达是否准确、输出格式是否符合请求、是否缺少当前会话必要信息。LLM 缓存的错命中常常看起来很像正确答案，只有对照证据才会发现偏差。

压测也有特殊点。缓存命中会改变系统负载形态：模型调用下降，KV 和向量库压力上升，元数据校验变多，后台刷新会形成异步峰值。压测要模拟热门问题集中访问，也要模拟大量低相似问题导致语义召回但最终拒绝的情况。后者容易被忽略，因为虽然没有命中，但已经消耗了向量检索和等价性判定成本。语义缓存如果设计不好，可能省不了钱，反而增加一段额外延迟。

灰度上线时建议先开启只读观测模式。系统照常生成答案，同时在后台计算“如果启用缓存会命中哪条”。对这些影子命中做离线评估，统计潜在节省和风险，再逐步放开私有精确缓存、公共精确缓存、私有语义缓存、公共语义缓存。每一步都要有回滚开关。缓存错误的影响会被命中次数放大，灰度节奏应该比普通性能优化更谨慎。

## 失败模式

第一个失败模式是把相似当等价。向量相似度高只是说明文本接近，不代表业务问题相同。尤其在政策、法律、财务、运维和权限场景里，少一个限定词就可能改变答案。解决办法是语义缓存必须经过等价性判定，并且保留拒绝路径。阈值可以帮助排序，不能单独决定返回。

第二个失败模式是权限哈希不完整。系统只按租户区分缓存，却忽略用户在租户内的部门、项目、角色和数据域。这样最容易发生跨团队泄露。解决办法是把权限摘要和策略版本纳入键，并在返回前再次校验。权限变化事件也要触发失效，不能等 TTL 自然过期。

第三个失败模式是提示词升级后旧答案继续命中。提示词升级往往是为了修复格式、口径或安全策略。如果缓存键没有包含提示词版本，旧答案会绕过新策略。解决办法是每个影响答案的提示词、输出 schema 和工具描述都有版本，版本变化后相关答案缓存失效或进入重验证。

第四个失败模式是缓存污染。一次错误生成、一次工具异常、一次检索漏召回，如果被写入缓存，就会被反复返回。解决办法是写入前做基础质量门，包括引用存在、结构完整、安全检查通过、模型 finish reason 正常、必要字段不为空。对于低置信答案可以只返回给当前用户，不进入共享缓存。

第五个失败模式是过期数据被包装成确定结论。很多答案没有说明数据时间，用户看不出它基于旧状态。解决办法是在缓存条目里保存数据时间，在界面或响应元数据里暴露必要的新鲜度信息。涉及实时状态、事故处理、价格库存和用户资产的问题，默认应该要求新鲜数据，除非用户明确接受历史视图。

第六个失败模式是缓存层不可观测。命中后用户拿到错误答案，工程只看到“cache hit”，不知道为什么命中。解决办法是记录命中模式、候选分数、等价判断、权限摘要版本、数据版本、条目年龄和质量分。排障时应该能回答：这条答案从哪里来，为什么认为可复用，哪些检查通过，哪个检查没有执行。

第七个失败模式是没有用户刷新通道。用户发现答案旧了，却只能重新问一遍。系统仍然可能继续命中同一缓存。解决办法是提供“重新查询”或“忽略缓存”能力，并把这个动作写入反馈。对于企业内部应用，运营或管理员还应该能按文档、数据源、提示词版本或缓存条目批量失效。

第八个失败模式是缓存节省了模型钱，却增加了产品风险。比如为了追求命中率，把公共语义缓存打开到所有问题，短期成本下降，长期投诉和信任损失上升。缓存策略必须有风险预算，关键业务场景宁可多花模型成本，也不要返回不可证明的答案。工程优化不能脱离产品责任。

## 上线 checklist

上线前建议按下面清单检查。缓存系统的问题通常不是语法错误，而是边界没讲清楚。

- 已明确缓存对象：提示词、检索、工具、最终答案和评测结果分别使用独立策略，不混成一个黑盒缓存。
- 缓存键包含租户、权限摘要、数据版本、提示词版本、输出格式和新鲜度要求，关键字段变化会导致旧缓存不可直接命中。
- 已区分私有缓存、租户共享缓存和公共缓存，默认从私有精确缓存开始灰度。
- 语义缓存只作为候选召回，必须经过规则门、等价性门、权限门、时效门和质量门。
- 用户显式要求最新、重新查询、刷新数据时，答案缓存会被跳过，并记录跳过原因。
- 最终答案写入缓存前通过安全检查、结构检查、引用检查和基础质量检查。
- 缓存条目保存引用、来源版本、创建时间、过期时间、提示词版本、模型信息、质量分和命中记录。
- 文档更新、索引重建、权限变化、提示词发布、工具 schema 变化和负反馈都会触发失效或降权。
- 响应元数据包含命中模式、条目年龄和必要的数据时间，方便产品按场景展示。
- 观测指标覆盖命中率、节省成本、节省延迟、错命中投诉、权限拒绝、stale 返回、后台刷新和写入失败。
- 已准备影子模式和灰度开关，可以快速关闭语义缓存或共享缓存，而不影响正常生成链路。
- 日志和存储策略经过隐私审查，问题、答案、引用和权限摘要不会在低等级日志里散落。
- 客服和工程能按缓存条目追踪命中原因，管理员能按数据源、文档、提示词版本或租户批量失效。
- 离线回归集包含相似但不等价问题、不同权限用户、不同时间窗口、强会话依赖和强制刷新样本。

对于早期团队，我建议先上线三件事：同用户同会话的精确答案缓存，检索和工具结果的短 TTL 缓存，带版本的缓存元数据和失效日志。等这些稳定后，再做共享缓存和语义缓存。这样既能拿到一部分成本收益，也能避免一开始就把安全边界拉得太宽。

## 总结

LLM 缓存不是简单的 `key -> answer`。它是一套围绕语义、权限、时效和质量建立的复用机制。缓存命中越靠近最终答案，产品风险越高；缓存越靠近底层工具和检索，复用边界越容易证明。工程上应该分层设计，保守放行，可观测，可失效，可回滚。

一个可靠的策略通常具备几个特征：请求先规范化，策略按请求动态生成，键里包含权限和版本，语义缓存只做候选，返回前做多重校验，写入前做质量门，失效由事件驱动，用户可以强制刷新，工程能解释每一次命中。做到这些，缓存才真正降低成本和延迟，而不是把错误答案缓存得更快。

最后要记住，缓存的本质是复用过去的计算。LLM 应用里的过去包含模型、提示词、数据、权限和用户语境。只要其中任何一项已经变化，复用就需要重新证明。把这个证明过程产品化和工程化，才是 LLM 缓存策略的核心。
